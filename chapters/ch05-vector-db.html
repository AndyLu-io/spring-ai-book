第 07 章

向量数据库

深入理解向量数据库原理，掌握 Chroma、Milvus、Pinecone 等主流向量数据库的使用与调优

预计 2 周 7 个知识点 中级难度

📚 开篇故事：图书馆找书的烦恼

想象你走进一座巨大的图书馆，想找一本关于"如何做好番茄炒蛋"的书。传统的方式是什么？你会去索引卡上搜索"番茄炒蛋"，然后找到对应的书架编号。

但问题来了 -- 如果一本书叫《家常菜入门》里面恰好有番茄炒蛋的做法呢？索引卡上没有"番茄炒蛋"这个词，你就找不到这本书。这就是关键词搜索的局限性。

现在想象一个超级图书馆：你走进去说"我想学做家常菜"，图书管理员（AI）会理解你的意思，然后从书架上找出所有相关的书 -- 《家常菜入门》、《厨房新手指南》、《30 分钟快手菜》，甚至《妈妈的味道》。它不是匹配关键词，而是理解了你"想学做饭"的意图。

这就是向量数据库的核心价值 -- 它是一座"超级图书馆"，能够理解语义相似度，找到意思相近的内容，而不仅仅是字面匹配。Spring AI 的 RAG（检索增强生成）功能，就是基于向量数据库来实现的。

📚 向量数据库原理 -- 超级图书馆的运作方式

📄 概念：为什么需要向量数据库？

传统数据库（MySQL、PostgreSQL）就像普通的图书索引卡 -- 只能按书名精确匹配。你搜"Spring"，只能找到书名里有"Spring"的书。但如果你想找"Java 框架入门"的书呢？

向量数据库就是"超级图书馆的智能管理员" -- 它把每本书的内容都转换成一串数字（向量），然后通过计算数字之间的距离来判断两本书有多"像"。这样，即使一本书叫《Spring Boot 实战》，另一本叫《Java Web 开发指南》，只要内容相似，它就能找到。

传统数据库                          向量数据库
SELECT * FROM docs                  search("如何学习 Spring Boot", topK=5)
WHERE title = 'Spring AI';          |
     |                              v
     v                              找到语义相似的文档:
只找到精确匹配的文档                  ├── "Spring Boot 快速上手" (0.95)
                                    ├── "Java 框架入门指南"   (0.88)
                                    ├── "Spring 实战教程"     (0.85)
                                    └── "Web 开发基础"        (0.72)

核心数据结构

概念说明比喻
向量 (Vector) 文本的高维数值表示，通常 384-3072 维 每本书的"指纹"
元数据 (Metadata) 附加信息，如来源、时间、标签 书的封面信息（作者、出版社）
Collection 向量集合，类似关系数据库的表 一个书架或一个图书馆分区
索引 (Index) 加速搜索的数据结构 图书馆的楼层导航系统

💻 代码示例：完整向量搜索流程

// ========================================== // 向量搜索完整流程演示 // ========================================== // 1. 定义文档（含元数据） Document doc1 = new Document( "Spring AI 提供了统一的 AI 模型抽象", Map.of( "source", "docs/spring-ai.md", "category", "framework" ) ); // 2. 存入向量数据库 vectorStore.accept(List.of(doc1)); // 3. 语义搜索 SearchRequest request = SearchRequest.query("AI 模型") .withTopK(5) // 返回 5 个结果 .withSimilarityThreshold(0.7); // 最低相似度 0.7 List<Document> results = vectorStore.similaritySearch(request); // 4. 输出结果 results.forEach(doc -> { System.out.println("内容: " + doc.getContent()); System.out.println("元数据: " + doc.getMetadata()); });

近似最近邻（ANN）-- 速度的秘密

精确搜索在高维数据上计算复杂度为 O(n*d)，无法满足实时需求。ANN 算法通过牺牲少量精度换取数量级的性能提升：

算法100万向量耗时精度 (Recall@10)内存占用
精确搜索 (KNN) ~2 秒 100% 基准
HNSW ~5 毫秒 (快 400x) 95-99% 较高
IVF ~10 毫秒 (快 200x) 90-95% 中等
IVF_PQ ~15 毫秒 85-92% 最低 (压缩)
💡 你知道吗？

HNSW 算法灵感来自"小世界网络"理论 -- 社交网络中任何两个人之间平均只需要 6 步就能建立联系。HNSW 把这种"六度分隔"的思路用在了向量搜索中，通过构建多层图结构实现快速导航。

📚 Chroma 使用 -- 轻量级图书角

📄 概念：Chroma 是什么？

Chroma 就像"轻量级图书角" -- 不需要复杂的安装配置，开箱即用，适合开发和原型验证。就像你在家里的书房放一个小型书架，随手就能拿到常用的书。

✔ 优势

  • 零配置启动（pip install 就能用）
  • Python/Java SDK 完善
  • 内嵌 Embedding 模型
  • 适合原型开发和学习

✘ 限制

  • 单机部署，不支持分布式
  • 数据量建议 < 1000万条
  • 不支持高可用
  • 生产环境性能有上限

💻 代码示例：Spring AI 集成 Chroma

# ========================================== # application.yml - Chroma 配置 # ========================================== spring.ai.vectorstore.chroma: url: http://localhost:8000 collection-name: my-docs initialize-schema: true
// ========================================== // ChromaService.java - Chroma 向量存储服务 // ========================================== @Service public class ChromaService { private final VectorStore vectorStore; /** * 添加文档到向量数据库 * 自动调用 Embedding 模型生成向量 */ public void addDocuments() { Document doc = new Document( "Spring AI 提供了统一的 AI 模型抽象", Map.of( "source", "docs/spring-ai.md", "category", "framework", "date", "2025-01-01" ) ); vectorStore.accept(List.of(doc)); // 存入 Chroma } /** * 相似度搜索 * @param query 搜索查询 * @return 最相似的 5 个文档 */ public List<Document> search(String query) { return vectorStore.similaritySearch( SearchRequest.query(query) .withTopK(5) // 返回前 5 个 .withSimilarityThreshold(0.7)); // 最低相似度 } }

💻 代码示例：元数据过滤

// ========================================== // 基于元数据过滤搜索结果 // 就像在图书馆按分类筛选 // ========================================== SearchRequest request = SearchRequest.query("AI 模型") .withTopK(5) // 最多 5 个 .withSimilarityThreshold(0.7) // 相似度 > 0.7 .withFilterExpression( // 元数据过滤 "category == 'framework' AND date >= '2025-01-01'" ); List<Document> results = vectorStore.similaritySearch(request);
💡 元数据过滤的作用

元数据过滤可以大幅缩小搜索范围，提升搜索速度。比如只搜索"2025年的框架文档"，而不是搜索所有文档。建议在向量搜索前先用元数据过滤缩小范围。

🔎 索引算法 -- 图书馆的楼层导航

📄 概念：索引算法是什么？

索引算法就像"图书馆的楼层导航系统" -- 你不需要走遍整个图书馆找书，导航系统会告诉你"这本书在 3 楼 A 区"。好的索引可以让你在百万级数据中快速找到目标。

HNSW（Hierarchical Navigable Small World）-- 首选方案

HNSW 是目前最主流的 ANN 索引算法，构建一个分层的图结构，就像图书馆的多层导航：

HNSW 分层结构

层 2 (最高层):  A ---- C ---- F          <-- 稀疏层，快速定位大方向
                 |     |
层 1 (中间层):  A - B - C - D - F        <-- 中等密度，逐步缩小范围
                 |   |   |   |   |
层 0 (最底层):  A - B - C - D - E - F    <-- 完整层，精确定位
                 每个节点都连接相邻的向量

HNSW 参数配置

// ========================================== // HNSW 参数调优指南 // ========================================== // M - 每个节点的最大连接数（默认 16） // 增大: 更准确但内存更多，搜索更慢 // 减小: 更快但可能漏掉近邻 M = 16 // efConstruction - 构建时的搜索范围（默认 200） // 增大: 索引质量更好但构建更慢 // 减小: 构建更快但索引质量下降 efConstruction = 200 // efSearch - 查询时的搜索范围（默认 100） // 增大: 搜索更准确但速度更慢 // 减小: 搜索更快但可能漏掉结果 efSearch = 100
💡 HNSW 最佳实践

HNSW 适合数据量在 100 万到 10 亿之间的场景。它是查询速度和准确率的最佳平衡。内存占用较高（向量数据需要全部加载到内存），这是用空间换时间的策略。

IVF（Inverted File Index）-- 内存受限时的选择

IVF 将向量空间用 K-Means 聚类分成多个簇，搜索时只在最近的几个簇中查找：

// ========================================== // IVF 参数配置 // ========================================== // nlist - 聚类中心数量（默认 1024） // 增大: 更精细的划分但索引更大 // 减小: 划分粗糙但索引更小 nlist = 1024 // nprobe - 搜索时访问的簇数（默认 10） // 增大: 更准确但速度更慢 // 减小: 更快但可能漏掉结果 nprobe = 10 // IVF + PQ 组合（压缩索引，适合超大规模数据） nlist = 1024 m = 32 // 子空间数量 nbits = 8 // 每个子空间的量化位数

索引算法选择指南

场景推荐算法原因
大多数场景 HNSW 速度和精度的最佳平衡
内存受限 IVF_FLAT 内存占用比 HNSW 低
超大规模 (>1 亿) IVF_PQ 牺牲精度换空间
需要 GPU 加速 GPU 索引 Milvus 支持 GPU 加速

📏 距离度量 -- 找最像的书

📄 概念：如何判断两本书"像不像"？

距离度量就是"判断相似度的标准" -- 就像图书馆管理员用不同标准判断两本书有多像：有的看封面风格（余弦相似度），有的看内容长度（欧氏距离），有的看关键词重叠（内积）。

// ========================================== // 三种常用距离度量 // ========================================== // 1. 余弦相似度（Cosine Similarity）-- 最常用 cosine(A, B) = (A . B) / (||A|| * ||B||) // 范围: [-1, 1], 1 表示完全相同 // 特点: 不受向量长度影响，只看方向 // 适合: 文本语义搜索 // 2. 欧氏距离（L2 Distance） l2(A, B) = sqrt(sum((Ai - Bi)^2)) // 范围: [0, +∞), 0 表示完全相同 // 特点: 直接计算空间距离 // 适合: 图像特征匹配 // 3. 内积（Inner Product / IP） ip(A, B) = sum(Ai * Bi) // 特点: 已归一化时等价于余弦 // 适合: 推荐系统

选择建议

应用场景推荐度量原因
文本搜索 余弦相似度 (COSINE) 默认推荐，不受文本长度影响
图像搜索 L2 或余弦 图像特征向量通常已归一化
推荐系统 内积 (IP) 适合已归一化的向量
已归一化向量 L2 (最快) 三者等价，L2 计算最快
💡 OpenAI 默认使用余弦相似度

OpenAI 的 text-embedding 模型输出的向量已经归一化（模为 1），所以使用 L2 距离和余弦相似度的结果完全相同。但 L2 计算更快，所以如果你确定向量已归一化，优先选择 L2。

💡 你知道吗？

余弦相似度计算的是两个向量在"方向"上的相似程度，而不是"长度"。这意味着"猫"和"猫咪"的余弦相似度很高，即使"猫咪"的文本更长。这就是为什么余弦相似度最适合文本搜索。

⚡ 性能调优

查询性能优化

  • 减小维度 -- 使用 PCA 等降维技术减少向量维度
  • 元数据预过滤 -- 先按元数据过滤，再进行向量搜索
  • 合理设置 TopK -- TopK 越大，召回越全但速度越慢
  • 相似度阈值 -- 设置最低阈值，过滤低质量结果
  • 连接池 -- 配置合适的数据库连接池大小

检查项建议原因

⚠️ 常见陷阱

生产环境不要使用 SimpleVectorStore（内存模式）。数据重启后丢失，且无法水平扩展。一定要使用 Chroma、Milvus 等持久化向量数据库。

📖 章节小结

本章我们深入探索了向量数据库的"超级图书馆"：

  • 向量数据库原理（超级图书馆）-- 语义相似度搜索，突破关键词匹配的局限
  • Chroma（轻量级图书角）-- 零配置启动，适合开发和原型验证
  • Milvus / Pinecone（企业级图书大厦）-- 生产级方案，支持海量数据

向量数据库是 RAG（检索增强生成）的基石。掌握它，你就能让 AI 应用真正"理解"用户的意图，而不仅仅是匹配关键词。

🎓 面试高频题

👨‍💻 Q1: 向量数据库和传统数据库的区别？

传统数据库基于精确匹配（SQL WHERE 条件），只能找"字面完全相同"的内容。向量数据库基于语义相似度（向量距离），能找到"意思相近"的内容。例如搜"如何学习编程"，传统数据库只能找包含这几个字的文档，而向量数据库能找到"编程入门指南"。

👨‍💻 Q2: HNSW 索引的原理是什么？

HNSW 构建一个多层图结构。顶层最稀疏，用于快速定位大致方向；底层最密集，用于精确定位。搜索时从顶层开始，逐层向下导航，类似"先看楼层索引，再看具体书架"。M 控制连接数，efControl 控制搜索范围。内存占用较高，但查询速度极快。

👨‍💻 Q3: 如何选择余弦相似度、L2 和内积？

文本搜索用余弦相似度（默认推荐）。如果向量已归一化（模为 1），三者等价，L2 计算最快。内积适合推荐系统。核心原则：选 Embedding 模型推荐的度量方式。OpenAI embedding 默认用余弦相似度。
