第 21 章

Agent 架构设计模式

掌握 Agent 架构设计的方法论，理解 ReAct、Plan-and-Execute、LATS 等架构模式，能根据场景选择合适的架构

预计 2-3 周 8 个知识点 高级难度

📚 开篇故事：建筑师的蓝图

想象你是一个城市的总建筑师。城市要扩建一座新城，你面前有无数种方案：是建一栋超高层的摩天大楼，把所有功能都塞进去？还是建一片由多个小建筑组成的园区，每个建筑各司其职？是先画好完整的蓝图再施工（Plan-Execute），还是一边建一边调整（ReAct）？是让一个全能团队包揽所有工作，还是分出专项小组各管一块？

这些问题的决策，决定了新城的质量、成本和未来的扩展能力。做对了，新城高效运转，居民满意；做错了，交通拥堵、功能缺失、改造成本高昂。

Agent 架构设计也是如此。选择 ReAct 还是 Plan-Execute，用单体还是分布式，采用同步还是异步 -- 这些架构决策就像建筑师的蓝图，一旦落地就很难推倒重来。本章将为你提供一套系统的架构决策框架和设计模式，让你能像经验丰富的建筑师一样，为不同的业务场景选择最合适的 Agent 架构。

🎯 学习目标

在本章结束后，你将能够：

  • 掌握架构决策框架 — 理解 ReAct、Plan-Execute、LATS、Reflection 等主流架构模式的适用场景
  • 选择合适架构 — 能根据业务场景、复杂度、成本等因素使用决策树选择最优架构
  • 设计可组合 Agent — 掌握 Agent 插件化、能力热插拔和微服务化架构设计
  • 构建 Agent-as-a-Service — 实现 Agent 注册发现、负载均衡和多租户隔离

📄 架构决策框架：选对路比跑得快更重要

概念比喻：医生的诊断思维

架构选型就像医生看病。一个好医生不会上来就开药，而是先问诊、检查、诊断，然后根据病人的具体情况选择最合适的治疗方案。同样，选择 Agent 架构时，你需要先理解任务特征、系统约束和质量要求，然后用决策树找到最佳方案。

很多团队犯的错误是"拿着锤子找钉子" -- 先选定一种架构，再试图把所有场景都塞进去。正确的做法是根据场景选择架构。

四大架构模式对比

架构模式核心思想适用场景复杂度延迟
ReAct 思考-行动-观察循环 简单问答、工具调用、单步决策 低 中

💡 经验法则

从简单开始：先用 ReAct 验证想法，当 ReAct 无法满足时再升级到 Plan-Execute 或 Reflection。过早引入复杂架构会增加开发和维护成本，就像盖一个报刊亭不需要请结构力学专家。

📄 核心架构模式详解

Actor-Critic 模式：执行者与评估者分离

比喻：就像拍电影，导演（Actor）负责拍戏，影评人（Critic）负责打分。导演可以尝试不同的表演方式，影评人会客观评估每种方式的好坏。好的电影就是两者不断磨合的产物。

Actor-Critic 模式将"做事"和"评估"分离为两个独立的 Agent。Actor 负责生成行动方案，Critic 负责评估方案质量并给出改进方向。这种分离使得评估标准可以独立于执行逻辑演进。

Actor-Critic 模式

  ┌───────────┐
  │  用户任务  │
  └─────┬─────┘
        v
  ┌───────────────┐         ┌───────────────┐
  │ Actor (执行者)  │ ────────> │ Critic (评估者) │
  │ 生成行动方案   │ <──────── │ 评估方案质量   │
  └───────┬───────┘   反馈    └───────┬───────┘
          │                          │
          └──────────┬───────────────┘
                     │
                     v  迭代直到满意
              ┌───────────┐
              │  最终输出  │
              └───────────┘

核心优势：Actor 专注于创造性输出，Critic 专注于质量把控，各司其职。适用于代码生成、论文写作、设计方案等需要高质量输出的场景。

Toolformer 模式：Agent 自主学习使用工具

比喻：传统 Agent 像一个只会按说明书操作的工人，工具怎么用全靠别人教。Toolformer 模式则像一个聪明的学徒，它会自己观察工具的使用方式，总结规律，然后自主决定何时、如何使用工具。

Toolformer 的核心思想是让 LLM 通过 few-shot 学习，自动判断何时调用工具、调用哪个工具、如何构造调用参数，而不需要开发者手动编写 tool-use 的决策逻辑。

Toolformer 自主工具学习

  输入文本: "法国的首都是哪里？人口多少？"
         │
         v
  ┌─────────────────────────────┐
  │  LLM 自主决策:              │
  │  1. 识别需要外部信息         │
  │  2. 选择工具: search_api     │
  │  3. 构造参数: "法国首都"     │
  │  4. 调用工具获取结果         │

实际应用中，Spring AI 通过 @Tool 注解和 ToolCallback 机制支持工具注册，而 Toolformer 模式强调的是让 LLM 在 Prompt 层面自主学习工具使用的模式。

MRKL 系统：模块化推理与知识学习

比喻：MRKL（Modular Reasoning, Knowledge and Language）就像一个现代化的手术团队。主刀医生（LLM）负责诊断和决策，麻醉师、护士、影像科医生（外部模块）各司其职，通过标准化接口配合完成手术。

MRKL 的核心贡献是将 Agent 系统模块化：每个推理步骤可以交给最适合的模块处理 -- LLM 负责自然语言理解，符号计算器负责精确数学，知识库负责事实检索，代码执行器负责复杂计算。

MRKL 模块化架构

  ┌──────────────────────────────────────┐
  │           Router (路由器)              │
  │     根据任务类型分发到对应模块          │
  └────┬──────────┬──────────┬───────────┘
       │          │          │
  ┌────▼────┐ ┌──▼────┐ ┌──▼──────┐ ┌────▼───────┐
  │  LLM    │ │计算器│ │知识库  │ │ 代码执行器 │
  │ 语言理解 │ │ 精确   │ │ 事实    │ │  复杂计算  │
  │ 推理规划 │ │ 数学   │ │ 检索    │ │  数据处理  │
  └─────────┘ └───────┘ └────────┘ └────────────┘

MRKL 的最大优势是可扩展性：新增能力只需添加模块并注册到路由器，无需修改现有代码。这与 Spring AI 的 ToolCallbackResolver 理念一脉相承。

Plan-and-Execute 模式：先谋后动

比喻：登山前要先看地图、规划路线、准备装备，而不是直接冲进山里。Plan-and-Execute 就是让 Agent 先做全局规划，再逐步执行，遇到意外时动态调整路线。

与 ReAct 的"走一步看一步"不同，Plan-Execute 的规划器会生成一个全局任务图，明确每一步的依赖关系和执行顺序。这大幅减少了无效的工具调用和重复探索。

📖 设计要点

Plan-Execute 的关键是规划器和执行器的解耦。规划器只负责"想"，执行器只负责"做"。当执行失败时，不是让执行器自行重试，而是将失败信息反馈给规划器重新规划，确保全局一致性。

LATS 模式：语言智能体树搜索

比喻：LATS 就像一个围棋高手下棋。高手不会只看眼前一步，而是在脑海中构建一棵"决策树"，对每种可能的走法进行推演（展开），用经验评估每种走法的好坏（模拟），然后选择最优的路径（选择）。遇到关键局面时，还会回头复盘，更新对每步棋的评估（回溯）。LATS 将这种树搜索思想引入 Agent 架构，通过系统性地探索多条解题路径，找到全局最优解。

LATS（Language Agent Tree Search）的核心创新在于将蒙特卡洛树搜索（MCTS）与LLM 的语言能力结合。传统 ReAct 只沿着一条路径走到底，而 LATS 会同时探索多条路径，通过 UCB（置信上界）公式在"探索新路径"和"利用已知好路径"之间取得平衡。

LATS 树搜索流程

  ┌────────────┐
  │  初始状态   │
  └──────┬─────┘
         │
         ▼  ① 选择 (UCB策略)
  ┌──────────────────────────────────────┐
  │           决策树                      │
  │                                      │
  │         ┌─────┐                      │
  │         │ 根节点 │ (state=初始)        │
  │         └──┬──┘                      │
  │      ┌─────┼─────┐                  │
  │      ▼     ▼     ▼                  │
  │   ┌─────┐┌─────┐┌─────┐             │
  │   │动作A││动作B││动作C│  ② 展开     │
  │   │v=0.8││v=0.3││v=0.6│             │
  │   └──┬──┘└─────┘└─────┘             │
  │      │                              │
  │   ┌──▼──┐                           │
  │   │动作D│  ③ 模拟评估               │
  │   │v=0.9│                           │
  │   └─────┘                           │
  │                                      │
  │  ④ 回溯更新路径上所有节点的 value    │
  └──────────────────────────────────────┘
         │
         ▼  重复直到迭代上限
  ┌────────────┐
  │  最优路径   │
  └────────────┘

LATS 的四个核心步骤：

  • 选择（Selection）：从根节点出发，用 UCB 公式选择"最有潜力"的子节点，平衡探索与利用
  • 展开（Expansion）：对选中的节点，让 LLM 生成多个可能的下一步动作，创建子节点
  • 模拟（Simulation）：对新展开的节点，用 LLM 评估其状态价值（不需要展开到底）
  • 回溯（Backpropagation）：将模拟得到的价值沿路径向上传播，更新所有祖先节点的统计信息
📖 LATS vs ReAct vs Plan-Execute

ReAct 是"单路径贪心"——每步选当前最优，容易陷入局部最优。Plan-Execute 是"全局规划+线性执行"——规划质量决定上限，执行偏差难以纠正。LATS 是"多路径搜索+动态评估"——通过树搜索探索多条路径，用模拟评估剪枝，找到全局最优解。LATS 的代价是更多的 LLM 调用和更高的延迟。

LATS 完整代码实现

// ============================================ // LATS 树节点 - 树搜索的基本单元 // ============================================ public class LATSTreeNode { private String id; private String state; // 当前状态描述 private String action; // 到达此节点执行的动作 private double value; // 节点累积价值 private int visits; // 访问次数 private double qValue; // 平均动作价值 (Q值) private List<LATSTreeNode> children; private LATSTreeNode parent; private String toolCallResult; // 工具调用结果 private boolean isTerminal; // 是否为终止节点 public LATSTreeNode(String id, String state, String action, LATSTreeNode parent) { this.id = id; this.state = state; this.action = action; this.parent = parent; this.children = new ArrayList<>(); this.visits = 0; this.qValue = 0.0; this.isTerminal = false; } /** * UCB 公式：选择最有潜力的子节点 * UCB = Q(s,a) + c * sqrt(ln(N) / n) * Q(s,a): 平均动作价值（利用） * c * sqrt(ln(N)/n): 探索奖励（探索） */ public double ucbScore(double explorationWeight) { if (visits == 0) { return Double.POSITIVE_INFINITY; // 未访问优先 } double parentVisits = (parent != null) ? parent.visits : 1; double exploration = explorationWeight * Math.sqrt(Math.log(parentVisits) / visits); return qValue + exploration; } // 判断是否为叶节点 public boolean isLeaf() { return children.isEmpty(); } // 获取最佳子节点（用于最终路径提取） public LATSTreeNode getBestChild() { return children.stream() .max(Comparator.comparingDouble(c -> c.qValue)) .orElse(null); } // getter/setter 省略... }
// ============================================ // LATS 搜索引擎 - 核心搜索算法 // ============================================ @Service public class LATASSearch { private final ChatClient chatClient; private final int maxIterations; private final double explorationWeight; private final int maxChildren; private final List<ToolCallback> tools; private int nodeCounter = 0; public LATASSearch(ChatClient chatClient, List<ToolCallback> tools) { this.chatClient = chatClient; this.tools = tools; this.maxIterations = 10; // 最大搜索轮次 this.explorationWeight = 1.414; // UCB探索系数 (√2) this.maxChildren = 3; // 每个节点最多展开3个子节点 } /** * 主搜索循环 - LATS 的入口 */ public LATSTreeNode search(String initialState, String task) { // 1. 创建根节点 LATSTreeNode root = createNode( initialState, "初始状态", null); // 2. 迭代搜索 for (int i = 0; i < maxIterations; i++) { // 2a. 选择：从根节点出发，用UCB选最优路径 LATSTreeNode selected = select(root); // 2b. 展开：为选中节点生成子节点 if (!selected.isTerminal) { List<LATSTreeNode> children = expand(selected, task); if (!children.isEmpty()) { selected = children.get(0); } } // 2c. 模拟：评估当前节点价值 double reward = simulate(selected, task); // 2d. 回溯：更新路径上所有节点 backpropagate(selected, reward); } // 3. 返回最优路径的最终节点 return extractBestPath(root); } // ---------- ① 选择阶段 ---------- /** * UCB 选择：从根到叶，选最有潜力的节点 */ private LATSTreeNode select(LATSTreeNode node) { while (!node.isLeaf() && !node.isTerminal) { LATSTreeNode best = node.children.stream() .max(Comparator.comparingDouble( c -> c.ucbScore(explorationWeight))) .orElse(node); node = best; } return node; } // ---------- ② 展开阶段 ---------- /** * 让 LLM 生成多个可能的下一步动作 */ private List<LATSTreeNode> expand( LATSTreeNode node, String task) { // 让 LLM 生成候选动作 List<ActionCandidate> candidates = generateCandidates(node, task); List<LATSTreeNode> children = new ArrayList<>(); for (ActionCandidate candidate : candidates) { LATSTreeNode child = createNode( candidate.newState(), candidate.action(), node); child.toolCallResult = candidate.result(); node.children.add(child); children.add(child); } return children; } private List<ActionCandidate> generateCandidates( LATSTreeNode node, String task) { return chatClient.prompt() .system(""" 你是一个问题解决助手。当前状态： %s 原始任务：%s 请生成 %d 个不同的下一步动作。 每个动作包含： - action: 动作描述 - reasoning: 选择该动作的理由 返回 JSON 数组格式。 """.formatted( node.state, task, maxChildren)) .call() .entity(new ParameterizedTypeReference <List<ActionCandidate>>() {}); } // ---------- ③ 模拟阶段 ---------- /** * 评估节点价值：用 LLM 打分 */ private double simulate( LATSTreeNode node, String task) { return chatClient.prompt() .system(""" 评估以下状态对完成任务的价值。 任务：%s 当前状态：%s 执行的动作：%s 工具返回：%s 请给出 0-1 之间的分数，其中： - 0.0 = 完全无关/失败 - 0.5 = 部分进展 - 1.0 = 完美达成目标 只返回一个数字。 """.formatted( task, node.state, node.action, node.toolCallResult)) .call() .content() .trim() .pipe(s -> { try { return Double.parseDouble(s); } catch (NumberFormatException e) { return 0.5; // 默认中等分数 } }); } // ---------- ④ 回溯阶段 ---------- /** * 从当前节点向根回溯，更新所有祖先的统计 */ private void backpropagate( LATSTreeNode node, double reward) { LATSTreeNode current = node; while (current != null) { current.visits++; current.value += reward; current.qValue = current.value / current.visits; current = current.parent; } } // ---------- 辅助方法 ---------- private LATSTreeNode createNode( String state, String action, LATSTreeNode parent) { return new LATSTreeNode( "node-" + (++nodeCounter), state, action, parent); } // 提取最优路径 public List<LATSTreeNode> getBestPath( LATSTreeNode root) { List<LATSTreeNode> path = new ArrayList<>(); LATSTreeNode current = root; path.add(current); while (!current.isLeaf()) { current = current.getBestChild(); if (current != null) { path.add(current); } } return path; } } // 候选动作数据结构 public record ActionCandidate( String action, String reasoning, String newState, String result ) {}
// ============================================ // LATS 工具集 - 为 LATS 搜索提供工具支持 // ============================================ @Component public class LATASToolset { private final ChatClient chatClient; public LATASToolset(ChatClient chatClient) { this.chatClient = chatClient; } // 搜索工具 - 用于信息检索 @Tool(description = "搜索相关信息") public String search(String query) { // 实际项目中调用搜索 API return "搜索结果: " + query; } // 计算工具 - 用于精确计算 @Tool(description = "执行数学计算") public String calculate(String expression) { // 实际项目中用表达式求值引擎 return "计算结果: " + expression; } // 代码执行工具 @Tool(description = "执行 Python 代码") public String executeCode(String code) { // 实际项目中调用代码沙箱 return "执行结果: 代码运行成功"; } } // ============================================ // LATS Agent - 封装 LATS 的完整 Agent // ============================================ @Service public class LATSAgent { private final LATASSearch search; private final ChatClient chatClient; public LATSAgent(ChatClient chatClient, LATASToolset toolset) { this.chatClient = chatClient; this.search = new LATASSearch( chatClient, List.of( wrapTool(toolset, "search"), wrapTool(toolset, "calculate"), wrapTool(toolset, "executeCode"))); } /** * 使用 LATS 解决任务 */ public String solve(String task) { // 1. 初始化状态 String initialState = "任务: " + task + "\n尚未采取任何行动。"; // 2. 执行树搜索 LATSTreeNode root = search.search(initialState, task); // 3. 提取最优路径 List<LATSTreeNode> bestPath = search.getBestPath(root); // 4. 生成最终答案 return synthesizeResult(task, bestPath); } private String synthesizeResult( String task, List<LATSTreeNode> path) { String pathDesc = path.stream() .map(n -> "- " + n.getAction() + " → " + n.getState()) .collect(Collectors.joining("\n")); return chatClient.prompt() .system(""" 根据以下搜索路径，为任务生成最终答案。 选择最优路径经过了以下步骤： %s 请综合这些步骤的结果，给出完整的答案。 """.formatted(pathDesc)) .user(task) .call() .content(); } private ToolCallback wrapTool( Object toolObj, String methodName) { // 实际项目中用 Spring 的反射工具包装 return new MethodToolCallback( toolObj, methodName); } }
// ============================================ // LATS REST 控制器 - 暴露 LATS Agent 接口 // ============================================ @RestController @RequestMapping("/api/lats") public class LATSController { private final LATSAgent agent; public LATSController(LATSAgent agent) { this.agent = agent; } @PostMapping("/solve") public ResponseEntity<Map<String, Object>> solve(@RequestBody SolveRequest req) { long start = System.currentTimeMillis(); String result = agent.solve(req.task()); long elapsed = System.currentTimeMillis() - start; return ResponseEntity.ok(Map.of( "task", req.task(), "result", result, "timeMs", elapsed)); } public record SolveRequest(String task) {} }
💡 LATS 调参建议

explorationWeight：值越大越倾向探索新路径，推荐 1.0-2.0（√2≈1.414 是理论最优）。maxIterations：搜索轮次越多结果越好但延迟越高，生产中建议 5-15 轮。maxChildren：每个节点的分支数，3-5 个是合理范围，太多会导致搜索空间爆炸。

⚠️ 成本控制

LATS 的 LLM 调用次数 = 迭代次数 x 每轮展开数 + 模拟评估次数。10 轮迭代 x 3 个分支 = 至少 30+ 次 LLM 调用。生产中务必设置调用上限和超时时间，避免成本失控。建议先用小模型（如 GPT-4o-mini）做初步筛选，再用大模型做精细评估。

📄 可组合 Agent 设计原则

Agent 插件化架构：乐高积木的哲学

比喻：乐高积木之所以经典，是因为每一块积木都是标准化的、可互换的、可自由组合的。你可以用同一套积木拼出房子、汽车、城堡。可组合 Agent 设计的目标也是如此 -- 让每个 Agent 能力成为一个"积木块"，按需组合成不同的系统。

插件化架构的核心原则：

  • 标准接口：每个 Agent 能力遵循统一的 AgentCapability 接口，包括输入类型、输出类型、能力描述
  • 依赖注入：通过 Spring 的 IoC 容器管理 Agent 的生命周期和依赖关系
  • 热插拔：运行时可以动态添加或移除 Agent 能力，无需重启服务
  • 组合优先：优先通过组合现有能力构建新功能，而非继承

能力组合与热插拔

热插拔意味着在不中断服务的情况下，动态添加或移除 Agent 的能力。这在生产环境中至关重要 -- 当某个工具 API 宕机时，可以临时移除该能力，让 Agent 降级运行而非完全不可用。

// Agent 能力接口定义 public interface AgentCapability { String getName(); // 能力名称 String getDescription(); // 能力描述 boolean isAvailable(); // 是否可用 ToolCallback[] getTools(); // 提供的工具集 } // 热插拔的 Agent 容器 @Service public class PluggableAgentContainer { private final ConcurrentHashMap<String, AgentCapability> capabilities = new ConcurrentHashMap<>(); // 热注册能力 public void registerCapability(AgentCapability cap) { capabilities.put(cap.getName(), cap); } // 热移除能力 public void unregisterCapability(String name) { capabilities.remove(name); } // 动态组合当前可用工具 public ToolCallback[] getActiveTools() { return capabilities.values().stream() .filter(AgentCapability::isAvailable) .flatMap(c -> Arrays.stream(c.getTools())) .toArray(ToolCallback[]::new); } }

微服务化 Agent 设计

比喻：微服务架构把一个大系统拆成多个独立部署的小服务。微服务化 Agent 也是如此 -- 每个 Agent 作为独立的服务运行，通过 API 网关统一暴露，通过消息队列异步通信。

微服务化 Agent 的优势：

  • 独立部署：每个 Agent 可以独立更新、扩缩容，不影响其他 Agent
  • 技术异构：不同 Agent 可以用不同语言实现（Python 做 ML 推理，Java 做业务逻辑）
  • 故障隔离：一个 Agent 崩溃不会拖垮整个系统
  • 弹性伸缩：根据各 Agent 的负载独立扩缩容

⚠️ 注意事项

微服务化 Agent 引入了网络延迟、序列化开销和运维复杂度。不要为了"微服务而微服务" -- 只有当 Agent 的独立部署和扩缩容确实带来价值时，才值得引入这一复杂度。

📄 Agent-as-a-Service 架构

Agent 注册与发现

比喻：Agent 注册中心就像一个人才市场。Agent 进入市场时"登记简历"（注册元数据：名称、能力、版本、SLA），需求方根据"岗位要求"（任务描述）在市场中"搜索匹配的 Agent"（服务发现），然后"雇佣"（调用）完成任务。

注册中心需要存储的信息：

  • Agent 身份：唯一 ID、名称、版本号
  • 能力描述：支持的任务类型、输入输出格式、支持的工具集
  • 运行状态：健康状态、当前负载、响应时间
  • SLA 约定：可用性、最大并发、超时时间

负载均衡策略

当多个 Agent 实例提供相同能力时，需要负载均衡策略来分配请求。不同的策略适用于不同场景：

策略原理适用场景
轮询 (Round Robin)依次分配给每个实例实例性能相近的无状态 Agent
最少连接分配给当前连接最少的实例请求处理时间差异大的场景
加权轮询按权重比例分配不同规格实例混合部署
基于能力路由根据任务类型路由到专项 Agent多 Agent 专业分工场景
基于成本路由选择成本最低的 Agent预算敏感的批量任务

多租户隔离

比喻：多租户隔离就像一个共享办公楼。不同公司（租户）在同一栋楼里办公，但各有独立的办公空间、门禁系统和网络。它们共享大楼的基础设施（电梯、空调），但互不干扰。

Agent 多租户隔离的关键维度：

  • 资源隔离：每个租户有独立的计算资源配额（CPU、内存、API 调用额度）
  • 数据隔离：租户间的对话历史、向量库数据严格隔离
  • 模型隔离：可以为不同租户配置不同的 LLM 模型和 Prompt 策略
  • 流量隔离：租户级别的限流和优先级控制，防止噪声邻居问题

// ============================================ // 架构决策器 - 根据任务特征自动选择 Agent 架构 // ============================================ @Service public class ArchitectureDecisionMaker { private final ChatClient chatClient; public ArchitectureDecisionMaker(ChatClient chatClient) { this.chatClient = chatClient; } /** * 分析任务特征，决定使用哪种架构模式 */ public ArchitectureChoice decide(String task) { // 1. 分析任务复杂度 TaskAnalysis analysis = analyzeTask(task); // 2. 决策树匹配 if (analysis.requiresMultiStep() && analysis.requiresPlanning()) { return new ArchitectureChoice( ArchitectureType.PLAN_EXECUTE, "多步骤+需要规划"); } if (analysis.requiresExploration() && analysis.needsMultipleCandidates()) { return new ArchitectureChoice( ArchitectureType.LATS, "探索性+需要多方案比较"); } if (analysis.requiresIteration() && analysis.qualityOverSpeed()) { return new ArchitectureChoice( ArchitectureType.REFLECTION, "需要迭代优化+质量优先"); } // 默认使用 ReAct return new ArchitectureChoice( ArchitectureType.REACT, "通用任务"); } private TaskAnalysis analyzeTask(String task) { return chatClient.prompt() .system(""" 分析以下任务的特征，返回 JSON: - requiresMultiStep: 是否需要多步骤 - requiresPlanning: 是否需要先规划 - requiresExploration: 是否需要探索 - requiresIteration: 是否需要迭代 - needsMultipleCandidates: 是否需要多方案 - qualityOverSpeed: 质量是否优先于速度 """) .user(task) .call() .entity(TaskAnalysis.class); } } // 架构类型枚举 public enum ArchitectureType { REACT, PLAN_EXECUTE, LATS, REFLECTION } // 架构选择结果 public record ArchitectureChoice( ArchitectureType type, String reason ) {}

📖 章节小结

  • 架构决策框架 — 用决策树根据任务复杂度、探索需求、质量要求选择 ReAct / Plan-Execute / LATS / Reflection，遵循"从简单开始"原则
  • 四大架构模式 — ReAct 适合单步简单任务；Plan-Execute 适合多步骤流程；LATS 适合探索性创意任务；Reflection 适合需要迭代优化的高质量输出
  • Actor-Critic 与 MRKL — Actor-Critic 将执行与评估分离；MRKL 通过模块化路由器将 LLM 与专用工具解耦，各自发挥优势
  • 可组合 Agent 设计 — 通过标准接口、依赖注入、热插拔实现 Agent 能力的自由组合；微服务化带来独立部署和弹性伸缩的能力，但也引入运维复杂度
  • Agent-as-a-Service — 注册中心管理 Agent 元数据和健康状态；负载均衡策略包括轮询、最少连接、基于能力路由等；多租户隔离需要从资源、数据、模型、流量四个维度保障

🎓 面试高频题

👨‍💻 Q1: ReAct 和 Plan-and-Execute 的核心区别是什么？分别适用于什么场景？

ReAct 是"边想边做"的模式，每一步都是 思考-行动-观察 的循环，适合简单、可预测的任务，如单步工具调用、问答。优势是实现简单、延迟低；劣势是在复杂多步骤任务中容易迷失方向或重复探索。

Plan-and-Execute 是"先想后做"的模式，先用 LLM 生成全局计划，再逐步执行。适合复杂多步骤任务，如数据分析流水线、自动化报告。优势是全局最优、减少无效调用；劣势是规划阶段延迟高，计划可能与实际执行不匹配需要 Re-Plan。

👨‍💻 Q2: 如何设计一个可热插拔的 Agent 系统？

关键设计要点：
1. 标准化接口：定义 AgentCapability 接口，包含 getName、getTools、isAvailable 方法
2. 能力注册中心：使用 ConcurrentHashMap 管理所有已注册的能力，支持运行时增删
3. 动态工具组合：每次执行前从注册中心获取当前可用的 ToolCallback 数组，传入 ChatClient
4. 健康检查：定时检查各能力的后端服务是否可用，自动标记不可用的能力
5. 降级策略：当某个能力不可用时，Agent 能自动调整行为，用剩余能力完成任务或返回降级结果

👨‍💻 Q3: Agent 多租户隔离需要从哪些维度设计？

四个核心维度：
1. 资源隔离：每个租户有独立的 API 调用配额、并发限制和超时配置，防止噪声邻居
2. 数据隔离：对话历史、向量数据库按 tenant_id 隔离，Prompt 模板和工具配置也按租户独立
3. 模型隔离：不同租户可以绑定不同的 LLM 模型（如 VIP 用 GPT-4，普通用户用 GPT-3.5）和自定义 Prompt 策略
4. 流量隔离：租户级别的限流和优先级队列，确保高优先级租户的请求优先处理

👨‍💻 Q4: MRKL 系统和简单的工具调用有什么区别？

简单工具调用是 LLM 直接决定调用哪个工具，所有工具都是"平级"的。MRKL 的核心区别在于引入了路由器（Router）模块，根据任务类型将请求分发到最适合的专用模块：LLM 处理自然语言，符号计算器处理精确数学，知识库处理事实检索。每个模块可以使用不同的技术栈和优化策略，而非全部依赖 LLM 的能力。MRKL 的优势在于可扩展性和精确性 -- 新增能力只需添加模块并注册，且数学计算等场景可以避免 LLM 的幻觉问题。

👨‍💻 Q5: 在生产环境中，如何选择 Agent 的负载均衡策略？

根据场景选择：
- 轮询：Agent 实例规格相同、请求处理时间相近时使用，简单高效
- 最少连接：请求处理时间差异大时使用，避免长请求阻塞某个实例
- 加权轮询：混合部署不同规格实例时使用，按 GPU/内存等权重分配
- 基于能力路由：多个 Agent 各有专长时使用，如搜索 Agent 处理检索任务、分析 Agent 处理数据任务
- 基于成本路由：预算敏感场景，优先路由到成本最低的 Agent（如本地小模型优先）
生产中通常组合使用：先按能力路由到专业 Agent 组，组内再按最少连接或加权轮询分发。

  • 为每种架构类型创建对应的 Agent 执行器（可以是简单的 stub 实现）
  • 编写单元测试，验证不同任务描述能正确路由到对应架构
✍️ 练习 2：构建 Agent 注册中心

实现一个支持服务发现的 Agent 注册中心：

  • 创建 AgentInfo record 和 AgentRegistry 服务
  • 实现 register、discover、healthCheck 方法
  • 注册 3 个不同能力的 Agent（如搜索、计算、翻译）
  • 实现基于能力的发现逻辑：输入能力名称，返回匹配的 Agent 列表
  • 模拟一个 Agent 宕机场景，验证健康检查和自动摘除功能
✍️ 练习 3：微服务化 Agent 部署

将你的 Agent 能力拆分为独立的 Spring Boot 微服务：

  • 创建 2-3 个独立的 Spring Boot 项目，每个实现一种 Agent 能力
  • 每个微服务暴露 REST API（如 /agent/search、/agent/calculate）
  • 创建一个网关服务，接收用户请求后路由到对应的 Agent 微服务
  • 实现简单的健康检查端点（/actuator/health）
  • 使用 Spring Cloud LoadBalancer 实现客户端负载均衡
