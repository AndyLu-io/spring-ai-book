第 07 章

高级特性

掌握 Function Calling、结构化输出、会话记忆等 Spring AI 高级能力，构建更智能的应用

预计 2-3 周 5 个知识点 中高级难度

📚 开篇故事：从"纸上谈兵"到"真刀真枪"

小李的 AI 客服系统终于上线了，用户反馈还不错。但领导又提了新需求："能不能让 AI 帮用户查订单？查天气？甚至帮用户下单？"

小李一脸懵："AI 不是只会聊天吗？它怎么查订单？"

直到他发现了 Function Calling。这项技术让 AI 从"纸上谈兵"变成了"真刀真枪" -- AI 不仅能回答问题，还能调用外部函数、访问数据库、调用 API，就像给 AI 装上了"手脚"。

更让他惊喜的是，AI 还能记住之前的对话（Chat Memory），像一个贴心的私人助理；能自动判断该调用哪个函数（Tool Use），像一个聪明的机器人管家；能返回结构化数据（Structured Output），像一个严谨的财务报表。

本章，我们就来给 AI 装上"手脚"、配上"笔记本"、教会它"管家技能"，让它从一个只会聊天的"话痨"，变成一个能真正帮你干活的"智能助手"。

🧹 Function Calling

Function Calling = "给 AI 装上手脚" 🧹

大语言模型有一个致命缺陷：它只能"想"，不能"做"。它能告诉你"北京今天 25 度"，但它自己并不知道现在北京几度 -- 它只是在"编"一个合理的答案。

Function Calling 就是给 AI 装上"手脚" -- 让它不仅能"想"（生成文本），还能"做"（调用函数、访问 API、查询数据库）。就像从一个只能坐轮椅的哲学家，变成了一个能跑能跳的全能助手。

工作流程图

用户: "北京今天天气怎么样？"
    |
    v
LLM 思考: 我不知道天气，但我有一个 getWeather 函数可以用！
    |
    v
LLM 返回: { "function": "getWeather", "args": { "city": "北京" } }
    |
    v
你的应用: 调用 getWeather("北京") --> 天气 API
    |
    v
函数结果: { temp: 25, condition: "晴" }
    |
    v
LLM 生成回答: "北京今天晴天，温度 25°C，适合外出。"

Spring AI Function Calling 实现

// 🎯 第一步：定义请求和响应类型（告诉 LLM 函数的"说明书"） public record WeatherRequest( @JsonPropertyDescription("城市名称，如 北京、上海") // 帮助 LLM 理解参数 String city ) {} public record WeatherResponse( String city, // 城市 double temperature, // 温度 String condition // 天气状况 ) {} // 🎯 第二步：注册函数（把"手脚"装上去） @Bean public Function<WeatherRequest, WeatherResponse> getWeather() { return request -> { // 这里执行真正的业务逻辑 var response = restTemplate.getForObject( "https://api.weather.com/city/" + request.city(), WeatherResponse.class); return response; }; } // 🎯 第三步：在对话中使用函数（让 AI 学会用"手脚"） String answer = chatClient.prompt() .user("北京今天天气怎么样？") .functions("getWeather") // 注册函数！LLM 会自动判断是否需要调用 .call() .content(); // 结果: "北京今天晴天，温度 25°C，非常适合户外活动。"
💡 原理揭秘
Spring AI 会自动将函数签名转换为 JSON Schema，注入到 Prompt 中告诉 LLM "你有哪些函数可用"。LLM 根据 Schema 判断何时调用哪个函数，返回函数名和参数。你的应用执行函数后，将结果返回给 LLM 生成最终回答。

多函数注册 -- AI 的"多功能工具箱" 🔧

你可以同时注册多个函数，就像给 AI 一个"多功能工具箱"。LLM 会根据用户的问题，自动判断该用哪个工具。

@Configuration public class FunctionConfig { @Bean public Function<WeatherRequest, WeatherResponse> getWeather() { return req -> weatherService.getWeather(req.city()); // 天气工具 } @Bean public Function<SearchRequest, SearchResponse> searchWeb() { return req -> searchService.search(req.query()); // 搜索工具 } @Bean public Function<CalcRequest, CalcResponse> calculate() { return req -> calcService.calculate(req.expression()); // 计算器工具 } } // 🎯 LLM 自动判断需要调用哪些函数 String answer = chatClient.prompt() .user("北京天气怎么样？顺便搜索一下最新的新闻") .functions("getWeather", "searchWeb") // 同时提供两个工具 .call() .content(); // LLM 会自动调用 getWeather 和 searchWeb 两个函数
💡 你知道吗？
LLM 可以在一个对话回合中调用多个函数。比如用户说"查一下北京天气和上海天气"，LLM 会返回两个函数调用请求，你的应用依次执行后，将两个结果一起返回给 LLM 生成综合回答。

🔧 方法引用方式注册 -- @Tool 注解

除了 Function Bean 方式，Spring AI 还支持用 @Tool 注解直接在 Service 方法上定义工具，更加简洁直观。

@Service public class ToolService { @Tool(description = "根据城市名称查询天气信息") // 告诉 LLM 这个工具干嘛的 public WeatherResponse getWeather( @ToolParam(description = "城市名称，如 北京") String city) { return weatherClient.fetch(city); } @Tool(description = "执行数学计算，支持加减乘除和括号") public double calculate( @ToolParam(description = "数学表达式，如 (3+5)*2") String expression) { return evalEngine.evaluate(expression); } @Tool(description = "查询数据库中的用户信息") public UserInfo findUser( @ToolParam(description = "用户ID") long userId) { return userRepository.findById(userId).orElse(null); } } // 🎯 使用 -- 传入工具类实例即可 String answer = chatClient.prompt() .user("计算 (123 + 456) * 789") .tools(toolService) // 传入整个工具类！所有 @Tool 方法都会被注册 .call() .content(); // LLM 会自动调用 calculate("(123 + 456) * 789")
💡 @Tool vs Function Bean
  • @Tool -- 更简洁，声明式定义，适合简单工具
  • Function Bean -- 更灵活，适合需要复杂逻辑的工具
  • 两者可以混合使用，根据场景选择

⚠️ Function Calling 常见陷阱

陷阱表现解决方案
描述不清晰 LLM 不知道何时调用 用自然语言清晰描述每个函数的用途
参数类型错误 LLM 传错参数类型 使用 @JsonPropertyDescription 描述参数
无限循环 函数调用结果又触发新调用 设置最大调用次数限制
安全风险 LLM 调用危险函数 只暴露必要函数，添加权限校验
// 注册时传入工具类实例 String answer = chatClient.prompt() .user("计算 123 * 456") .tools(toolService) .call() .content();

🔧 Tool Use 模式

Tool Use = "AI 的工具箱" 🔧

如果说 Function Calling 是"给 AI 装上手脚"，那 Tool Use 就是"给 AI 配上一个完整的工具箱" -- 里面有锤子、螺丝刀、电钻、万用表...AI 根据任务需要，自动选择合适的工具。

Spring AI 中 Tool Use 是 Function Calling 的增强版，支持更丰富的工具定义和编排：

特性Function CallingTool Use
定义方式Function Bean + 手动注册@Tool 注解 + 自动扫描
参数描述@JsonPropertyDescription@ToolParam（更丰富）
安全控制需要手动实现内置拦截机制
适用场景简单函数调用复杂工具编排

数据库查询工具

@Service public class DatabaseTools { private final JdbcTemplate jdbcTemplate; @Tool(description = "查询数据库中的用户信息") public List<Map<String, Object>> queryUsers( @ToolParam(description = "查询条件，如 name='张三'") String condition) { // ⚠️ 生产环境必须用参数化查询！这里仅为示例 String sql = "SELECT id, name, email FROM users WHERE " + condition; return jdbcTemplate.queryForList(sql); } @Tool(description = "统计指定表的记录数") public long countRecords( @ToolParam(description = "表名") String tableName) { // ⚠️ 白名单校验表名，防止 SQL 注入 if (!Set.of("users", "orders", "products").contains(tableName)) { throw new IllegalArgumentException("不允许查询该表"); } return jdbcTemplate.queryForObject( "SELECT COUNT(*) FROM " + tableName, Long.class); } } // 🎯 使用 -- LLM 会自动判断该调用哪个工具 String answer = chatClient.prompt() .user("查询所有名字包含张的用户") .tools(databaseTools) // 注册工具类 .call() .content();
⚠️ 安全警告

数据库查询工具存在 SQL 注入风险！生产环境必须：(1) 使用参数化查询；(2) 只允许 SELECT 操作；(3) 表名白名单校验；(4) 设置查询超时；(5) 限制返回行数。

外部 API 调用工具

@Service public class ApiTools { @Tool(description = "调用翻译API将文本翻译为目标语言") public String translate( @ToolParam(description = "要翻译的文本") String text, @ToolParam(description = "目标语言代码，如 en/zh/ja") String targetLang) { return translationService.translate(text, targetLang); } @Tool(description = "获取指定城市的实时汇率") public ExchangeRate getExchangeRate( @ToolParam(description = "源货币代码，如 USD/CNY") String from, @ToolParam(description = "目标货币代码") String to) { return exchangeRateClient.getRate(from, to); } }

📊 结构化输出

结构化输出 = "AI 的报表生成器" 📊

LLM 的输出通常是自由文本，像一篇散文。但在程序中，我们往往需要结构化数据 -- JSON、XML 或 Java 对象。结构化输出就是让 AI 按照你指定的格式输出，就像让 AI 帮你"填表单"。

工作原理图

你的 Java Record (CodeReview)
    |
    v  转换为 JSON Schema
System Prompt: "请按照以下 Schema 输出 JSON: ..."
    |
    v  LLM 按 Schema 生成
LLM 输出: { "summary": "...", "issues": [...], "score": 85 }
    |
    v  Jackson 反序列化
Java 对象: CodeReview 实例 ✔

代码审查 -- 结构化输出实战

// 🎯 第一步：定义输出结构 -- 告诉 AI "按这个格式填表" public record CodeReview( String summary, // 审查摘要 List<Issue> issues, // 问题列表 int score, // 评分 (0-100) String suggestion // 改进建议 ) {} public record Issue( String severity, // 严重程度: HIGH/MEDIUM/LOW String description, // 问题描述 String line // 出问题的代码行 ) {} // 🎯 第二步：使用 entity() 自动解析 CodeReview review = chatClient.prompt() .user("审查这段代码：\n" + codeSnippet) .call() .entity(CodeReview.class); // 自动解析为 Java 对象！ // 🎯 第三步：像使用普通 Java 对象一样使用 System.out.println("评分: " + review.score()); // 85 System.out.println("问题数: " + review.issues().size()); // 3 review.issues().forEach(issue -> { System.out.println("[" + issue.severity() + "] " + issue.description()); });

结构化输出 vs 自由文本

对比维度自由文本 (.content())结构化输出 (.entity())
输出格式纯文本字符串类型安全的 Java 对象
后续处理需要手动解析直接使用
可靠性格式不固定JSON Schema 约束
适用场景聊天、写作数据分析、报表、API
推荐度⭐⭐⭐⭐⭐⭐⭐⭐（程序调用首选）
💡 建议
只要你的程序需要消费 AI 的输出，就优先使用 .entity()。它能避免大量的 JSON 解析和字段映射代码。

流式结构化输出

// 流式返回结构化对象 Flux<ChatResponse> flux = chatClient.prompt() .user("分析这段代码的安全问题") .options(OpenAiChatOptions.builder() .responseFormat(new ResponseFormat(ResponseFormat.Type.JSON_SCHEMA)) .build()) .stream() .chatResponse();

📝 会话记忆管理

Chat Memory = "AI 的笔记本" 📝

默认情况下，LLM 是无状态的 -- 每次对话都是独立的，它不记得你上一句说了什么。就像一个没有笔记本的侦探，每次问话都要从头开始。

Chat Memory 就是给 AI 配上一个"笔记本" -- 它会把之前的对话记录下来，下次聊天时先翻翻笔记本，回忆一下之前说过什么。

有记忆 vs 无记忆对比

🔴 无记忆 AI                    🟢 有记忆 AI

用户: "我叫张三"                用户: "我叫张三"
AI: "你好！"                   AI: "你好，张三！"
                               | (记录到笔记本)
用户: "我叫什么？"              用户: "我叫什么？"
AI: "我不知道你叫什么"  ❌     AI: "你叫张三"  ✅
                               | (从笔记本回忆)

ChatMemory 实现

@Service public class MemoryChatService { private final ChatClient chatClient; private final ChatMemory chatMemory; // 笔记本 public String chat(String userId, String message) { return chatClient.prompt() .system("你是一个有帮助的助手") .advisors(new MessageChatMemoryAdvisor( chatMemory, // 笔记本 userId, // 每个用户一个笔记本 10 // 保留最近 10 条记录（笔记本只有 10 页） )) .user(message) .call() .content(); } }

笔记本存在哪里？存储方案对比

方案特点适用场景数据安全
InMemoryChatMemory 内存存储，零配置 开发测试 🔴 重启丢失
JdbcChatMemory 关系数据库存储 生产环境 🟢 持久化
CassandraChatMemory 分布式存储 大规模、高可用 🟢 持久化 + 高可用

@Bean public ChatMemory chatMemory() { return new InMemoryChatMemory(); }

JdbcChatMemory（生产环境）

@Bean public ChatMemory chatMemory(DataSource ds) { return new JdbcChatMemory(ds); }

记忆窗口管理 -- 笔记本只有 10 页

笔记本不能无限写，否则会"装满"。记忆窗口控制 AI "记住最近多少条消息"。

// 🎯 控制记忆窗口大小 -- 只保留最近 10 条 MessageChatMemoryAdvisor advisor = new MessageChatMemoryAdvisor( chatMemory, userId, 10 // 保留最近 10 条消息（笔记本只有 10 页） ); // 🎯 配合定时清理过期记忆 @Scheduled(fixedRate = 3600000) // 每小时清理一次 public void cleanExpiredMemory() { chatMemory.clearExpired(Duration.ofHours(24)); // 清理 24 小时前的记忆 }
场景建议窗口大小Token 预估
客服对话5-10 条约 500-1000 tokens
个人助手10-20 条约 1000-2000 tokens
代码审查3-5 条约 300-500 tokens
深度分析20-50 条约 2000-5000 tokens
💡 选择建议
记忆窗口越大，AI 的上下文越丰富，但 Token 消耗也越多。建议根据对话类型动态调整：客服场景保留 5-10 条，助手场景保留 10-20 条。

🎯 完整实战：带记忆的聊天 REST API

@RestController @RequestMapping("/api/v1/chat") public class ChatController { private final ChatClient chatClient; private final ChatMemory chatMemory; @PostMapping("/memory") public ResponseEntity<ChatResponse> chatWithMemory( @RequestBody ChatRequest request) { String reply = chatClient.prompt() .system("你是一个有帮助的助手，记住用户说过的话。") .advisors(new MessageChatMemoryAdvisor( chatMemory, request.userId(), // 每个用户独立的记忆 10 // 保留最近 10 条 )) .user(request.message()) .call() .content(); return ResponseEntity.ok(new ChatResponse(reply, Instant.now())); } } // 测试对话： // 1. POST /api/v1/chat/memory {"userId":"u1", "message":"我叫张三"} // -> "你好，张三！有什么可以帮你的？" // 2. POST /api/v1/chat/memory {"userId":"u1", "message":"我叫什么？"} // -> "你叫张三。" (AI 记住了！)

📈 Observability 监控

Observability = "AI 的体检报告" 📈

AI 应用上线后，你需要知道：延迟多少？Token 消耗多少？错误率多高？成本花了多少？Observability 就是给 AI 做"体检"，让你随时掌握 AI 的"健康状况"。

监控指标一览

📈 AI 应用体检报告

  ⏱ 请求延迟    P50: 800ms  P95: 2.1s  P99: 4.5s
  💰 Token 消耗   输入: 1.2M  输出: 3.8M  总计: 5.0M
  💲 成本估算    今日: $12.5  本月: $285.3
  ⚠️ 错误率      0.3% (12/4000 请求)
  📱 模型分布    GPT-4o: 60%  Claude: 30%  Gemini: 10%

Spring AI 可观测性配置

# application.yml - 开启监控 management: tracing: sampling: probability: 1.0 # 100% 采样（生产环境建议 0.1） observability: metrics: enabled: true # 开启指标采集 zipkin: tracing: endpoint: http://localhost:9411/api/v2/spans

自定义指标采集

@Service public class AiMetricsService { private final MeterRegistry meterRegistry; /** * 记录 AI 调用的关键指标 */ public void recordChatMetrics(ChatResponse response, long durationMs) { // 📈 记录延迟（直方图） meterRegistry.timer("ai.chat.duration") .record(durationMs, TimeUnit.MILLISECONDS); // 💰 记录 Token 用量（计数器） TokenUsage usage = response.getMetadata().getUsage(); meterRegistry.counter("ai.chat.tokens.input") .increment(usage.getPromptTokens()); meterRegistry.counter("ai.chat.tokens.output") .increment(usage.getGenerationTokens()); // 📱 记录模型使用分布（带标签） meterRegistry.counter("ai.chat.requests", "model", response.getMetadata().getModel()) .increment(); } /** * 计算成本估算 * GPT-4o: 输入 $2.5/M tokens, 输出 $10/M tokens */ public double estimateCost(TokenUsage usage) { double inputCost = usage.getPromptTokens() * 2.5 / 1_000_000; double outputCost = usage.getGenerationTokens() * 10.0 / 1_000_000; return inputCost + outputCost; } }

监控面板关键指标

指标含义告警阈值建议
⏱ P50 延迟50% 请求的响应时间< 1s
⏱ P99 延迟99% 请求的响应时间< 10s
💰 Token 消耗输入/输出 Token 数趋势日环比增长 > 50%
⚠️ 错误率API 调用失败比例> 1%
💲 成本基于 Token 用量计算费用日费用 > $50
📱 模型分布不同模型的使用比例-
💡 省钱技巧

监控发现 80% 的请求都是简单问答，GPT-4o-mini 就够用了。通过模型路由，简单任务用便宜模型，复杂任务用强模型，成本可以降低 60%+。

📖 章节小结

本章我们掌握了 Spring AI 的四大高级能力：

  • Function Calling = "给 AI 装上手脚" -- 让 AI 能调用外部函数和 API
  • Tool Use = "AI 的工具箱" -- 声明式工具定义 + 安全控制
  • Chat Memory = "AI 的笔记本" -- 让 AI 记住之前的对话
  • Structured Output = "AI 的报表生成器" -- 输出类型安全的 Java 对象
  • Observability = "AI 的体检报告" -- 监控延迟、Token、成本

这些能力让 AI 从一个"只会聊天的话痨"变成了一个"能真正干活的智能助手"。掌握它们，你就能构建出功能强大、安全可靠、可监控的 AI 应用。

🎓 面试高频题

👨‍💻 Q1: Function Calling 的工作原理是什么？

(1) 应用将函数签名（JSON Schema）注入 Prompt；(2) LLM 根据用户问题判断是否需要调用函数；(3) 如果需要，LLM 返回函数名和参数（JSON）；(4) 应用执行函数，将结果返回给 LLM；(5) LLM 基于函数结果生成最终回答。关键点：LLM 本身不执行函数，只是"决定"调用哪个函数。

👨‍💻 Q2: Chat Memory 有哪些存储方案？如何选择？

InMemoryChatMemory（开发测试）、JdbcChatMemory（生产环境）、CassandraChatMemory（大规模高可用）。选择依据：数据量、持久化需求、运维能力。注意设置记忆窗口大小，避免 Token 溢出。生产环境建议保留最近 10-20 条消息。

👨‍💻 Q3: 如何防止 Function Calling 的安全风险？

(1) 输入校验 -- 防止注入攻击；(2) 权限控制 -- 只暴露必要函数；(3) 输出过滤 -- 防止敏感数据泄露；(4) 执行沙箱 -- 限制函数的执行环境；(5) 审计日志 -- 记录所有函数调用。数据库工具必须禁止 DROP/DELETE/UPDATE。

👨‍💻 Q4: 结构化输出的原理是什么？

Spring AI 将 Java Record/Class 转换为 JSON Schema，注入到 System Prompt 中引导 LLM 输出符合 Schema 的 JSON。LLM 生成 JSON 后，用 Jackson 反序列化为 Java 对象。关键：Record 类型效果最好，因为字段名和类型信息会被准确映射。
