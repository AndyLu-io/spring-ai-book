Agent 规划与推理

掌握主流推理范式，理解 CoT、ToT、GoT 等推理技术，设计和实现具备自主规划能力的 Agent 系统

预计 2-3 周 8 个知识点 高级难度

📚 开篇故事：从厨师到主厨

小李是一家餐厅的学徒厨师。刚开始，他只会照着菜谱一步一步做菜 — 放多少盐、炒几分钟、什么时候翻面，全都写在纸上。这就是Chain-of-Thought（思维链）：一步一步、线性思考。

一年后，小李升为了副厨。他开始在脑中同时考虑多条路线："如果用红烧，要40分钟；如果清蒸，只要15分钟，但需要先准备蒸锅..." 他会在几个方案中同时探索、比较优劣，然后选最好的那条路。这就是Tree-of-Thought（思维树）。

三年后，小李成为了主厨。他不再只是比较几个方案，而是把不同的烹饪技法、食材搭配、客人反馈编织成一张知识网络。他知道红烧可以借鉴日式炖煮的技巧，清蒸的摆盘可以从法餐里学 — 各种知识互相连接、互相增强。这就是Graph-of-Thought（思维图）。

从学徒到主厨的进化之路，正是 Agent 推理能力的进化之路。本章将带你深入理解这三种推理范式，以及如何让 Agent 具备自主规划和反思的能力。

🎯 学习目标

在本章结束后，你将能够：

  • 掌握推理范式 — 理解 CoT、ToT、GoT 三种推理方法的原理、适用场景和优劣对比
  • 设计规划系统 — 能够设计 HTN 分层任务网络、目标分解、动态重规划等规划架构
  • 实现反思机制 — 掌握自我反思、置信度校准、知道何时放弃等元认知能力
  • 工程化落地 — 使用 Spring AI 实现规划器模式、反思循环和多步推理链

🧠 推理范式：从线性到网状

概念比喻：三种思考方式

想象你在解一道数学应用题。Chain-of-Thought 就像在草稿纸上从第一步推到第二步，线性前进；Tree-of-Thought 就像同时尝试三种解法，比较哪条路走得通；Graph-of-Thought 则像把多种解法的知识碎片互相串联，形成一张推理网络。

三种推理范式对比

Chain-of-Thought (CoT) — 线性链
  A ──> B ──> C ──> D ──> 答案
  一步一步，前因后果

Tree-of-Thought (ToT) — 分支树
                    ┌─── B1 ──> C1 ──> D1
          A ──> ├─── B2 ──> C2
          |       └─── B3 ──> C3 ──> D2
  多条路径并行探索，回溯剪枝

Graph-of-Thought (GoT) — 网状图
          ┌─── B1 ──> C1 ───┐
  A ──> ├─── B2 ──> C2 ──> D ──> 答案
          |       ^       |
          └─── B3 ────────┘
  节点可合并、可循环、形成网络

Chain-of-Thought（思维链）

CoT 是最基础也最常用的推理范式。它要求 LLM 在给出最终答案之前，先展示完整的推理步骤。就像一个学生在考试时被要求"写出解题过程"。

核心思想：将复杂问题分解为一系列中间推理步骤，每个步骤的输出作为下一步的输入。

典型 Prompt："Let's think step by step." — 这句经典的提示词就能显著提升推理准确率。

适用场景：数学计算、逻辑推理、多步骤判断等有明确因果链的问题。

局限性：只能沿一条路径前进，如果中间某步推理错误，后续全部偏离，无法回溯。

Tree-of-Thought（思维树）

ToT 在 CoT 基础上引入了分支探索和回溯。每一步推理时，LLM 会生成多个候选思路，通过评估函数（可以是 LLM 自身或外部验证器）打分，保留最优分支，剪掉劣质路径。

核心思想：把推理过程建模为一棵搜索树，每个节点是一个中间状态，通过 BFS 或 DFS 探索最优路径。

搜索策略：

  • BFS（广度优先） — 每层保留 Top-K 候选，适合搜索空间不大的问题
  • DFS（深度优先） — 沿一条路走到底，不通则回溯，适合路径较长的问题

适用场景：24点游戏、创意写作、规划类问题等需要探索多种可能性的任务。

局限性：Token 消耗是 CoT 的 3-5 倍，且需要设计可靠的评估函数来判断每个分支的质量。

Graph-of-Thought（思维图）

GoT 是最强大的推理范式。它将推理过程建模为一个有向图，节点是推理状态，边是推理操作。与 ToT 不同的是，GoT 支持节点合并（将多个思路综合为一个更优的思路）和循环推理（反复验证和精炼）。

核心思想：推理不是一棵树，而是一张网。不同的推理路径可以相互借鉴、合并增强。

关键操作：

  • Generate（生成） — 从一个节点扩展出多个候选子节点
  • Aggregate（聚合） — 将多个节点的信息合并为一个更完整的节点
  • Refine（精炼） — 对节点内容进行迭代改进

适用场景：需要综合多源信息的复杂推理，如多跳问答、跨文档推理、创意综合等。

三种范式对比

维度 CoT（思维链） ToT（思维树） GoT（思维图）
推理结构 线性链 分支树 有向图
探索能力 单路径 多路径 + 回溯 多路径 + 合并 + 循环
Token 消耗 低 中高（3-5x） 高（5-10x）
实现复杂度 低 中 高
容错能力 差（一步错全错） 中（可回溯） 强（可合并修正）
典型应用 数学推理、逻辑判断 游戏求解、创意生成 多跳问答、知识综合
💡 小窍门

实际工程中，三种范式往往组合使用。比如：用 CoT 处理每个单独步骤，用 ToT 在关键决策点做分支探索，用 GoT 在最终阶段综合所有信息。不要为了"高级"而盲目选择 GoT — 简单任务用 CoT 就足够了。

📋 规划策略：让 Agent 学会"先想后做"

概念比喻：项目经理的工具箱

一个优秀的项目经理不会上来就让团队干活。他会先把大目标拆成小任务（Goal Decomposition），理清任务之间的依赖关系（谁先做谁后做），然后按优先级排序执行。执行中遇到突发状况，还要动态调整计划（Re-Planning）。Agent 的规划策略，就是借鉴了这套项目管理方法论。

分层任务网络（HTN）

HTN 的核心思想是逐层分解：把一个高层任务不断拆解为更小的子任务，直到每个子任务都足够简单、可以直接执行。

类比来说，就像建筑施工：先拆成"地基 → 框架 → 装修"，再把"框架"拆成"浇筑混凝土 → 搭建钢筋 → 养护"，直到每个动作都是一个工人能完成的原子操作。

HTN 的优势：分解过程中天然形成了任务的层次结构，便于管理和追踪进度。上层任务负责"做什么"，下层任务负责"怎么做"。

在 Agent 中的应用：让 LLM 先生成高层计划，再逐层细化。每一层的分解都可以是一次 LLM 调用，输出结构化的子任务列表。

目标分解（Goal Decomposition）

目标分解关注的是任务之间的逻辑关系：

  • 顺序依赖 — B 必须在 A 之后执行（如：先搜索数据，再分析数据）
  • 并行独立 — A 和 B 可以同时执行（如：同时搜索中英文资料）
  • 条件分支 — 根据 A 的结果决定执行 B 还是 C（如：如果数据量大用方案A，否则用方案B）
  • 循环迭代 — 重复执行直到满足条件（如：反复修改直到通过审查）

在工程实现中，通常用DAG（有向无环图）来建模子任务之间的依赖关系，然后用拓扑排序确定执行顺序。

子目标排序与依赖分析

排好依赖关系后，还需要考虑执行顺序的优化：

  • 关键路径法 — 找出耗时最长的依赖链，优先保证关键路径上的资源供给
  • 尽早开始 — 没有前置依赖的子任务立刻并行启动，不浪费等待时间
  • 资源冲突检测 — 如果多个子任务需要同一资源（如同一个 API），需要排队或错峰

DAG 拓扑排序实现

下面用 Java 实现完整的 DAG（有向无环图）数据结构和 Kahn 拓扑排序算法，用于 Agent 的任务依赖调度。

1. 任务节点与 DAG 数据结构

// 任务节点：表示 DAG 中的一个顶点 public class TaskNode { private final String id; private final String description; private final List<String> dependencies; // 依赖的前置任务 ID private volatile TaskStatus status; public TaskNode(String id, String description, List<String> dependencies) { this.id = id; this.description = description; this.dependencies = List.copy(dependencies); // 不可变拷贝 this.status = TaskStatus.PENDING; } public String getId() { return id; } public String getDescription() { return description; } public List<String> getDependencies() { return dependencies; } public TaskStatus getStatus() { return status; } public void setStatus(TaskStatus status) { this.status = status; } public enum TaskStatus { PENDING, // 等待执行 READY, // 依赖已满足，可以执行 RUNNING, // 正在执行 COMPLETED, // 已完成 FAILED // 执行失败 } }

2. DAG 调度器 — Kahn 算法拓扑排序

// DAG 调度器：管理任务依赖、拓扑排序、循环检测 public class DAGScheduler { private final Map<String, TaskNode> tasks = new LinkedHashMap<>(); private final Map<String, Set<String>> adjList = new HashMap<>(); // 邻接表：任务 -> 依赖它的后续任务 private final Map<String, Integer> inDegree = new HashMap<>(); // 入度表 /** * 添加任务到 DAG * @throws IllegalArgumentException 如果添加会导致循环依赖 */ public void addTask(TaskNode task) { tasks.put(task.getId(), task); inDegree.putIfAbsent(task.getId(), 0); adjList.putIfAbsent(task.getId(), new LinkedHashSet<>()); // 建立反向边：前置任务 -> 当前任务 for (String depId : task.getDependencies()) { adjList.computeIfAbsent(depId, k -> new LinkedHashSet<>()).add(task.getId()); inDegree.merge(task.getId(), 1, Integer::sum); } } /** * Kahn 算法拓扑排序 * 返回按依赖关系排序的执行序列 * @return 拓扑序列表，若存在循环依赖则抛出异常 */ public List<TaskNode> topologicalSort() { List<TaskNode> sorted = new ArrayList<>(); Queue<String> queue = new LinkedList<>(); // 第一步：将所有入度为 0 的节点（无依赖）加入队列 for (Map.Entry<String, Integer> entry : inDegree.entrySet()) { if (entry.getValue() == 0) { queue.add(entry.getKey()); } } // 第二步：依次处理入度为 0 的节点 while (!queue.isEmpty()) { String currentId = queue.poll(); TaskNode currentNode = tasks.get(currentId); sorted.add(currentNode); // 将当前节点的所有后续任务入度减 1 for (String nextId : adjList.getOrDefault(currentId, Set.of())) { int newDegree = inDegree.merge(nextId, -1, Integer::sum); if (newDegree == 0) { queue.add(nextId); // 所有前置依赖完成，加入执行队列 } } } // 第三步：如果排序结果数量不等于任务总数，说明存在循环依赖 if (sorted.size() != tasks.size()) { throw new IllegalStateException( "检测到循环依赖！" + (tasks.size() - sorted.size()) + " 个任务无法执行"); } return List.copy(sorted); // 返回不可变副本 } /** * 获取当前可执行的任务（所有前置依赖已完成） */ public List<TaskNode> getReadyTasks() { return tasks.values().stream() .filter(t -> t.getStatus() == TaskNode.TaskStatus.PENDING) .filter(t -> t.getDependencies().stream() .allMatch(depId -> { TaskNode dep = tasks.get(depId); return dep != null && dep.getStatus() == TaskNode.TaskStatus.COMPLETED; })) .toList(); } /** * 标记任务完成，触发后续任务的依赖更新 */ public void completeTask(String taskId) { TaskNode task = tasks.get(taskId); if (task == null) throw new IllegalArgumentException("任务不存在: " + taskId); task.setStatus(TaskNode.TaskStatus.COMPLETED); } /** * 检测 DAG 中是否存在循环依赖（DFS 三色标记法） */ public boolean hasCycle() { Set<String> WHITE = new HashSet<>(tasks.keySet()); // 未访问 Set<String> GRAY = new HashSet<>(); // 正在访问（在递归栈中） Set<String> BLACK = new HashSet<>(); // 已完成 for (String nodeId : tasks.keySet()) { if (WHITE.contains(nodeId) && dfsDetect(nodeId, WHITE, GRAY, BLACK)) { return true; } } return false; } private boolean dfsDetect(String node, Set<String> white, Set<String> gray, Set<String> black) { white.remove(node); gray.add(node); for (String neighbor : adjList.getOrDefault(node, Set.of())) { if (gray.contains(neighbor)) return true; // 发现回边 → 有环 if (white.contains(neighbor) && dfsDetect(neighbor, white, gray, black)) { return true; } } gray.remove(node); black.add(node); return false; } }

3. 并行任务执行器

// 并行任务执行器：按拓扑层级并行调度任务 @Service public class ParallelTaskExecutor { private final ChatClient chatClient; private final ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor(); /** * 按拓扑层级并行执行所有任务 * 同一层级的任务互不依赖，可以并行执行 */ public CompletableFuture<Void> executeInOrder(DAGScheduler scheduler) { return CompletableFuture.runAsync(() -> { Set<String> completed = new ConcurrentHashMap<String, Boolean>().keySet(true); while (true) { // 获取所有前置依赖已完成的就绪任务 List<TaskNode> ready = scheduler.getReadyTasks(); if (ready.isEmpty() && completed.size() < count(scheduler)) { throw new IllegalStateException("死锁：存在无法满足的依赖"); } if (ready.isEmpty()) break; // 所有任务完成 // 并行执行当前层级的所有就绪任务 List<CompletableFuture<Void>> futures = ready.stream() .map(task -> CompletableFuture.runAsync(() -> { task.setStatus(TaskNode.TaskStatus.RUNNING); executeTask(task); synchronized (completed) { scheduler.completeTask(task.getId()); completed.add(task.getId()); } }, executor)) .toList(); // 等待当前层级所有任务完成后再进入下一层 CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join(); } }, executor); } /** * 执行单个任务：调用 LLM 完成任务描述 */ private void executeTask(TaskNode task) { try { String result = chatClient.prompt() .user("请执行以下任务：" + task.getDescription()) .call() .content(); System.out.println("任务完成 [" + task.getId() + "]: " + result); } catch (Exception e) { task.setStatus(TaskNode.TaskStatus.FAILED); throw new RuntimeException("任务执行失败: " + task.getId(), e); } } }

4. 完整使用示例

// 调研报告生成 — DAG 任务调度示例 @Service public class ResearchReportAgent { private final ParallelTaskExecutor executor; public String generateReport(String topic) { // 1. 构建 DAG 任务图 DAGScheduler scheduler = new DAGScheduler(); scheduler.addTask(new TaskNode("search_cn", "搜索中文资料", List.of())); scheduler.addTask(new TaskNode("search_en", "搜索英文资料", List.of())); scheduler.addTask(new TaskNode("analyze", "综合分析资料", List.of("search_cn", "search_en"))); scheduler.addTask(new TaskNode("write_draft", "撰写初稿", List.of("analyze"))); scheduler.addTask(new TaskNode("review", "质量审查", List.of("write_draft"))); scheduler.addTask(new TaskNode("polish", "润色定稿", List.of("review"))); // 2. 循环依赖检测 if (scheduler.hasCycle()) { throw new IllegalStateException("任务规划存在循环依赖，请检查"); } // 3. 拓扑排序，输出执行序列 List<TaskNode> order = scheduler.topologicalSort(); order.forEach(t -> System.out.println( "→ " + t.getId() + ": " + t.getDescription())); // 4. 并行调度执行 executor.executeInOrder(scheduler).join(); return "报告生成完成"; } } /* * 执行拓扑图： * * search_cn ──┐ * ├──▶ analyze ──▶ write_draft ──▶ review ──▶ polish * search_en ──┘ * * 执行层级： * Level 0: search_cn, search_en (并行) * Level 1: analyze (串行) * Level 2: write_draft (串行) * Level 3: review (串行) * Level 4: polish (串行) */

动态规划调整（Re-Planning）

现实世界是不确定的。Agent 执行计划时经常会遇到意外：API 调用失败、数据不符合预期、发现新的约束条件。Re-Planning 就是让 Agent 在执行过程中根据实际情况动态调整计划。

Re-Planning 的触发条件：

  • 某个步骤执行失败且无法重试恢复
  • 执行结果与预期偏差过大
  • 发现了新的信息改变了问题的前提假设
  • 外部环境发生变化（如截止时间提前）

局部重规划 vs 全局重规划：局部重规划只调整失败步骤及其后续依赖，成本低但可能遗漏全局影响；全局重规划从头重新制定所有步骤，成本高但更全面。实际中推荐先尝试局部重规划，失败后再全局重规划。

⚠️ 陷阱警告

规划不是越细越好。过度规划会导致计划僵化 — 当环境变化时，过于详细的计划反而难以调整。好的规划应该在"足够详细以指导执行"和"足够灵活以应对变化"之间取得平衡。推荐滚动式规划：只细化即将执行的步骤，远期步骤保持粗粒度。

💡 反思与元认知：Agent 的"自我觉察"

概念比喻：大脑的"审查委员会"

人类之所以能做出高质量的决策，不仅因为我们会"思考"，更因为我们能思考自己的思考 — 这就是"元认知"。一个优秀的棋手不只是计算下一步走哪，还会评估"我的计算是否可靠？这个局面我是否真的看懂了？" Agent 的反思与元认知，就是在推理过程中加入一层自我审查和自我校准。

Self-Reflection（自我反思）

自我反思是让 Agent 在完成推理后，回头审视自己的推理过程和结论，找出潜在的问题和改进点。

自我反思循环

  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ 推理/执行 │──>│ 生成结果 │──>│ 自我审查 │
  └──────────┘    └──────────┘    └────┬─────┘
       ^                               |
       |         ┌──────────┐          |
       +─────────│ 改进重试  │<──────────┘
                 └──────────┘  有问题
                  无问题 ──> 输出最终结果

反思的核心问题：

  • "我的推理过程中有没有逻辑跳跃？"
  • "有没有遗漏重要的信息或反面证据？"
  • "如果换一个角度思考，结论会不同吗？"
  • "这个结论的适用边界是什么？"

Reflexion 框架：将反思结果存储为语言化的"经验教训"，在后续任务中作为上下文传入，让 Agent 能从过去的错误中学习。这就像一个人写"错题本" — 不只是知道错了，更要知道为什么错、下次怎么避免。

Confidence Calibration（置信度校准）

一个好的推理系统不仅要知道"答案是什么"，还要知道"我对这个答案有多确定"。置信度校准就是让 Agent 给出推理结论的同时，评估自己结论的可靠程度。

为什么重要：如果 Agent 对自己的错误答案"迷之自信"，下游系统就会盲目信任错误结果。校准后的置信度可以用来决定：

  • 置信度高（> 0.9）— 直接采纳，自动执行
  • 置信度中等（0.5-0.9）— 作为参考，建议人工确认
  • 置信度低（< 0.5）— 标记为不确定，触发替代方案

实现方式：在 Prompt 中要求 LLM 同时输出答案和置信度分数（0-1），或者用多次采样（Self-Consistency）— 如果多次独立推理得到相同答案，说明置信度高。

Knowing When to Abstain（知道何时放弃）

这可能是 Agent 元认知中最重要也最难的能力。一个成熟的 Agent 应该知道自己的能力边界，在以下情况主动"举白旗"：

Agent 的决策矩阵

                    我有能力处理    超出能力范围
                   ┌──────────────┬──────────────┐
  信息充分        │  自信执行     │  请求人类帮助  │
                   ├──────────────┼──────────────┤
  信息不足        │  先补充信息   │  明确拒绝     │
                   └──────────────┴──────────────┘

  核心原则：不确定时，宁可不答也不乱答

需要放弃的信号：

  • 反复尝试同一方法多次仍然失败
  • 推理过程中发现前提假设不成立
  • 涉及超出知识范围的领域（如：最新的法律条款、个人隐私数据）
  • 置信度持续低于阈值
  • 推理步骤数超过合理上限

放弃不等于失败 — 一个能诚实说"我不确定"的 Agent，比一个胡乱给出答案的 Agent 可靠得多。在生产系统中，Agent 应该将不确定性传递给上层决策系统或人类审查者。

💡 你知道吗？

心理学中有一个著名的"达克效应"（Dunning-Kruger Effect）：能力越低的人越容易高估自己的能力。LLM 也有类似问题 — 它有时会对明显错误的答案给出高置信度。因此，外部校验机制（如事实核查、单元测试、人类反馈）是置信度校准的重要补充，不能完全依赖 LLM 的自我评估。

💻 Spring AI 实战：规划器与反思循环

规划器模式（Plan-and-Solve）

下面实现一个基于 Spring AI 的规划器，它会将复杂任务分解为步骤，然后逐步执行。

// 基于 Spring AI 的任务规划器 @Service public class TaskPlannerAgent { private final ChatClient chatClient; /** * 将复杂任务分解为可执行的步骤列表 */ public Plan createPlan(String task) { return chatClient.prompt() .system(""" 你是一个专业的任务规划专家。 请将复杂任务分解为可执行的步骤列表。 每个步骤应该是原子的、可验证的。 步骤之间标注依赖关系。 返回 JSON 格式: { "steps": [ { "id": 1, "description": "步骤描述", "dependsOn": [], "estimatedTokens": 500 } ] } """) .user(task) .call() .entity(Plan.class); } /** * 按照依赖关系执行计划 */ public String executePlan(String task, int maxReplans) { Plan plan = createPlan(task); Map<Integer, String> results = new HashMap<>(); for (int replan = 0; replan <= maxReplans; replan++) { for (Step step : plan.steps()) { // 检查依赖是否已完成 if (!dependenciesMet(step, results)) continue; // 执行步骤 String context = buildContext(step, results); String output = chatClient.prompt() .user("任务: " + step.description() + "\n上下文: " + context) .call() .content(); results.put(step.id(), output); } // 反思：评估计划执行效果 if (isSatisfactory(task, results)) break; // 不满意则重新规划剩余步骤 plan = replan(task, results, plan); } return synthesize(results); } }

反思循环（Reflection Loop）

实现一个自我反思机制，让 Agent 在生成结果后自动评估和改进。

// 反思循环：生成 → 评估 → 改进 @Service public class ReflectionLoopAgent { private final ChatClient chatClient; private final int maxIterations = 3; public ReflectionResult solveWithReflection(String task) { String output = ""; List<String> critiques = new ArrayList<>(); for (int i = 0; i < maxIterations; i++) { // 1. 生成/改进输出 output = generate(task, critiques); // 2. 自我反思评估 Reflection reflection = chatClient.prompt() .system(""" 你是一个严格的评审者。请评估以下结果: 1. 给出 0-1 的质量分数 2. 指出存在的问题 3. 提供改进建议 用 JSON 格式回复。 """) .user("任务: " + task + "\n结果: " + output) .call() .entity(Reflection.class); // 3. 置信度校准：分数达标则输出 if (reflection.score() >= 0.85) { return new ReflectionResult( output, reflection.score(), i + 1); } // 4. 记录批评，用于下一轮改进 critiques.add(reflection.feedback()); } // 达到最大迭代次数，返回当前最佳结果 return new ReflectionResult(output, 0, maxIterations); } } public record Reflection( double score, String issues, String feedback ) {} public record ReflectionResult( String output, double confidence, int iterations ) {}

多步推理链（Multi-step Reasoning Chain）

结合 CoT 推理和反思机制，实现一个完整的多步推理链。

// 多步推理链：CoT + 反思 @Service public class ReasoningChainAgent { private final ChatClient chatClient; public ReasoningChain reason(String question) { List<ReasoningStep> steps = new ArrayList<>(); // 步骤 1: 初始 CoT 推理 String initial = chatClient.prompt() .system(""" 请一步一步地推理以下问题。 每一步都要明确写出你的思考过程。 如果某步不确定，标注置信度。 """) .user(question) .call() .content(); steps.add(new ReasoningStep("初始推理", initial)); // 步骤 2: 自我检验 String critique = chatClient.prompt() .system("检查以下推理过程，找出逻辑漏洞。") .user(initial) .call() .content(); steps.add(new ReasoningStep("自我检验", critique)); // 步骤 3: 综合得出最终结论 String conclusion = chatClient.prompt() .system(""" 基于原始推理和检验结果， 给出最终答案，标注整体置信度。 """) .user("推理: " + initial + "\n检验: " + critique) .call() .content(); return new ReasoningChain(question, steps, conclusion); } } public record ReasoningStep(String name, String content) {} public record ReasoningChain( String question, List<ReasoningStep> steps, String conclusion ) {}
💡 你知道吗？

Spring AI 的 ChatClient 支持结构化输出（.entity()），这让规划器和反思模块能直接输出 Java 对象，无需手动解析 JSON。这是 Spring AI 相比原生 Prompt 调用的重要优势 — 类型安全的 LLM 输出。

📖 章节小结

本章我们深入探讨了 Agent 的推理与规划能力：

  • CoT / ToT / GoT — 三种推理范式从线性到网状，适用于不同复杂度的问题，实际中往往组合使用
  • 规划策略 — HTN 逐层分解、目标依赖分析、动态 Re-Planning 让 Agent 具备"先想后做"的能力
  • 反思与元认知 — 自我反思避免重复犯错，置信度校准让 Agent 知道"自己有多确定"，知道何时放弃则是最可靠的安全阀
  • 工程化落地 — Spring AI 的结构化输出让规划器和反思循环的实现简洁且类型安全

记住：推理是 Agent 的大脑，规划是 Agent 的骨架，反思是 Agent 的免疫系统。三者协同，才能构建出真正可靠的智能体。

🎓 面试高频题

👨‍💻 Q1: CoT、ToT、GoT 的核心区别是什么？各自适合什么场景？

CoT（Chain-of-Thought）是线性推理，每步推导基于上一步，适合有明确因果链的数学和逻辑推理问题。ToT（Tree-of-Thought）在每步生成多个候选思路并通过评估函数选择最优分支，支持回溯剪枝，适合需要探索多种可能性的规划和创意任务。GoT（Graph-of-Thought）将推理建模为有向图，支持节点合并和循环精炼，适合需要综合多源信息的复杂推理任务。Token 消耗：CoT 最低，ToT 约 3-5 倍，GoT 约 5-10 倍。

👨‍💻 Q2: 什么是 Re-Planning？局部重规划和全局重规划有什么区别？

Re-Planning 是 Agent 在执行计划过程中遇到意外（如步骤失败、新信息出现）时，动态调整计划的能力。局部重规划只调整失败步骤及其后续依赖的子任务，成本低、速度快，但可能遗漏全局影响。全局重规划则从头重新制定完整计划，更全面但成本更高。工程实践中推荐"先局部后全局"策略：先尝试局部调整，如果局部调整无法解决问题再进行全局重规划。

👨‍💻 Q3: 如何实现 Agent 的置信度校准？为什么不能完全依赖 LLM 的自我评估？

置信度校准的实现方式包括：1）Prompt 要求 LLM 同时输出答案和置信度分数；2）Self-Consistency 方法，对同一问题多次采样，答案一致性越高置信度越高；3）用外部验证器（如单元测试、事实核查）客观评估。不能完全依赖 LLM 自我评估的原因是 LLM 存在"过度自信"问题 — 对明显错误的答案也可能给出高置信度（类似达克效应）。因此需要外部校验机制作为补充。

👨‍💻 Q4: Agent 的"知道何时放弃"为什么重要？有哪些实现策略？

"知道何时放弃"是 Agent 可靠性的关键保障。一个对所有问题都强行作答的 Agent 会在边界场景产出错误结果，且下游系统无法识别。实现策略包括：1）设置最大推理步数/时间限制；2）监控置信度阈值，低于阈值时主动放弃；3）检测重复失败模式（如反复尝试同一方法）；4）知识边界检测，识别超出训练数据范围的问题。放弃后应明确输出"不确定"信号，交由上层系统或人类处理。
