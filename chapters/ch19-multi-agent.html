第 15 章

多 Agent 编排框架

从"独狼"到"狼群" — 掌握 AutoGen、CrewAI、MetaGPT 等多 Agent 协作框架，让多个 Agent 分工合作完成复杂任务。

⏱ 预计 3-4 周 📖 6 个知识点 💻 高级难度

📚 开篇故事：从独狼到狼群

想象你是一个创业公司的 CEO。刚开始只有你一个人：既要写代码、做产品设计、写文案、做客服、搞运维。你是一个"全能选手"，但每个方面都只是"勉强能用"。

后来你招了几个合伙人：一个负责技术架构，一个负责产品设计，一个负责市场运营。你们不再是"一个人干所有事"，而是各司其职、分工协作。CEO（你）变成了"协调者"，分配任务、整合成果。

这正是 单 Agent 到多 Agent 的进化。一个 Agent 再强大，也有它的局限性 — 上下文窗口有限、专业知识有边界、并行能力不足。多 Agent 系统就像一支特种部队：每个 Agent 有自己的专长，通过编排框架协调合作，完成单个 Agent 无法胜任的复杂任务。

AutoGen 的"对话圆桌"、CrewAI 的"角色扮演"、MetaGPT 的"软件公司模拟" — 每个框架都在用不同的方式组织这场"团队协作"。本章将带你认识这些框架，学会搭建自己的 Agent 团队。

💬 AutoGen — Agent 的"对话圆桌"

什么是 AutoGen？

AutoGen（微软出品）的核心思想是多 Agent 对话：多个 Agent 围坐"圆桌"，通过消息传递协作完成任务。每个 Agent 可以是 LLM、人类、或工具，它们通过对话来协调工作。

比喻

就像一个圆桌会议：产品经理提出需求，架构师评估可行性，开发者给出实现方案，测试工程师指出潜在问题。大家通过对话达成共识，而不是一个人说了算。

AutoGen 对话模式：

  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ 产品经理  │    │ 架构师   │    │ 开发者   │
  │ (Agent)  │    │ (Agent)  │    │ (Agent)  │
  └────┬─────┘    └────┬─────┘    └────┬─────┘
       |               |               |
       v               v               v
  ┌─────────────────────────────────────────┐
  │         群聊（Group Chat）              │
  │                                         │
  │  产品经理: "需要用户认证功能"           │
  │  架构师:   "建议用 JWT + Redis"        │
  │  开发者:   "我来实现，预计 3 天"        │
  │  产品经理: "好的，优先级高"             │
  │                                         │
  └─────────────────────────────────────────┘

AutoGen 工作流实现

/** * AutoGen 风格的多 Agent 对话 * 基于 Spring AI 实现 Agent 间消息传递 */ @Service public class AutoGenOrchestrator { private final ChatClient chatClient; // 定义 Agent 角色 public record AgentConfig( String name, String systemPrompt, String model ) {} public AutoGenOrchestrator(ChatClient chatClient) { this.chatClient = chatClient; } /** * 单个 Agent 发言 */ private String agentSpeak(AgentConfig agent, String conversationHistory) { return chatClient.prompt() .system(agent.systemPrompt()) .user(conversationHistory) .options(ChatOptions.builder() .model(agent.model()) .temperature(0.7) .build()) .call() .content(); } /** * 群聊模式：多个 Agent 轮流发言 */ public String groupChat( List<AgentConfig> agents, String task, int maxRounds) { StringBuilder history = new StringBuilder(); history.append("任务：").append(task).append("\n\n"); for (int round = 0; round < maxRounds; round++) { for (AgentConfig agent : agents) { // 每个 Agent 基于对话历史发言 String response = agentSpeak(agent, history.toString()); history.append("[").append(agent.name()).append("]: ") .append(response).append("\n\n"); System.out.printf("[Round %d] %s: %s%n", round + 1, agent.name(), response); // 检查是否达成共识 if (response.contains("[DONE]")) { return response; } } } return history.toString(); } /** * 使用示例：代码审查多 Agent 协作 */ public String codeReview(String code) { List<AgentConfig> reviewers = List.of( new AgentConfig("安全专家", "你是安全专家。审查代码中的安全漏洞。", "gpt-4o"), new AgentConfig("性能专家", "你是性能专家。审查代码的性能问题。", "gpt-4o"), new AgentConfig("代码质量专家", "你是代码质量专家。审查代码的可读性和可维护性。", "gpt-4o") ); return groupChat(reviewers, "请审查以下代码并给出改进建议：\n" + code, 2); } }
💡 要点小结
  • AutoGen 的核心：多 Agent 通过对话协作
  • 支持三种 Agent 类型：LLM Agent、Human Agent、Tool Agent
  • 群聊模式让多个 Agent 轮流发言，达成共识
  • 适合需要多角度分析的任务（审查、讨论、辩论）
💡 你知道吗？
AutoGen 的论文（2023，微软研究院）提出了"Conversable Agent"的概念，认为 Agent 之间最自然的协作方式就是对话。这与人类团队的工作方式高度一致。

🎯 CrewAI — Agent 的"特种部队"

什么是 CrewAI？

CrewAI 的核心思想是角色化 Agent 协作：每个 Agent 有明确的角色（Role）、目标（Goal）和背景故事（Backstory），通过预定义的任务流程协作完成复杂工作。

比喻

就像一支特种部队：队长负责指挥、狙击手负责远程打击、爆破手负责开路、医疗兵负责救治。每个人都有明确的职责和专业技能，通过战术流程（SOP）协调行动。

CrewAI 编排模式：

  ┌─────────────────────────────────────────┐
  │              Crew（团队）                │
  │                                         │
  │  ┌─────────┐  ┌─────────┐  ┌─────────┐ │
  │  │ 研究员  │  │ 写作员  │  │ 审核员  │ │
  │  │ - 搜索  │  │ - 撰写  │  │ - 校对  │ │
  │  │ - 分析  │  │ - 排版  │  │ - 修正  │ │
  │  └────┬────┘  └────┬────┘  └────┬────┘ │
  │       |            |            |       │
  │       v            v            v       │
  │  ┌─────────────────────────────────────┐│
  │  │        Task Pipeline（任务流）      ││
  │  │  研究 ──> 写作 ──> 审核 ──> 输出   ││
  │  └─────────────────────────────────────┘│
  └─────────────────────────────────────────┘

CrewAI 任务构建

/** * CrewAI 风格的角色化 Agent 协作 * 基于 Spring AI 的实现 */ @Service public class CrewAIOrchestrator { private final ChatClient chatClient; // Agent 定义 public record CrewAgent( String role, String goal, String backstory, String model ) {} // 任务定义 public record CrewTask( String description, String expectedOutput, CrewAgent assignedAgent ) {} public CrewAIOrchestrator(ChatClient chatClient) { this.chatClient = chatClient; } /** * 执行单个任务 */ private String executeTask(CrewTask task, String context) { CrewAgent agent = task.assignedAgent(); return chatClient.prompt() .system(""" 你的角色：{role} 你的目标：{goal} 你的背景：{backstory} 请严格按照你的角色和专业领域完成任务。 """.replace("{role}", agent.role()) .replace("{goal}", agent.goal()) .replace("{backstory}", agent.backstory())) .user(""" 任务描述：{description} 期望输出：{expectedOutput} 上下文：{context} """.replace("{description}", task.description()) .replace("{expectedOutput}", task.expectedOutput()) .replace("{context}", context)) .options(ChatOptions.builder() .model(agent.model()) .temperature(0.7) .build()) .call() .content(); } /** * 执行 Crew（按顺序执行任务） */ public String executeCrew(List<CrewTask> tasks) { String context = ""; for (int i = 0; i < tasks.size(); i++) { CrewTask task = tasks.get(i); System.out.printf("[Crew] 执行任务 %d/%d: %s%n", i + 1, tasks.size(), task.description()); String result = executeTask(task, context); context += "\n\n## 任务 " + (i + 1) + " 结果：\n" + result; System.out.printf("[Crew] 任务 %d 完成%n%n", i + 1); } return context; } /** * 使用示例：技术博客创作团队 */ public String writeBlogPost(String topic) { // 定义团队成员 CrewAgent researcher = new CrewAgent( "技术研究员", "深入调研技术主题，收集权威资料", "你是一位有10年经验的技术研究员，擅长从多个来源收集和验证信息。", "gpt-4o" ); CrewAgent writer = new CrewAgent( "技术写作员", "将研究成果转化为通俗易懂的技术博客", "你是一位知名技术博主，擅长用比喻和代码示例解释复杂概念。", "gpt-4o" ); CrewAgent reviewer = new CrewAgent( "内容审核员", "确保文章质量、准确性和可读性", "你是一位严格的内容编辑，关注技术准确性和文章结构。", "gpt-4o" ); // 定义任务流程 List<CrewTask> tasks = List.of( new CrewTask( "调研关于 " + topic + " 的最新技术资料", "一份包含关键概念、代码示例和最佳实践的研究报告", researcher ), new CrewTask( "基于研究报告撰写技术博客文章", "一篇约 1500 字的 Markdown 格式技术博客", writer ), new CrewTask( "审核博客文章的技术准确性和可读性", "修改建议列表 + 最终版本", reviewer ) ); return executeCrew(tasks); } }
💡 要点小结
  • CrewAI 的核心：角色化 Agent + 任务管道
  • 每个 Agent 有明确的 Role、Goal、Backstory
  • 任务按顺序执行，前一个任务的输出作为后一个的上下文
  • 适合流程化工作（内容创作、数据处理、报告生成）

💻 MetaGPT — Agent 的"软件公司"

什么是 MetaGPT？

MetaGPT 的核心思想是SOP 驱动的多 Agent 协作：模拟一家软件公司的组织架构，用标准化的流程（SOP）来协调多个 Agent 的工作。

比喻

就像一家软件公司：产品经理写 PRD，架构师出技术方案，开发者写代码，测试工程师写测试用例。每个角色都遵循标准工作流程（SOP），产出标准化的文档和代码。

MetaGPT 软件公司架构：

  ┌─────────────────────────────────────────┐
  │           MetaGPT 软件公司              │
  │                                         │
  │  产品经理 ──> PRD 文档                  │
  │     |                                   │
  │     v                                   │
  │  架构师 ──> 系统设计文档                │
  │     |                                   │
  │     v                                   │
  │  项目经理 ──> 任务分配                  │
  │     |                                   │
  │     v                                   │
  │  ┌──────────┐  ┌──────────┐            │
  │  │ 工程师 A │  │ 工程师 B │            │
  │  │ (前端)   │  │ (后端)   │            │
  │  └────┬─────┘  └────┬─────┘            │
  │       |            |                   │
  │       v            v                   │
  │  测试工程师 ──> 测试用例 + Bug 报告   │
  │                                         │
  └─────────────────────────────────────────┘

MetaGPT 的 SOP 流程

/** * MetaGPT 风格的 SOP 驱动多 Agent 协作 * 模拟软件公司的工作流程 */ @Service public class MetaGPTOrchestrator { private final ChatClient chatClient; // 标准化文档结构 public record PRD( String projectName, String requirements, List<String> userStories, List<String> acceptanceCriteria ) {} public record SystemDesign( String architecture, List<String> apiEndpoints, List<String> dataModels, List<String> techStack ) {} public MetaGPTOrchestrator(ChatClient chatClient) { this.chatClient = chatClient; } /** * Step 1: 产品经理生成 PRD */ public PRD generatePRD(String projectBrief) { String response = chatClient.prompt() .system(""" 你是一位资深产品经理。 根据项目简述，生成标准 PRD 文档，包含： 1. 项目名称 2. 需求描述 3. 用户故事列表（3-5个） 4. 验收标准 用 JSON 格式输出。 """) .user(projectBrief) .options(ChatOptions.builder().temperature(0.3).build()) .call() .content(); // 解析 PRD（简化） return new PRD( "AI Chat App", projectBrief, List.of("用户可以与 AI 对话", "支持历史记录"), List.of("响应时间 < 3s", "支持流式输出") ); } /** * Step 2: 架构师生成系统设计 */ public SystemDesign generateDesign(PRD prd) { String response = chatClient.prompt() .system(""" 你是一位资深架构师。 根据 PRD 生成系统设计文档，包含： 1. 架构图描述 2. API 端点列表 3. 数据模型 4. 技术栈选择 用 JSON 格式输出。 """) .user("PRD：" + prd.requirements()) .options(ChatOptions.builder().temperature(0.3).build()) .call() .content(); return new SystemDesign( "微服务架构", List.of("POST /api/chat", "GET /api/history"), List.of("ChatMessage", "Conversation"), List.of("Spring Boot", "Spring AI", "Redis") ); } /** * Step 3: 开发者生成代码 */ public String generateCode(SystemDesign design, String module) { return chatClient.prompt() .system(""" 你是一位资深 Java 开发者。 根据系统设计，为指定模块生成生产级别的代码。 使用 Spring Boot + Spring AI。 代码需要包含完整的注释和错误处理。 """) .user("模块：" + module + "\n设计：" + design.architecture() + "\nAPI：" + design.apiEndpoints() + "\n数据模型：" + design.dataModels()) .options(ChatOptions.builder().temperature(0.3).build()) .call() .content(); } /** * Step 4: 测试工程师生成测试 */ public String generateTests(String code, PRD prd) { return chatClient.prompt() .system(""" 你是一位资深测试工程师。 根据代码和 PRD，生成完整的单元测试和集成测试。 测试覆盖率目标 80%+。 """) .user("代码：\n" + code + "\n验收标准：" + prd.acceptanceCriteria()) .options(ChatOptions.builder().temperature(0.3).build()) .call() .content(); } /** * 完整的 MetaGPT 流程 */ public String buildSoftware(String projectBrief) { System.out.println("[MetaGPT] Step 1: 产品经理生成 PRD..."); PRD prd = generatePRD(projectBrief); System.out.println("[MetaGPT] Step 2: 架构师生成设计..."); SystemDesign design = generateDesign(prd); System.out.println("[MetaGPT] Step 3: 开发者生成代码..."); String code = generateCode(design, "ChatService"); System.out.println("[MetaGPT] Step 4: 测试工程师生成测试..."); String tests = generateTests(code, prd); return "## PRD\n" + prd + "\n\n## 设计\n" + design + "\n\n## 代码\n" + code + "\n\n## 测试\n" + tests; } }
💡 要点小结
  • MetaGPT 的核心：SOP 驱动 + 标准化文档
  • 每个角色产出标准化的中间文档（PRD、设计、代码、测试）
  • 文档驱动的协作让 Agent 之间的接口更清晰
  • 适合软件开发流程（从需求到代码的全链路）
⚠️ 陷阱警告

MetaGPT 的多 Agent 协作延迟很高（每个步骤都需要 LLM 调用）。对于简单项目，单 Agent 可能更高效。MetaGPT 适合中大型项目的原型生成。

🔧 自定义多 Agent 编排

基于 Spring AI 的自定义实现

在 Java/Spring 生态中，我们可以从零构建多 Agent 编排系统，完全掌控 Agent 的行为和交互方式。

/** * 自定义多 Agent 编排引擎 * 支持消息传递、任务分发、并行执行 */ @Service public class CustomAgentOrchestrator { private final ChatClient chatClient; private final AgentRegistry agentRegistry; private final MessageBroker messageBroker; public CustomAgentOrchestrator(ChatClient chatClient, AgentRegistry agentRegistry, MessageBroker messageBroker) { this.chatClient = chatClient; this.agentRegistry = agentRegistry; this.messageBroker = messageBroker; } // ===== 消息传递 ===== /** * Agent 间消息格式 */ public record AgentMessage( String from, String to, String content, MessageType type, Map<String, Object> metadata ) {} public enum MessageType { TASK_ASSIGN, // 任务分配 TASK_RESULT, // 任务结果 QUERY, // 查询请求 RESPONSE, // 查询响应 BROADCAST // 广播消息 } /** * 发送消息给指定 Agent */ public void sendMessage(AgentMessage message) { messageBroker.publish(message); } // ===== 任务分发 ===== /** * 将任务分发给最合适的 Agent */ public String dispatchTask(String task, String preferredRole) { // 查找匹配的 Agent Agent agent = agentRegistry.findByRole(preferredRole) .orElseThrow(() -> new IllegalStateException( "No agent found for role: " + preferredRole)); // 分配任务 AgentMessage assignment = new AgentMessage( "orchestrator", agent.id(), task, MessageType.TASK_ASSIGN, Map.of("priority", "high", "timestamp", Instant.now()) ); sendMessage(assignment); return agent.execute(task); } // ===== 并行执行 ===== /** * 并行分发多个任务 */ public Map<String, String> parallelDispatch( Map<String, String> taskByRole) { return taskByRole.entrySet().stream() .collect(Collectors.toMap( Map.Entry::getKey, entry -> dispatchTask(entry.getValue(), entry.getKey()) )); } /** * 共享记忆：Agent 间的状态同步 */ public static class SharedMemory { private final ConcurrentHashMap<String, Object> store = new ConcurrentHashMap<>(); public void write(String key, Object value) { store.put(key, value); } public <T> T read(String key, Class<T> type) { return type.cast(store.get(key)); } public Map<String, Object> snapshot() { return new HashMap<>(store); } } }
💡 要点小结
  • 自定义编排的核心组件：消息传递、任务分发、并行执行、共享记忆
  • Spring AI 提供了灵活的底层能力，编排逻辑可以完全自定义
  • 并行执行使用 CompletableFuture 或 Reactor
  • 共享记忆使用 ConcurrentHashMap 保证线程安全
💡 你知道吗？
Spring AI 的 Advisor 机制可以用来实现 Agent 间的消息拦截和路由。每个 Agent 可以看作一个 Advisor，通过拦截 Prompt 请求来实现协作逻辑。

📡 多 Agent 通信协议

Agent 间消息格式

/** * 标准化的 Agent 间通信协议 */ // 消息格式 public record InterAgentMessage( String messageId, // 消息唯一 ID String conversationId, // 会话 ID（同一任务的所有消息） String fromAgent, // 发送方 Agent ID String toAgent, // 接收方 Agent ID（null = 广播） String type, // 消息类型 String content, // 消息内容 Map<String, String> headers, // 元数据头 Instant timestamp ) {} // 消息类型定义 public interface MessageTypes { String TASK_REQUEST = "task.request"; String TASK_RESPONSE = "task.response"; String STATUS_UPDATE = "status.update"; String ERROR_REPORT = "error.report"; String HEARTBEAT = "heartbeat"; } // 消息总线实现 @Service public class AgentMessageBus { private final ConcurrentHashMap<String, FluxSink<InterAgentMessage>> subscribers = new ConcurrentHashMap<>(); /** * 发布消息 */ public void publish(InterAgentMessage message) { // 广播给所有订阅者（除发送方外） subscribers.forEach((agentId, sink) -> { if (!agentId.equals(message.fromAgent())) { sink.next(message); } }); } /** * 订阅消息 */ public Flux<InterAgentMessage> subscribe(String agentId) { return Flux.create(sink -> { subscribers.put(agentId, sink); sink.onDispose(() -> subscribers.remove(agentId)); }); } /** * 发送请求-响应 */ public Mono<InterAgentMessage> request( String from, String to, String content) { InterAgentMessage request = new InterAgentMessage( UUID.randomUUID().toString(), UUID.randomUUID().toString(), from, to, MessageTypes.TASK_REQUEST, content, Map.of(), Instant.now() ); publish(request); // 等待响应（超时 30 秒） return subscribe(from) .filter(msg -> msg.conversationId() .equals(request.conversationId())) .next() .timeout(Duration.ofSeconds(30)); } }

共享记忆与状态同步

/** * Agent 共享记忆管理 * 支持跨 Agent 的状态同步和知识共享 */ @Service public class AgentSharedMemory { private final RedisTemplate<String, Object> redis; public AgentSharedMemory(RedisTemplate<String, Object> redis) { this.redis = redis; } /** * 写入共享记忆 */ public void write(String conversationId, String agentId, String key, Object value) { String fullKey = "agent:shared:" + conversationId + ":" + key; redis.opsForHash().put(fullKey, agentId, value); redis.expire(fullKey, Duration.ofHours(1)); } /** * 读取共享记忆（所有 Agent 写入的值） */ public Map<String, Object> read(String conversationId, String key) { String fullKey = "agent:shared:" + conversationId + ":" + key; return redis.opsForHash().entries(fullKey); } /** * 获取最新的上下文快照 */ public String getContextSnapshot(String conversationId) { String fullKey = "agent:shared:" + conversationId + ":*"; Set<String> keys = redis.keys(fullKey); StringBuilder snapshot = new StringBuilder(); for (String key : keys) { Map<Object, Object> entries = redis.opsForHash().entries(key); snapshot.append("## ").append(key).append("\n"); entries.forEach((k, v) -> snapshot.append(" ").append(k).append(": ").append(v).append("\n")); } return snapshot.toString(); } }
💡 要点小结
  • Agent 间通信需要标准化的消息格式（ID、类型、内容、元数据）
  • 使用 Reactor Flux 实现响应式消息订阅
  • 共享记忆使用 Redis 实现跨 Agent 的状态同步
  • 上下文快照让 Agent 能获取全局任务状态
⚠️ 陷阱警告

多 Agent 系统的调试非常困难。建议：1) 为每条消息添加唯一 ID 和时间戳；2) 实现消息日志记录；3) 使用分布式追踪（如 OpenTelemetry）跟踪消息流。

🏢 企业级多 Agent 高级场景

1. Agent 失败恢复机制

在生产环境中，Agent 可能因为各种原因失败（LLM 超时、工具调用错误、网络中断）。需要设计完善的失败恢复机制。

// Agent 失败恢复策略 public class AgentRetryPolicy { private final int maxRetries = 3; private final Duration retryDelay = Duration.ofSeconds(2); public Mono<AgentResult> executeWithRetry( Agent agent, Task task) { return Mono.defer(() -> agent.execute(task)) .retryWhen(Retry.backoff(maxRetries, retryDelay) .filter(ex -> ex instanceof AgentException)) .onErrorResume(ex -> Mono.just( AgentResult.fallback("Agent 不可用，请稍后重试"))); } }
💡 失败恢复策略要点
  • 指数退避重试：避免在 Agent 故障时立即重试造成雪崩
  • 熔断降级：当失败率超过阈值时，自动切换到备用 Agent 或降级策略
  • 优雅降级：返回有意义的 fallback 结果，而非直接报错
  • 失败隔离：单个 Agent 失败不应导致整个多 Agent 系统崩溃

2. 跨 Agent 状态同步

当多个 Agent 协作处理复杂任务时，需要保持状态一致性。

分布式 Agent 状态同步：

  ┌─────────────────────────────────────────────────────┐
  │                                                     │
  │  Agent A ──────┐                                    │
  │                │                                    │
  │  Agent B ──────┼──> 状态协调器 ──> Redis/MongoDB    │
  │                │                                    │
  │  Agent C ──────┘                                    │
  │                                                     │
  │  同步机制：                                          │
  │  • 事件驱动：Agent 状态变更发布事件                    │
  │  • 最终一致性：接受短暂的不一致                       │
  │  • 冲突解决：基于时间戳或优先级                       │
  └─────────────────────────────────────────────────────┘
/** * 分布式 Agent 状态协调器 * 使用事件驱动实现最终一致性 */ @Service public class AgentStateCoordinator { private final RedisTemplate<String, Object> redis; private final ApplicationEventPublisher eventPublisher; // 状态变更事件 public record StateChangeEvent( String agentId, String taskContext, Map<String, Object> state, Instant timestamp ) extends ApplicationEvent { public StateChangeEvent(Object src, String agentId, String taskContext, Map<String, Object> state) { super(src); this.agentId = agentId; this.taskContext = taskContext; this.state = state; this.timestamp = Instant.now(); } } /** * 发布状态变更 */ public void publishStateChange(String agentId, String taskContext, Map<String, Object> state) { // 持久化到 Redis String key = "state:" + taskContext + ":" + agentId; redis.opsForHash().putAll(key, state); redis.expire(key, Duration.ofHours(1)); // 发布事件通知其他 Agent eventPublisher.publishEvent( new StateChangeEvent(this, agentId, taskContext, state)); } /** * 获取全局状态快照（带冲突解决） */ public Map<String, Object> getGlobalState( String taskContext) { String pattern = "state:" + taskContext + ":*"; Set<String> keys = redis.keys(pattern); Map<String, Object> globalState = new HashMap<>(); keys.forEach(key -> { Map<Object, Object> entries = redis.opsForHash().entries(key); globalState.putAll(entries); }); return globalState; } }

3. 分布式 Agent 事务

在涉及多个 Agent 的复杂业务流程中，需要保证事务的完整性。

// Saga 模式实现分布式 Agent 事务 public class AgentSaga { private final List<SagaStep> steps; private final List<SagaStep> compensations; public Mono<SagaResult> execute(SagaContext ctx) { return Flux.fromIterable(steps) .concatMap(step -> step.execute(ctx) .onErrorResume(ex -> { // 执行补偿操作 return compensate(ctx, ex); })) .then(Mono.just(SagaResult.success())); } private Mono<SagaResult> compensate( SagaContext ctx, Exception ex) { return Flux.fromIterable( List.of(compensations).reversed()) .concatMap(comp -> comp.execute(ctx)) .then(Mono.error(ex)); } }
💡 Saga 模式核心概念
  • 正向步骤：每个 Agent 按顺序执行各自的业务逻辑
  • 补偿步骤：当某步失败时，逆序执行已完成步骤的补偿操作
  • 状态持久化：每步执行结果持久化，支持断点续传
  • 超时控制：每步设置超时阈值，避免长时间阻塞

4. 多 Agent 负载均衡

当相同类型的 Agent 有多个实例时，需要设计智能的负载均衡策略。

多 Agent 负载均衡策略：

  ┌─────────────────────────────────────────────────────┐
  │                                                     │
  │  任务输入 ──> 负载均衡器                            │
  │                │                                    │
  │       ┌───────┼───────┐                            │
  │       ▼       ▼       ▼                            │
  │    Agent-1  Agent-2  Agent-3                       │
  │                                                     │
  │  策略选择：                                          │
  │  • 轮询（Round Robin）：简单公平                     │
  │  • 加权轮询：根据 Agent 能力分配                     │
  │  • 最少连接：选择负载最低的 Agent                    │
  │  • 一致性哈希：相同任务路由到相同 Agent              │
  │  • 基于能力：根据任务类型选择最优 Agent              │
  └─────────────────────────────────────────────────────┘
/** * 多 Agent 负载均衡器 * 支持多种负载均衡策略 */ @Service public class AgentLoadBalancer { private final Map<String, List<Agent>> agentPools = new ConcurrentHashMap<>(); private final AtomicInteger roundRobinCounter = new AtomicInteger(0); /** * 轮询策略 */ public Agent roundRobin(String agentType) { List<Agent> pool = agentPools.get(agentType); int index = roundRobinCounter.getAndIncrement() % pool.size(); return pool.get(index); } /** * 最少连接策略 */ public Agent leastConnections(String agentType) { return agentPools.get(agentType).stream() .min(Comparator.comparingInt(Agent::activeTaskCount)) .orElseThrow(); } /** * 加权轮询策略 */ public Agent weightedRoundRobin(String agentType) { List<WeightedAgent> pool = agentPools.get(agentType) .stream() .map(a -> new WeightedAgent(a, a.weight())) .toList(); int totalWeight = pool.stream() .mapToInt(WeightedAgent::weight).sum(); int random = new Random().nextInt(totalWeight); int cumulative = 0; for (WeightedAgent wa : pool) { cumulative += wa.weight(); if (random < cumulative) { return wa.agent(); } } return pool.get(pool.size() - 1).agent(); } }

5. 多 Agent 监控与告警

生产环境需要对多 Agent 系统进行全面的监控和告警。

📊 监控指标体系
  • 任务级指标：任务完成率、平均延迟、失败率
  • Agent 级指标：Agent 利用率、响应时间、错误率
  • 系统级指标：CPU/内存使用率、网络延迟、队列长度
  • 成本级指标：Token 消耗、API 调用成本、每任务成本
/** * 多 Agent 监控服务 * 基于 Micrometer + Prometheus 实现 */ @Service public class AgentMonitoringService { private final MeterRegistry registry; public void recordTaskExecution( String agentId, String taskType, Duration duration, boolean success) { // 任务延迟指标 Timer.builder("agent.task.duration") .tag("agent", agentId) .tag("task_type", taskType) .tag("status", success ? "success" : "failure") .register(registry) .record(duration); // 任务计数器 Counter.builder("agent.task.count") .tag("agent", agentId) .tag("status", success ? "success" : "failure") .register(registry) .increment(); } /** * Token 消耗追踪 */ public void recordTokenUsage( String agentId, int inputTokens, int outputTokens) { registry.gauge("agent.tokens.input", Tags.of("agent", agentId), inputTokens); registry.gauge("agent.tokens.output", Tags.of("agent", agentId), outputTokens); } }
⚠️ 告警规则建议
  • P0 紧急：Agent 无响应超过 5 分钟，任务失败率 > 50%
  • P1 重要：任务平均延迟 > 30s，Token 消耗异常激增
  • P2 一般：Agent 利用率 < 10%（资源浪费），队列积压 > 100

🎓 面试高频题

👨‍💻 Q1: AutoGen 和 CrewAI 的核心区别是什么？

AutoGen 基于多 Agent 对话，Agent 间通过消息传递自由交流；CrewAI 基于角色化任务管道，Agent 按预定义流程顺序执行任务。AutoGen 更灵活，CrewAI 更结构化。

👨‍💻 Q2: 多 Agent 系统的主要挑战是什么？

1) 调试困难 — 多个 Agent 的交互难以追踪；2) 延迟累积 — 每次 Agent 间通信都增加延迟；3) 状态同步 — 保持 Agent 间状态一致；4) 错误传播 — 一个 Agent 的错误可能影响整个系统。

👨‍💻 Q3: 在 Java/Spring 项目中如何实现多 Agent 编排？

使用 Spring AI 的 ChatClient 作为 Agent 的核心能力，Spring Integration 或 Reactor 实现消息传递，Redis 实现共享记忆，CompletableFuture 实现并行执行。完全自定义编排逻辑。

👨‍💻 Q4: MetaGPT 的 SOP 模式有什么优势和局限？

优势：标准化（每个角色产出固定格式文档）、可追溯（中间文档可审查）、可复用（SOP 可模板化）。局限：灵活性低（流程固定）、延迟高（多步 LLM 调用）、不适合简单项目。
