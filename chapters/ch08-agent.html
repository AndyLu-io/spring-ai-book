    📚 开篇故事：机器人管家的诞生

    想象你请了一位私人管家。第一天，你告诉他："帮我准备明天的商务晚宴。"他不会呆站着问你"具体要做什么"，而是自己思考：需要预订餐厅、采购食材、安排司机、发送邀请函。

    接着他主动行动：拿起电话预订了米其林餐厅，打开冰箱检查食材，用车载系统安排接送。每做完一件事，他会观察结果 -- "餐厅已订好，但包间只有6个座位，需要确认客人人数。"然后他调整计划，继续执行。

    这个管家就是一个 AI Agent。普通的 LLM 就像一个只会回答问题的图书管理员 -- 你问一句，它答一句。而 Agent 是一个能自主思考、主动行动、持续观察的智能管家，它不等你下达每一个指令，而是自己分解任务、调用工具、验证结果。

    在本章中，我们将亲手打造这样的"机器人管家"，让它学会思考-行动-观察的循环，拥有自己的"工具百宝箱"，甚至能像项目管理大师一样拆解复杂任务。

    🤖 Agent 概念与架构

    📄 概念：Agent = 机器人管家

    定义：AI Agent 是一个能够自主感知环境、做出决策并执行行动的智能实体。它不像普通的 LLM 调用那样"一问一答"，而是能自主规划、调用工具、观察反馈并迭代优化。

    生活比喻：普通的 LLM 是一个只会回答问题的图书管理员 -- 你问什么它答什么，但不会主动帮你找书。而 Agent 是一个全能机器人管家 -- 你说"帮我准备下周的客户演示"，它会自己列清单、查资料、做 PPT、发邮件，遇到问题还会自己调整方案。

    📄 概念：Agent vs 普通 LLM 调用

    维度普通 LLM 调用AI Agent
    交互模式一问一答自主循环执行
    工具使用无按需调用外部工具
    决策能力被动回答主动规划和决策
    错误处理返回错误自动重试和调整
    状态保持无状态（或简单对话历史）有状态记忆系统
    复杂任务需要多轮人工协调自动分解并执行

    ⚡ 实战场景：智能助手 Agent

    想象你在构建一个个人 AI 助手。用户说："帮我查一下明天北京的天气，如果会下雨就提醒我带伞，然后帮我预约明天下午2点的会议室。"

    Agent 会自动：

    • 调用天气 API 查询北京明天天气
    • 判断是否下雨，如果是则在回复中提醒带伞
    • 调用日历 API 预约会议室
    • 返回综合结果给用户

    这一切不需要用户逐步指令，Agent 自主完成了整个流程。

    ⚠️ 常见陷阱

    陷阱 1：Agent 不是万能的。 Agent 的能力取决于 LLM 的推理能力和工具的质量。不要让 Agent 执行需要精确计算的任务（LLM 擅长语言，不擅长数学），而是为它配备计算工具。

    陷阱 2：缺少终止条件。 Agent 循环可能无限执行。必须设置最大迭代次数或明确的完成判断条件，否则会陷入死循环。

    陷阱 3：System Prompt 过于模糊。 "你是一个助手"不如 "你是一个数据分析助手，擅长SQL查询和图表生成"。明确的角色定义让 Agent 决策更准确。

    ✔ 要点小结

    Agent = LLM + 工具 + 自主决策。它通过"感知-决策-行动"循环来完成任务，核心优势在于能自主规划和调用外部工具。设计 Agent 时要明确角色、配备合适工具、设置终止条件。

    🧠 ReAct 模式：思考-行动-观察循环

    📄 概念：ReAct = 思考-行动-观察循环

    定义：ReAct（Reasoning + Acting）是让 LLM 交替进行推理（Thought）和行动（Action），并观察（Observation）结果的模式。

    生活比喻：想象你在解一道数学应用题。你不会直接写答案，而是先"读题分析"（Thought），然后"列算式计算"（Action），接着"检查计算结果"（Observation）。如果结果不对，你会重新分析、重新计算。这就是 ReAct 循环。

    ReAct 的核心价值在于让 LLM 的推理过程透明化 -- 你可以清楚地看到 Agent 在想什么、做什么、得到了什么结果。

    🔧 原理图：ReAct 循环

                ┌───────────────────────────────────┐
            │         ReAct 循环                │
            │                                   │
            │   ┌─────────┐                     │
            │   │ Thought │  1. 分析当前情况      │
            │   │ (思考)  │  2. 决定下一步行动    │
            │   └────┬────┘                     │
            │        │                          │
            │        ▼                          │
            │   ┌─────────┐                     │
            │   │ Action  │  3. 选择并调用工具    │
            │   │ (行动)  │  4. 执行具体操作      │
            │   └────┬────┘                     │
            │        │                          │
            │        ▼                          │
            │   ┌─────────┐                     │
            │   │Observe  │  5. 获取工具返回结果  │
            │   │ (观察)  │  6. 评估是否完成      │
            │   └────┬────┘                     │
            │        │                          │
            │        ▼                          │
            │   ┌─────────┐                     │
            │   │ 判断    │  完成？→ 返回答案     │
            │   │         │  未完成→回到 Thought │
            │   └─────────┘                     │
            └───────────────────────────────────┘

    📄 代码示例：ReAct Agent 完整实现

    // ============================================ // ReAct Agent 实现 // 核心：让 LLM 按 Thought/Action/Observation 格式输出 // ============================================ @Service public class ReActAgent { private final ChatClient chatClient; private static final int MAX_ITERATIONS = 5; // 最大迭代次数，防止死循环 public ReActAgent(ChatClient chatClient) { this.chatClient = chatClient; } /** * 使用 ReAct 模式解决问题 * * Prompt 中明确要求 LLM 按特定格式输出： * Thought → Action → Observation → ... → Final Answer */ public String solve(String task) { return chatClient.prompt() .system(""" 你是一个 ReAct 智能体。对于每个任务，严格按以下格式思考和行动： Thought: [分析当前情况，解释为什么需要采取下一步行动] Action: [选择要使用的工具名称和参数] Observation: [工具返回的结果] ... (可以重复 Thought/Action/Observation 多轮) Final Answer: [最终答案，当所有信息都收集完毕后给出] 可用工具： - search(query): 搜索网络信息 - calculate(expr): 执行数学计算 - lookup(name): 查询数据库记录 注意事项： 1. 每轮只能调用一个工具 2. 必须先观察结果再决定下一步 3. 最多进行 """ + MAX_ITERATIONS + """ 轮迭代 4. 当收集到足够信息时，给出 Final Answer """) .user(task) .tools(searchTool, calcTool, lookupTool) .call() .content(); } }

    📄 代码示例：ReAct 执行轨迹记录

    在生产环境中，记录 Agent 的每一步决策至关重要，便于调试和优化。

    // ============================================ // ReAct 执行轨迹记录 // 记录每一步的 Thought/Action/Observation // ============================================ public record ReActTrace( String taskId, List<Step> steps, String finalAnswer, long totalDurationMs, boolean completed ) { // 单步记录 public record Step( int iteration, String thought, String action, String observation, long durationMs, boolean success ) {} } // ============================================ // Agent 执行时记录轨迹 // ============================================ @Service public class ReActAgentWithTrace { private final ChatClient chatClient; private final ReActTraceRepository traceRepository; public ReActTrace solveWithTrace(String task) { long startTime = System.currentTimeMillis(); List<ReActTrace.Step> steps = new ArrayList<>(); // 使用 Advisor 拦截每次工具调用并记录 String result = chatClient.prompt() .system("""...""") .user(task) .tools(searchTool, calcTool, lookupTool) .advisors(new ToolCallLoggingAdvisor(steps)) // 记录工具调用 .call() .content(); // 保存执行轨迹 ReActTrace trace = new ReActTrace( UUID.randomUUID().toString(), steps, result, System.currentTimeMillis() - startTime, true ); traceRepository.save(trace); return trace; } }

    ⚡ 实战案例：ReAct 解决多步问题

    场景：用户问"今年诺贝尔物理学奖得主的母校在哪个城市？"

    Step 1 - Thought: 需要先查今年诺贝尔物理学奖得主是谁
Step 1 - Action: search("2024 诺贝尔物理学奖得主")
Step 1 - Observation: 得主是 John Hopfield 和 Geoffrey Hinton

Step 2 - Thought: 已知得主，需要查他们的母校
Step 2 - Action: search("Geoffrey Hinton 教育背景")
Step 2 - Observation: Hinton 毕业于剑桥大学国王学院

Step 3 - Thought: 已知母校是剑桥大学，需要查所在城市
Step 3 - Action: search("剑桥大学所在城市")
Step 3 - Observation: 剑桥大学位于英国剑桥市

Final Answer: Geoffrey Hinton 的母校剑桥大学位于英国剑桥市。

    💡 你知道吗？

    ReAct 模式来自 2022 年的论文 "ReAct: Synergizing Reasoning and Acting in Language Models"。它证明了让 LLM "边想边做" 比 "先想完再做" 或 "只做不想" 都要有效得多。OpenAI 的 function calling 本质上就是 ReAct 模式的工程化实现。

    ⚠️ 常见陷阱

    陷阱 1：迭代次数过多。 没有设置 MAX_ITERATIONS，Agent 可能陷入"思考-行动"的死循环。生产环境中务必设置上限。

    陷阱 2：工具描述不清晰。 LLM 根据工具的 description 选择工具。如果描述含糊（如 "处理数据"），Agent 可能选错工具。确保每个工具的 description 准确描述其功能。

    陷阱 3：忽略 Observation 的反馈。 有些 Agent 在 Observation 返回错误后仍继续执行相同操作。应在 System Prompt 中明确要求 Agent 根据 Observation 调整策略。

    ✔ 要点小结

    ReAct 模式让 LLM 按"思考-行动-观察"循环执行任务。关键设计点：(1) 在 System Prompt 中明确输出格式；(2) 设置最大迭代次数；(3) 记录执行轨迹便于调试；(4) 确保 Observation 的反馈能影响后续决策。

    🔧 多工具编排：工具百宝箱

    📄 概念：多工具编排 = 工具百宝箱

    定义：多工具编排是指 Agent 根据任务需求，自主选择和组合使用多个工具的能力。

    生活比喻：想象一个装修工人的工具箱。锤子、螺丝刀、电钻、水平仪... 他不会一次拿出所有工具，而是根据当前步骤选择合适的工具。拧螺丝用螺丝刀，钉钉子用锤子，开孔用电钻。Agent 的工具编排就是这样的"按需取用"。

    📄 代码示例：工具注册与管理

    // ============================================ // 工具配置类 - 定义 Agent 的"工具百宝箱" // 每个 @Tool 方法就是一个可用工具 // ============================================ @Service public class AgentTools { /** * 搜索工具 - 搜索网络信息 * description 会发送给 LLM，让它理解这个工具能做什么 */ @Tool(description = "搜索网络获取最新信息。输入搜索关键词，返回相关网页摘要。") public String searchWeb( @ToolParam(description = "搜索关键词") String query) { // 调用搜索 API return webSearchService.search(query); } /** * 计算工具 - 执行数学计算 * Agent 不擅长心算，但可以调用这个工具精确计算 */ @Tool(description = "执行数学计算表达式。支持加减乘除、幂运算等。") public String calculate( @ToolParam(description = "数学表达式，如 2+3*4") String expression) { // 安全的数学表达式求值 return mathEvaluator.evaluate(expression); } /** * 数据库查询工具 */ @Tool(description = "查询公司内部数据库。返回查询结果的 JSON 格式。") public String queryDatabase( @ToolParam(description = "SQL 查询语句，只允许 SELECT") String sql) { if (!sql.trim().toUpperCase().startsWith("SELECT")) { throw new SecurityException("安全限制：只允许 SELECT 查询"); } return objectMapper.writeValueAsString( jdbcTemplate.queryForList(sql)); } /** * 邮件发送工具 */ @Tool(description = "发送邮件给指定收件人") public String sendEmail( @ToolParam(description = "收件人邮箱") String to, @ToolParam(description = "邮件主题") String subject, @ToolParam(description = "邮件正文") String body) { return emailService.send(to, subject, body); } }

    ⚠️ 常见陷阱

    陷阱 1：工具描述是给 LLM 看的。 不是给人看的文档，而是 LLM 选择工具的依据。描述必须准确、具体、无歧义。"处理数据"不如 "执行SQL查询并返回JSON格式结果"。

    陷阱 2：工具数量过多。 工具太多会让 LLM 选择困难。建议每个 Agent 不超过 10 个工具，超过则考虑拆分为多个专业 Agent。

    陷阱 3：没有错误处理。 工具调用可能失败（网络超时、参数错误等）。在工具内部和 Agent 层面都要做好错误处理。

    ✔ 要点小结

    多工具编排的核心是让 Agent 根据任务自主选择和组合工具。设计要点：(1) 工具 description 是 LLM 选择工具的依据，必须清晰准确；(2) 控制工具数量（建议不超过10个）；(3) 支持并行调用提升效率；(4) 做好错误处理和安全限制。

    📄 代码示例：动态任务调整

    // ============================================ // 动态任务调整 - 当步骤失败时重新规划 // ============================================ @Service public class AdaptivePlanAgent { private final ChatClient chatClient; public String executeWithAdaptation(String goal) { List<PlanStep> currentPlan = generatePlan(goal); StringBuilder results = new StringBuilder(); int maxRetries = 3; for (int i = 0; i < currentPlan.size(); i++) { PlanStep step = currentPlan.get(i); boolean success = false; for (int retry = 0; retry < maxRetries; retry++) { try { String result = executeStep(step); results.append(result).append("\n"); success = true; break; } catch (Exception e) { if (retry < maxRetries - 1) { // 让 LLM 生成替代方案 String alternative = chatClient.prompt() .system("步骤执行失败，请生成替代方案。") .user(""" 原步骤: %s 错误: %s 请提供替代执行方案。 """.formatted(step.action(), e.getMessage())) .call() .content(); step = new PlanStep(step.step(), alternative, step.tool(), step.dependsOn()); } } } if (!success) { // 重新规划剩余步骤 currentPlan = replan(goal, results.toString(), currentPlan.subList(i, currentPlan.size())); i = -1; // 从头开始执行新计划 } } return results.toString(); } }

    ⚠️ 常见陷阱

    陷阱 1：过度规划。 简单任务不需要复杂的分层规划。"今天天气怎么样" 不需要 Plan-and-Execute，直接调用天气工具即可。根据任务复杂度选择合适的策略。

    任务规划让 Agent 能处理复杂任务。核心模式是 Plan-and-Execute：先制定计划，再逐步执行。关键设计：(1) 识别步骤间的依赖关系；(2) 支持并行执行无依赖步骤；(3) 实现动态调整机制应对意外；(4) 根据任务复杂度选择是否需要规划。

    👥 多 Agent 协作：团队协作

    ⚠️ 常见陷阱

    陷阱 1：Agent 角色重叠。 如果两个 Agent 的职责相似，它们可能产生矛盾的输出。确保每个 Agent 有明确、不重叠的职责边界。

    陷阱 2：通信开销过大。 Agent 之间的每次通信都消耗 Token。设计清晰的数据传递格式，避免冗余的来回沟通。

    陷阱 3：单点故障。 如果编排 Agent 出错，整个流程崩溃。实现超时和降级机制，某个 Agent 失败时能回退到简单方案。

    // ============================================ // Agent 记忆系统 - 短期记忆 + 长期记忆 // ============================================ @Service public class MemoryAgent { private final ChatClient chatClient; private final ChatMemory chatMemory; // 短期记忆（对话历史） private final VectorStore longTermMemory; // 长期记忆（向量数据库） public String chat(String userId, String message) { // ===== Step 1: 检索长期记忆 ===== // 从向量数据库中搜索与当前问题相关的记忆 List<Document> relevantMemories = longTermMemory .similaritySearch( SearchRequest.query(message).withTopK(3)); String memoryContext = relevantMemories.stream() .map(Document::getContent) .collect(Collectors.joining("\n")); // ===== Step 2: 使用记忆增强对话 ===== return chatClient.prompt() .system(""" 你是智能助手。 以下是相关的长期记忆（用户历史偏好和经验）： %s 请基于记忆和当前对话回答问题。 如果用户提到了新的重要信息（如偏好、习惯）， 请在回答末尾标记 [MEMORIZE: 信息内容]。 """.formatted(memoryContext)) .advisors(new MessageChatMemoryAdvisor( chatMemory, userId, 10)) // 保留最近 10 条对话 .user(message) .call() .content(); } // ===== Step 3: 定期保存重要记忆到长期存储 ===== @Scheduled(fixedRate = 300000) // 每 5 分钟执行一次 public void saveMemories() { // 从最近对话中提取重要信息 List<String> memories = extractImportantInfo(); for (String info : memories) { Document doc = new Document(info, Map.of( "type", "user_preference", "timestamp", Instant.now().toString() )); longTermMemory.accept(List.of(doc)); } } }

    // ============================================ // 记忆管理策略 // ============================================ @Service public class MemoryManager { private final ChatClient chatClient; private final VectorStore vectorStore; /** * 策略 1: 摘要压缩 * 定期将长对话压缩为摘要，节省 Token */ public String compressConversation(List<ChatMessage> messages) { return chatClient.prompt() .system("将以下对话压缩为简短摘要，保留关键信息。") .user(messages.toString()) .call() .content(); } /** * 策略 2: 重要性评分 * 根据信息重要性决定是否保存到长期记忆 */ public double scoreImportance(String info) { String scoreStr = chatClient.prompt() .system(""" 评估以下信息的重要性（0.0-1.0）。 高分标准：用户偏好、重要决定、长期有效的信息。 低分标准：临时信息、闲聊、一次性查询。 只返回数字。 """) .user(info) .call() .content(); return Double.parseDouble(scoreStr.trim()); } /** * 策略 3: 遗忘机制 * 清理过期或低价值的记忆 */ @Scheduled(cron = "0 0 3 * * ?") // 每天凌晨 3 点 public void cleanupMemories() { // 删除超过 30 天的低重要性记忆 List<Document> oldDocs = vectorStore .search(FilterExpressionBuilder .lt("timestamp", Instant.now().minus(Duration.ofDays(30)))); for (Document doc : oldDocs) { double importance = scoreImportance(doc.getContent()); if (importance < 0.3) { vectorStore.delete(List.of(doc.getId())); } } } }

    记忆类型对比

    📋 短期记忆（Working Memory）

    • 存储：Redis / 内存
    • 容量：最近 5-20 条对话
    • 生命周期：会话期间
    • 用途：保持对话上下文连贯
    • 比喻：工作台上的便签纸

    📔 长期记忆（Long-term Memory）

    • 存储：向量数据库（Chroma/Milvus）
    • 容量：理论上无限
    • 生命周期：持久化存储
    • 用途：用户偏好、历史经验
    • 比喻：私人日记本

    • 长期记忆：记住学生的学习偏好（"这个学生偏好通过代码示例学习"、"在 Java 基础方面需要加强"）
    • 记忆检索：新会话开始时，自动加载相关记忆，提供个性化辅导

    记忆系统是 Agent 从"工具"进化为"助手"的关键。研究显示，带有长期记忆的 Agent 在重复性任务中的表现提升 40% 以上。Mem0 是一个专门为 AI Agent 设计的记忆层框架，它自动处理记忆的提取、存储和检索。

    ⚠️ 常见陷阱

    陷阱 1：记忆无限膨胀。 不加限制地保存记忆会导致存储成本飙升和检索效率下降。务必实现遗忘机制和存储容量限制。

    陷阱 2：记忆污染。 错误信息或临时信息被保存为长期记忆，导致 Agent 后续决策出错。实现重要性评分机制，只保存高质量的记忆。

    陷阱 3：隐私泄露。 长期记忆可能包含敏感信息（如用户密码、个人隐私）。实现数据脱敏和访问控制，确保记忆系统的安全性。

    ✔ 要点小结

    记忆系统让 Agent 能"记住过去"。设计要点：(1) 短期记忆（Redis）保持对话上下文，长期记忆（VectorDB）存储经验；(2) 实现重要性评分，只保存高质量记忆；(3) 定期清理过期数据；(4) 注意隐私保护和数据安全。

    🎓 面试高频题

    👨‍💻 Q1: AI Agent 和普通的 LLM API 调用有什么区别？

    LLM API 调用是无状态的一问一答模式，用户需要自己协调多步操作。AI Agent 是一个自主决策系统，具备四个核心能力：(1) 自主规划 -- 将任务分解为子任务；(2) 工具使用 -- 调用外部 API、数据库等完成操作；(3) 反思迭代 -- 根据结果动态调整策略；(4) 记忆保持 -- 在多轮交互中保持上下文。简单说，LLM 是"大脑"，Agent 是"完整的人"。

    👨‍💻 Q2: ReAct 模式的核心思想是什么？为什么比纯推理或纯行动更好？

    ReAct 的核心是让 LLM 交替进行推理（Thought）和行动（Action），通过观察（Observation）获取反馈。纯推理（Chain-of-Thought）容易产生"幻觉"，因为没有外部事实验证；纯行动（只调用工具不思考）缺乏方向性。ReAct 结合了两者优势：推理提供方向，行动获取真实数据，观察验证推理。这使 Agent 的决策更加可靠和可解释。

    👨‍💻 Q3: 如何设计一个生产级的 Agent 记忆系统？

    分层设计：(1) 短期记忆用 Redis 或内存存储当前会话的最近 N 条消息，保证低延迟；(2) 长期记忆用向量数据库（如 Milvus/Chroma）存储历史经验和用户偏好，支持语义检索；(3) 实现记忆压缩（定期将长对话摘要化）；(4) 重要性评分（只将高价值信息存入长期记忆）；(5) 遗忘机制（定期清理过期低价值数据）；(6) 隐私保护（敏感信息脱敏）。

    👨‍💻 Q4: 多 Agent 协作有哪些常见模式？各自适用什么场景？

    主要有三种模式：(1) 流水线模式 -- Agent A → B → C 顺序执行，适合有明确阶段划分的任务（如文章生产）；(2) 辩论模式 -- 多个 Agent 从不同角度分析同一问题，最终汇总，适合需要多视角的任务（如代码审查）；(3) 主从模式 -- 一个编排 Agent 分配任务给多个工作 Agent，适合可并行的子任务。选择模式时要考虑任务的依赖关系和并行度。
