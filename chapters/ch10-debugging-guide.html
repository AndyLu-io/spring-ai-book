第 10 章

调试排错指南

AI 应用的 Bug 不像传统程序那样有明确的错误堆栈。学会像"AI 医生"一样诊断问题，是每个 AI 开发者的必备技能。

⏱ 预计 1 周 📖 6 个知识点 💻 中级难度
本章目录

    📚 开篇故事：AI 医生的诊断室

    传统程序出了 Bug，就像得了感冒 — 症状明确（报错信息），病因清楚（代码逻辑错误），治疗方案直接（改代码、加断点、看堆栈）。但 AI 应用出了问题，更像是一种疑难杂症 — 你说"不舒服"（输出不对），但到底是哪里不舒服？是 Prompt 写得不好（"药方"有问题），还是模型理解有偏差（"患者体质"不同），还是检索到了错误的文档（"化验单"拿错了）？

    更麻烦的是，AI 应用的 Bug 往往是概率性的 — 同样的输入，可能这次对、下次错。你不能像调试普通程序那样"设断点、单步执行"，因为模型的推理过程是一个"黑箱"。这就像中医看病，不能只看一个症状就下结论，需要"望闻问切"，从多个维度综合诊断。

    本章将教你一套完整的 AI 应用调试方法论。从错误分类到 Prompt 调试，从 RAG 诊断到性能优化，再到日志追踪系统的设计。掌握这些技能，你就能从"AI 应用的用户"升级为"AI 应用的医生"。

    🔍 常见错误分类

    AI 应用的四大类错误

    调试的第一步是分类。知道是哪一类问题，才能对症下药。

    比喻：Agent 的"病历本"

    就像医院的病历本会记录患者的基本信息、症状分类、诊断结果一样，我们也要给 AI 应用的错误建立一个"病历"。把错误分门别类，下次遇到类似问题就能快速定位。

    错误类别典型症状比喻常见原因
    API 错误 401/403/429/500 "门卫不让进" API Key 错误、额度用尽、限流
    模型错误 输出不达预期、幻觉 "答非所问" Prompt 不清晰、模型选型不当
    配置错误 Bean 注入失败、连接超时 "电线接错了" 配置文件、依赖版本、环境变量
    运行时错误 OOM、超时、数据格式错误 "半路抛锚" Token 超限、网络问题、数据异常

    错误码速查表

    HTTP 状态码含义解决方案
    401Unauthorized检查 API Key 是否正确、是否过期
    403Forbidden检查 API Key 权限、是否绑定了正确的 IP
    429Rate Limited降低请求频率、实现指数退避重试
    400Bad Request检查请求参数、Token 是否超限
    500Server Error服务商端问题，稍后重试或联系支持
    503Service Unavailable模型服务过载，等待或切换备用模型
    // Spring AI 中的异常处理最佳实践 @RestControllerAdvice public class AiExceptionHandler { @ExceptionHandler(AIRequestException.class) public ResponseEntity<ErrorResponse> handleAiRequest( AIRequestException e) { return switch (e.getStatusCode()) { case 401, 403 -> ResponseEntity.status(HttpStatus.UNAUTHORIZED) .body(new ErrorResponse("AUTH_ERROR", "API 认证失败，请检查 API Key")); case 429 -> ResponseEntity.status(HttpStatus.TOO_MANY_REQUESTS) .body(new ErrorResponse("RATE_LIMIT", "请求过于频繁，请稍后重试")); default -> ResponseEntity.status(HttpStatus.BAD_GATEWAY) .body(new ErrorResponse("AI_ERROR", "AI 服务暂时不可用")); }; } @ExceptionHandler(TokenLimitExceededException.class) public ResponseEntity<ErrorResponse> handleTokenLimit( TokenLimitExceededException e) { return ResponseEntity.badRequest() .body(new ErrorResponse("TOKEN_LIMIT", "输入内容过长，请缩短后重试。当前 Token 数: " + e.getTokenCount())); } }
    ⚠️ 踩坑警告

    API 限流（429 错误）是 AI 应用最常见的"线上故障"之一。务必在生产环境中实现指数退避重试机制，不要无限重试（会让限流更严重），也不要遇到 429 就直接报错给用户。

    指数退避重试示例

    // 使用 Spring Retry 实现指数退避 @Service public class ResilientChatService { private final ChatClient chatClient; @Retryable( retryFor = AIRequestException.class, maxAttempts = 3, backoff = @Backoff( multiplier = 2, // 每次等待时间翻倍 delay = 1000, // 首次等待 1 秒 maxDelay = 10000 // 最大等待 10 秒 ) ) public String chatWithRetry(String message) { return chatClient.prompt() .user(message) .call() .content(); } @Recover public String recover(AIRequestException e, String message) { log.error("AI 服务调用失败，已重试 3 次", e); return "抱歉，AI 服务暂时不可用，请稍后再试。"; } }

    🔎 Prompt 调试技巧

    输出不达预期的排查

    Prompt 问题是 AI 应用中最常见的"疑难杂症"。模型没有报错，但就是不按你想的做。

    比喻：Prompt 的"听诊器"

    调试 Prompt 就像医生用听诊器听心跳 — 你不能只看表面症状（输出结果），还要听内在的声音（模型的推理过程）。通过系统化的排查流程，找到"病根"所在。

    Prompt 调试流程

输出不符合预期
       |
       v
Step 1: 是否有明确的错误信息？
  是 ──> 根据错误码处理（见错误速查表）
  否 ──> 继续
       |
       v
Step 2: 输出格式是否正确？
  否 ──> 在 Prompt 中明确指定输出格式
  是 ──> 继续
       |
       v
Step 3: 内容质量是否达标？
  否 ──> 优化 Prompt（加示例、加约束）
  是 ──> 继续
       |
       v
Step 4: 是否存在幻觉？
  是 ──> 添加事实性约束、使用 RAG
  否 ──> 问题已解决

    Prompt 调试清单

    // ===== Prompt 调试清单 ===== // 1. 角色定义是否清晰？ // 差: "你是一个助手" // 好: "你是一位有 10 年经验的 Java 架构师，擅长 Spring Boot 微服务" // 2. 任务描述是否具体？ // 差: "分析这段代码" // 好: "分析这段 Java 代码的以下方面：\n1. 性能瓶颈\n2. 安全漏洞\n3. 代码异味" // 3. 输出格式是否明确？ // 差: "给出建议" // 好: "请用以下 JSON 格式输出：\n{\"issues\": [...], \"suggestions\": [...]}" // 4. 约束条件是否充分？ // 差: "翻译这段话" // 好: "将以下中文翻译成英文，要求：\n1. 保持技术术语准确\n2. 语气正式\n3. 不要添加解释" // 5. Few-shot 示例是否提供？ // 对于复杂任务，提供 2-3 个输入-输出示例 // 6. 温度参数是否合适？ // 代码生成: temperature=0.2（精确） // 创意写作: temperature=0.8（多样） // 数据分析: temperature=0.0（确定性）
    💡 小窍门

    Prompt 调试的黄金法则：每次只改一个变量。同时改了 Prompt、换了模型、调了 temperature，你永远不知道是哪个改动起了作用。

    实战案例：修复一个"幻觉"问题

    // 问题：AI 生成了不存在的 API 方法名 // 用户问："Spring AI 怎么用 ChatMemory？" // AI 回答了一个不存在的 ChatMemory.builder() 方法 // 修复前的 Prompt： "回答关于 Spring AI 的问题" // 修复后的 Prompt： """ 你是一位 Spring AI 技术专家。 回答用户关于 Spring AI 的问题时，请遵守以下规则： 1. 只基于 Spring AI 1.0+ 的官方 API 回答 2. 如果不确定某个 API 是否存在，请明确说明 3. 不要编造方法名或类名 4. 回答格式： - 先给出简明结论 - 再给出代码示例（使用实际存在的 API） - 最后给出参考文档链接 已知的核心 API： - ChatClient: 对话客户端 - ChatMemory: 对话记忆（使用 Advisor 模式） - VectorStore: 向量存储 - Advisor: 拦截器模式 """ // 关键改动： // 1. 明确了"只基于官方 API"的约束 // 2. 要求"不确定时明确说明"（减少幻觉） // 3. 列出了已知 API 作为参考

    📽 RAG 调试技巧

    检索质量诊断

    RAG 系统的输出质量取决于两个环节：检索（Retrieval）和生成（Generation）。调试时需要先确定是哪个环节出了问题。

    比喻：检索的"X光"

    调试 RAG 就像医生看 X 光片 — 你不能只看最终的诊断结果（AI 的回答），还要看"片子"（检索到的文档）。如果"片子"拍得不清楚（检索质量差），再好的医生（模型）也难以给出准确诊断。

    RAG 调试决策树

AI 回答质量差
       |
       v
检查检索到的文档
       |
       +---> 检索质量差 ──> 优化检索
       |       |
       |       +---> 相关文档未命中 ──> 调整分块策略
       |       +---> 噪声文档过多 ──> 调整 TopK / 过滤
       |       +---> 向量表示不准 ──> 换嵌入模型
       |
       +---> 检索质量好 ──> 优化生成
               |
               +---> 上下文太长 ──> 减少 TopK / 压缩
               +---> 模型未利用上下文 ─> 优化 Prompt

    分块策略调优

    策略分块大小重叠适用场景
    固定大小512 tokens50 tokens通用文本
    递归分割按段落/句子20%结构化文档
    语义分割按语义边界可选长文档、书籍
    代码分割按函数/类无代码仓库
    // Spring AI 中配置分块策略 @Bean public TextReader documentReader() { return new TextReader("classpath:docs/spring-ai-guide.md"); } // 使用递归分割器 @Bean public DocumentTransformer textSplitter() { return new RecursiveCharacterTextSplitter( 500, // 分块大小（字符数） 100, // 重叠大小 List.of("\n\n", "\n", " ", "") // 分割符优先级 ); } // 调试：检查分块结果 List<Document> chunks = textSplitter.apply(documents); for (int i = 0; i < chunks.size(); i++) { log.info("Chunk {}: {} chars", i, chunks.get(i).getContent().length()); }
    ⚠️ 踩坑警告

    分块太大会导致检索到无关内容（噪声），分块太小会导致上下文不完整。建议先用不同大小测试，找到最佳平衡点。通常 256-1024 tokens 是一个合理的范围。

    RAG 调试代码示例

    // RAG 调试工具：打印检索结果 @Service public class RagDebugger { private final VectorStore vectorStore; public void debugRetrieval(String query, int topK) { SearchRequest request = SearchRequest.query(query) .withTopK(topK) .withSimilarityThreshold(0.7); // 相似度阈值 List<Document> results = vectorStore.similaritySearch(request); log.info("===== RAG 调试: query='{}' =====", query); log.info("检索到 {} 个文档", results.size()); for (int i = 0; i < results.size(); i++) { Document doc = results.get(i); log.info("--- 文档 {} ---", i + 1); log.info(" 来源: {}", doc.getMetadata().get("source")); log.info(" 相似度: {}", doc.getMetadata().get("similarity")); log.info(" 内容: {}", doc.getContent().substring(0, Math.min(200, doc.getContent().length()))); } } public void debugFullRag(String query) { // 1. 打印检索结果 debugRetrieval(query, 3); // 2. 构建 Prompt 并打印完整内容 String systemPrompt = """ 根据以下检索到的文档回答问题。 如果文档中没有相关信息，请明确说明。 文档内容： [此处会插入检索结果] """; log.info("System Prompt:\n{}", systemPrompt); // 3. 记录 AI 的回答 // 4. 对比：检索内容 vs AI 回答 } }

    🧠 Agent 调试技巧

    工具调用失败排查

    Agent 系统中最复杂的调试场景是工具调用链 — 模型决定调用哪个工具、传什么参数、如何处理返回结果，任何一个环节出错都可能导致整个流程失败。

    比喻：Agent 的"脑电图"

    调试 Agent 就像做脑电图 — 你不能直接"打开"大脑看里面在想什么，但可以通过监测电信号（日志、追踪）来推断思维过程。Agent 的"思维链"就是它的"脑电波"。

    // Agent 调试：启用详细日志 # application.yml logging: level: org.springframework.ai: DEBUG org.springframework.ai.chat.model: DEBUG org.springframework.ai.tool: DEBUG // 自定义工具调用拦截器 @Component public class ToolCallLogger implements ToolCallback { private final Logger log = LoggerFactory.getLogger(ToolCallLogger.class); @Override public String call(String toolInput) { log.info("====== 工具调用 ======"); log.info("输入参数: {}", toolInput); try { String result = this.delegate.call(toolInput); log.info("返回结果: {}", result); return result; } catch (Exception e) { log.error("工具调用失败: {}", e.getMessage()); throw e; } } }

    Agent 循环检测

    Agent 可能会陷入无限循环 — 不断调用同一个工具，或者在两个工具之间反复切换。

    正常的 Agent 执行流程
用户提问 -> 调用工具A -> 获得结果 -> 回答用户 [完成]

异常：循环调用
用户提问 -> 调用工具A -> 结果不满意 ->
 调用工具A -> 结果不满意 ->
  调用工具A -> ... (无限循环)

异常：工具乒乓
用户提问 -> 工具A -> 工具B ->
 工具A -> 工具B ->
  工具A -> ... (乒乓循环)
    // 防止 Agent 循环：设置最大迭代次数 @Service public class SafeAgentService { private static final int MAX_ITERATIONS = 10; private static final int MAX_SAME_TOOL_CALLS = 3; public String safeAgentCall(String query, ChatClient client) { List<String> toolCallHistory = new ArrayList<>(); for (int i = 0; i < MAX_ITERATIONS; i++) { String result = client.prompt() .user(query) .call() .content(); // 检测循环：同一个工具连续调用超过阈值 toolCallHistory.add(extractToolName(result)); if (isToolLoop(toolCallHistory)) { log.warn("检测到工具调用循环，强制终止"); return "处理过程中遇到问题，请尝试换个方式提问。"; } } return "处理超时，请简化问题后重试。"; } private boolean isToolLoop(List<String> history) { if (history.size() < MAX_SAME_TOOL_CALLS) return false; List<String> recent = history.subList( history.size() - MAX_SAME_TOOL_CALLS, history.size()); return recent.stream().distinct().count() == 1; } }
    💡 小窍门

    在开发阶段，为每个工具调用添加调用日志，记录：工具名、输入参数、输出结果、耗时。这些日志是调试 Agent 行为的"黑匣子"。

    📈 性能问题排查

    延迟高

    AI 应用的延迟通常来自三个地方：网络（API 调用）、模型推理（生成时间）、检索（向量搜索）。

    比喻：性能的"血液检查"

    排查性能问题就像做血液检查 — 你需要逐项检测每个指标，找出"异常值"。不能只看总体延迟，要拆分成各个阶段分别测量。

    // 性能诊断工具 @Service public class PerformanceDiagnostics { private final ChatClient chatClient; private final VectorStore vectorStore; public DiagnosticsResult diagnose(String query) { var result = new DiagnosticsResult(); // 1. 测量检索延迟 long retrievalStart = System.currentTimeMillis(); List<Document> docs = vectorStore.similaritySearch( SearchRequest.query(query).withTopK(3)); result.retrievalTime = System.currentTimeMillis() - retrievalStart; // 2. 测量模型推理延迟 long inferenceStart = System.currentTimeMillis(); String response = chatClient.prompt() .user(query) .call() .content(); result.inferenceTime = System.currentTimeMillis() - inferenceStart; // 3. 估算 Token 使用量 result.estimatedTokens = query.length() / 2 + response.length() / 2; // 4. 总耗时 result.totalTime = result.retrievalTime + result.inferenceTime; return result; } public record DiagnosticsResult( long retrievalTime, long inferenceTime, long totalTime, int estimatedTokens ) {} }

    优化清单

    问题症状优化方案
    延迟高响应时间 > 5s使用流式输出、减少 TopK、用更快的模型
    成本高Token 消耗大压缩上下文、使用更小的模型、缓存常见问题
    质量差回答不准确优化 Prompt、改善检索质量、使用更好的模型
    并发低QPS 上不去异步处理、连接池、本地模型部署
    不稳定间歇性失败重试机制、熔断器、降级策略
    // 常用优化策略代码 // 1. 流式输出 — 用户体验优化 chatClient.prompt() .user(message) .stream() // 流式输出，用户感知延迟降低 .content() .subscribe(chunk -> { // 实时推送给前端 sseEmitter.send(chunk); }); // 2. 缓存常见问题 @Cacheable(value = "ai-responses", key = "#message.hashCode()") public String cachedChat(String message) { return chatClient.prompt().user(message).call().content(); } // 3. 降级策略 — 使用更小的模型 public String chatWithFallback(String message) { try { return chatClient.prompt() .options(ChatOptions.builder().model("gpt-4o").build()) .user(message).call().content(); } catch (Exception e) { return chatClient.prompt() .options(ChatOptions.builder().model("gpt-4o-mini").build()) .user(message).call().content(); } }

    📷 日志与追踪

    结构化日志

    AI 应用的日志不同于传统应用 — 你需要记录的不只是"调用了什么接口"，还有"Prompt 是什么"、"模型返回了什么"、"Token 消耗了多少"。

    比喻：问题的"监控录像"

    结构化日志就像超市的监控录像 — 不只拍到"有人来了"（请求进来了），还拍到"他买了什么"（调用了什么工具）、"花了多少钱"（Token 消耗）、"什么时候走的"（响应时间）。出问题时可以回放录像，定位根因。

    // 结构化日志配置（application.yml） logging: level: org.springframework.ai: INFO pattern: console: "%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n" // 自定义 AI 调用日志 @Aspect @Component public class AiCallLoggingAspect { private static final Logger log = LoggerFactory.getLogger(AiCallLoggingAspect.class); @Around("execution(* org.springframework.ai.chat.client.ChatClient..*(..))") public Object logAiCall(ProceedingJoinPoint joinPoint) throws Throwable { long start = System.currentTimeMillis(); String traceId = UUID.randomUUID().toString().substring(0, 8); log.info("[AI-{}] 开始调用: {}", traceId, joinPoint.getSignature()); try { Object result = joinPoint.proceed(); long duration = System.currentTimeMillis() - start; log.info("[AI-{}] 成功完成 | 耗时: {}ms", traceId, duration); return result; } catch (Throwable e) { long duration = System.currentTimeMillis() - start; log.error("[AI-{}] 调用失败 | 耗时: {}ms | 错误: {}", traceId, duration, e.getMessage()); throw e; } } }

    OpenTelemetry 集成

    OpenTelemetry 是一个可观测性框架，可以统一追踪、指标和日志。对于 AI 应用，它可以帮你可视化整个调用链路。

    生产环境监控建议

    • Token 使用量监控 — 设置每日/每月 Token 消耗告警
    • 延迟监控 — P50/P95/P99 延迟指标，超过阈值自动告警
    • 错误率监控 — 429/500 错误率超过 1% 触发告警
    • 成本追踪 — 按用户/项目维度统计 API 调用成本
    • 质量评估 — 定期抽样检查 AI 输出质量（可用另一个模型打分）

    建立"AI 健康看板"：用 Grafana 搭建一个实时看板，展示关键指标：当前 QPS、平均延迟、错误率、Token 消耗趋势。这是团队了解 AI 服务状态的"仪表盘"。

    📖 章节小结

    本章我们掌握了一套完整的 AI 应用调试方法论：

    • 错误分类 — API 错误、模型错误、配置错误、运行时错误四大类
    • Prompt 调试 — 系统化排查流程，每次只改一个变量
    • RAG 调试 — 先诊断检索质量，再优化生成质量
    • Agent 调试 — 工具调用日志 + 循环检测 + 最大迭代限制
    • 性能优化 — 分阶段测量，流式输出、缓存、降级策略
    • 日志追踪 — 结构化日志 + OpenTelemetry 追踪

    AI 应用的调试比传统应用更需要耐心和方法论。掌握了这些技能，你就能从容应对生产环境中的各种"疑难杂症"。

    🎓 面试高频题

    👨‍💻 Q1: AI 应用的输出不稳定（同样的输入，不同结果），如何排查？

    这是 LLM 的概率性本质决定的。排查步骤：1) 检查 temperature 参数，设为 0 可获得确定性输出；2) 检查是否使用了随机种子；3) 如果是 RAG 系统，检查检索结果是否稳定；4) 如果是 Agent，检查工具调用是否有随机因素。生产环境中可以通过多次调用取平均/投票来提高稳定性。

    👨‍💻 Q2: 如何防止 Agent 陷入无限循环？

    设置最大迭代次数限制（如 10 次）；检测同一工具的连续重复调用（如超过 3 次相同工具调用则终止）；记录工具调用历史，检测乒乓循环（A->B->A->B）；在 Prompt 中明确限制工具调用次数；使用 Spring Retry 的 maxAttempts 限制。

    👨‍💻 Q3: RAG 系统检索质量差，可能的原因和优化方案？

    可能原因：1) 分块策略不合理（太大/太小）；2) 嵌入模型不适合当前领域；3) 查询与文档的语义不匹配；4) TopK 设置不当；5) 缺少元数据过滤。优化方案：调整分块大小（256-1024 tokens）；尝试不同的嵌入模型；添加查询重写/扩展；使用混合检索（向量+关键词）；添加相似度阈值过滤。
