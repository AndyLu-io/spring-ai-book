Prompt 工程

掌握 Prompt 设计的艺术与科学，通过模板化、Few-shot、思维链等技术提升 LLM 输出质量

预计 1-2 周 7 个知识点 中级难度

📚 开篇故事：同一个人，两种回答

小王和小李同时问 AI 一个简单的问题："帮我写一份请假邮件"。

小王直接输入："写请假邮件"。AI 回复："好的，请假邮件如下：我因身体不适需要请假一天。" -- 干巴巴的，连收件人、日期、事由都没有。

小李却精心设计了 Prompt："你是一位专业的行政助理。请为我写一份正式的请假邮件，收件人是张经理，请假日期是6月5日，原因是家中有事需要处理。语气要礼貌正式，包含交接安排。"。AI 回复了一封完美的请假邮件，格式规范、措辞得体、还主动提到了工作交接。

同一个 AI 模型，因为Prompt 不同，输出质量天壤之别。这就是 Prompt 工程的魅力 -- 它就像"魔法咒语"，说对了话，AI 就能施展强大的魔法；说错了话，它就只会给你平庸的答案。

🔑 Prompt 设计原则 -- 魔法咒语的规则

📄 概念：什么是好的 Prompt？

Prompt 就是你发给 AI 的"魔法咒语"。咒语说得越精确、越完整，AI 的"魔法"就越强大。好的 Prompt 不是随便写几个字，而是需要遵循一套精心设计的规则。

坏咒语                              好咒语
"写代码"                            "用 Spring Boot 3.x + Java 17
                                      编写 REST API:
                                        - GET /api/users
                                        - 使用 Record DTO
                                        - 包含异常处理"
     |                                    |
     v                                    v
AI: 写什么? 什么语言?               AI: 精准输出代码

五大核心原则

原则说明比喻示例
📝 明确性 指令清晰无歧义 点菜要说清楚口味 "翻译成英文" vs "翻译成美式英语"
📏 具体性 提供约束和格式要求 下厨要有菜谱 "输出 JSON，包含 name 和 age"
📋 结构化 用 Markdown/JSON 组织内容 写信要有格式 使用编号列表、分隔线
🔄 分步骤 复杂任务拆分为多步 做菜要分步骤 "第一步: 分析, 第二步: 实现"
💡 提供上下文 给出背景信息和示例 给演员看剧本 "参考这个示例的风格"

💻 代码示例：好 Prompt vs 坏 Prompt 对比

✘ 坏 Prompt

帮我写一段关于Spring的代码

问题：太模糊，不清楚要什么功能、什么语言版本、什么框架、什么输出格式

✔ 好 Prompt

使用 Spring Boot 3.x + Java 17， 编写一个 REST API： - GET /api/users 返回用户列表 - 使用 Record 定义 DTO - 包含全局异常处理 - 返回 JSON 格式 - 代码有详细注释

优点：具体、结构化、有约束条件、有格式要求

💡 你知道吗？

研究表明，好的 Prompt 可以让同一个模型的表现提升 50% 以上。Google 在 2023 年发布的一篇论文显示，在特定任务上，精心设计的 Prompt 甚至能让小模型超越大模型的原始表现。Prompt Engineering 已经成为一个专门的职业方向。

💻 试试看

对比以下三种质量的 Prompt，分别请求同一个问题"如何学习 Java"，观察输出差异：
1. "学 Java" (太简短)
2. "给我一些学习 Java 的建议" (基本可用)
3. "你是一位有 10 年经验的 Java 架构师。请为一位有 Python 基础的程序员制定 Java 学习路线，包含：推荐书籍、实战项目、学习时间规划。用 Markdown 格式输出。" (高质量)

📝 Prompt 模板 -- 可复用的咒语库

📄 概念：为什么需要模板？

如果每次都要手写完整的 Prompt，既费时又容易出错。Prompt 模板就像"咒语模板" -- 把固定的部分写好，每次只需要填入变量就行。Spring AI 提供了优雅的模板语法。

模板定义:                              模板实例化:
"你是{role}。                           role = "翻译助手"
 将文本从{source}翻译成{target}。       source = "中文"
 文本：{text}"                          target = "英文"
                                        text = "你好世界"
     |
     v
完整 Prompt: "你是翻译助手。将文本从中文翻译成英文。文本：你好世界"

💻 代码示例：Spring AI PromptTemplate

// ========================================== // TranslationTemplateConfig.java // 定义一个翻译 Prompt 模板 // ========================================== @Bean public PromptTemplate translationTemplate() { // 使用花括号 {variable} 定义模板变量 return new PromptTemplate(""" 你是一个专业的翻译助手。 将以下文本从 {sourceLanguage} 翻译成 {targetLanguage}。 保持原文的语气和风格。 文本：{text} """); } // ========================================== // 使用模板 - 填入变量 // ========================================== // 1. 创建 Prompt 对象 Prompt prompt = translationTemplate().create( Map.of( "sourceLanguage", "中文", // 源语言 "targetLanguage", "英文", // 目标语言 "text", "Spring AI 是一个强大的框架" // 待翻译文本 ) ); // 2. 发送给 AI String result = chatClient.prompt(prompt).call().content(); System.out.println(result); // 输出: "Spring AI is a powerful framework"

💻 代码示例：ChatClient 内联模板

// ========================================== // 在 ChatClient 中直接使用模板变量 // 更简洁，适合简单场景 // ========================================== String answer = chatClient.prompt() .system("你是{role}，专注于{domain}领域", Map.of( "role", "技术顾问", // 角色变量 "domain", "Java/Spring" // 领域变量 )) .user("请解释{concept}", Map.of("concept", "依赖注入")) // 概念变量 .call() .content(); System.out.println(answer); // 输出: "依赖注入（Dependency Injection）是一种设计模式... // 在 Spring 框架中，它通过 @Autowired 注解实现..."
💡 模板变量最佳实践

模板变量支持动态替换，非常适合构建可复用的 Prompt 库。建议将常用的 Prompt 模板存储在 application.yml 配置文件或数据库中，实现 Prompt 的版本管理和热更新。

💻 代码示例：Prompt 模板库（YAML 配置）

# ========================================== # application.yml - Prompt 模板配置 # 把 Prompt 集中管理，方便维护和迭代 # ========================================== prompts: # 翻译模板 translation: | 你是一个专业的翻译助手。 语言对：{source} -> {target} 保持专业术语准确。 输出格式：仅返回翻译结果，无额外说明。 文本：{text} # 代码审查模板 code-review: | 你是资深代码审查员。 请审查以下 {language} 代码： {code} 重点关注： 1. 安全性（30%权重） 2. 性能（25%权重） 3. 可读性（25%权重） 4. 最佳实践（20%权重） 输出格式：JSON
⚠️ 常见陷阱

1. 变量名拼写错误 -- 模板变量 {source} 写成 {souce} 不会报错但不会替换
2. 变量值包含花括号 -- 如果用户输入包含 {xxx}，可能被误解析为变量
3. 模板过长 -- 超过模型上下文窗口限制会导致截断

💡 Few-shot 学习 -- 示范教学法

📄 概念：什么是 Few-shot？

Few-shot 就像"示范教学" -- 你不需要教 AI 什么是情感分析，只需要给它几个例子，它就能"学会"。这就像老师在黑板上写三道例题，学生就能举一反三。

Zero-shot (不给示例)              Few-shot (给示例)
"分析情感：好吃"                    "示例1: 好吃 -> 正面
 AI: ??? (不确定)"                  "示例2: 难吃 -> 负面
                                    "示例3: 还行 -> 中性"
                                    "分析：好吃 -> 正面"

💻 代码示例：Few-shot 情感分析

// ========================================== // Few-shot 情感分析 Prompt // ========================================== String prompt = """ 你是一个情感分析助手。请分析以下评论的情感。 示例 1： 评论：这个产品质量很好，物超所值！ 情感：正面 置信度：0.95 示例 2： 评论：物流太慢了，等了一周才到 情感：负面 置信度：0.85 示例 3： 评论：包装还行，功能一般 情感：中性 置信度：0.70 现在请分析： 评论：{userComment} 情感： """; // 使用 ChatClient 执行 String result = chatClient.prompt() .user(prompt) .call() .content();

运行结果说明：AI 会按照示例的格式（情感 + 置信度）输出结果。示例越多、覆盖越全面，分类准确率越高。

💻 代码示例：Spring AI Few-shot 服务

// ========================================== // FewShotService.java - 客服意图分类器 // ========================================== @Service public class FewShotService { private final ChatClient chatClient; /** * 意图分类：根据用户输入判断客服工单类型 * 使用 Few-shot 示例引导 AI 输出正确类别 */ public String classifyIntent(String userInput) { return chatClient.prompt() .system(""" 你是一个意图分类器。根据用户输入判断意图类别。 可选类别：ORDER_QUERY, RETURN_REQUEST, PRODUCT_INFO, COMPLAINT, OTHER 示例： 输入："我的订单什么时候发货？" -> ORDER_QUERY 输入："我想退掉这个商品" -> RETURN_REQUEST 输入："这个产品有什么功能？" -> PRODUCT_INFO 输入："你们服务太差了" -> COMPLAINT 输入："今天天气真好" -> OTHER """) .user(userInput) .call() .content(); } } // 使用示例： // classifyIntent("我想退货") -> "RETURN_REQUEST" // classifyIntent("我的包裹到哪了") -> "ORDER_QUERY"

💻 代码示例：动态 Few-shot（从数据库加载示例）

// ========================================== // DynamicFewShotService.java - 动态示例加载 // ========================================== @Service public class DynamicFewShotService { private final ExampleRepository exampleRepository; // 示例数据库 private final ChatClient chatClient; /** * 动态构建 Few-shot Prompt * 根据任务类别从数据库加载相关示例 */ public String classify(String input, String category) { // 1. 从数据库获取该类别的示例（最多 5 个） List<FewShotExample> examples = exampleRepository .findByCategory(category); // 2. 将示例格式化为 Prompt 文本 String exampleText = examples.stream() .map(e -> "输入：\"" + e.input() + "\" -> " + e.output()) .collect(Collectors.joining("\n")); // 3. 动态构建完整的 Prompt return chatClient.prompt() .system("分类规则：\n" + exampleText) .user(input) .call() .content(); } }
💡 Few-shot 最佳实践

示例数量通常 3-5 个效果最佳。太少不够学习，太多会占用上下文窗口。示例应覆盖边界情况，格式保持一致，最好包含正面、负面和中性的样本。

💡 你知道吗？

Few-shot 这个概念来自 2020 年 GPT-3 的论文。研究发现，只需要在 Prompt 中给出 2-5 个示例，大模型就能"学会"新任务，无需重新训练。这被称为 "In-context Learning"（上下文学习），是大模型最神奇的能力之一。

💻 试试看

使用 Few-shot 技术实现一个代码审查分类器：给 5 个示例（安全问题、性能问题、可读性问题、最佳实践、无问题），输入一段代码片段，让 AI 输出审查类别和严重程度。

🔄 Chain-of-Thought -- 解题步骤

📄 概念：让 AI "展示解题过程"

Chain-of-Thought (CoT) 就像"解题步骤" -- 你不是让 AI 直接给答案，而是让它先写出推理过程。这就像数学考试要求"写出解题过程"，不只看最终答案。CoT 能显著提升 AI 在复杂推理任务上的准确率。

普通 Prompt                     CoT Prompt
"125 x 48 = ?"                  "125 x 48 = ?
                                 请一步步思考："
     |                                |
     v                                v
"6000" (可能算错)              "步骤1: 125 x 48 = 125 x 40 + 125 x 8
                                步骤2: 125 x 40 = 5000
                                步骤3: 125 x 8 = 1000
                                步骤4: 5000 + 1000 = 6000
                                答案: 6000"

💻 代码示例：Spring AI CoT 实现

// ========================================== // CotService.java - 思维链推理服务 // ========================================== @Service public class CotService { private final ChatClient chatClient; /** * CoT 推理：要求 AI 展示推理过程 * 适用于：数学推理、逻辑判断、代码调试 */ public String reason(String question) { return chatClient.prompt() .system(""" 你是一个擅长逻辑推理的助手。 对于每个问题，请： 1. 首先分析问题的关键信息 2. 逐步推理 3. 给出最终答案 使用以下格式： 分析：... 推理步骤： 步骤 1：... 步骤 2：... 最终答案：... """) .user(question) .call() .content(); } } // 使用示例： // reason("一个水池有两个进水管...") // 分析：这是一个工程问题... // 步骤1：管A每小时注入... // 步骤2：管B每小时注入... // 最终答案：需要 6 小时

💻 代码示例：Zero-shot CoT（最简单的 CoT）

// ========================================== // Zero-shot CoT - 只需要加一句话 // 最简单的思维链技巧 // ========================================== String answer = chatClient.prompt() .user(""" {question} 让我们一步一步思考： """, Map.of("question", question)) .call() .content(); // 示例： // question = "小明有 5 个苹果，给了小红 2 个，又买了 3 个，他现在有几个？" // // 普通 Prompt: "8" (直接回答, 可能出错) // CoT Prompt: "步骤1: 小明开始有 5 个苹果 // 步骤2: 给了小红 2 个, 剩下 5-2=3 个 // 步骤3: 又买了 3 个, 现在有 3+3=6 个 // 最终答案: 6 个"

CoT 的适用场景对比

场景是否需要 CoT原因
数学推理 ✔ 强烈推荐 多步计算容易出错
逻辑判断 ✔ 推荐 需要逐步分析条件
代码调试 ✔ 推荐 需要逐行追踪执行
简单问答 ✘ 不需要 增加延迟但无收益
翻译任务 ✘ 不需要 AI 已经擅长翻译
💡 CoT 使用建议

CoT 特别适合需要多步推理的场景。但对于简单问答，CoT 会增加不必要的延迟和 Token 消耗。建议先用简单 Prompt 测试，如果准确率不够再加 CoT。

💡 你知道吗？

"Let's think step by step"（让我们一步一步思考）这句话出自 2022 年 Google 的论文《Chain-of-Thought Prompting Elicits Reasoning in Large Language Models》。仅仅加上这 8 个单词，就能让模型在数学推理任务上的准确率从 17% 提升到 78%。这就是 CoT 的魔力。

🎭 角色设定 -- 给 AI "戴面具"

📄 概念：System Prompt 的魔力

角色设定就是给 AI "戴上面具" -- 告诉它"你现在是谁"。不同的角色会带来完全不同的回答风格和专业程度。就像同一个人，穿上白大褂就是医生，穿上西装就是商务人士。

同一个问题: "什么是依赖注入?"

角色 A: 资深架构师           角色 B: 教学助手
"DI 是 IoC 的实现方式,       "想象你是一个送快递的小哥。
 通过构造器注入或              以前你要自己去仓库取快递（new 对象），
 字段注入来解耦..."            现在快递直接送到你手里（Spring 注入）..."

角色 C: 代码审查员
"依赖注入是必须的。没有 DI,
 你的代码会直接 new 对象，
 这违反了 DIP 原则..."

💻 代码示例：三种角色的 Prompt 设计

// ========================================== // 不同角色的 System Prompt 设计 // ========================================== // 角色 1：技术专家 -- 简洁、专业、直接 String techExpert = """ 你是一位有 10 年经验的 Java 架构师。 特长：Spring 生态、微服务、分布式系统。 回答风格：简洁专业，直接给出技术方案。 不确定时说明原因，不要编造答案。 """; // 角色 2：教学助手 -- 循序渐进、用类比解释 String teacher = """ 你是一位耐心的编程教师。 目标受众：有基础编程经验但刚学 Java 的学生。 回答风格：循序渐进，先概念后代码。 用简单的类比解释复杂概念。 """; // 角色 3：代码审查员 -- 严格、量化、可操作 String reviewer = """ 你是一位严格的代码审查员。 审查标准： 1. 代码安全性（30%权重） 2. 性能优化（25%权重） 3. 可读性（25%权重） 4. 最佳实践（20%权重） 输出格式：评分 + 具体问题列表 + 改进建议 """;

💻 代码示例：多角色切换服务

// ========================================== // MultiRoleService.java - 动态角色切换 // ========================================== @Service public class MultiRoleService { private final ChatClient chatClient; /** * 根据角色动态切换 System Prompt * @param message 用户消息 * @param role 角色类型: expert/teacher/reviewer */ public String chat(String message, String role) { // 根据角色选择不同的 System Prompt String systemPrompt = switch (role) { case "expert" -> "你是技术专家。回答简洁专业，直接给出方案。"; case "teacher" -> "你是教学助手。循序渐进，用类比解释概念。"; case "reviewer" -> "你是代码审查员。严格审查，输出评分和建议。"; default -> "你是一个有帮助的助手。"; }; // 使用选择的角色发送请求 return chatClient.prompt() .system(systemPrompt) .user(message) .call() .content(); } }
💻 试试看

设计 3 种不同角色（产品经理、设计师、测试工程师），让它们分别审查同一段代码。对比三个角色的审查视角和输出差异。

📄 输出格式控制

💻 代码示例：JSON 结构化输出

// ========================================== // 强制 AI 输出 JSON 格式 // 使用 Spring AI 的 ResponseFormat // ========================================== String answer = chatClient.prompt() .system("请以 JSON 格式输出，包含以下字段：name, age, skills（数组）") .user("描述一个 Java 高级工程师") .options(OpenAiChatOptions.builder() .responseFormat(new ResponseFormat(Type.JSON_OBJECT)) .build()) .call() .content(); // 输出示例： // {"name":"张三","age":30,"skills":["Java","Spring","微服务"]} // 使用 entity() 方法自动解析为 Java 对象 Engineer engineer = chatClient.prompt() .system("描述一个 Java 高级工程师") .user("请输出") .call() .entity(Engineer.class); // 自动反序列化为 Record

💻 代码示例：Markdown 格式输出

对于结构化输出，优先使用 Spring AI 的 entity() 方法，它能自动将 LLM 输出解析为 Java 类型安全的对象，避免手动解析 JSON 的麻烦。

🎨 Temperature -- 创造力旋钮

📄 概念：控制 AI 的"创造力"

Temperature 就像"创造力旋钮" -- 旋钮调到 0，AI 像参加考试一样精确保守；调到 2，AI 像灵魂画手一样天马行空。不同的场景需要不同的"创造力级别"。

Temperature 旋钮

  0.0        0.7        1.0        2.0
  |-----------|-----------|-----------|
  考试模式     通用模式     创意模式     疯狂模式
  (精确)      (平衡)      (有趣)      (随机)

  适合:       适合:        适合:        适合:
  代码生成     日常对话     写诗/故事    头脑风暴
  数据提取     知识问答     广告文案     创意实验

💻 代码示例：Temperature 调优

// ========================================== // TemperatureService.java - 不同场景的 Temperature // ========================================== @Service public class TemperatureService { private final ChatClient chatClient; /** * 代码生成 - Temperature = 0（精确模式） * 代码需要精确，不能有"创意" */ public String generateCode(String requirement) { return chatClient.prompt() .user(requirement) .options(OpenAiChatOptions.builder() .temperature(0.0) // 最精确 .build()) .call() .content(); } /** * 创意写作 - Temperature = 0.9（高创造力） * 写诗需要灵感和创意 */ public String writePoem(String topic) { return chatClient.prompt() .user("以「" + topic + "」为主题写一首现代诗") .options(OpenAiChatOptions.builder() .temperature(0.9) // 高创造力 .build()) .call() .content(); } /** * 日常问答 - Temperature = 0.7（平衡模式） * 既准确又有一定灵活性 */ public String chat(String message) { return chatClient.prompt() .user(message) .options(OpenAiChatOptions.builder() .temperature(0.7) // 平衡模式 .build()) .call() .content(); } }

Temperature 调优指南

场景推荐 Temperature原因
💻 代码生成 0.0 - 0.2 代码需要精确，不能有"创意发挥"
🔍 数据提取 0.0 结构化数据必须严格准确
💬 日常对话 0.5 - 0.7 平衡准确性和自然度
🎨 创意写作 0.8 - 1.2 需要灵感和多样性
💡 头脑风暴 1.0 - 1.5 尽可能多地产生不同想法
⚠️ 常见陷阱

Temperature 过高（>1.5）可能导致输出胡言乱语。Temperature 过低（=0）每次输出完全相同，缺乏多样性。建议从 0.7 开始，根据效果微调。

💡 你知道吗？

Temperature 的数学原理：它在 softmax 函数中作为分母的缩放因子。softmax(logits/T)。T 越小，概率分布越尖锐（倾向最高概率）；T 越大，分布越平坦（更随机）。这就是为什么 T=0 时输出确定，T>1 时输出多样。

📖 章节小结

本章我们掌握了 Prompt 工程的"魔法咒语"：

  • 设计原则（咒语规则）-- 明确性、具体性、结构化、分步骤、提供上下文
  • Prompt 模板（可复用咒语库）-- Spring AI PromptTemplate + YAML 配置
  • Few-shot（示范教学）-- 3-5 个示例引导 AI 输出，支持动态加载
  • Chain-of-Thought（解题步骤）-- 让 AI 展示推理过程，提升准确率
  • 角色设定（戴面具）-- System Prompt 控制回答风格和专业程度
  • Temperature（创造力旋钮）-- 0 精确到 2 疯狂，按场景调节

掌握这些技巧，你就能让同一个 AI 模型输出质量提升 50% 以上。Prompt 工程是 AI 开发者最核心的技能之一。

🎓 面试高频题

👨‍💻 Q1: Few-shot 和 Zero-shot 的区别是什么？

Zero-shot 不提供示例，直接给指令（"翻译这段话"）。Few-shot 在 Prompt 中提供 2-5 个示例，让 AI 学习输出格式和模式。Few-shot 在分类、格式化等任务上表现更好，但会占用上下文窗口。最佳实践是 3-5 个示例，覆盖正负样本。

👨‍💻 Q2: Chain-of-Thought 为什么能提升推理准确率？

CoT 利用了 LLM 的"自回归"特性 -- 生成的每一步推理都会成为下一步的上下文。这让模型可以"分步思考"，而不是试图一步到位。数学推理准确率可从 17% 提升到 78%（Google 2022 年论文）。

👨‍💻 Q3: Temperature 和 Top-P 的区别？

Temperature 通过缩放概率分布来控制随机性（全局调节）。Top-P（核采样）只从累积概率前 P 的 token 中采样（局部裁剪）。两者通常不同时调，一般只调一个。Temperature=0 等价于贪心搜索（每次选最高概率）。
