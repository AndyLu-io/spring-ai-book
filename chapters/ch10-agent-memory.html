第 10 章

Agent 记忆系统（深度）

深入探索 Agent 的四种记忆类型——工作记忆、情景记忆、语义记忆和程序性记忆，掌握记忆架构模式与管理策略，构建具备持久化能力的智能 Agent

预计 2-3 周 8 个知识点 高级难度

📚 开篇故事：大脑的记忆迷宫

你有没有这样的经历？走进厨房，突然忘了自己要拿什么。站在冰箱前，脑子一片空白。但当你转身走回客厅，瞥见茶几上的遥控器时，突然想起来——"哦，我要拿的是啤酒。"

这个看似尴尬的瞬间，其实揭示了人类记忆系统最精妙的设计。你的工作记忆（正在思考的事情）只有7±2个信息块的容量，就像大脑的一块小黑板。你的情景记忆记录了"上周二在酒吧和朋友聊天"这样的事件序列。你的语义记忆存储着"啤酒是用麦芽酿造的"这样的知识。你的程序性记忆则让骑自行车这件事不再需要刻意思考。

AI Agent 面临完全相同的挑战。当一个智能助手需要记住"用户昨天说他喜欢喝IPA精酿"、"上周的订单还没发货"、"用户公司叫XX科技"这些信息时，它需要一套完整的记忆系统——而不仅仅是一段聊天历史。没有记忆的 Agent，就像一个每天早上失忆的管家，你不得不每次从头开始解释你是谁、你要什么、你讨厌什么。

本章将带你深入构建 Agent 的记忆系统——从最短暂的工作记忆到持久化的长期记忆，让 Agent 真正成为一个"记性好"的智能伙伴。

📄 核心内容（第1节）：记忆类型体系

📄 工作记忆（Working Memory）—— 办公桌上的便签纸

定义：工作记忆是 Agent 在当前任务执行过程中，临时存储和操作信息的区域。它就像你办公桌上那块小黑板——你只能同时关注有限的几件事，一旦黑板写满了，最早的信息就会被覆盖。

核心特征：

  • 容量限制：心理学研究发现人类工作记忆容量约为 7±2 个信息块。Agent 的工作记忆通常也有窗口限制（如最近 10-20 条消息）
  • 注意力窗口：只有被"注意到"的信息才能进入工作记忆，类似于 Agent 的 system prompt 中只能放入有限的上下文
  • 快速衰减：没有被持续关注的信息会迅速消失，就像你转身就忘了刚才要拿什么

// 工作记忆：简单的滑动窗口实现 public class WorkingMemory { private final int windowSize; private final Deque<Message> window; public WorkingMemory(int windowSize) { this.windowSize = windowSize; this.window = new ArrayDeque<>(windowSize); } // 添加新消息，超出窗口的自动丢弃 public void add(Message message) { window.addLast(message); if (window.size() > windowSize) { window.pollFirst(); // 丢弃最早的 } } // 获取当前工作记忆快照 public List<Message> snapshot() { return new ArrayList<>(window); } }

📄 情景记忆（Episodic Memory）—— 日记本

定义：情景记忆存储 Agent 与用户交互的事件序列，带有时间戳和上下文标签。就像你的日记本，记录了"2024年3月15日，用户投诉了配送延迟，我帮他申请了退款"这样的完整事件。

核心特征：

  • 事件序列：以 (时间, 事件, 结果) 三元组形式存储
  • 时间索引：支持按时间范围检索，如"上周发生的所有事件"
  • 上下文标记：每个事件附带情绪标签、任务类型、用户满意度等元数据

📄 语义记忆（Semantic Memory）—— 百科全书

定义：语义记忆存储 Agent 从交互中提取的知识和事实，以结构化的实体-关系形式组织。就像一本不断更新的百科全书，记录着"用户张三是一名Java开发工程师"、"Spring AI 支持多种向量数据库"这样的结构化知识。

核心特征：

  • 知识图谱：以 (实体, 关系, 实体) 三元组形式组织
  • 事实提取：从对话中自动提取并归纳知识
  • 语义检索：支持基于含义的模糊匹配，而非精确关键词

📄 程序性记忆（Procedural Memory）—— 操作手册

定义：程序性记忆存储 Agent 学到的操作流程和技能模板。就像一个厨师不需要每次做菜都想"先放油、再放蒜、然后放菜"——这些步骤已经内化为本能。对于 Agent 来说，处理退款、生成报告、部署代码等流程一旦学会，就可以反复使用。

核心特征：

  • 技能模板：可复用的步骤序列，如"处理用户投诉"的标准流程
  • 操作序列：经过验证的行动链，带有条件分支和异常处理
  • 经验积累：从成功案例中提炼出的最佳实践
// 程序性记忆：可复用的技能模板 public record ProceduralMemory( String skillId, String skillName, List<Step> steps, Map<String, String> conditions, int successCount, int failureCount ) { // 计算技能可靠性评分 public double reliability() { int total = successCount + failureCount; return total == 0 ? 0.0 : (double) successCount / total; } public record Step( int order, String action, String toolRequired, String fallbackAction ) {} } // 示例：退款处理流程 // ProceduralMemory refundProcess = new ProceduralMemory( // "SKILL-001", "退款处理", // List.of( // new Step(1, "验证订单状态", "orderService", "手动查询"), // new Step(2, "计算退款金额", "paymentCalc", "默认全额"), // new Step(3, "执行退款", "paymentGateway", "转人工"), // new Step(4, "通知用户", "notificationService", "记录待发送") // ), // Map.of("前置条件", "用户已提供订单号"), // 47, 3 // );
💡 四种记忆的关系

四种记忆并非孤立存在。工作记忆是"当前正在使用的"，情景记忆是"过去发生了什么"，语义记忆是"我知道什么"，程序性记忆是"我会做什么"。它们互相协作：工作记忆从情景记忆中检索相关信息，语义记忆为情景提供知识背景，程序性记忆指导行动决策。

📄 核心内容（第2节）：记忆架构模式

📄 MemoryStream 模式 —— 流式记忆

比喻：就像一条河，信息源源不断地流过。你不需要存储每一滴水，只需要记住最近的水流状态。

MemoryStream 是最简单的记忆模式，本质上就是滑动窗口（Sliding Window）。它只保留最近 N 轮对话，超出窗口的旧消息自动丢弃。这是 Spring AI 默认的 ChatMemory 行为。

MemoryStream 模式

  时间线 ───────────────────────────────────────────>

  消息流: [M1] [M2] [M3] [M4] [M5] [M6] [M7] [M8]
                                       |           |
                              窗口起点  |           | 窗口终点
                                       v           v
                              ┌───────────────────────┐
  LLM 实际看到的上下文:        │ [M4] [M5] [M6] [M7] [M8] │
                              └───────────────────────┘
                              M1~M3 → 已丢弃（不可恢复）
// MemoryStream：Spring AI 内置的滑动窗口记忆 @Bean public ChatMemory chatMemory() { // 基础用法：只保留最近 20 条消息 return MessageWindowChatMemory.builder() .maxMessages(20) .build(); } // 使用时 chatClient.prompt() .advisors(a -> a .advisors(new MessageChatMemoryAdvisor(chatMemory)) .param(ChatMemory.CONVERSATION_ID, "user-123")) .user("今天天气怎么样？") .call() .content();
⚡ 适用场景

简单对话、短期会话、对上下文要求不高的场景。优点是实现简单、内存占用可控。缺点是丢失历史上下文，无法处理跨会话的长期关系。

📄 MemoryStore 模式 —— 存储式记忆

比喻：就像一个笔记本。你不会遗忘，但需要的时候必须翻到正确的页码才能找到想要的信息。

MemoryStore 将所有对话历史持久化存储到数据库或文件中，检索时通过语义搜索或关键词匹配找到最相关的记忆片段。

// MemoryStore：基于向量数据库的持久化记忆 @Service public class VectorMemoryStore { private final VectorStore vectorStore; private final EmbeddingModel embeddingModel; // 存储一条记忆 public void store(String conversationId, String content, Map<String, Object> metadata) { Document doc = Document.builder() .id(UUID.randomUUID().toString()) .text(content) .metadata(Metadata.from(metadata)) .build(); doc.getMetadata().put("conversationId", conversationId); doc.getMetadata().put("timestamp", Instant.now().toString()); vectorStore.add(List.of(doc)); } // 检索相关记忆 public List<Document> recall(String query, int topK) { SearchRequest request = SearchRequest.builder() .query(query) .topK(topK) .build(); return vectorStore.similaritySearch(request); } }

📄 MemoryGraph 模式 —— 图式记忆

比喻：就像维基百科。每个概念是一个页面，页面之间通过超链接互相连接。你可以从"张三"跳转到"XX科技"再跳转到"Spring Boot项目"，形成一条知识路径。

MemoryGraph 以知识图谱的形式组织记忆，存储实体之间的关系。它支持复杂的推理查询，如"张三的同事都用了什么技术栈"。

// ============================================ // MemoryGraph：基于知识图谱的记忆存储 // 支持关系推理、路径追踪、图遍历查询 // ============================================ // 记忆节点 public record MemoryNode( String id, String content, String type, // episodic, semantic, procedural LocalDateTime timestamp, double importance, Map<String, Object> metadata ) {} // 记忆边（关系） public record MemoryEdge( String sourceId, String targetId, String relation, // caused_by, related_to, part_of, works_at, likes double weight, Map<String, Object> metadata ) {} // 图式记忆核心类 public class MemoryGraph { private final Map<String, MemoryNode> nodes = new ConcurrentHashMap<>(); private final Map<String, List<MemoryEdge>> adjacency = new ConcurrentHashMap<>(); // ---- 写入操作 ---- /** 添加记忆节点 */ public void addNode(MemoryNode node) { nodes.put(node.id(), node); } /** 添加关系边（双向索引，支持双向遍历） */ public void addEdge(MemoryEdge edge) { // 校验端点是否存在 if (!nodes.containsKey(edge.sourceId()) || !nodes.containsKey(edge.targetId())) { throw new IllegalArgumentException( "边的端点必须是已存在的节点"); } adjacency .computeIfAbsent(edge.sourceId(), k -> new ArrayList<>()) .add(edge); // 反向索引，支持反向查询（如"被谁喜欢"） MemoryEdge reverse = new MemoryEdge( edge.targetId(), edge.sourceId(), edge.relation() + "_reverse", edge.weight(), edge.metadata()); adjacency .computeIfAbsent(edge.targetId(), k -> new ArrayList<>()) .add(reverse); } // ---- 读取操作 ---- /** 获取节点 */ public Optional<MemoryNode> getNode(String nodeId) { return Optional.ofNullable(nodes.get(nodeId)); } /** 获取节点的所有邻居边 */ public List<MemoryEdge> getEdges(String nodeId) { return adjacency.getOrDefault(nodeId, List.of()); } // ---- 图查询 ---- /** * 广度优先搜索：查找指定深度内的相关记忆 * @param startId 起始节点ID * @param depth 遍历深度（1 = 直接邻居，2 = 邻居的邻居） * @return 按距离排序的相关记忆节点 */ public List<MemoryNode> findRelated(String startId, int depth) { List<MemoryNode> result = new ArrayList<>(); Set<String> visited = new HashSet<>(); Queue<Map.Entry<String, int>> queue = new LinkedList<>(); visited.add(startId); queue.add(Map.entry(startId, 0)); while (!queue.isEmpty()) { var current = queue.poll(); String nodeId = current.getKey(); int currentDepth = current.getValue(); if (currentDepth > 0) { nodes.get(nodeId).ifPresent(result::add); } if (currentDepth < depth) { for (MemoryEdge edge : getEdges(nodeId)) { if (!visited.contains(edge.targetId())) { visited.add(edge.targetId()); queue.add(Map.entry( edge.targetId(), currentDepth + 1)); } } } } return result; } /** * 路径追踪：找到两个节点之间的最短路径 * 使用 BFS 实现，返回路径上的所有节点 */ public List<MemoryNode> tracePath(String fromId, String toId) { Map<String, String> parentMap = new HashMap<>(); Set<String> visited = new HashSet<>(); Queue<String> queue = new LinkedList<>(); visited.add(fromId); queue.add(fromId); while (!queue.isEmpty()) { String current = queue.poll(); if (current.equals(toId)) { return rebuildPath(parentMap, fromId, toId); } for (MemoryEdge edge : getEdges(current)) { if (!visited.contains(edge.targetId())) { visited.add(edge.targetId()); parentMap.put(edge.targetId(), current); queue.add(edge.targetId()); } } } return List.of(); // 无可达路径 } private List<MemoryNode> rebuildPath( Map<String, String> parentMap, String fromId, String toId) { LinkedList<MemoryNode> path = new LinkedList<>(); String current = toId; while (current != null) { path.addFirst(nodes.get(current)); current = parentMap.get(current); } return path; } /** * 按关系类型查找：找出所有通过特定关系连接的节点 * 例：找出所有 "works_at" XX科技 的人 */ public List<MemoryNode> findByRelation( String sourceId, String relationType) { return getEdges(sourceId).stream() .filter(e -> e.relation().equals(relationType)) .map(e -> nodes.get(e.targetId())) .filter(Optional::isPresent) .map(Optional::get) .toList(); } } // ---- Spring AI 集成示例 ---- @Service public class GraphMemoryService { private final MemoryGraph memoryGraph = new MemoryGraph(); private final ChatClient chatClient; /** * 从对话中自动提取实体和关系，构建图记忆 */ public void extractAndStore(String conversation) { GraphExtract extract = chatClient.prompt() .system(""" 从对话中提取实体和关系，返回 JSON。 格式：{"nodes": [{"id","content","type"}], "edges": [{"sourceId","targetId","relation","weight"}]} """) .user(conversation) .call() .entity(GraphExtract.class); for (var n : extract.nodes()) { memoryGraph.addNode(new MemoryNode( n.id(), n.content(), n.type(), LocalDateTime.now(), 0.5, Map.of())); } for (var e : extract.edges()) { memoryGraph.addEdge(new MemoryEdge( e.sourceId(), e.targetId(), e.relation(), e.weight(), Map.of())); } } /** * 结合图查询和 LLM 的推理式回答 * 例："张三的同事都用什么技术栈？" */ public String answerWithGraph(String question) { // 1. 从问题中提取实体 String entityId = extractEntityId(question); // 2. 图查询获取相关记忆（深度2） List<MemoryNode> related = memoryGraph .findRelated(entityId, 2); // 3. 将图谱上下文注入 Prompt String graphContext = related.stream() .map(n -> n.content()) .collect(Collectors.joining("\n")); return chatClient.prompt() .system("基于以下知识图谱信息回答：\n" + graphContext) .user(question) .call() .content(); } }

📄 核心内容（第3节）：记忆管理策略

📄 Importance Scoring（重要性评分）

比喻：你不会记住今天午餐吃了什么，但你会清楚地记得昨天的面试表现。大脑会自动给记忆打"重要性分数"——越重要的事情，越不容易被遗忘。

在 Agent 系统中，每条记忆在存入时会被赋予一个重要性分数。分数由以下因素决定：

重要性评分模型

  重要性分数 = w1 * 情感强度 + w2 * 事实密度 + w3 * 用户关注度 + w4 * 任务关联度

  ┌─────────────────────────────────────────────────┐
  │  消息内容                                        │
  │  "我的订单延迟了一周，我非常生气！"               │
  ├─────────────────────────────────────────────────┤
  │  情感强度:  0.9  (用户明确表达负面情绪)            │
  │  事实密度:  0.7  (包含订单、延迟时间等关键事实)    │
  │  用户关注度: 0.8  (用户主动提及，非被动回答)       │
  │  任务关联度: 0.9  (直接关系到客服核心任务)         │
  ├─────────────────────────────────────────────────┤
  │  最终分数: 0.83 (高重要性，应长期保留)            │
  └─────────────────────────────────────────────────┘

  对比:
  "好的，谢谢" → 0.1 (低重要性，可以快速遗忘)
// 重要性评分实现 @Service public class ImportanceScorer { private final ChatClient chatClient; /** * 使用 LLM 评估消息的重要性 (0.0 ~ 1.0) * 适用于需要高质量评分但调用成本较高的场景 */ public double scoreWithLLM(String message) { String response = chatClient.prompt() .system(""" 评估以下消息的重要性，返回 0.0-1.0 的分数。 考虑因素：情感强度、信息密度、用户关注度。 只返回数字。""") .user(message) .call() .content(); return Double.parseDouble(response.trim()); } /** * 基于规则的快速评分 (不调用 LLM，性能更好) * 适用于需要快速处理大量消息的场景 */ public double scoreWithRules(String message) { double score = 0.3; // 基础分 // 包含情感词加分 if (message.matches(".*[生气|感谢|满意|投诉|着急].*")) score += 0.2; // 包含实体信息（订单号、金额等）加分 if (message.matches(".*[A-Z]{2}-\\d{4,}|\\d+元|订单].*")) score += 0.2; // 消息长度适中加分（太短通常不重要） if (message.length() > 20) score += 0.1; // 包含行动请求加分 if (message.matches(".*[帮我|请|能不能|需要].*")) score += 0.2; return Math.min(score, 1.0); } }

📄 Recency Weighting（近因加权）

比喻：昨天发生的事你记得最清楚，上周的事有点模糊，上个月的事只能想起大概。近因加权就是让最近的记忆在检索时获得更高的权重。

近因衰减曲线 (Recency Decay)

  权重
  1.0 ┤ ████
      │ ████ ███
  0.8 ┤ ████ ███
      │ ████ ███ ██
  0.6 ┤ ████ ███ ██
      │ ████ ███ ██ █
  0.4 ┤ ████ ███ ██ █ █
      │ ████ ███ ██ █ █ █
  0.2 ┤ ████ ███ ██ █ █ █ █ █
      │ ████ ███ ██ █ █ █ █ █ █ █
  0.0 ┤──────────────────────────────
      1h  2h  3h  6h  12h 24h 48h 7d

  衰减公式: weight = e^(-λ * timeDelta)
  λ = 0.1 (衰减速率，可调)
// 近因加权计算 public class RecencyWeighter { private final double decayRate; // λ: 衰减速率 public RecencyWeighter(double decayRate) { this.decayRate = decayRate; } /** * 计算时间衰减权重 * @param timestamp 记忆创建时间 * @return 0.0 ~ 1.0 的权重值 */ public double weight(Instant timestamp) { double hoursAgo = Duration.between( timestamp, Instant.now()).toHours(); return Math.exp(-decayRate * hoursAgo); } // 对一批记忆按近因加权排序 public List<ScoredMemory> rankByRecency( List<MemoryEntry> memories) { return memories.stream() .map(m -> new ScoredMemory(m, weight(m.timestamp()))) .sorted(Comparator .comparingDouble(ScoredMemory::score).reversed()) .toList(); } }

📄 Relevance Retrieval（相关性检索）

比喻：你不会翻遍整本日记去找某件事。你会先回忆"大概是关于什么话题的"，然后直接翻到相关的页码。相关性检索就是让 Agent 只检索与当前问题最相关的记忆。

// 综合检索：重要性 * 近因 * 相关性 @Service public class HybridMemoryRetriever { private final VectorStore vectorStore; private final RecencyWeighter recencyWeighter; /** * 综合检索：三因子加权 * * finalScore = α * 语义相关性 * + β * 重要性分数 * + γ * 近因权重 */ public List<ScoredMemory> retrieve( String query, int topK, double alpha, double beta, double gamma) { // 1. 语义检索候选集（取较多候选） List<Document> candidates = vectorStore .similaritySearch( SearchRequest.builder() .query(query) .topK(topK * 3) // 多取一些，后面重排 .build()); // 2. 三因子加权重排 return candidates.stream() .map(doc -> { double relevance = doc.getScore(); double importance = doc.getMetadata() .get("importance", double.class, 0.5); double recency = recencyWeighter.weight( Instant.parse( doc.getMetadata().get("timestamp", String.class))); double finalScore = alpha * relevance + beta * importance + gamma * recency; return new ScoredMemory( doc, finalScore, relevance, importance, recency); }) .sorted(Comparator .comparingDouble(ScoredMemory::finalScore).reversed()) .limit(topK) .toList(); } }

📄 记忆 Consolidation（记忆固化）

比喻：你在课堂上做的笔记（工作记忆），经过整理和复习后变成你长期掌握的知识（长期记忆）。记忆固化就是将临时性的工作记忆提炼、压缩并转入持久化存储的过程。

记忆固化流程

  ┌──────────┐      ┌──────────┐      ┌──────────┐
  │ 工作记忆  │ ───> │   固化    │ ───> │ 长期记忆  │
  │ (最近消息) │      │ (LLM提炼)│      │ (向量DB)  │
  └──────────┘      └─────┬────┘      └──────────┘
                          │
                          v
                    ┌──────────────┐
                    │ LLM 总结提炼 │
                    │              │
                    │ 原始: 30轮   │
                    │ 对话记录      │
                    │       v      │
                    │ 精华: 3条    │
                    │ 关键记忆      │
                    └──────────────┘
// 记忆固化：将对话历史提炼为关键记忆 @Service public class MemoryConsolidator { private final ChatClient chatClient; private final VectorMemoryStore memoryStore; /** * 将一段对话历史固化为关键记忆 */ public void consolidate(String conversationId, List<Message> messages) { String conversation = messages.stream() .map(m -> m.getRole() + ": " + m.getText()) .collect(Collectors.joining("\n")); // 用 LLM 从对话中提取关键记忆 MemoryExtract extract = chatClient.prompt() .system(""" 从以下对话中提取关键记忆，返回 JSON 数组。 每条记忆包含： - content: 记忆内容（简洁的事实或偏好） - type: 类型 (fact/preference/emotion/event) - importance: 重要性 0.0-1.0 只提取值得长期记住的信息，忽略寒暄。""") .user(conversation) .call() .entity(MemoryExtract.class); // 持久化到向量数据库 for (MemoryItem item : extract.memories()) { memoryStore.store(conversationId, item.content(), Map.of( "type", item.type(), "importance", item.importance(), "consolidatedAt", Instant.now().toString() )); } } }

📄 遗忘曲线在 Agent 中的应用

比喻：艾宾浩斯遗忘曲线告诉我们，记忆的遗忘速度是先快后慢的。刚学的东西如果不复习，一天后就忘掉60%。但对于 Agent 来说，"遗忘"反而是一种优势——自动清理不再重要的记忆，可以节省存储空间并提升检索效率。

Agent 记忆的遗忘策略

  策略              说明                         触发条件
  ─────────────    ──────────────────────        ──────────────
  时间衰减遗忘    超过阈值且低重要性的记忆       每日定时任务
                  自动降级或删除

  容量淘汰遗忘    存储满时，优先删除低分记忆      存储使用率 > 80%

  主动遗忘        用户要求删除特定记忆           用户隐私请求

  语义合并遗忘    相似记忆合并为更精炼的版本     定期优化任务
// 遗忘策略实现 @Service public class MemoryForgetting { private final VectorStore vectorStore; /** * 策略1: 时间衰减遗忘 * 重要性低于阈值且超过保留期的记忆自动删除 */ public void forgetByDecay(int retainDays, double minImportance) { Instant cutoff = Instant.now() .minus(Duration.ofDays(retainDays)); List<Document> oldMemories = vectorStore .similaritySearch( SearchRequest.builder() .query("*") // 全量检索 .build()); oldMemories.stream() .filter(doc -> Instant.parse( doc.getMetadata().get("timestamp", String.class)) .isBefore(cutoff)) .filter(doc -> doc.getMetadata() .get("importance", double.class, 0.0) < minImportance) .forEach(doc -> vectorStore.delete( List.of(doc.getId()))); } /** * 策略2: 用户主动遗忘（GDPR 合规） */ public void forgetByUser(String conversationId) { vectorStore.delete( SearchRequest.builder() .query("*") .build(), Metadata.from( Map.of("conversationId", conversationId))); } }
⚠️ 隐私边界设计

Agent 记忆系统必须遵守数据隐私法规。设计时需要考虑：(1) 用户有权要求删除所有个人记忆；(2) 敏感信息（密码、身份证号等）不应被存储；(3) 记忆存储应有明确的保留期限；(4) 跨用户记忆严格隔离，绝不能让 Agent A 的记忆影响 Agent B 的回答。

📄 ChatMemory 增强实现

以下是一个完整的增强型记忆管理器，整合了工作记忆、情景记忆和记忆固化：

// ============================================ // Spring AI 增强记忆管理器 // 整合滑动窗口 + 向量检索 + 记忆固化 // ============================================ @Service public class EnhancedMemoryManager { private final ChatMemory workingMemory; private final VectorStore longTermMemory; private final ChatClient summarizer; public EnhancedMemoryManager( ChatMemory workingMemory, VectorStore longTermMemory, ChatClient summarizer) { this.workingMemory = workingMemory; this.longTermMemory = longTermMemory; this.summarizer = summarizer; } /** * 构建增强型 Prompt * 1. 检索相关长期记忆 * 2. 加入工作记忆窗口 * 3. 组合成完整上下文 */ public Prompt buildPrompt(String conversationId, String userMessage) { // 1. 从长期记忆中检索相关信息 List<Document> relevantMemories = longTermMemory .similaritySearch( SearchRequest.builder() .query(userMessage) .topK(5) .build()); // 2. 构建记忆上下文 String memoryContext = relevantMemories.stream() .map(Document::getText) .collect(Collectors.joining("\n")); // 3. 组合成完整 Prompt String systemPrompt = """ 你是一个有记忆的智能助手。 === 你记住的关于用户的信息 === %s ================================= 请基于以上记忆和当前对话来回答问题。 如果记忆中有相关信息，请自然地引用。 """.formatted(memoryContext); return Prompt.builder() .system(systemPrompt) .user(userMessage) .build(); } /** * 对话结束后，固化重要记忆 */ public void onConversationEnd(String conversationId) { List<Message> history = workingMemory .get(conversationId); // 用 LLM 提取关键信息 String summary = summarizer.prompt() .system("从对话中提取用户的关键偏好和事实信息。") .user(history.toString()) .call() .content(); // 存入长期记忆 Document doc = Document.builder() .id(UUID.randomUUID().toString()) .text(summary) .metadata(Metadata.from(Map.of( "conversationId", conversationId, "type", "consolidated", "timestamp", Instant.now().toString() ))) .build(); longTermMemory.add(List.of(doc)); } }

💡 生产环境建议

生产环境中，工作内存用 Spring AI 内置的 MessageWindowChatMemory 即可，长期记忆建议使用 PgVector 或 Milvus 等生产级向量数据库。记忆固化建议在异步任务中执行，避免阻塞用户请求。重要性评分可以先用规则引擎快速评分，再用 LLM 异步校准。

📖 章节小结

  • 四种记忆类型 — 工作记忆（当前上下文窗口）、情景记忆（事件时间线）、语义记忆（知识图谱）、程序性记忆（操作技能模板），各有分工、协同工作
  • 三种架构模式 — MemoryStream（滑动窗口，简单高效）、MemoryStore（向量持久化，语义检索）、MemoryGraph（知识图谱，关系推理），生产系统通常采用混合架构
  • 四项管理策略 — 重要性评分（决定哪些值得记住）、近因加权（最近的记忆更重要）、相关性检索（只检索相关记忆）、记忆固化（将临时记忆转为长期知识）
  • 遗忘与隐私 — 遗忘不是缺陷而是特性，通过时间衰减、容量淘汰、主动删除等策略保持记忆系统高效运行，同时遵守隐私法规
  • Spring AI 实践 — ChatMemory 提供工作记忆，VectorStore 提供长期记忆，ChatMemoryAdvisor 自动将记忆注入 Prompt

🎓 面试高频题

👨‍💻 Q1: Agent 的四种记忆类型分别是什么？它们之间有什么区别和联系？

参考答案：四种记忆类型借鉴了认知心理学的分类：
工作记忆：当前对话的上下文窗口，容量有限（滑动窗口），用于即时推理。
情景记忆：过去交互的事件序列，带时间戳和上下文标签，支持按事件检索。
语义记忆：从交互中提取的知识和事实，以知识图谱或向量形式存储，支持语义检索。
程序性记忆：操作流程和技能模板，记录验证过的行动序列，可复用。

联系：工作记忆从情景和语义记忆中检索相关信息；语义记忆为情景提供知识背景；程序性记忆指导行动决策。在实际系统中，四种记忆通过记忆检索层统一管理。

👨‍💻 Q2: 如何设计一个高效的 Agent 记忆检索系统？需要考虑哪些因素？

参考答案：高效的记忆检索需要考虑三个核心因子：
(1) 语义相关性：使用 Embedding 模型计算查询与记忆的语义相似度。
(2) 重要性评分：每条记忆存入时评估其重要性（情感强度、事实密度、用户关注度），重要记忆权重更高。
(3) 时间衰减：最近的记忆权重更高，使用指数衰减函数 e^(-λt) 计算。

最终检索公式：score = α * 相关性 + β * 重要性 + γ * 近因权重。其中 α、β、γ 是可调参数，根据业务场景调整比例。
此外还需考虑：检索延迟（向量数据库索引优化）、容量上限（定期遗忘）、跨会话隔离（用户数据安全）。

👨‍💻 Q3: Spring AI 中如何实现有状态的对话记忆？ChatMemoryAdvisor 的工作原理是什么？

参考答案：Spring AI 通过 ChatMemory 接口 + ChatMemoryAdvisor 实现有状态对话。
(1) MessageWindowChatMemory 是默认实现，维护一个滑动窗口，只保留最近 N 条消息。
(2) MessageChatMemoryAdvisor 是一个 CallAdvisor，在每次调用 LLM 前自动注入历史消息到 Prompt 中。
(3) 使用时通过 .param(ChatMemory.CONVERSATION_ID, id) 指定会话 ID，不同会话的记忆自动隔离。

工作原理：请求进入 → Advisor 从 ChatMemory 中取出该 conversationId 的历史消息 → 将历史消息拼接到 system 或 user message 中 → 发送给 LLM → 响应后将新的 user+assistant 消息存回 ChatMemory。

生产环境可以自定义 ChatMemory 实现，将记忆持久化到 Redis/数据库，实现跨实例的记忆共享。

👨‍💻 Q4: 记忆固化（Consolidation）和向量检索有什么区别？为什么需要两者结合？

参考答案：
向量检索：原始对话直接存入向量数据库，检索时通过语义相似度找到相关片段。优点是保留了原始信息，缺点是噪声多、存储冗余。
记忆固化：用 LLM 从对话中提炼关键信息（事实、偏好、情感），只存储精华。优点是噪声低、存储效率高，缺点是可能丢失细节。

两者结合的原因：固化后的记忆作为"核心知识"长期保存；原始对话作为"事件记录"在短期保留。检索时先查固化记忆（快、准），再查原始对话（全、细）。这就像你既有百科全书（固化知识），也有会议录音（原始记录），根据需要选择查阅。

👨‍💻 Q5: 如何处理 Agent 记忆中的隐私问题？GDPR 等法规对记忆系统有什么要求？

参考答案：Agent 记忆系统必须满足以下隐私要求：
(1) 遗忘权：实现"被遗忘权"（Right to be Forgotten），用户可以要求删除所有个人记忆。需要提供按 conversationId 或 userId 批量删除的接口。
(2) 数据最小化：只存储必要的信息，不存储敏感数据（密码、身份证号等），可通过正则过滤实现。
(3) 保留期限：为记忆设置 TTL，超期自动清理。
(4) 跨用户隔离：严格按用户隔离记忆，检索时加上 user_id 过滤条件。
(5) 审计日志：记录记忆的存取日志，便于合规审查。
(6) 本地化存储：某些法规要求数据不出境，需要选择合适的数据中心区域。

✍️ 练习 1：构建带记忆的客服 Agent

实现一个客服 Agent，具备以下记忆能力：

  • 使用 Spring AI 的 MessageWindowChatMemory 作为工作记忆，窗口大小设为 15
  • 将每次对话结束后的重要信息固化到 VectorStore（使用 SimpleVectorStore 即可）
  • 下次对话开始时，自动检索与当前问题相关的 3 条历史记忆，注入到 system prompt 中
  • 实现一个简单的遗忘策略：超过 30 天且重要性低于 0.3 的记忆自动清理
✍️ 练习 2：实现混合记忆检索

基于本章的三因子检索模型，实现一个 HybridMemoryRetriever：

  • 从向量数据库中取 15 个候选记忆
  • 使用三因子公式（相关性 * 0.5 + 重要性 * 0.3 + 近因 * 0.2）进行重排
  • 返回 Top-5 的最终结果
  • 编写单元测试验证：当调高重要性权重时，低重要性的近期记忆排名上升；当调高近因权重时，最近的记忆排名上升
✍️ 练习 3：记忆隐私过滤器

实现一个记忆写入前的隐私过滤器 MemoryPrivacyFilter：

  • 使用正则表达式过滤身份证号、手机号、银行卡号等敏感信息
  • 将敏感信息替换为占位符（如 [手机号已脱敏]）后再存储
  • 编写测试用例，验证各种敏感信息格式（11位手机号、18位身份证号等）均被正确过滤
  • 考虑性能：过滤器应在微秒级完成，不影响记忆写入延迟
