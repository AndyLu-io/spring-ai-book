    ☕ Java 17+ 新特性

    为什么需要 Java 17？

    Spring Boot 3.x 要求 Java 17+。这不是微软的"强制升级"，而是因为 Java 17 带来了一些革命性的特性，让代码量减少 30-50% 的同时更安全、更易读。

    把它想象成从"功能机"升级到"智能手机" — 不只是多了几个按钮，而是整个交互方式都变了。

    📄 Record — 不可变数据类的"终极形态"

    在以前，定义一个 DTO 需要写 getter、equals、hashCode、toString，一个简单的类就要 50+ 行。Record 把这些全部自动化了。

    生活比喻

    Record 就像一张快递面单 — 收件人、地址、电话号码，一旦打印出来就不会变，你也不需要问"这个面单的收件人是谁"，信息就直接印在上面。

    // 以前：50 行的 POJO public class ChatMessage { private final String role; private final String content; private final Instant timestamp; // 构造器、getter、equals、hashCode、toString... 省略 40 行 } // 现在：1 行搞定 public record ChatMessage( String role, String content, Instant timestamp ) {} // 自动拥有： // - 全参构造器 // - role() / content() / timestamp() 访问器 // - equals() / hashCode() / toString() // 使用示例 var msg = new ChatMessage("user", "你好", Instant.now()); System.out.println(msg.role()); // user System.out.println(msg.content()); // 你好
    💡 小窍门
    Record 默认是不可变的。如果需要修改某个字段，用 withXxx() 模式创建新实例：
    // Record 没有 setter，需要用 compact constructor 做校验 public record ChatMessage(String role, String content, Instant timestamp) { // 紧凑构造器：在赋值前校验 public ChatMessage { if (role == null || role.isBlank()) { throw new IllegalArgumentException("role 不能为空"); } } }

    🔐 Sealed Classes — 给继承加上"围栏"

    以前的类继承是"开放"的，任何人都可以 extends 你的类。Sealed Classes 让你精确控制谁能继承你。

    生活比喻

    就像一个私人俱乐部 — 你只允许特定的会员进入，其他人想加入？对不起，门关着。

    // 定义一个密封接口：只有 TextResponse 和 ErrorResponse 可以实现它 public sealed interface AIResponse permits TextResponse, ErrorResponse, ToolCallResponse {} public record TextResponse(String text, TokenUsage usage) implements AIResponse {} public record ErrorResponse(String code, String message) implements AIResponse {} public record ToolCallResponse(String name, String result) implements AIResponse {} // 配合 Pattern Matching，编译器会帮你检查所有情况 public String describe(AIResponse response) { return switch (response) { case TextResponse t -> "文本: " + t.text(); case ErrorResponse e -> "错误: " + e.message(); case ToolCallResponse tc -> "工具调用: " + tc.name(); // 不需要 default！编译器知道你已经覆盖了所有情况 }; }
    💡 你知道吗？
    在 Spring AI 的源码中，大量使用了 Sealed Classes 来定义 Response 的类型层次。当你看到 permits 关键字时，就知道这个类型体系是经过精心设计的。

    🔎 Pattern Matching — 告别 instanceof + 强转

    // 以前：又臭又长 if (response instanceof TextResponse) { TextResponse textResp = (TextResponse) response; // 手动强转 process(textResp.text()); } // 现在：一行搞定 if (response instanceof TextResponse text) { process(text.text()); // 直接使用，自动解构 } // 配合 null 检查更好用 if (response instanceof TextResponse(String content, _)) { // 直接解构 Record 的字段！content 就是 text 字段的值 System.out.println("内容: " + content); }

    📄 Text Blocks & Switch 表达式

    // Text Blocks — 再也不用拼接多行字符串了 String systemPrompt = """ 你是一位资深的 Java 架构师。 请根据用户的问题，给出： 1. 架构方案 2. 技术选型 3. 注意事项 请用 Markdown 格式输出。 """; // Switch 表达式 — 简洁且不会漏掉分支 String model = switch (complexity) { case SIMPLE -> "gpt-4o-mini"; // 简单任务用便宜模型 case MEDIUM -> "gpt-4o"; // 中等任务 case COMPLEX -> "claude-sonnet"; // 复杂任务用最强模型 };

    🌻 Spring Boot 核心原理

    Spring Boot 的"魔法"：自动配置

    Spring Boot 最让人惊叹的特性就是"约定优于配置"。你只要加一个 @SpringBootApplication，它就自动帮你配好了 Tomcat、JSON 解析、日志系统...

    生活比喻

    自动配置就像智能酒店。你走进房间，灯已经开着、空调温度刚好、WiFi 自动连接 — 不是有人跟着你，而是酒店预设了默认配置，大多数客人不需要改动就能直接入住。如果你需要调温，也可以手动覆盖。

    依赖注入的"快递员模式"

    依赖注入（DI）的核心思想：你不需要自己去找依赖，有人会送上门。

    // Spring 就像一个"快递员"：它知道 ChatClient 需要什么，自动送过来 @Service public class ChatService { private final ChatClient chatClient; private final ChatMemory chatMemory; // 构造器注入 — 最推荐的方式 // Spring 看到这个构造器，会自动把 ChatClient 和 ChatMemory 实例"快递"过来 public ChatService(ChatClient chatClient, ChatMemory chatMemory) { this.chatClient = chatClient; this.chatMemory = chatMemory; } public String chat(String sessionId, String message) { return chatClient.prompt() .system("你是一个友好的助手") .user(message) .advisors(a -> a.param(ChatMemory.CONVERSATION_ID, sessionId)) .call() .content(); } }
    💡 为什么推荐构造器注入？

    1. 字段可以声明为 final，保证不可变
    2. 依赖一目了然（看构造器就知道需要什么）
    3. 单元测试时可以直接 new，不依赖 Spring 容器

    🔌 REST API 设计

    设计一个 AI 对话 API

    让我们设计一个完整的 AI 对话接口，涵盖请求、响应、错误处理：

    // ============ 请求/响应 DTO ============ // 请求体 public record ChatRequest( @NotBlank String message, @Nullable String model, @Nullable Double temperature, @Nullable List<String> history ) {} // 响应体 public record ChatResponse( String id, String content, String model, TokenUsage usage, Instant timestamp ) {} // Token 使用统计 public record TokenUsage( int promptTokens, int completionTokens, int totalTokens ) {} // 错误响应 public record ErrorResponse( String code, String message, Instant timestamp ) {}

    Controller 完整实现

    @RestController @RequestMapping("/api/v1/chat") @Validated public class ChatController { private final ChatService chatService; public ChatController(ChatService chatService) { this.chatService = chatService; } /** * POST /api/v1/chat * 发送消息并获取 AI 回复 */ @PostMapping public ResponseEntity<ChatResponse> chat( @Valid @RequestBody ChatRequest request) { var response = chatService.chat(request); return ResponseEntity.ok(response); } /** * GET /api/v1/chat/stream * 流式输出 — SSE（Server-Sent Events） */ @GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE) public Flux<String> stream(@RequestParam String message) { return chatService.stream(message); } }

    全局异常处理

    @RestControllerAdvice public class GlobalExceptionHandler { @ExceptionHandler(ValidationException.class) public ResponseEntity<ErrorResponse> handleValidation(ValidationException e) { return ResponseEntity.badRequest() .body(new ErrorResponse("INVALID_INPUT", e.getMessage(), Instant.now())); } @ExceptionHandler(AIException.class) public ResponseEntity<ErrorResponse> handleAIError(AIException e) { return ResponseEntity.status(HttpStatus.BAD_GATEWAY) .body(new ErrorResponse("AI_SERVICE_ERROR", e.getMessage(), Instant.now())); } }

    🧠 AI 与大模型核心概念

    大语言模型（LLM）是什么？

    LLM 可以理解为一个"压缩了人类互联网知识的超级学霸"。它通过阅读海量文本，学会了语言的规律，能理解和生成自然语言。

    你输入的问题  ──────────────────────>  LLM 大模型
                                       |
                                       |  "嗯...这个问题，
                                       |   让我想想训练时
                                       ||
AI 的回答  <──────────────────────   v

    核心能力：

    • 文本理解 — 理解你说了什么（语义分析、意图识别）
    • 文本生成 — 给出回答（这是最核心的能力）
    • 推理能力 — 简单的逻辑推理（但不是真的"思考"）
    • 代码生成 — 理解并生成代码（Spring AI 就是用这个能力）

    核心参数详解

    参数作用生活比喻推荐值
    Temperature 控制输出随机性 "创造力旋钮"：0 = 考试模式（精确），1 = 灵魂画手模式（随机） 0 ~ 0.7
    Top-P 核采样范围 "选项范围"：0.1 = 只考虑最优选项，1.0 = 所有选项都有可能 0.9
    Max Tokens 最大输出长度 "答题纸大小"：限制 AI 回答的长度 1024 ~ 4096
    Frequency Penalty 降低重复用词 "换个说法"：让 AI 不要老用同一个词 0 ~ 0.5
    // Spring AI 中设置参数 chatClient.prompt() .user("写一首关于编程的诗") .options(ChatOptions.builder() .model("gpt-4o") .temperature(0.8) // 创意写作，给高一点 .maxTokens(500) .build()) .call() .content();

    Token — AI 的"语言货币"

    LLM 按 Token 计费。Token 是模型处理文本的最小单位，大约：

    文本类型Token 换算示例
    英文1 word ≈ 1-2 tokens"Hello world" ≈ 2 tokens
    中文1 字 ≈ 1-2 tokens"你好世界" ≈ 4 tokens
    代码1 行 ≈ 3-5 tokens一个方法约 20-50 tokens
    ⚠️ 踩坑警告

    输入和输出的 Token 都计费！GPT-4o 的输入价格约 $2.5/M tokens，输出约 $10/M tokens。输出比输入贵 4 倍，所以让 AI 简洁回答也是省钱之道。

    Prompt — 和 AI 沟通的"魔法咒语"

    Prompt 就是你发给 AI 的指令。好的 Prompt 就像好的问题 — 越清晰，答案越好。

    // 差的 Prompt "帮我写个东西" // AI: 写什么？给谁看？什么风格？ // 好的 Prompt """ 你是一位资深的技术博主。 请为初学者写一篇关于 Spring AI 入门的博客文章。 要求： - 标题吸引人 - 用通俗易懂的比喻解释概念 - 包含完整的代码示例 - 字数约 1500 字 - 使用 Markdown 格式 """
    💡 你知道吗？
    Prompt Engineering（提示词工程）已经发展成一个专门的领域。好的 Prompt 可以让同一个模型的表现提升 50% 以上。这也是 Spring AI 的核心价值之一 — 让你用 Java 代码优雅地构建和管理 Prompt。

    🎓 面试高频题

    👨‍💻 Q1: Java Record 和 Class 有什么区别？

    Record 是不可变的（没有 setter），自动生成构造器、访问器、equals/hashCode/toString。Class 是可变的，需要手动编写这些方法。Record 适合做 DTO/值对象，Class 适合需要状态变化的场景。

    👨‍💻 Q2: Spring Boot 自动配置的原理是什么？

    通过 @EnableAutoConfiguration 读取 META-INF/spring.factories（Spring Boot 3.x 改为 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports）中的配置类列表，再根据 @Conditional 注解判断是否加载。

    👨‍💻 Q3: Temperature 参数如何影响模型输出？

    Temperature 控制采样时的概率分布平滑度。值越低（趋近 0），模型越倾向于选择概率最高的 token，输出越确定；值越高，概率分布越平坦，输出越随机多样。数学上，它在 softmax 中作为分母的缩放因子。
