第 28 章

Agent 成本优化与性能调优

掌握 Agent 系统的成本优化策略，理解 Token 经济学与缓存策略，设计和实现端到端的性能调优方案

预计 1-2 周 6 个知识点 中级难度

📚 开篇故事：精打细算的 AI 管家

想象你经营一家快递公司，每天有成千上万个包裹要处理。有的包裹只需要送到隔壁街，有的要跨越整个城市。如果你对每个包裹都派出一辆大卡车、一个高级快递员，成本会高得离谱。聪明的做法是：近的包裹用自行车送，普通的用小货车，只有大件才派大卡车。而且，如果发现有5个人都要往同一个地址送东西，不如合并一车一起送。

Agent 系统面临的挑战与此惊人地相似。每次调用大模型 API 都像是派出一辆大卡车 -- 费用按 Token 计算，一个简单的"你好"可能要花几毛钱，而一个复杂的推理任务可能要花几十块。如果你的 Agent 处理1000万次请求，其中80%都是简单的问答，却每次都调用 GPT-4 级别的模型，那么每月的账单会让你心痛不已。

好消息是，就像快递公司可以优化路线、合并包裹、分级配送一样，Agent 系统也有一整套成本优化策略：用小模型处理简单任务，用缓存避免重复计算，用批量处理摊薄固定成本，用精简的 Prompt 减少每次对话的 Token 消耗。本章将带你掌握这些"省钱秘籍"，让你的 Agent 系统既聪明又经济。

🎯 章节概述与学习目标

学习目标

在本章结束后，你将能够：

  • 理解成本结构 -- 掌握 Token 计费模型，能准确计算 Agent 系统的运行成本
  • 设计模型路由 -- 能构建智能路由器，根据任务复杂度自动选择合适的模型
  • 实现语义缓存 -- 掌握基于向量相似度的语义缓存，避免重复的 LLM 调用
  • 优化 Token 消耗 -- 能实施 Prompt 精简、输出压缩、流式输出等 Token 优化技术
  • 实施并发控制 -- 掌握连接池管理、请求限流、批量处理等并发优化策略

📄 核心内容（第1节）：模型路由策略

📄 概念：模型路由 = 快递分级配送

定义：模型路由（Model Routing）是一种根据任务的复杂度、类型和质量要求，自动将请求分发到不同模型的策略。就像快递公司根据包裹的紧急程度和体积选择不同的运输方式。

生活比喻：想象你是一个快递调度员。来了一个包裹，你需要快速判断：这是送到隔壁小区的普通信件（用自行车就行），还是需要冷链运输的生鲜食品（必须用专车）？模型路由器就是这个调度员 -- 它分析用户的问题，判断需要 GPT-4 这样的"大卡车"，还是 GPT-3.5-Turbo 这样的"小货车"就够了。

📄 路由决策的核心维度

模型路由的决策通常基于以下维度：

维度简单任务（小模型）复杂任务（大模型）
任务类型分类、翻译、格式化推理、创作、代码生成
上下文长度< 500 tokens> 2000 tokens
质量要求容忍小误差需要高精度
响应延迟要求快速（<1s）可接受较慢
成本预算每次 < $0.001可接受 > $0.01

📄 代码示例：模型路由器

// ============================================ // 模型路由器 - 根据任务复杂度选择模型 // ============================================ @Component public class ModelRouter { private final ChatClient classifierClient; private final ChatClient simpleClient; private final ChatClient complexClient; public ModelRouter( @Qualifier("classifier") ChatClient classifierClient, @Qualifier("simple") ChatClient simpleClient, @Qualifier("complex") ChatClient complexClient) { this.classifierClient = classifierClient; this.simpleClient = simpleClient; this.complexClient = complexClient; } /** * 智能路由：分析任务并选择合适的模型 * * 路由策略： * 1. 用小模型（如 GPT-4o-mini）做分类 * 2. 根据分类结果选择对应模型 * 3. 成本降低 40-60% */ public String route(String userMessage) { // Step 1: 用小模型分类任务复杂度 TaskComplexity complexity = classify(userMessage); // Step 2: 根据复杂度选择模型 return switch (complexity) { case SIMPLE -> simpleClient.prompt() .user(userMessage).call().content(); case COMPLEX -> complexClient.prompt() .user(userMessage).call().content(); }; } private TaskComplexity classify(String message) { String result = classifierClient.prompt() .system(""" 分类用户消息的复杂度。 返回 SIMPLE 或 COMPLEX。 SIMPLE: 简单问答、格式转换、翻译 COMPLEX: 推理分析、代码生成、创意写作 """) .user(message) .call() .content(); return result.contains("COMPLEX") ? TaskComplexity.COMPLEX : TaskComplexity.SIMPLE; } }
💡 成本对比实例

假设每月处理 100万次请求，其中 70% 为简单任务。全用 GPT-4o 成本约 $12,500/月；使用路由策略后，70% 走 GPT-4o-mini，成本降至约 $4,300/月，节省 65%。

⚠️ 常见陷阱

分类器本身的成本。 每次路由都要调一次分类模型，如果分类模型选得太大（如用 GPT-4 做分类），分类成本可能抵消路由节省。建议分类器使用最轻量的模型（GPT-4o-mini 或微调的小模型），甚至可以用规则引擎替代。

📄 核心内容（第2节）：语义缓存

📄 概念：语义缓存 = 问过的答案不再重复回答

定义：语义缓存（Semantic Cache）是一种基于语义相似度而非精确匹配的缓存机制。当用户的查询与之前某个已缓存的查询在语义上相似时，直接返回缓存结果，无需再次调用 LLM。

生活比喻：想象你是一个班主任老师。第一个学生问"1+1等于几"，你认真回答了"2"。第二个学生问"一加一等于多少"，第三个学生问"1+1=?"，你不需要再思考了，因为这些问题意思相同。传统缓存（Redis exact match）只能识别完全一样的问题，而语义缓存能识别"意思相同"的问题。

📄 缓存命中率优化策略

  • 相似度阈值调优 -- 阈值太高会漏掉语义相近的问题，太低会返回不相关的答案。建议从 0.92 开始，根据业务场景调整
  • 多粒度缓存 -- 对 system prompt 相同的对话做分组缓存，减少不必要的向量搜索范围
  • TTL 管理 -- 为缓存设置过期时间，确保信息时效性。知识型查询可设较长 TTL（24h），时效型查询设较短（1h）
  • 缓存预热 -- 系统启动时预先加载高频查询，减少冷启动时的缓存未命中

📄 代码示例：语义缓存 Advisor

// ============================================ // 语义缓存 Advisor - Spring AI 实现 // ============================================ @Component public class SemanticCacheAdvisor implements CallAdvisor { private final EmbeddingModel embeddingModel; private final VectorStore vectorStore; private final double similarityThreshold; public SemanticCacheAdvisor(EmbeddingModel embeddingModel, VectorStore vectorStore, @Value("${cache.threshold:0.92}") double threshold) { this.embeddingModel = embeddingModel; this.vectorStore = vectorStore; this.similarityThreshold = threshold; } @Override public CallResponse aroundCall(CallAdvisorChain chain, CallRequest request) { String userMessage = extractUserMessage(request); // Step 1: 在向量库中搜索相似查询 List<Document> results = vectorStore.similaritySearch( SearchRequest.query(userMessage) .withTopK(1) .withSimilarityThreshold(similarityThreshold) ); // Step 2: 如果命中缓存，直接返回 if (!results.isEmpty()) { String cachedResponse = results.get(0) .getMetadata().get("response").toString(); return new CallResponse( ChatResponse.from(createAssistantMessage(cachedResponse)) ); } // Step 3: 未命中，调用 LLM CallResponse response = chain.nextCall(request); // Step 4: 将结果存入缓存 String assistantResponse = response.getResult() .getOutput().getText(); Document doc = new Document(userMessage, Map.of("response", assistantResponse, "timestamp", Instant.now().toString())); vectorStore.accept(List.of(doc)); return response; } }
💡 语义缓存 vs 精确缓存

精确缓存（Redis exact match）命中率通常仅 5-15%，因为用户的措辞几乎不会完全一致。语义缓存通过向量相似度匹配，可将命中率提升到 30-60%，具体取决于业务场景中问题的重复程度。

📄 核心内容（第3节）：Token 优化

📄 概念：Token 优化 = 精简信件内容

定义：Token 优化是通过减少每次 LLM 调用中输入和输出的 Token 数量，来降低 API 调用成本的技术手段。1个 Token 大约是 0.75 个英文单词或 0.5 个中文字。

生活比喻：你寄快递时，包裹越重运费越贵。Token 优化就像是精简包裹里的物品 -- 去掉不必要的填充物（冗余 Prompt），压缩物品体积（输出压缩），甚至改变包装方式（使用更高效的 Prompt 结构）。

📄 Prompt 精简策略

一个典型的 Agent 系统中，System Prompt 可能占用 500-2000 Token。通过以下策略可以大幅缩减：

策略示例节省
去除冗余指令"请仔细地、认真地、全面地分析..." -> "分析..."10-20%
使用缩写格式完整句子 -> 关键词列表30-50%
动态上下文加载只加载当前任务需要的上下文40-60%
对话历史裁剪只保留最近N轮对话 + 关键摘要20-40%

📄 Token 预算管理器

// ============================================ // Token 预算管理器 - 精细化控制 Token 消耗 // ============================================ @Component public class TokenBudgetManager { private final int maxInputTokens; private final int maxOutputTokens; private final Tokenizer tokenizer; public TokenBudgetManager( @Value("${token.budget.input:4000}") int maxInput, @Value("${token.budget.output:2000}") int maxOutput) { this.maxInputTokens = maxInput; this.maxOutputTokens = maxOutput; this.tokenizer = new SimpleTokenizer(); } /** * 裁剪对话历史，确保不超过输入 Token 预算 * 策略：保留 System Prompt + 最近对话 + 摘要 */ public List<Message> trimConversationHistory( List<Message> messages, String systemPrompt) { int systemTokens = tokenizer.count(systemPrompt); int remaining = maxInputTokens - systemTokens; List<Message> trimmed = new ArrayList<>(); // 从最新的消息向前保留 for (int i = messages.size() - 1; i >= 0; i--) { int tokens = tokenizer.count( messages.get(i).getText()); if (remaining - tokens >= 0) { trimmed.add(0, messages.get(i)); remaining -= tokens; } else { break; } } return trimmed; } /** * 检查预算是否超支 */ public boolean isOverBudget(String input, String output) { return tokenizer.count(input) > maxInputTokens || tokenizer.count(output) > maxOutputTokens; } }

📄 流式输出优化

流式输出（Streaming）不仅改善用户体验，也是成本优化的重要手段：

  • 提前终止 -- 用户看到足够信息后主动取消，减少输出 Token 消耗
  • 首字节延迟降低 -- 用户感知的响应时间从"完整生成时间"降为"首字节时间"
  • 并发处理 -- 流式输出允许在等待响应的同时处理其他请求
💡 实战经验

在客服场景中，将 System Prompt 从 2000 Token 精简到 500 Token（使用关键词列表替代完整句子），配合对话历史裁剪，可将每次对话的输入成本降低 60-70%，而回答质量几乎无损失。

📄 核心内容（第4节）：并发与批量处理

📄 概念：并发优化 = 高效调度快递车队

定义：并发与批量处理优化是通过管理连接池、实施限流策略、合并请求等方式，最大化利用 API 配额和网络资源，降低单位请求的固定成本。

生活比喻：快递公司不会为每个包裹单独派一辆车，而是把同一方向的包裹装满一车再出发（批量处理）。同时，车队有自己的车辆数限制（连接池），如果所有车都在路上，新包裹就排队等候（背压机制），而不是无限增加车辆导致交通瘫痪。

📄 核心策略

连接池管理

复用 HTTP 连接，避免每次请求都建立新连接。对于 LLM API，连接池大小通常设为 10-50，具体取决于 API 的并发限制。

请求限流

使用令牌桶或滑动窗口算法控制请求速率，避免触发 API 的限流机制（429 错误），减少重试带来的额外成本。

批量处理

将多个独立请求合并为一次 API 调用。某些模型提供商支持 Batch API，价格可降低 50%。

异步处理

非实时任务使用异步模式，在 API 低峰期发送请求，通常可享受更低的价格。

📄 代码示例：限流器与批量处理

// ============================================ // 限流器 - 基于令牌桶算法 // ============================================ @Component public class RateLimiter { private final Semaphore semaphore; private final int maxConcurrent; private final long timeoutMs; public RateLimiter( @Value("${rate-limit.max-concurrent:20}") int maxConcurrent, @Value("${rate-limit.timeout-ms:5000}") long timeoutMs) { this.semaphore = new Semaphore(maxConcurrent); this.maxConcurrent = maxConcurrent; this.timeoutMs = timeoutMs; } public <T> T execute(Supplier<T> task) { try { if (semaphore.tryAcquire(timeoutMs, TimeUnit.MILLISECONDS)) { try { return task.get(); } finally { semaphore.release(); } } throw new RateLimitExceededException( "并发数已达上限: " + maxConcurrent); } catch (InterruptedException e) { Thread.currentThread().interrupt(); throw new RuntimeException(e); } } } // ============================================ // 批量处理器 - 合并多个请求一次处理 // ============================================ @Component public class BatchProcessor { private final ChatClient chatClient; private final BlockingQueue<BatchRequest> queue; /** * 将多个独立请求合并为一次调用 * 适用于非实时场景（如离线分析、报告生成） */ public List<String> processBatch(List<String> inputs) { // 将多个输入合并为一个 Prompt String batchPrompt = inputs.stream() .map((input, idx) -> "[" + idx + "] " + input) .collect(Collectors.joininging("\n")); String result = chatClient.prompt() .system("批量处理以下请求，" + "对每个输入给出独立回答，" + "用 [编号] 分隔。") .user(batchPrompt) .call() .content(); // 解析批量响应 return parseBatchResponse(result, inputs.size()); } }
⚠️ 注意事项

批量处理不适合实时场景。 批量处理通过延迟响应换取成本降低，只适用于对延迟不敏感的任务（如日报生成、数据标注、内容审核）。实时对话、代码补等场景必须使用单次调用。

📄 成本追踪 Advisor

// ============================================ // 成本追踪 Advisor - 实时监控每次调用成本 // ============================================ @Component public class CostTrackingAdvisor implements CallAdvisor { private final MeterRegistry meterRegistry; // 价格表（美元/百万 Token） private static final Map<String, ModelPrice> PRICE_TABLE = Map.of( "gpt-4o", new ModelPrice(2.50, 10.00), "gpt-4o-mini", new ModelPrice(0.15, 0.60) ); public CostTrackingAdvisor(MeterRegistry meterRegistry) { this.meterRegistry = meterRegistry; } @Override public CallResponse aroundCall(CallAdvisorChain chain, CallRequest request) { long start = System.currentTimeMillis(); CallResponse response = chain.nextCall(request); // 提取 Token 使用量 Usage usage = response.getResult().getMetadata() .getUsage(); int inputTokens = usage.getPromptTokens(); int outputTokens = usage.getGenerationTokens(); // 计算成本 String model = extractModel(request); ModelPrice price = PRICE_TABLE .getOrDefault(model, new ModelPrice(0, 0)); double cost = (inputTokens * price.input() + outputTokens * price.output()) / 1_000_000.0; // 记录指标 meterRegistry.gauge("llm.cost.dollars", cost); meterRegistry.counter("llm.tokens.input") .increment(inputTokens); meterRegistry.counter("llm.tokens.output") .increment(outputTokens); long latency = System.currentTimeMillis() - start; meterRegistry.timer("llm.latency.ms") .record(latency, TimeUnit.MILLISECONDS); return response; } record ModelPrice(double input, double output) {} }

📖 章节小结

  • 模型路由 -- 根据任务复杂度自动选择模型，简单任务用小模型，复杂任务用大模型，可节省 40-60% 调用成本
  • 语义缓存 -- 基于向量相似度匹配重复查询，避免不必要的 LLM 调用，将命中率从精确匹配的 5-15% 提升到 30-60%
  • Token 优化 -- 通过 Prompt 精简、上下文裁剪、输出压缩等手段减少每次调用的 Token 消耗，可降低 15-30%
  • 并发控制 -- 连接池复用、请求限流、批量处理等策略最大化利用 API 配额，降低固定成本和延迟
  • 成本监控 -- 通过 Advisor 链实现全链路成本追踪，实时监控每次调用的 Token 消耗和费用

🎓 面试高频题

👨‍💻 Q1: 如何设计一个 Agent 系统的成本优化方案？

答：完整的成本优化方案包含四个层次：
(1) 模型路由层：用轻量分类器分析任务复杂度，将简单任务（分类、翻译）路由到小模型，复杂任务（推理、创作）路由到大模型；
(2) 缓存层：实现语义缓存，基于向量相似度匹配重复查询，设置合理的 TTL 和相似度阈值；
(3) Token 优化层：精简 System Prompt（去除冗余指令），动态裁剪对话历史，使用关键词列表替代完整句子；
(4) 并发控制层：连接池管理、请求限流（令牌桶算法）、非实时任务使用批量处理。通过这四层策略组合，通常可将总成本降低 50-70%。

👨‍💻 Q2: 语义缓存和传统缓存（Redis）有什么区别？各自的适用场景？

答：传统缓存基于精确匹配（key-value），只有完全相同的查询才能命中，命中率通常 5-15%。语义缓存基于向量相似度，只要两个查询在语义上相似（如"什么是机器学习"和"机器学习的定义"）就能命中，命中率可达 30-60%。

适用场景：传统缓存适合结构化查询（API 调用、数据库查询），延迟低（亚毫秒级）。语义缓存适合自然语言问答、客服对话等措辞多变的场景，但需要额外的 Embedding 计算成本和向量数据库存储。实际项目中常两者结合使用：先精确匹配，未命中再语义匹配。

👨‍💻 Q3: Token 优化有哪些具体技术手段？如何评估优化效果？

答：主要技术手段包括：
(1) Prompt 精简：去除冗余指令，用关键词列表替代完整句子，可节省 30-50% 输入 Token；
(2) 对话历史裁剪：只保留最近 N 轮对话 + 关键摘要，避免上下文爆炸；
(3) 动态上下文加载：根据用户问题只加载相关的知识片段，而非全量注入；
(4) 输出格式约束：要求 JSON 格式输出，比自由文本节省 20-40% 输出 Token；
(5) 流式输出：支持用户提前终止，减少不必要的输出生成。

评估方法：建立 Token 监控指标（输入/输出 Token 数、每请求平均 Token、总成本），A/B 测试对比优化前后的回答质量（准确率、用户满意度），确保成本降低不以质量损失为代价。

👨‍💻 Q4: 模型路由中的分类器如何设计？有什么注意事项？

答：分类器的设计要点：
(1) 模型选择：分类器本身必须足够轻量，建议使用 GPT-4o-mini 或微调的小模型（如 DistilBERT），避免分类成本抵消路由节省；
(2) 特征设计：基于输入长度、关键词、意图类型等特征做快速判断，复杂度高的特征（如代码生成、多步推理）路由到大模型；
(3) 降级策略：分类器不确定时默认使用大模型，宁可多花一点成本也不牺牲质量；
(4) 缓存分类结果：对相似查询缓存分类决策，减少分类器调用次数；
(5) 监控与调优：持续监控路由分布和各模型的响应质量，动态调整路由阈值。

👨‍💻 Q5: 在生产环境中如何监控和控制 Agent 系统的成本？

答：生产环境的成本监控体系包含：
(1) 实时追踪：通过 Advisor 拦截每次 LLM 调用，记录模型名、输入/输出 Token 数、延迟、成本，使用 Micrometer 输出到 Prometheus/Grafana；
(2) 预算告警：设置每日/每月成本上限，超过阈值时触发告警，极端情况下自动降级到小模型或拒绝服务；
(3) 用户级配额：为每个用户/租户设置 Token 预算，防止单用户消耗过多资源；
(4) 异常检测：监控 Token 消耗异常（如某类请求突然消耗大量 Token），及时发现 Prompt 注入或死循环；
(5) 成本分析报告：按模型、用户、时间段维度分析成本分布，识别优化机会。
