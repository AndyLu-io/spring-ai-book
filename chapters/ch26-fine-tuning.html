模型微调与定制化

从"通用助手"到"领域专家" — 掌握 LoRA、QLoRA、PEFT 等参数高效微调技术，让大模型真正理解你的业务领域。

本章目录

    📚 开篇故事：从万能翻译到专业翻译

    想象你是一家跨国医药公司的翻译部门主管。公司雇了一个"万能翻译"——他能说 50 种语言，什么类型的文本都能翻，速度快、效率高，看起来非常完美。

    但问题很快出现了。当你把一份心脏外科手术记录交给他的时候，他把"主动脉夹层"翻成了"大血管裂缝"；当你让他翻译一份新药临床试验报告时，他把"I期临床试验"翻成了"第一个阶段的临床测试"。虽然听起来都能理解，但任何一个医学专业人士都会皱眉——这些翻译完全不专业。

    这不是因为他不够聪明，而是因为他是一个通才，而不是一个专才。他什么都懂一点，但什么都不精通。

    于是你做了一个决定：给他放一个长假，让他闭门研读你们公司十年来积累的所有医学文献、翻译规范、术语表和经典译文。三个月后，他回来了。这一次，他不仅能准确翻译"主动脉夹层（Aortic Dissection）"，还能根据上下文判断应该使用"急性"还是"慢性"的表述，甚至能在翻译新药名称时自动遵循 WHO 国际非专利药品名称（INN）的命名规则。

    这个"闭门修炼"的过程，就是微调（Fine-Tuning）。通用大模型就像那个万能翻译，知识渊博但缺少专业深度。微调就是让它沉浸式学习你的领域知识，从"什么都能聊"进化为"什么都能精通"。

    本章将带你掌握模型微调的完整知识体系——从理解"为什么需要微调"，到掌握 LoRA、QLoRA 等参数高效微调技术，再到完成从数据准备到生产部署的全流程实战。

    💡 为什么需要微调？— Prompt Engineering 的天花板

    Prompt Engineering 的局限性

    Prompt Engineering 是最轻量的定制化手段，但它有明确的天花板：

    • 上下文窗口限制 — 即使模型支持 128K tokens，真正有效的上下文远小于理论值。超过一定长度后，模型的"大海捞针"能力急剧下降，无法在超长上下文中可靠地提取和利用信息
    • 领域知识深度不够 — Prompt 只能"激活"模型已有的知识，无法注入模型训练时未见过的专业知识。你无法通过提示词让模型理解你公司的内部缩写和专有术语体系
    • 输出格式不稳定 — 复杂的结构化输出（如医疗报告模板、法律文书格式）在 Prompt Engineering 下经常出现格式漂移，今天输出完美格式，明天可能缺少字段或改变排列
    • 推理能力瓶颈 — Prompt 能引导思维链，但无法从根本上提升模型在特定领域的推理能力。面对需要深层专业知识的复杂问题，Prompt 的引导力有上限

    微调 vs RAG vs Prompt Engineering 对比

    什么场景适合微调

    ✓ 适合微调的场景

    • 领域术语体系 — 医疗、法律、金融等领域有严格的术语规范，微调能让模型内化这些术语
    • 输出风格统一 — 企业品牌语言、客服话术、技术文档风格等需要高度一致
    • 格式严格约束 — 生成的 JSON、XML 等结构化数据必须严格符合 schema
    • 性能要求高 — 需要低延迟、高质量的推理，减少 Prompt 开销

    ✗ 不需要微调的场景

    • RAG 够用 — 只需要知识问答，RAG + 向量检索完全满足需求
    • 数据量不足 — 少于 100 条高质量数据，微调效果不如精心设计的 Prompt
    • 成本敏感 — 微调需要 GPU 训练资源和持续维护，成本显著高于 Prompt
    • 任务简单 — 分类、摘要等简单任务，Few-shot Prompt 通常足够
    💡 关键决策原则

    微调不是万能药，也不是银弹。先尝试 Prompt Engineering，再评估 RAG，最后才考虑微调。只有当你的需求确实需要"改变模型行为"时，微调才是正确的选择。很多看似需要微调的问题，其实用精心设计的 Prompt + RAG 就能解决 80%。

    🔧 参数高效微调（PEFT）：站在巨人肩膀上

    Full Fine-Tuning vs PEFT

    传统的全量微调（Full Fine-Tuning）需要更新模型的所有参数。对于一个 7B 参数的模型，全量微调需要：

    • 显存需求：至少 64GB+（模型参数 + 优化器状态 + 梯度），通常需要 4-8 张 A100 80GB
    • 训练时间：数天到数周
    • 灾难性遗忘：模型可能"忘记"通用能力，只在特定任务上表现好

    PEFT（Parameter-Efficient Fine-Tuning）的核心思想是：冻结预训练模型的大部分参数，只训练一小部分"新增参数"。这就像给一个经验丰富的医生只发一本科室手册，而不是让他重新读一遍医学院——他只需要学习你科室的特殊流程和规范。

    Full Fine-Tuning：
  预训练模型（6.7B 参数）====>====>====>====>  全部参数更新
  显存：~60GB | 训练数据：需要大量 | 风险：灾难性遗忘

PEFT（LoRA）：
  预训练模型（6.7B 参数）── 冻结，不更新 ──
  |
  v
  新增低秩矩阵（~4M 参数） ====>====>  只训练这部分
  显存：~16GB | 训练数据：少量即可 | 风险：保留通用能力

    LoRA 原理：低秩分解

    LoRA（Low-Rank Adaptation）是目前最流行的 PEFT 方法。它的核心思想来自一个观察：微调时的权重更新矩阵是低秩的。也就是说，微调产生的变化可以用两个小矩阵的乘积来近似。

    LoRA 数学原理：

  原始权重更新：  W' = W + ΔW     其中 ΔW 的维度是 [d × d]
  LoRA 近似：     W' = W + B × A   其中 B 是 [d × r], A 是 [r × d]

  当 d = 4096, r = 16 时：
    ΔW 参数量 = 4096 × 4096 = 16,777,216
    B × A 参数量 = 4096 × 16 + 16 × 4096 = 131,072
    压缩比 = 128:1    只需训练 0.06% 的参数！

    LoRA 通常应用在 Transformer 的注意力层权重上：Q（Query）、K（Key）、V（Value）和 O（Output）投影矩阵。在微调时，原始权重被冻结（不更新），只训练 LoRA 新增的小矩阵 A 和 B。

    LoRA 的关键超参数

    • r（秩）：低秩矩阵的维度，通常取 8、16、32、64。r 越大，表达能力越强，但参数越多。一般 16 是个不错的起点
    • lora_alpha：缩放因子，控制 LoRA 更新的强度。通常设为 r 的 2 倍（如 r=16, alpha=32）。alpha/r 决定了学习率的有效缩放
    • lora_dropout：防止过拟合的正则化手段，通常取 0.05-0.1
    • target_modules：要应用 LoRA 的模块。通常包括 ["q_proj", "v_proj", "k_proj", "o_proj"]，更多模块可以提升性能但增加参数量

    QLoRA：量化 + LoRA 的终极组合

    QLoRA（Quantized LoRA）将模型量化到 4-bit 再应用 LoRA，使得消费级 GPU 也能微调大模型。核心技术包括：

    • 4-bit NormalFloat（NF4）量化 — 比普通 4-bit 量化更精确的数据类型，专为正态分布的神经网络权重设计
    • 双重量化（Double Quantization） — 对量化常数再次量化，进一步减少内存占用，每个参数平均节省约 0.37 bit
    • 分页优化器（Paged Optimizers） — 使用 CPU 内存作为 GPU 内存的溢出缓冲，避免 OOM

    其他 PEFT 方法速览

    Adapter Tuning

    在 Transformer 每一层的 FFN 之后插入小型 Adapter 层。参数量比 LoRA 稍多，但结构更简单。适合需要逐层精细控制的场景。

    Prefix Tuning

    在每一层的注意力计算前添加可训练的"虚拟 Token"（Prefix）。不修改模型权重，只学习连续的向量表示。适合生成式任务。

    P-Tuning v2

    在每一层的输入前添加可训练的嵌入向量。比 P-Tuning v1 更稳定，适合中文 NLU 任务。参数量最少（每层仅几个嵌入向量）。

    IA3（Infused Adapter by Inhibiting and Amplifying）

    通过学习向量来缩放模型的激活值，而非添加新的参数矩阵。参数量极少（<0.01%），但表达能力有限，适合快速实验。

    PEFT 方法选型决策树

    PEFT 方法选型决策树：

  你的 GPU 显存是多少？
      |
      +---> < 8 GB ──> QLoRA（4-bit 量化 + LoRA）
      |
      +---> 8-16 GB ──> QLoRA 或 LoRA
      |
      +---> 16-24 GB ──> LoRA（标准 16-bit）
      |
      +---> 24 GB+ ──> LoRA 或 Full Fine-Tuning
      |
      v
  你的数据量是多少？
      |
      +---> < 500 条 ──> LoRA（r=8, 强正则化）
      |
      +---> 500-5000 条 ──> LoRA（r=16-32）
      |
      +---> 5000+ 条 ──> LoRA 或 Full Fine-Tuning

    💡 要点小结
    • PEFT 的核心优势：只训练 0.06% 的参数，达到接近全量微调 95%+ 的效果
    • QLoRA 让消费级 GPU（RTX 3060）也能微调 7B 模型
    • LoRA r=16, alpha=32 是通用的起点配置，根据任务效果逐步调整
    • 选型原则：能用 LoRA 就不用 Full Fine-Tuning，能用 QLoRA 就不用标准 LoRA

    📋 训练数据准备：垃圾进垃圾出

    数据格式：instruction-input-output

    微调数据的核心格式是三元组：instruction（指令）、input（可选输入）、output（期望输出）。这模拟了用户与模型的交互方式，让模型学会"听到指令后应该如何响应"。

    训练数据格式示例：

  {
    "instruction": "将以下医疗报告翻译为通俗语言",
    "input": "患者CT显示右肺上叶3mm磨玻璃结节，建议随访观察。",
    "output": "您的CT检查发现右肺上方有一个3毫米的小结节，
             目前看来问题不大，建议半年后复查一次。"
  }

  instruction: 模型应该执行的任务（必填）
  input:      任务的上下文信息（可选，有时为空）
  output:     模型应该生成的目标回答（必填）

    数据清洗流程

    "垃圾进，垃圾出"——训练数据的质量直接决定微调效果。一个完整的数据清洗流程包括：

    • 去重 — 完全重复和近似重复（SimHash / MinHash）都会导致模型过拟合到特定模式。通常需要去除 30-50% 的近似重复数据
    • 质量过滤 — 过滤掉长度异常（太短/太长）、内容混乱（乱码、无意义文本）、格式错误的数据
    • 格式标准化 — 统一编码格式（UTF-8）、去除多余空白字符、规范标点符号
    • 安全过滤 — 去除含有有害内容、PII 信息、版权争议内容的数据

    数据量参考

    数据量效果预期适用场景
    100-500 条轻度风格调整、简单格式约束快速验证、内部工具
    500-2000 条中等领域适配、输出格式统一企业级应用、客服系统
    2000-10000 条深度领域定制、复杂推理提升专业领域、核心产品
    10000+ 条接近全量微调效果需要极致性能的场景

    数据增强策略

    Self-Instruct

    用 LLM 自动生成新的 instruction-output 对。先提供少量种子样本，让 GPT-4 生成更多类似的训练数据，再人工审核和修正。适合快速扩充数据量，但需要严格的质量控制。

    Evol-Instruct

    对现有 instruction 进行"进化"：增加约束、深化推理、复杂化场景。从简单指令逐步演化为复杂指令，覆盖更广的任务空间。WizardLM 论文的核心方法。

    数据质量评估

    • 一致性 — 同类任务的 input-output 应该遵循一致的规则。如"翻译为通俗语言"的输出不应今天口语化、明天书面化
    • 多样性 — 覆盖不同难度级别、不同子领域、不同表述方式。单一模式的数据会导致模型行为单一
    • 准确性 — output 中的事实必须正确。一条错误数据可能导致模型"学会"错误知识

    ⚠️ 数据质量陷阱

    最危险的不是"少"，而是"错"。一条包含错误事实的数据，模型会以 100% 的确信度"学会"错误知识。在医疗、法律等高风险领域，建议对训练数据进行领域专家审核，宁可少用 500 条数据，也不能留下一条错误数据。

    🚀 微调实战：从数据到模型

    训练框架选择

    对于 Spring AI 生态中的 Java 开发者，推荐使用 Python 生态的 Hugging Face 工具链完成微调，再将模型导出为通用格式（如 SafeTensors）供 Spring AI 通过 vLLM/Ollama 调用。核心工具链：

    • Hugging Face Transformers — 模型加载、分词器、推理接口
    • PEFT — LoRA/QLoRA 配置与模型包装
    • TRL（Transformer Reinforcement Learning） — SFTTrainer、DPOTrainer 等训练器
    • Accelerate — 分布式训练、混合精度、设备映射

    SFT（Supervised Fine-Tuning）训练流程

    # 完整的 SFT 训练流程 import torch from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments from peft import LoraConfig, get_peft_model, TaskType from trl import SFTTrainer from datasets import load_dataset # ==================== 1. 加载基础模型 ==================== model_name = "Qwen/Qwen2-7B" model = AutoModelForCausalLM.from_pretrained( model_name, torch_dtype=torch.bfloat16, device_map="auto", trust_remote_code=True ) tokenizer = AutoTokenizer.from_pretrained(model_name, trust_remote_code=True) tokenizer.pad_token = tokenizer.eos_token # ==================== 2. 配置 LoRA ==================== lora_config = LoraConfig( task_type=TaskType.CAUSAL_LM, r=16, lora_alpha=32, lora_dropout=0.1, target_modules=["q_proj", "v_proj", "k_proj", "o_proj"] ) model = get_peft_model(model, lora_config) # ==================== 3. 加载训练数据 ==================== dataset = load_dataset("json", data_files="training_data.json") train_dataset = dataset["train"].shuffle(seed=42).select(range(int(len(dataset["train"]) * 0.9))) eval_dataset = dataset["train"].shuffle(seed=42).select(range(int(len(dataset["train"]) * 0.9), len(dataset["train"]))) # ==================== 4. 训练配置 ==================== training_args = TrainingArguments( output_dir="./output", num_train_epochs=3, per_device_train_batch_size=4, gradient_accumulation_steps=4, learning_rate=2e-4, warmup_steps=100, logging_steps=10, evaluation_strategy="steps", eval_steps=50, save_strategy="steps", save_steps=100, fp16=True, report_to="tensorboard", load_best_model_at_end=True, metric_for_best_model="eval_loss" ) # ==================== 5. 开始训练 ==================== trainer = SFTTrainer( model=model, args=training_args, train_dataset=train_dataset, eval_dataset=eval_dataset, tokenizer=tokenizer, max_seq_length=2048 ) trainer.train() # ==================== 6. 保存 LoRA 权重 ==================== trainer.save_model("./output/final_checkpoint")

    超参数调优指南

    超参数推荐值说明
    learning_rate1e-4 ~ 3e-4LoRA 的学习率通常比 Full Fine-Tuning 高一个数量级
    num_train_epochs1-3LoRA 通常 1-2 epoch 即可，过多 epoch 容易过拟合
    per_device_train_batch_size2-8根据 GPU 显存调整，QLoRA 可以用更大的 batch
    gradient_accumulation_steps4-16等效 batch size = per_device * accumulation * GPU数
    warmup_steps总步数的 5-10%避免训练初期梯度爆炸
    weight_decay0.01-0.1正则化，防止过拟合
    💡 超参数调优经验

    最重要的超参数是 learning_rate。如果 loss 不收敛，先调学习率（减半或加倍）；如果过拟合，减少 epochs 或增加 dropout；如果欠拟合，增大 batch size 或提高学习率。建议使用 WandB 或 TensorBoard 实时监控训练曲线。

    🎯 RLHF 与 DPO：让模型对齐人类偏好

    RLHF 三阶段流程

    RLHF（Reinforcement Learning from Human Feedback）是让模型"说人话"的关键技术。ChatGPT 之所以能流畅、安全、有帮助地对话，RLHF 功不可没。它分为三个阶段：

    RLHF 三阶段：

  阶段 1：SFT（监督微调）
  高质量对话数据 ───────────> 训练基础对话能力
  目标：让模型学会"怎么回答"

  阶段 2：Reward Model 训练
  人类标注偏好对 (回答A > 回答B) ──> 训练奖励模型
  目标：让模型学会"什么是好的回答"

  阶段 3：PPO 强化学习
  奖励模型打分 ───────────> PPO 优化模型策略
  目标：让模型"生成人类喜欢的回答"

    RLHF 的关键创新在于：它让模型不只是"学习正确答案"，而是"学习人类的偏好标准"。同样的问题可能有多个正确答案，但人类对不同答案有偏好排序——RLHF 让模型理解这种偏好。

    DPO：绕过 Reward Model

    DPO（Direct Preference Optimization）是 RLHF 的简化版本。它发现了一个数学等价关系：可以跳过 Reward Model 的训练，直接用偏好数据优化模型策略。

    RLHF vs DPO：

  RLHF：数据 → Reward Model → PPO → 最终模型
       （需要训练两个模型，PPO 不稳定，超参数敏感）

  DPO： 数据 → 直接优化 → 最终模型
       （只训练一个模型，更稳定，更简单）

    DPO 的核心公式用一个简单的类比理解：想象你在训练一只狗。RLHF 是先训练一个"评判狗表现好坏的裁判"（Reward Model），再让狗根据裁判的打分调整行为（PPO）。DPO 是直接告诉狗"这个行为好，那个行为差"，不需要中间的裁判。

    RLHF vs DPO 对比

    维度RLHF（PPO）DPO
    实现复杂度高 — 需要 Reward Model + PPO低 — 只需要偏好数据
    训练稳定性低 — PPO 超参数敏感，训练不稳定高 — 类似 SFT，稳定可控
    资源消耗高 — 需要同时加载 3 个模型中 — 只需 2 个模型（model + ref）
    效果上限更高，但需要精细调优大多数场景下效果接近 RLHF
    适合场景大规模产品级对齐（如 ChatGPT）中小规模偏好对齐、快速实验

    DPO 实战代码

    # DPO 训练流程 import torch from transformers import AutoModelForCausalLM, AutoTokenizer from trl import DPOTrainer, DPOConfig from datasets import load_dataset # 加载基础模型 model_name = "Qwen/Qwen2-7B" model = AutoModelForCausalLM.from_pretrained( model_name, torch_dtype=torch.bfloat16, device_map="auto" ) # 参考模型（冻结的原始模型） ref_model = AutoModelForCausalLM.from_pretrained( model_name, torch_dtype=torch.bfloat16, device_map="auto" ) tokenizer = AutoTokenizer.from_pretrained(model_name) # 加载偏好数据集（chosen/rejected 对） # 每条数据包含同一个 instruction 的两个回答， # chosen 是人类偏好的回答，rejected 是较差的回答 preference_dataset = load_dataset("json", data_files="preference_data.json") # DPO 训练配置 dpo_config = DPOConfig( output_dir="./dpo_output", per_device_train_batch_size=2, learning_rate=5e-7, num_train_epochs=1, beta=0.1, # KL散度系数，控制与参考模型的偏离程度 logging_steps=10, eval_strategy="steps", eval_steps=50, save_steps=100, report_to="tensorboard", max_length=1024, max_prompt_length=512, ) # 创建 DPO Trainer dpo_trainer = DPOTrainer( model=model, ref_model=ref_model, args=dpo_config, train_dataset=preference_dataset["train"], tokenizer=tokenizer ) # 开始训练 dpo_trainer.train() # 保存模型 dpo_trainer.save_model("./dpo_output/final")
    💡 DPO beta 参数解析

    beta（KL 散度系数）控制模型偏离原始行为的程度。beta 越大，模型越保守（接近原始模型）；beta 越小，模型越激进地学习偏好。通常从 0.1 开始，如果模型变化太大就增大，变化太小就减小。

    什么时候用 RLHF，什么时候用 DPO

    选择 RLHF 当：

    • 你有大规模标注团队和充足的 GPU 资源
    • 需要极致的对齐效果（如面向消费者的产品）
    • 偏好标准复杂且需要动态调整
    • 已有成熟的 RLHF 训练流水线

    选择 DPO 当：

    • 团队规模有限，想要快速迭代
    • GPU 资源有限（DPO 只需要 2 个模型）
    • 偏好对齐是辅助需求而非核心竞争力
    • 想要更稳定、可复现的训练过程

    🚀 微调效果评估与生产部署

    评估指标

    微调效果的评估需要多维度、多方法。单一指标无法全面反映模型质量：

    • Perplexity（困惑度） — 衡量模型对文本的"惊讶程度"，值越低表示模型对训练数据的理解越好。适合评估语言建模质量
    • BLEU/ROUGE — 基于 n-gram 匹配的自动评估指标。BLEU 侧重精确率（生成的词中有多少在参考答案中），ROUGE 侧重召回率（参考答案中的词有多少被生成了）
    • 人工评估 — 最可靠的评估方式，但成本高、速度慢。通常采样 200-500 条进行人工打分（正确性、流畅性、有用性）
    • 领域基准测试 — 在特定领域的测试集上评估。如医疗微调模型在 MedQA、CMExam 等基准上的表现

    A/B 测试：微调模型 vs 基础模型

    生产环境中最有效的评估方式是A/B 测试：将一部分用户流量导流到微调模型，对比两个模型在真实业务场景中的表现。

    • 质量指标 — 用户满意度评分、回答采纳率、人工审核通过率
    • 效率指标 — 首 Token 延迟（TTFT）、每秒 Token 数（TPS）、总响应时间
    • 成本指标 — 每次调用的 Token 消耗、GPU 利用率、整体运营成本
    ⚠️ A/B 测试注意事项

    微调模型可能在特定领域表现更好，但在通用能力上有所退化。A/B 测试时不仅要测试目标场景，还要测试通用场景的"回归"——确保微调没有让模型在不该变化的地方发生变化。

    模型合并：LoRA 权重合并到基础模型

    LoRA 微调后的模型有两个部分：基础模型 + LoRA 权重。生产部署时，通常需要将 LoRA 权重合并到基础模型中，形成一个完整的模型文件。

    # LoRA 权重合并 from peft import PeftModel # 加载基础模型和 LoRA 权重 base_model = AutoModelForCausalLM.from_pretrained( "Qwen/Qwen2-7B", torch_dtype=torch.bfloat16 ) model = PeftModel.from_pretrained(base_model, "./output/final_checkpoint") # 合并 LoRA 权重到基础模型 merged_model = model.merge_and_unload() # 保存合并后的完整模型 merged_model.save_pretrained("./merged_model") # 保存分词器 tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen2-7B") tokenizer.save_pretrained("./merged_model")

    生产部署：vLLM + LoRA 热切换

    对于需要同时服务多个微调模型的场景，vLLM 的 LoRA 热切换是最佳方案：基础模型只加载一次，多个 LoRA 权重按需加载和切换，极大节省显存。

    💡 部署最佳实践
    • 合并部署 — 只有一个微调模型时，合并后直接部署最简单
    • LoRA 热切换 — 多个微调模型共享基础模型时，vLLM LoRA 热切换是最佳选择
    • 监控与回滚 — 部署后持续监控模型质量指标，保留快速回滚到基础模型的能力
    • 版本管理 — 为每个微调模型建立版本号，记录训练数据、超参数和评估结果

    📖 章节小结

    • Prompt Engineering 有天花板 — 当领域知识深度、输出格式一致性、推理能力成为瓶颈时，微调是突破手段。但先尝试 Prompt + RAG，微调是最后的手段
    • PEFT 是微调的正确姿势 — LoRA 只训练 0.06% 的参数，QLoRA 让消费级 GPU 也能微调。全量微调仅在资源充足且效果要求极致时才考虑
    • 数据质量决定一切 — "垃圾进，垃圾出"在微调领域尤其严重。数据清洗、去重、质量审核是微调成功的前提，宁少勿错
    • SFT 是微调的起点 — 使用 Hugging Face Transformers + PEFT + TRL 工具链，遵循 loss 监控、超参调优的标准流程
    • DPO 比 RLHF 更实用 — 对于大多数团队，DPO 是更简单、更稳定的偏好对齐方案。RLHF 适合大规模产品级对齐
    • 部署要考虑可扩展性 — 单模型用合并部署，多模型用 vLLM LoRA 热切换。建立版本管理和回滚机制

    👨‍💻 面试高频题

    👨‍💻 Q1: LoRA 的核心原理是什么？为什么只训练 0.06% 的参数就能达到接近全量微调的效果？

    答：LoRA 的核心原理是低秩分解。它发现微调时的权重更新矩阵（ΔW）是低秩的，可以用两个小矩阵的乘积 B×A 来近似（B: d×r, A: r×d）。当 r=16, d=4096 时，参数量从 16M 压缩到 128K，压缩比 128:1。

    之所以有效，原因有三：(1) 预训练模型已经学到了丰富的特征表示，微调只需要在这些特征上做微小的方向调整，这种调整本身就是低秩的；(2) LoRA 的更新通过残差连接加到原始权重上（W' = W + B×A），保留了预训练知识；(3) 实验表明，微调的 ΔW 确实集中在前几个主成分上，低秩近似的误差很小（< 1%）。

    👨‍💻 Q2: 微调 vs RAG，在实际项目中如何选择？请给出决策框架。

    答：决策框架遵循"Prompt → RAG → 微调"的渐进路径：
    (1) 先尝试 Prompt Engineering — 如果精心设计的系统提示词 + Few-shot 示例能达到 80%+ 的效果，就不需要额外手段；
    (2) 然后评估 RAG — 如果需要注入外部知识或实时数据，RAG 是最佳选择。RAG 适合知识问答、信息检索等场景，且更新灵活（更新文档即可）；
    (3) 最后考虑微调 — 当需求是"改变模型行为"而非"注入知识"时，才需要微调。典型信号：输出格式不稳定、领域术语使用不当、推理能力不足、Prompt 开销过大。

    实际项目中，三者往往组合使用：微调模型 + RAG + 精心设计的 Prompt 是最强大的组合。微调提供领域适配能力，RAG 提供知识检索能力，Prompt 提供任务引导能力。

    👨‍💻 Q3: QLoRA 是如何实现的？为什么 4-bit 量化不会严重影响模型质量？

    答：QLoRA 结合了三项关键技术：
    (1) NF4（4-bit NormalFloat）量化 — 针对神经网络权重近似正态分布的特性设计的数据类型，比均匀的 INT4 量化更精确，信息损失更小；
    (2) 双重量化 — 对量化的缩放因子（每 64 个参数共享一个 FP32 的 scale）再次量化为 INT8，将量化开销从 0.5 bit/param 降低到 0.127 bit/param；
    (3) 分页优化器 — 使用 NVIDIA 统一内存特性，在 GPU 显存不足时自动将优化器状态卸载到 CPU 内存。

    4-bit 量化不影响模型质量的原因：预训练模型的权重中有大量冗余，量化主要影响的是这些冗余信息。而 LoRA 训练时，梯度更新和优化器状态仍使用 BF16/FP16 精度，只有权重存储使用 NF4。反向传播时，NF4 权重被反量化为 BF16 进行计算，因此训练精度不受影响。

    👨‍💻 Q4: DPO 和 RLHF 的区别是什么？为什么 DPO 更适合大多数团队？

    答：核心区别在于是否需要训练 Reward Model：
    RLHF 是三阶段流程：SFT → 训练 Reward Model → PPO 强化学习。Reward Model 学习人类偏好的评分函数，PPO 策略梯度算法利用 Reward Model 的打分来优化模型。
    DPO 发现了一个数学等价关系：可以通过对比学习直接用偏好数据优化策略，无需中间的 Reward Model。

    DPO 适合大多数团队的原因：
    (1) 实现简单 — 只需要 SFT 之后的偏好数据，不需要额外训练 Reward Model；
    (2) 训练稳定 — DPO 的训练过程类似 SFT，不需要调 PPO 的众多超参数（clip ratio、KL 系数、GAE lambda 等）；
    (3) 资源节省 — 只需加载 2 个模型（当前模型 + 参考模型），而非 RLHF 的 3 个（模型 + 参考模型 + Reward Model）；
    (4) 效果接近 — 在大多数任务上，DPO 的效果与 RLHF 相当，仅在超大规模（如 ChatGPT 级别）时 RLHF 有明显优势。

    👨‍💻 Q5: 微调后如何进行生产部署？如何支持多个微调模型的高效服务？

    答：生产部署有两种主要策略：
    (1) 合并部署 — 将 LoRA 权重通过 merge_and_unload() 合并到基础模型中，形成一个完整模型，用 vLLM/Ollama 直接部署。优点是简单高效，缺点是每个微调模型都要单独占用一份基础模型的显存；
    (2) LoRA 热切换 — 使用 vLLM 的 --enable-lora 功能，基础模型只加载一次，多个 LoRA 权重按需加载。调用时通过 model 参数指定使用哪个 LoRA 模块。显存效率高，适合多租户或多场景。

    推荐的最佳实践：(a) 建立模型版本管理，记录每个微调模型的训练数据、超参数、评估结果；(b) 部署后持续监控质量指标（准确率、用户满意度），设置自动告警；(c) 保留快速回滚到基础模型的能力；(d) 使用 A/B 测试逐步放量，而非一次性全量切换。

    为你的微调模型添加 DPO 偏好对齐：

    • 构造 200+ 条偏好数据（chosen/rejected 对），可以基于 SFT 模型生成候选回答，人工标注偏好
    • 使用 TRL 的 DPOTrainer 完成偏好对齐训练
    • 对比 SFT 模型和 DPO 模型在主观质量（流畅性、有用性、安全性）上的差异
    • 调整 beta 参数（0.05、0.1、0.2），观察不同值对模型行为的影响
    • 思考：在你的业务场景中，哪些偏好是最重要的？如何设计偏好数据的标注指南？
