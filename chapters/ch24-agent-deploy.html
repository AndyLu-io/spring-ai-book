第 24 章

Agent 规模化部署

从"路边摊"到"连锁店" — 掌握 Kubernetes、Serverless、边缘部署等云原生技术，让 AI Agent 在生产环境中稳定运行。

⏱ 预计 3-4 周 📖 6 个知识点 💻 高级难度

📚 开篇故事：从路边摊到连锁店

想象你开了一家路边摊，卖的是你独家秘方的 AI 智能问答服务。每天只有几十个顾客，一台笔记本电脑就能搞定所有事情。你亲手处理每一个请求，就像厨师亲自掌勺 — 品质有保证，但规模上不去。

突然有一天，你的 AI 服务火了。顾客排起了长队，一台电脑根本处理不过来。你需要扩大规模：雇更多厨师（多实例部署）、租更大的店面（容器化）、开连锁店（多区域部署）、甚至在商场里设自动售货机（边缘部署）。

这就是 AI Agent 规模化部署的故事。从单机开发到云原生部署，从手动运维到自动伸缩，从单点服务到高可用架构 — 每一步都是对"路边摊"的升级。

本章将带你完成从"个体户"到"连锁企业"的蜕变，掌握 Kubernetes、Serverless、边缘计算等现代部署技术。

📋 Kubernetes 部署模式

为什么用 Kubernetes？

Kubernetes（K8s） 是容器编排的事实标准。对于 AI Agent 部署，它提供了：自动扩缩容（应对流量波动）、服务发现（多个 Agent 实例间通信）、滚动更新（零停机部署）。

比喻

K8s 就像一个集装箱码头：每个容器是一个标准化的集装箱，K8s 负责把这些集装箱运到正确的位置、监控它们的状态、在出问题时自动替换。

Kubernetes 部署架构：

  ┌─────────────────────────────────────────┐
  │            K8s Cluster                  │
  │                                         │
  │  ┌──────────┐  ┌──────────┐            │
  │  │ Agent   │  │ Agent   │  ...        │
  │  │ Pod 1    │  │ Pod 2    │            │
  │  └────┬─────┘  └────┬─────┘            │
  │       |            |                   │
  │       v            v                   │
  │  ┌─────────────────────────────────────┐│
  │  │        Service（负载均衡）           ││
  │  └─────────────────────┬───────────────┘│
  │                        |                │
  │                        v                │
  │  ┌─────────────────────────────────────┐│
  │  │        HPA（自动扩缩容）            ││
  │  └─────────────────────────────────────┘│
  └─────────────────────────────────────────┘

💡 要点小结
  • K8s 部署三件套：Deployment（管理 Pod）、Service（负载均衡）、HPA（自动扩缩容）
  • API Key 等敏感信息使用 Secret 管理
  • 配置健康检查（liveness + readiness）确保服务可用性
  • 设置合理的资源请求和限制避免资源竞争
⚠️ 陷阱警告

AI Agent 的内存使用波动大（LLM 调用可能消耗大量内存）。建议 limits.memory 设置为 requests.memory 的 2-3 倍，并启用 Pod Disruption Budget 确保升级时至少保持 2/3 的 Pod 可用。

☁️ Serverless 部署

什么是 Serverless？

Serverless 让你专注于代码，不需要管理服务器。按请求计费，空闲时不花钱。非常适合流量波动大的 AI Agent。

比喻

就像弹性工位 — 你不需要租一整间办公室，只需要"按需"使用工位。有人来开会就开一间，没人用就关掉，只为实际使用的资源付费。

/** * Serverless Agent 部署 * 基于 AWS Lambda + Spring Cloud Function */ // Spring Cloud Function 配置 @Bean public Function<AgentRequest, AgentResponse> agent( ChatClient chatClient) { return request -> { String reply = chatClient.prompt() .system(request.systemPrompt()) .user(request.message()) .options(ChatOptions.builder() .model(request.model()) .temperature(request.temperature()) .build()) .call() .content(); return new AgentResponse(reply, Instant.now()); }; } public record AgentRequest( String message, String systemPrompt, String model, double temperature ) {} public record AgentResponse( String reply, Instant timestamp ) {}

💡 Serverless 优化技巧
  • 使用 GraalVM Native Image 减少冷启动时间（从秒级到毫秒级）
  • 设置 minInstances=1 保持至少一个热实例（Cloud Run）
  • 将向量数据库连接池化，避免 Lambda 实例间连接管理问题
  • 使用 Provisioned Concurrency 预热关键路径

📡 边缘部署

什么是边缘部署？

边缘部署 将 AI 推理能力推送到离用户最近的位置，减少延迟、保护隐私、降低带宽成本。

比喻

就像便利店的"前哨站" — 你不需要去市中心的总店买东西，街角的便利店就能满足日常需求。核心商品在总店（云端），但常用商品在便利店（边缘）。

边缘部署架构：

  ┌─────────────────────────────────────────┐
  │            云端（总店）                  │
  │  ┌──────────┐  ┌──────────┐            │
  │  │ 大模型   │  │ 知识库   │            │
  │  │ (GPT-4)  │  │ (向量DB) │            │
  │  └────┬─────┘  └────┬─────┘            │
  └───────┼──────────────┼─────────────────┘
          |              |
          v              v
  ┌─────────────────────────────────────────┐
  │            边缘层（便利店）              │
  │  ┌──────────┐  ┌──────────┐            │
  │  │ 小模型   │  │ 本地缓存 │            │
  │  │ (7B量化) │  │ (Redis)  │            │

                        |
                        v

边缘推理实现

/** * 边缘 Agent：本地小模型 + 云端大模型混合 */ @Service public class EdgeAgentService { private final ChatClient cloudClient; // 云端大模型 private final LocalModelRunner localModel; // 本地小模型 private final RedisTemplate<String, String> cache; public EdgeAgentService(ChatClient cloudClient, LocalModelRunner localModel, RedisTemplate<String, String> cache) { this.cloudClient = cloudClient; this.localModel = localModel; this.cache = cache; } /** * 智能路由：根据任务复杂度选择模型 */ public String answer(String query) { // 1. 检查缓存 String cached = cache.opsForValue().get("cache:" + query); if (cached != null) { System.out.println("[Edge] 缓存命中"); return cached; } // 2. 用小模型评估任务复杂度 String complexity = localModel.classifyComplexity(query); String answer; if ("SIMPLE".equals(complexity)) { // 简单任务：本地小模型处理 System.out.println("[Edge] 使用本地小模型"); answer = localModel.generate(query); } else { // 复杂任务：云端大模型处理 System.out.println("[Edge] 使用云端大模型"); answer = cloudClient.prompt().user(query).call().content(); } // 3. 缓存结果 cache.opsForValue().set("cache:" + query, answer, Duration.ofMinutes(30)); return answer; } }
💡 要点小结
  • 边缘部署的核心：本地小模型 + 云端大模型混合
  • 使用 模型量化（INT4/INT8）减少本地模型大小
  • 智能路由根据任务复杂度选择模型
  • 本地缓存减少重复调用，降低延迟和成本
💡 你知道吗？
Spring AI 支持多种本地模型运行时（如 Ollama、ONNX Runtime），可以在边缘设备上运行小型语言模型（如 Phi-3、Gemma 2B），实现真正的端侧 AI。

🌐 混合部署策略

云端 + 边缘 + 本地

生产环境通常是混合部署：核心推理在云端，低延迟需求在边缘，隐私敏感数据在本地。

混合部署架构：

  ┌─────────────────────────────────────────────────┐
  │                    负载均衡器                    │
  └───────────────────────┬─────────────────────────┘
                          |
            ┌─────────────┼─────────────┐
            v             v             v
  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
  │  云端集群   │ │  边缘节点   │ │  本地服务   │
  │  (K8s)      │ │  (Cloud Run) │ │  (Docker)    │
  │             │ │              │ │              │
  │ - 复杂推理  │ │ - 低延迟响应│ │ - 隐私数据  │
  │ - 大模型    │ │ - 小模型    │ │ - 离线能力  │
  │ - 全局知识  │ │ - 本地缓存  │ │ - 本地推理  │
  └──────────────┘ └──────────────┘ └──────────────┘

混合部署配置

/** * 混合部署路由策略 */ @Service public class HybridDeploymentRouter { public enum DeployTarget { CLOUD, EDGE, LOCAL } /** * 根据请求特征路由到不同的部署目标 */ public DeployTarget route(AgentRequest request) { // 1. 隐私敏感数据 → 本地 if (request.containsSensitiveData()) { return DeployTarget.LOCAL; } // 2. 低延迟要求 → 边缘 if (request.maxLatencyMs() < 200) { return DeployTarget.EDGE; } // 3. 复杂任务 → 云端 if (request.complexity() == Complexity.HIGH) { return DeployTarget.CLOUD; } // 4. 默认：边缘 return DeployTarget.EDGE; } /** * 故障转移：主目标不可用时自动切换 */ public String executeWithFallback(AgentRequest request) { DeployTarget primary = route(request); try { return executeOn(request, primary); } catch (Exception e) { System.out.printf("[Hybrid] %s 不可用，尝试回退...%n", primary); // 回退策略：LOCAL -> EDGE -> CLOUD for (DeployTarget fallback : List.of( DeployTarget.EDGE, DeployTarget.CLOUD)) { if (fallback != primary) { try { return executeOn(request, fallback); } catch (Exception ignored) {} } } throw new RuntimeException("所有部署目标均不可用"); } } private String executeOn(AgentRequest request, DeployTarget target) { // 根据目标调用不同的服务 return switch (target) { case LOCAL -> localAgent.answer(request.message()); case EDGE -> edgeAgent.answer(request.message()); case CLOUD -> cloudAgent.answer(request.message()); }; } }
💡 要点小结
  • 混合部署的核心：智能路由 + 自动故障转移
  • 根据数据敏感度、延迟要求、任务复杂度选择部署目标
  • 回退策略确保服务连续性
  • 三种部署目标互补：云端强大、边缘快速、本地安全

🚀 弹性伸缩

自动扩缩容策略

AI Agent 的流量通常有明显的峰谷模式（工作日高峰、夜间低谷）。自动伸缩能确保在高峰期不崩溃、低谷期不浪费。

比喻

就像餐厅的弹性排班 — 午餐高峰期多雇几个服务员，下午茶时间减少人手，晚上打烊前只留值班人员。

# ========================================== # K8s HPA + VPA + CronHPA 组合配置 # ========================================== # 1. 基于 CPU 的自动扩缩容 apiVersion: autoscaling/v2 kind: HorizontalPodAutoscaler metadata: name: agent-cpu-hpa spec: scaleTargetRef: apiVersion: apps/v1 kind: Deployment name: ai-agent minReplicas: 2 maxReplicas: 20 metrics: - type: Resource resource: name: cpu target: type: Utilization averageUtilization: 70 --- # 2. 基于自定义指标（请求队列长度）的扩缩容 apiVersion: autoscaling/v2 kind: HorizontalPodAutoscaler metadata: name: agent-queue-hpa spec: scaleTargetRef: apiVersion: apps/v1 kind: Deployment name: ai-agent minReplicas: 2 maxReplicas: 30 metrics: - type: Pods pods: metric: name: http_requests_per_second target: type: AverageValue averageValue: "100" --- # 3. CronHPA：定时扩缩容 apiVersion: autoscaling/v2 kind: HorizontalPodAutoscaler metadata: name: agent-cron-hpa spec: scaleTargetRef: apiVersion: apps/v1 kind: Deployment name: ai-agent minReplicas: 3 maxReplicas: 15 metrics: - type: Resource resource: name: cpu target: type: Utilization averageUtilization: 60 
弹性伸缩策略组合：

  CPU HPA      ─── 应对突发流量（反应式）
  Queue HPA    ─── 根据实际负载伸缩（精确式）
  CronHPA      ─── 提前应对已知高峰（预测式）
  VPA          ─── 自动调整单 Pod 资源（优化式）

  三者结合 = 智能弹性伸缩
💡 要点小结
  • CPU HPA：基础扩缩容，应对突发流量
  • Queue HPA：基于实际请求量精确扩缩
  • CronHPA：定时扩缩容，提前应对已知高峰
  • 三种策略组合使用效果最佳

🛡️ 灾备与高可用

多区域部署架构

AI Agent 的高可用需要多区域部署：当一个区域出问题时，流量自动切换到其他区域。

比喻

就像安全网 — 你在高空走钢丝（单区域部署），下面有安全网（多区域备份）。即使失足（区域故障），安全网也能接住你。

多区域高可用架构：

  ┌─────────────────────────────────────────────┐
  │              全局负载均衡（GSLB）            │
  │         智能路由 + 健康检查 + 故障转移       │
  └────────────────┬────────────────┬────────────┘
                   |                |
          ┌────────┼────────┐      |
          v        v        v      v
  ┌────────────┐ ┌────────────┐ ┌────────────┐
  │ 区域 A    │ │ 区域 B    │ │ 区域 C    │
  │ (华东)     │ │ (华南)     │ │ (华北)     │
  │            │ │            │ │            │
  │ - 3 副本   │ │ - 3 副本   │ │ - 2 副本   │
  │ - 主数据库 │ │ - 从数据库 │ │ - 从数据库 │
  │ - 向量索引 │ │ - 向量索引 │ │ - 向量索引 │
  └────────────┘ └────────────┘ └────────────┘

故障转移配置

/** * 故障转移与健康检查 */ @Service public class HighAvailabilityService { private final List<AgentEndpoint> endpoints; private final HealthChecker healthChecker; public record AgentEndpoint( String region, String url, int weight, AtomicBoolean healthy ) {} /** * 带故障转移的请求执行 */ public String executeWithFailover(String query) { // 按权重排序，优先使用健康的端点 List<AgentEndpoint> sorted = endpoints.stream() .filter(e -> e.healthy().get()) .sorted(Comparator.comparingInt( AgentEndpoint::weight).reversed()) .toList(); for (AgentEndpoint endpoint : sorted) { try { String result = callEndpoint(endpoint, query); System.out.printf("[HA] 成功: %s%n", endpoint.region()); return result; } catch (Exception e) { System.out.printf("[HA] 失败: %s - %s%n", endpoint.region(), e.getMessage()); endpoint.healthy().set(false); } } throw new RuntimeException("所有区域均不可用"); } /** * 定期健康检查 */ @Scheduled(fixedRate = 30_000) // 每 30 秒检查一次 public void healthCheck() { for (AgentEndpoint endpoint : endpoints) { boolean healthy = healthChecker.check(endpoint.url()); endpoint.healthy().set(healthy); if (!healthy) { System.out.printf("[HA] 区域 %s 不健康%n", endpoint.region()); } } } }
💡 要点小结
  • 高可用的核心：多区域部署 + 自动故障转移
  • 使用 GSLB（全局服务器负载均衡）实现智能路由
  • 定期健康检查确保发现故障端点
  • 向量数据库需要跨区域同步（如 Weaviate 多区域集群）
⚠️ 陷阱警告

向量数据库的跨区域同步是最大的挑战。高维度向量的同步延迟和带宽成本都很高。建议使用最终一致性模型，接受秒级的同步延迟。

📖 章节小结

本章我们掌握了 AI Agent 的 6 种部署模式：

  • Kubernetes — 容器编排的标准方案，适合中大型部署
  • Serverless — 按需付费，适合流量波动大的场景
  • 边缘部署 — 低延迟 + 隐私保护，适合终端设备
  • 混合部署 — 云端 + 边缘 + 本地的智能组合
  • 弹性伸缩 — 多种 HPA 策略组合，智能应对流量波动
  • 灾备与高可用 — 多区域部署 + 自动故障转移

规模化部署就像从路边摊到连锁企业 — 每一步升级都需要更专业的运维能力和更完善的技术架构。

🎓 面试高频题

👨‍💻 Q1: K8s HPA 和 VPA 的区别是什么？

HPA（Horizontal Pod Autoscaler） 水平扩缩容，增加/减少 Pod 数量；VPA（Vertical Pod Autoscaler） 垂直扩缩容，调整单个 Pod 的 CPU/内存资源。HPA 适合无状态服务，VPA 适合有状态服务。

👨‍💻 Q2: Serverless 的冷启动问题如何解决？

1) 使用 GraalVM Native Image 编译为原生可执行文件；2) 设置 minInstances=1 保持热实例；3) 使用 Provisioned Concurrency 预热；4) 精简依赖减少初始化时间。

👨‍💻 Q3: 如何设计 AI Agent 的高可用架构？

1) 多区域部署（至少 2 个区域）；2) 全局负载均衡（智能路由 + 健康检查）；3) 自动故障转移（主区域故障时自动切换）；4) 数据同步（向量数据库跨区域复制）。

👨‍💻 Q4: 混合部署的路由策略如何设计？

根据三个维度路由：1) 数据敏感度（敏感数据→本地）；2) 延迟要求（低延迟→边缘）；3) 任务复杂度（复杂任务→云端）。加上自动故障转移确保服务连续性。
