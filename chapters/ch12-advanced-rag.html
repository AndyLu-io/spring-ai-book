第 12 章

高级 RAG 技术

从"能检索"到"会思考" — 掌握 HyDE、Self-RAG、CRAG、GraphRAG 等前沿技术，让 Agent 的检索能力产生质的飞跃。

⏱ 预计 3-4 周 📖 6 个知识点 💻 高级难度

📚 开篇故事：从普通图书馆到智能图书馆

想象你走进一座普通图书馆。你问管理员："我想找关于量子计算的书。"管理员翻开卡片目录，按关键词"量子计算"搜索，把所有包含这四个字的书搬给你 — 其中有些是量子计算专著，有些只是在附录里提了一句量子计算，甚至有些是"量子力学"的书被错误匹配了。

这就是基础 RAG — 基于关键词或简单向量相似度检索。它能工作，但远不够聪明。

现在想象走进一座智能图书馆。你还没开口，AI 助手已经根据你之前借阅的书、正在做的研究课题，推断出你可能想要的资料方向。当你问出问题时，它不会傻傻地按字面匹配，而是先在心里模拟一个理想的答案，然后去找最匹配这个答案的资料。找到资料后，它还会自我审查："这份资料够好吗？质量行不行？"如果不行，它会换个角度重新搜索。

这座智能图书馆，就是高级 RAG 技术的目标。从 HyDE 的预判答题、Self-RAG 的自我判断、CRAG 的自我纠错，到 GraphRAG 的关系网络和多模态的五感检索 — 每一项技术都在让 Agent 变得更"聪明"、更"自主"。

本章将带你从"普通图书馆管理员"进化为"智能图书馆架构师"。

💡 HyDE — Agent 的"预判答题"

什么是 HyDE？

HyDE（Hypothetical Document Embeddings） 的核心思想出人意料地简单：在检索之前，先让 LLM 生成一个假设性的答案，然后用这个假设答案的向量去检索真实文档。

比喻

就像一个学霸在考试前的"押题" — 他还没看到题目，但根据经验先猜一个可能的答案方向，然后带着这个方向去找对应的参考资料。因为"答案方向"比"问题本身"和"参考资料"在语义空间上更接近，所以检索准确率大幅提升。

传统 RAG：
  用户问题  ─────────────>  向量化  ─────────────>  检索  ─────────────>  LLM 生成答案
                                                                      ^
  问题和文档之间可能有语义鸿沟  ───────────────────────────────────────┘

HyDE RAG：
  用户问题  ─────────────>  LLM 生成假设答案  ─────────────>  向量化假设答案
                                                                    |
                                                              v
  LLM 生成最终答案  <──────────────────────────  检索真实文档
  ^                                                    |
  └──── 用检索结果 增强 ─────────────────────────────────┘

HyDE 完整实现

/** * HyDE（Hypothetical Document Embeddings）实现 * 原理：先用 LLM 生成假设答案，再用假设答案检索真实文档 */ @Service public class HyDERetriever { private final ChatClient chatClient; private final VectorStore vectorStore; public HyDERetriever(ChatClient chatClient, VectorStore vectorStore) { this.chatClient = chatClient; this.vectorStore = vectorStore; } /** * 第一步：生成假设答案 * 这个答案不需要正确，只需要在语义上接近真实答案 */ public String generateHypotheticalAnswer(String query) { return chatClient.prompt() .system(""" 你是一个专业的文档生成器。 根据用户的问题，生成一个假设性的、详细的回答文档。 这个文档不需要完全准确，但应该在语义上与真实答案接近。 请用专业、正式的语气撰写。 """) .user("请为以下问题生成一个假设性答案文档：" + query) .options(ChatOptions.builder() .temperature(0.7) // 适度的创造性 .maxTokens(500) .build()) .call() .content(); } /** * 第二步：用假设答案检索真实文档 */ public List<Document> retrieve(String query) { // 1. 生成假设答案 String hypotheticalAnswer = generateHypotheticalAnswer(query); // 2. 用假设答案的向量去检索 SearchRequest searchRequest = SearchRequest.query(hypotheticalAnswer) .withTopK(5); return vectorStore.similaritySearch(searchRequest); } /** * 第三步：用检索结果增强 LLM 生成最终答案 */ public String answerWithHyDE(String query) { // 检索相关文档 List<Document> documents = retrieve(query); // 构建增强 Prompt String context = documents.stream() .map(Document::getContent) .collect(Collectors.joining("\n\n---\n\n")); return chatClient.prompt() .system("你是一个专业的知识助手。请基于以下参考文档回答问题。") .user("参考文档：\n" + context + "\n\n问题：" + query) .call() .content(); } }
💡 要点小结
  • HyDE 的关键洞察：假设答案的向量比问题的向量更接近真实答案
  • 假设答案不需要正确，只需要在语义空间上"接近"真实文档
  • HyDE 特别适合语义鸿沟大的场景（如技术术语 vs 通俗描述）
  • 代价：多一次 LLM 调用，增加延迟和成本
⚠️ 陷阱警告

HyDE 的假设答案如果偏离太远，反而会降低检索质量。建议 temperature 设在 0.5-0.7 之间，并限制假设答案长度在 300-500 tokens。

🧠 Self-RAG — Agent 的"自我判断"

什么是 Self-RAG？

Self-RAG（Self-Reflective RAG） 让 Agent 自主决定三件事：是否需要检索、检索结果是否有用、生成的答案是否可靠。它通过特殊的"反思标记"（Reflection Tokens）来实现自我评估。

比喻

就像一个经验丰富的研究员：面对一个问题时，他不会每次都跑去图书馆（有些问题他自己就知道答案）；找到一本书后，他会快速翻阅判断是否值得细读；读完后，他还会自问"我确定这个答案是对的吗？"。

Self-RAG 决策循环：

  用户提问
      |
      v
  [判断 1] 需要检索吗？
      |
      +---> 不需要 ──> 直接生成答案 ──> [判断 3] 答案可靠吗？
      |                                             |
      v                                             +---> 可靠 ──> 输出答案
  检索文档                                            |
      |                                             v
      v                                          不可靠 ──> 重新检索或补充信息
  [判断 2] 检索结果有用吗？
      |
      +---> 有用 ──> 基于文档生成答案 ──> [判断 3]
      |
      v
  无用 ──> 换个 query 重新检索

Self-RAG 决策循环实现

/** * Self-RAG：自适应检索决策 * Agent 自主判断是否需要检索、检索质量、答案可靠性 */ @Service public class SelfRAGEngine { private final ChatClient chatClient; private final VectorStore vectorStore; // 检索判断阈值 private static final double RETRIEVAL_THRESHOLD = 0.6; private static final double RELEVANCE_THRESHOLD = 0.7; private static final int MAX_RETRY = 3; public SelfRAGEngine(ChatClient chatClient, VectorStore vectorStore) { this.chatClient = chatClient; this.vectorStore = vectorStore; } /** * 判断是否需要检索（IsRelrelevant） */ private boolean shouldRetrieve(String query) { String response = chatClient.prompt() .system(""" 判断以下问题是否需要检索外部文档来回答。 只回答 "YES" 或 "NO"。 判断标准： - 事实性问题、专业知识问题 → YES - 简单问候、闲聊、常识问题 → NO """) .user(query) .options(ChatOptions.builder().temperature(0.0).build()) .call() .content(); return response.trim().toUpperCase().startsWith("YES"); } /** * 评估检索结果的相关性（IsUseful） */ private double assessRelevance(String query, Document doc) { String response = chatClient.prompt() .system("评估文档与问题的相关性，给出 0-1 的分数。只输出数字。") .user("问题：" + query + "\n文档片段：" + doc.getContent()) .options(ChatOptions.builder().temperature(0.0).maxTokens(10).build()) .call() .content(); try { return Double.parseDouble(response.trim()); } catch (NumberFormatException e) { return 0.0; } } /** * 评估答案的可靠性（IsSupported） */ private boolean isAnswerSupported(String query, String answer, List<Document> docs) { String docContext = docs.stream() .map(Document::getContent) .collect(Collectors.joining("\n")); String response = chatClient.prompt() .system(""" 判断以下答案是否被参考文档支持。 只回答 "SUPPORTED" 或 "UNSUPPORTED"。 """) .user("问题：" + query + "\n答案：" + answer + "\n参考文档：" + docContext) .options(ChatOptions.builder().temperature(0.0).build()) .call() .content(); return response.trim().toUpperCase().startsWith("SUPPORTED"); } /** * Self-RAG 主循环 */ public String answer(String query) { // Step 1: 判断是否需要检索 if (!shouldRetrieve(query)) { String directAnswer = chatClient.prompt().user(query).call().content(); System.out.println("[Self-RAG] 跳过检索，直接回答"); return directAnswer; } // Step 2: 检索 + 评估 for (int retry = 0; retry < MAX_RETRY; retry++) { List<Document> docs = vectorStore.similaritySearch(query); // 过滤低相关性文档 List<Document> relevantDocs = docs.stream() .filter(doc -> assessRelevance(query, doc) >= RELEVANCE_THRESHOLD) .toList(); if (relevantDocs.isEmpty()) { System.out.println("[Self-RAG] 重试 #" + (retry + 1) + ": 无相关文档"); continue; } // Step 3: 生成答案并验证 String answer = generateAnswer(query, relevantDocs); if (isAnswerSupported(query, answer, relevantDocs)) { System.out.println("[Self-RAG] 答案可靠，输出结果"); return answer; } System.out.println("[Self-RAG] 答案不可靠，重试 #" + (retry + 1)); } return "抱歉，无法找到可靠的答案。"; } private String generateAnswer(String query, List<Document> docs) { String context = docs.stream() .map(Document::getContent) .collect(Collectors.joining("\n---\n")); return chatClient.prompt() .system("基于参考文档回答问题。") .user("文档：\n" + context + "\n问题：" + query) .call() .content(); } }
💡 要点小结
  • Self-RAG 的三个关键判断：是否检索、结果有用、答案可靠
  • 对于简单问题，跳过检索直接回答，节省延迟和成本
  • 通过多次重试和验证，确保输出质量
  • 适用于需要高准确度的场景（如医疗、法律咨询）
💡 你知道吗？
Self-RAG 论文（2023，Akari Asai 等）提出了一种通用的框架，通过训练特殊的反思标记（[Retrieve]、[IsRellevant]、[IsUseful]、[IsSupported]）来实现自适应检索。我们这里的实现是其思想的工程化简化版本。

🔄 CRAG — Agent 的"自我纠错"

什么是 CRAG？

CRAG（Corrective RAG） 在检索后增加一个知识精炼步骤：评估检索到的文档质量，如果质量不够，自动触发重新检索或知识精炼。

比喻

就像一个严格的编辑：收到一篇稿件后，先审查引用的参考资料质量。如果参考资料不可靠（"Correct"），直接使用；如果部分可靠（"Ambiguous"），提取有用部分并补充新资料；如果完全不靠谱（"Incorrect"），扔掉重来。

CRAG 流程：

  用户问题
      |
      v
  检索文档
      |
      v
  知识精炼器（Retriever Evaluator）
      |
      +---> Correct ──> 直接使用 ──> LLM 生成答案
      |
      +---> Ambiguous ──> 提取有用部分 + Web 搜索补充 ──> LLM 生成答案
      |
      v
  Incorrect ──> 丢弃所有文档 ──> Web 搜索新文档 ──> LLM 生成答案

CRAG 完整实现

/** * CRAG（Corrective RAG）实现 * 检索后评估质量，不满意则重新检索或精炼 */ @Service public class CRAGEngine { private final ChatClient chatClient; private final VectorStore vectorStore; private final WebSearchService webSearchService; // Web 搜索服务 private static final double CORRECT_THRESHOLD = 0.8; private static final double AMBIGUOUS_THRESHOLD = 0.4; public CRAGEngine(ChatClient chatClient, VectorStore vectorStore, WebSearchService webSearchService) { this.chatClient = chatClient; this.vectorStore = vectorStore; this.webSearchService = webSearchService; } /** * 评估检索文档的质量 * 返回：Correct / Ambiguous / Incorrect */ private String evaluateRetrieval(String query, List<Document> docs) { String docContent = docs.stream() .map(Document::getContent) .collect(Collectors.joining("\n---\n")); String evaluation = chatClient.prompt() .system(""" 评估以下检索结果对回答问题的质量。 返回 JSON：{"score": 0.0-1.0, "reason": "评估理由"} - score >= 0.8: Correct（文档质量高，可直接使用） - 0.4 <= score < 0.8: Ambiguous（部分有用，需要精炼） - score < 0.4: Incorrect（文档质量低，需要重新检索） """) .user("问题：" + query + "\n检索结果：\n" + docContent) .options(ChatOptions.builder().temperature(0.0).build()) .call() .content(); // 解析分数 double score = parseScore(evaluation); if (score >= CORRECT_THRESHOLD) return "CORRECT"; if (score >= AMBIGUOUS_THRESHOLD) return "AMBIGUOUS"; return "INCORRECT"; } /** * 知识精炼：提取有用部分并补充新信息 */ private String refineKnowledge(String query, List<Document> docs) { String docContent = docs.stream() .map(Document::getContent) .collect(Collectors.joining("\n---\n")); // Step 1: 提取有用信息 String refined = chatClient.prompt() .system("从以下文档中提取与问题相关的有用信息，去除无关内容。") .user("问题：" + query + "\n文档：\n" + docContent) .call() .content(); // Step 2: Web 搜索补充 String webResults = webSearchService.search(query); return "## 从知识库提取的信息：\n" + refined + "\n\n## 从互联网补充的信息：\n" + webResults; } /** * CRAG 主流程 */ public String answer(String query) { // 1. 检索 List<Document> docs = vectorStore.similaritySearch(query); // 2. 评估质量 String quality = evaluateRetrieval(query, docs); System.out.println("[CRAG] 检索质量评估: " + quality); // 3. 根据质量决定处理策略 String context = switch (quality) { case "CORRECT" -> docs.stream() .map(Document::getContent) .collect(Collectors.joining("\n\n")); case "AMBIGUOUS" -> refineKnowledge(query, docs); case "INCORRECT" -> webSearchService.search(query); default -> ""; }; // 4. 生成最终答案 return chatClient.prompt() .system("基于以下参考信息回答问题。") .user("参考信息：\n" + context + "\n\n问题：" + query) .call() .content(); } private double parseScore(String json) { // 简化的 JSON 分数提取 int idx = json.indexOf("score"); if (idx == -1) return 0.0; String sub = json.substring(idx); int start = sub.indexOf(":") + 1; int end = sub.indexOf(","); return Double.parseDouble(sub.substring(start, end).trim()); } }
💡 要点小结
  • CRAG 的三级质量评估：Correct（直接用）、Ambiguous（精炼+补充）、Incorrect（重新检索）
  • 引入 Web 搜索作为兜底检索源
  • 知识精炼步骤能去除噪声，保留核心信息
  • 适合对答案质量要求极高的生产环境
⚠️ 陷阱警告

CRAG 的质量评估本身依赖 LLM 调用，可能出现评估不准确的情况。建议在生产环境中结合规则过滤（如文档来源可信度、时效性）来辅助判断。

🌐 GraphRAG — Agent 的"关系网络"

什么是 GraphRAG？

GraphRAG（Graph-based RAG） 将知识图谱与向量检索融合，利用实体关系来增强检索效果。传统的向量检索擅长找"语义相似"的内容，但不擅长找"关联关系"。

比喻

向量检索就像按主题找书：找"人工智能"的书，会找到所有关于 AI 的书。但如果你想知道"深度学习之父 Hinton 的学生们现在在哪家公司"，向量检索就懵了。GraphRAG 就像关系网络：它知道 Hinton → 学生 → Ilya Sutskever → OpenAI，能沿着关系链找到答案。

GraphRAG 架构：

  用户问题："Hinton 的学生在哪家公司？"
      |
      v
  ┌─────────────────────────────────────────┐
  │  实体识别：Hinton, 学生, 公司          │
  │  |                                     │
  │  v                                     │
  │  知识图谱查询：                         │
  │  Hinton --学生--> Ilya Sutskever        │
  │  Ilya --联合创始人--> OpenAI            │
  │  Hinton --学生--> Yann LeCun           │
  │  Yann --首席科学家--> Meta              │
  │  |                                     │
  │  v                                     │
  │  向量检索补充：相关背景文档             │
  └─────────────────────────────────────────┘
      |
      v
  LLM 综合生成答案

GraphRAG 实现

/** * GraphRAG：知识图谱 + 向量检索融合 * 利用实体关系增强检索效果 */ @Service public class GraphRAGEngine { private final ChatClient chatClient; private final VectorStore vectorStore; private final KnowledgeGraphStore graphStore; // 知识图谱存储 public GraphRAGEngine(ChatClient chatClient, VectorStore vectorStore, KnowledgeGraphStore graphStore) { this.chatClient = chatClient; this.vectorStore = vectorStore; this.graphStore = graphStore; } /** * 从问题中提取实体 */ private List<String> extractEntities(String query) { String response = chatClient.prompt() .system("从问题中提取关键实体（人名、公司名、技术名等）。以 JSON 数组格式输出。") .user(query) .options(ChatOptions.builder().temperature(0.0).build()) .call() .content(); // 解析 JSON 数组 return List.of(response.replaceAll("[\\[\\]\"\\s]", "").split(",")); } /** * 在知识图谱中查询实体关系 */ private String queryGraph(List<String> entities) { StringBuilder graphContext = new StringBuilder(); for (String entity : entities) { // 查询实体的直接关系 List<Triple<String, String, String>> relations = graphStore.findRelations(entity); for (var rel : relations) { graphContext.append(rel.getLeft()) .append(" --[").append(rel.getMiddle()).append("]--> ") .append(rel.getRight()).append("\n"); } } return graphContext.toString(); } /** * GraphRAG 主流程 */ public String answer(String query) { // 1. 提取实体 List<String> entities = extractEntities(query); // 2. 查询知识图谱 String graphContext = queryGraph(entities); // 3. 向量检索补充 List<Document> vectorDocs = vectorStore.similaritySearch(query); String vectorContext = vectorDocs.stream() .map(Document::getContent) .collect(Collectors.joining("\n---\n")); // 4. 融合生成答案 return chatClient.prompt() .system(""" 你是知识图谱增强的 AI 助手。 综合知识图谱关系和向量检索结果回答问题。 优先使用知识图谱中的结构化关系信息。 """) .user("知识图谱关系：\n" + graphContext + "\n\n参考文档：\n" + vectorContext + "\n\n问题：" + query) .call() .content(); } }
💡 要点小结
  • GraphRAG 的核心优势：实体关系推理（向量检索做不到的事）
  • 两种检索方式互补：向量检索找语义相似，图谱检索找关系链
  • 需要预先构建知识图谱（可以用 LLM 自动从文档中抽取）
  • 适合多跳推理场景（"A 的老板的公司做什么？"）

Microsoft 的 GraphRAG 项目（2024）使用 LLM 自动从文本中构建知识图谱，然后利用社区检测算法来回答需要全局理解的问题。这使得 RAG 系统不仅能回答"关于 X 的细节"，还能回答"整个数据集的全局主题是什么"。

👁 多模态 RAG — Agent 的"五感检索"

什么是多模态 RAG？

多模态 RAG 扩展了传统文本检索的边界，支持图片、表格、代码、PDF等多种格式的混合检索。现实世界的信息不只是纯文本。

比喻

就像人类的五感 — 你不仅能读文字（视觉），还能看图片（视觉增强）、听声音（听觉）。多模态 RAG 让 Agent 也能"看"图片、"读"表格、"理解"代码，而不仅仅是处理纯文本。

多模态 RAG 实现

/** * 多模态 RAG：支持文本、图片、表格的混合检索 */ @Service public class MultimodalRAGEngine { private final ChatClient chatClient; private final VectorStore vectorStore; /** * 处理图片：用 Vision Model 生成描述文本 */ public String processImage(String imageUrl) { return chatClient.prompt() .user(UserMessage.builder() .media(MimeTypeUtils.IMAGE_PNG, new URL(imageUrl)) .text("请详细描述这张图片的内容，包括图表类型、数据趋势、关键发现。") .build()) .call() .content(); } /** * 处理表格：转换为结构化 Markdown */ public String processTable(String tableData) { // 将 CSV/JSON 表格转换为 Markdown 格式 return chatClient.prompt() .system("将以下表格数据转换为易读的 Markdown 表格，并总结关键数据趋势。") .user(tableData) .call() .content(); } /** * 处理代码：生成摘要和文档 */ public String processCode(String code, String language) { return chatClient.prompt() .system(""" 分析以下代码，生成： 1. 功能摘要（一句话） 2. 主要方法/函数列表 3. 关键技术点 代码语言：""" + language) .user(code) .call() .content(); } /** * 多模态文档索引 */ public void indexDocument(MultimodalDocument doc) { switch (doc.type()) { case TEXT -> vectorStore.add(List.of( new Document(doc.content(), doc.metadata()))); case IMAGE -> { String desc = processImage(doc.url()); vectorStore.add(List.of( new Document(desc, doc.metadata()))); } case TABLE -> { String structured = processTable(doc.content()); vectorStore.add(List.of( new Document(structured, doc.metadata()))); } case CODE -> { String summary = processCode(doc.content(), doc.language()); vectorStore.add(List.of( new Document(summary, doc.metadata()))); } } } /** * 多模态检索 */ public List<Document> search(String query, MediaType preferType) { SearchRequest request = SearchRequest.query(query) .withTopK(5); return vectorStore.similaritySearch(request); } } // 多模态文档类型 public enum DocType { TEXT, IMAGE, TABLE, CODE } // 多模态文档 public record MultimodalDocument( DocType type, String content, String url, // 图片 URL String language, // 代码语言 Map<String, Object> metadata ) {}
💡 要点小结
  • 多模态 RAG 的核心：将所有模态转换为文本表示后统一索引
  • 图片用 Vision Model 生成描述，表格转 Markdown，代码生成摘要
  • 统一向量存储简化了检索流程
  • 元数据可用于按类型过滤检索结果
⚠️ 陷阱警告

Vision Model 的描述可能遗漏图片中的细节（如精确数值）。对于关键数据，建议同时保留原始数据的结构化表示（如 JSON），并将其与描述文本一起索引。

📈 RAG 评估框架

为什么需要评估？

没有评估的 RAG 系统就像没有考试的学生 — 你不知道它到底学得好不好。RAGAS（Retrieval Augmented Generation Assessment）提供了标准化的评估指标。

指标评估内容生活比喻
Faithfulness答案是否忠于检索到的文档"引用是否真实？"
Answer Relevancy答案是否与问题相关"答非所问了吗？"
Context Precision检索到的文档中相关文档的比例"找的书对路吗？"
Context Recall是否检索到了所有相关文档"漏掉关键资料了吗？"

构建 RAG 评估流水线

/** * RAG 评估框架 * 使用 RAGAS 指标评估 RAG 系统质量 */ @Service public class RAGEvaluationService { private final ChatClient chatClient; private final CRAGEngine ragEngine; public RAGEvaluationService(ChatClient chatClient, CRAGEngine ragEngine) { this.chatClient = chatClient; this.ragEngine = ragEngine; } /** * 评估 Faithfulness（答案忠于文档的程度） */ public double evaluateFaithfulness(String question, String answer, String context) { String response = chatClient.prompt() .system(""" 评估答案是否忠于提供的上下文。 分解答案为独立声明，判断每个声明是否被上下文支持。 输出 JSON：{"score": 0.0-1.0, "unsupported_claims": [...]} """) .user("问题：" + question + "\n答案：" + answer + "\n上下文：" + context) .options(ChatOptions.builder().temperature(0.0).build()) .call() .content(); return parseScore(response); } /** * 评估 Answer Relevancy（答案相关性） */ public double evaluateRelevancy(String question, String answer) { // 用答案反向生成可能的问题 String generatedQuestions = chatClient.prompt() .system("根据以下答案，生成 3 个可能被回答的问题。") .user(answer) .call() .content(); // 计算原始问题与生成问题的语义相似度 double similarity = calculateSimilarity(question, generatedQuestions); return similarity; } /** * 完整评估流水线 */ public EvaluationResult evaluate(List<EvalCase> testCases) { List<Double> faithfulnessScores = new ArrayList<>(); List<Double> relevancyScores = new ArrayList<>(); for (EvalCase testCase : testCases) { String answer = ragEngine.answer(testCase.question()); double faithfulness = evaluateFaithfulness( testCase.question(), answer, testCase.expectedContext()); double relevancy = evaluateRelevancy( testCase.question(), answer); faithfulnessScores.add(faithfulness); relevancyScores.add(relevancy); System.out.printf("[评估] Q: %s%n", testCase.question()); System.out.printf(" Faithfulness: %.2f%n", faithfulness); System.out.printf(" Relevancy: %.2f%n", relevancy); } return new EvaluationResult( avg(faithfulnessScores), avg(relevancyScores) ); } private double avg(List<Double> scores) { return scores.stream() .mapToDouble(Double::doubleValue) .average().orElse(0.0); } private double parseScore(String json) { int idx = json.indexOf("score"); if (idx == -1) return 0.0; String sub = json.substring(idx); int start = sub.indexOf(":") + 1; int end = sub.indexOf(","); if (end == -1) end = sub.indexOf("}"); return Double.parseDouble(sub.substring(start, end).trim()); } private double calculateSimilarity(String text1, String text2) { // 使用嵌入模型计算余弦相似度 // 这里简化为 LLM 判断 String response = chatClient.prompt() .system("评估以下两段文本的语义相关性，0-1 分。只输出数字。") .user("文本1：" + text1 + "\n文本2：" + text2) .options(ChatOptions.builder().temperature(0.0).maxTokens(10).build()) .call() .content(); try { return Double.parseDouble(response.trim()); } catch (NumberFormatException e) { return 0.0; } } } // 评估数据结构 public record EvalCase( String question, String expectedAnswer, String expectedContext ) {} public record EvaluationResult( double avgFaithfulness, double avgRelevancy ) {}
💡 要点小结
  • RAG 评估四大指标：Faithfulness、Relevancy、Precision、Recall
  • 评估流水线应该在CI/CD 中自动化运行
  • 使用 golden dataset（标准测试集）确保评估的可重复性
  • 定期评估以监控 RAG 系统的退化

📖 章节小结

本章我们掌握了 6 项高级 RAG 技术：

  • HyDE — 先生成假设答案再检索，缩小语义鸿沟
  • Self-RAG — Agent 自主判断是否检索、结果是否有用、答案是否可靠
  • CRAG — 检索后质量评估，不满意则精炼或重新检索
  • GraphRAG — 知识图谱 + 向量检索融合，支持关系推理
  • 多模态 RAG — 图片、表格、代码的统一检索
  • RAG 评估 — RAGAS 指标体系，确保系统质量

这些技术就像给 Agent 装上了"智慧大脑" — 从盲目检索到精准检索，从被动回答到主动判断。

🎓 面试高频题

👨‍💻 Q1: HyDE 和传统 RAG 的核心区别是什么？

传统 RAG 用问题的向量直接检索，而 HyDE 先让 LLM 生成假设答案，用假设答案的向量检索。假设答案在语义空间上比问题更接近真实答案，因为它们共享相似的文档风格和术语。

👨‍💻 Q2: Self-RAG 的三个反思标记分别是什么？

[Retrieve] 判断是否需要检索；[IsRellevant] 评估检索结果的相关性；[IsSupported] 验证生成答案是否被文档支持。这三个标记让 Agent 能自主决定检索策略。

👨‍💻 Q3: GraphRAG 在什么场景下比传统 RAG 更有优势？

在需要多跳推理的场景（如"A 的导师在哪里工作？"）、关系查询（如"和 X 有合作关系的人有哪些？"）、以及需要全局理解的场景（如"这些论文的整体研究趋势是什么？"）。

👨‍💻 Q4: 如何评估一个 RAG 系统的质量？

使用 RAGAS 框架的四大指标：Faithfulness（答案忠于文档）、Answer Relevancy（答案相关性）、Context Precision（检索精度）、Context Recall（检索召回率）。需要构建标准测试集并定期评估。
