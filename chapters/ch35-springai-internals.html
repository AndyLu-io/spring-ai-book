Spring AI 内部原理与源码剖析

拆开手表看机芯 -- 深入 Spring AI 核心类的内部实现机制，从架构设计到源码级分析，让你成为真正的 Spring AI 专家。

⏱ 预计 3-4 天 📖 5 个核心模块 💻 高级难度 📦 基于 Spring AI 1.0.0

📚 开篇故事：拆开手表看机芯

你有没有拆开过一块机械手表？当你拧下表壳、揭开表盘，看到的不是混乱的零件堆，而是一台精密的微型机器 -- 每个齿轮咬合的角度、每根游丝振动的频率、每个宝石轴承的定位，都经过深思熟虑。正是这些你平时看不见的内部结构，决定了手表走时的精准与持久。

Spring AI 就是这样一块"精密手表"。我们在前面 30 章里学会了"调时间"（使用 API）、"换表带"（配置参数），但如果你要造出自己的手表，或者在手表出故障时能自己修，你就必须理解机芯的构造。

本章我们将扮演"钟表师"的角色，用螺丝刀一层层拆解 Spring AI 的核心类：ChatClient 如何构建调用链、ChatModel 如何与 LLM 通信、ToolCallback 如何发现和执行工具、VectorStore 如何管理向量索引。理解了这些"机芯零件"，你就能自定义组件、排查深层问题、做出架构级优化。

📄 三层职责详解

层次核心包职责设计原则
API 层spring-ai-model定义接口契约，如 ChatModel、EmbeddingModel与具体 LLM 无关，纯接口定义
实现层spring-ai-openai 等实现接口，对接具体 LLM API一个模块对应一个 LLM 提供商
基础设施层spring-boot-starterHTTP 通信、JSON 序列化、自动配置复用 Spring Boot 生态能力

关键设计思想：API 层只定义"做什么"，实现层决定"怎么做"。当你切换 OpenAI 为 Ollama 时，只需更换一个 starter 依赖，上层代码零修改。这就是依赖倒置原则（DIP）的经典实践。

📄 核心设计模式

1. Builder 模式 -- ChatClient 的构建

ChatClient 不直接 new，而是通过 Builder 逐步配置。这解决了"构造函数参数爆炸"的问题：

// ChatClient.Builder 的工作方式 ChatClient client = ChatClient.builder(chatModel) .defaultSystem("你是一个助手") // 链式配置系统提示词 .defaultAdvisors(retrievalAdv) // 链式配置默认Advisor .defaultFunctions(weatherTool) // 链式配置默认工具 .build(); // 构建不可变实例

2. Strategy 模式 -- ChatModel 的多实现

ChatModel 是一个策略接口，不同的 LLM 提供商各自实现。Spring 通过自动配置选择当前激活的策略：

// 策略模式：同一个接口，不同的实现 public interface ChatModel { ChatResponse call(ChatRequest request); Flux<ChatResponse> stream(ChatRequest request); } // 实现1：对接 OpenAI public class OpenAiChatModel implements ChatModel { ... } // 实现2：对接 Ollama public class OllamaChatModel implements ChatModel { ... } // 实现3：对接 Anthropic public class AnthropicChatModel implements ChatModel { ... }

3. Adapter 模式 -- 工具函数的适配

你的 Java 方法签名千差万别，但 LLM 期望的标准工具格式是固定的 JSON Schema。ToolCallback 就是中间的适配器：

你的 Java 方法          ToolCallback 适配器        LLM 期望的工具格式
─────────────        ──────────────────         ─────────────────
String               →  提取参数类型        →   {
  getWeather(           →  生成 JSON Schema  →     "name": "getWeather",
    String city,        →  映射返回值        →     "parameters": {
    int days            →                     →       "city": {"type":"string"},
  )                     →                     →       "days": {"type":"integer"}
                                              →     }
                                              →   }

4. 责任链模式 -- Advisor 链

每个 Advisor 就像流水线上的一个工位，请求经过所有 Advisor 的"加工"后才到达 ChatModel：

用户请求
  │
  ▼
Advisor[0].aroundCall() ── before ──→ 校验输入
  │
  ▼
Advisor[1].aroundCall() ── before ──→ 注入上下文
  │
  ▼
ChatModel.call()         ── 执行 LLM 调用
  │
  ▼
Advisor[1].aroundCall() ── after ─── 处理响应
  │
  ▼
Advisor[0].aroundCall() ── after ─── 日志记录
  │
  ▼
最终响应返回给用户

📄 依赖注入与自动配置

Spring AI 大量使用 Spring Boot 的自动配置机制。当你引入 spring-ai-openai-spring-boot-starter 时，OpenAiAutoConfiguration 会自动创建所有必要的 Bean：

// OpenAiAutoConfiguration 的核心逻辑（简化版） @AutoConfiguration @ConditionalOnClass(OpenAiChatModel.class) @EnableConfigurationProperties(OpenAiProperties.class) public class OpenAiAutoConfiguration { @Bean @ConditionalOnMissingBean public OpenAiChatModel chatModel(OpenAiProperties properties, ObjectProvider<RestClient.Builder> builder) { // 从 application.yml 读取配置 // 创建 RestClient 用于 HTTP 通信 // 注入重试模板和拦截器 return new OpenAiChatModel( builder.getIfAvailable(), properties.getApi(), properties.getChat(), /* metadata... */ ); } @Bean @ConditionalOnMissingBean public ChatClient.Builder chatClientBuilder(OpenAiChatModel model) { // 自动用上面创建的 ChatModel 构建 ChatClient return new ChatClient.Builder(model); } }
✔ 要点小结

Spring AI 采用 API 层 / 实现层 / 基础设施层三层架构，通过 Builder、Strategy、Adapter、责任链四种核心设计模式实现高内聚低耦合。自动配置让你"零代码"完成 Bean 注册，而 @ConditionalOnMissingBean 保证你总能覆盖默认实现。

📄 ChatClient.Builder 的构建过程

ChatClient 的创建遵循不可变 Builder 模式。关键点在于 default 前缀的配置会被所有后续调用继承：

// ============================================ // ChatClient.Builder 内部结构（简化源码） // ============================================ public class ChatClient { private final ChatModel chatModel; private final List<ChatClientRequestListener> listeners; private final Map<String, ChatClientResponseFilter> filters; private final List<Advisor> advisors; // Advisor 链 private final Map<String, ToolCallback> toolCallbacks; // 工具注册表 public static class Builder { private final ChatModel chatModel; private String defaultSystemText; private final List<Advisor> defaultAdvisors = new ArrayList<>(); private final Map<String, ToolCallback> defaultToolCallbacks = new HashMap<>(); public Builder(ChatModel chatModel) { this.chatModel = chatModel; // 必须提供 ChatModel } // default 前缀：这些配置被所有后续调用共享 public Builder defaultSystem(String text) { this.defaultSystemText = text; return this; } public Builder defaultAdvisors(Advisor... advisors) { this.defaultAdvisors.addAll(Arrays.asList(advisors)); return this; } public ChatClient build() { // 构建不可变的 ChatClient 实例 return new ChatClient(this); } } }
⚠️ 关键区分

defaultAdvisors() 设置的 Advisor 会被所有 prompt 调用共享（如日志、记忆管理）。而 .advisors() 在 prompt 链式调用中设置的 Advisor 只对当前调用生效。理解这个区别，是掌握 Advisor 机制的关键。

📄 流式响应处理机制

当调用 .stream() 时，ChatClient 返回 Flux<ChatResponse>。内部通过 SSE（Server-Sent Events）逐块接收数据：

  .call() 流程（同步）                    .stream() 流程（异步流式）
  ─────────────────                    ──────────────────────
  ┌──────────┐                        ┌──────────┐
  │ 发送请求  │                        │ 发送请求  │
  └────┬─────┘                        └────┬─────┘
       │ 等待...                            │
       │ 等待...                         ┌──▼──────────────┐
       │ 等待...                         │ SSE 事件流       │
  ┌────▼─────┐                        │                  │
  │ 收到完整  │                        │ data: {"delta"   │
  │ 响应      │                        │   "你好"} ───────│──→ 发射 chunk
  └──────────┘                        │ data: {"delta"   │
                                      │   "世界"} ───────│──→ 发射 chunk
                                      │ data: [DONE]     │
                                      │   ────────────── │──→ 完成信号
                                      └──────────────────┘

在底层，Spring AI 使用 WebFlux 的 Flux 或 RestClient 的 SSE 解析来处理流式响应。每个 SSE 事件被解析为一个 ChatResponse delta，通过 .map() 操作符逐个发射到下游。

📄 Advisor 链的执行原理

Advisor 链的核心是一个递归包装器。每个 Advisor 的 aroundCall() 方法接收一个 CallAdvisorChain 参数，调用 chain.nextCall() 就会触发下一个 Advisor：

// ============================================ // AdvisorChain 的递归执行机制（简化源码） // ============================================ class DefaultCallAdvisorChain implements CallAdvisorChain { private final List<Advisor> advisors; private int current index = 0; private final ChatModel chatModel; // 最终的执行者 @Override public ChatResponse nextCall(ChatClientRequest request) { if (currentIndex >= advisors.size()) { // 所有 Advisor 都执行完了，最终调用 ChatModel return chatModel.call(request.toChatRequest()); } Advisor currentAdvisor = advisors.get(currentIndex); currentIndex++; // 递归：当前 Advisor 的 aroundCall 内部会再次调用 nextCall return currentAdvisor.aroundCall(request, this); } } // Advisor 的典型实现 class MessageChatMemoryAdvisor implements CallAdvisor { @Override public ChatClientResponse aroundCall( ChatClientRequest request, CallAdvisorChain chain) { // before: 从记忆存储中加载历史消息 List<Message> memory = loadMemory(request); ChatClientRequest enriched = request.withMessages(memory); // 传递给下一个 Advisor / ChatModel ChatClientResponse response = chain.nextCall(enriched); // after: 将本次对话存入记忆 saveToMemory(request, response); return response; } }
✔ 要点小结

ChatClient 通过 Builder 模式组装配置，通过递归的 Advisor 链实现请求拦截，最终委托 ChatModel 执行 LLM 调用。default 前缀的配置全局共享，非 default 配置仅当前调用生效。流式响应通过 SSE + Flux 实现逐块传输。

📄 ChatModel 接口设计

ChatModel 的接口极其简洁，只有两个方法 -- call() 同步调用和 stream() 流式调用：

// ============================================ // ChatModel 接口定义（Spring AI 源码） // ============================================ public interface ChatModel extends Model<ChatRequest, ChatResponse> { /** * 同步调用 LLM * @param chatRequest 包含 messages、options、tools 的请求 * @return 包含 generatedMessage、usage、metadata 的响应 */ ChatResponse call(ChatRequest chatRequest); /** * 流式调用 LLM * @return 响应的 Flux 流 */ Flux<ChatResponse> stream(ChatRequest chatRequest); } // ChatRequest 的结构 public class ChatRequest { private final List<Message> messages; // 对话消息列表 private final ChatOptions chatOptions; // 模型参数 private final ToolDefinition[] toolDefinitions; // 工具定义 private final ToolCallback[] toolCallbacks; // 工具实现 }

📄 请求/响应转换过程

Spring AI 内部使用 两套数据模型：一套是通用的（Message、ChatRequest），一套是供应商特定的（OpenAiMessage、OpenAiRequest）。转换过程是 ChatModel 实现的核心工作：

// ============================================ // OpenAiChatModel.call() 核心流程（简化源码） // ============================================ public ChatResponse call(ChatRequest request) { // ① 合并默认 Options 和请求级 Options OpenAiChatOptions mergedOptions = mergeOptions( request.getChatOptions(), this.defaultOptions); // ② 将通用 Message 转为 OpenAI 格式 List<OpenAiMessage> openAiMessages = request.getMessages().stream() .map(this::convertMessage) .collect(toList()); // ③ 将 ToolDefinition 转为 OpenAI function schema List<OpenAiTool> tools = convertToolDefinitions( request.getToolDefinitions()); // ④ 构建 OpenAI API 请求体 OpenAiRequest apiRequest = new OpenAiRequest( mergedOptions.getModel(), openAiMessages, tools, mergedOptions.getTemperature(), mergedOptions.getMaxTokens() ); // ⑤ 发送 HTTP 请求 OpenAiResponse apiResponse = callWithRetry(apiRequest); // ⑥ 处理工具调用（如果 LLM 决定调用工具） if (apiResponse.hasToolCalls()) { return handleToolCalls(request, apiResponse); } // ⑦ 转换响应 return convertResponse(apiResponse); }

📄 工具调用的自动循环

当 LLM 返回 tool_calls 时，OpenAiChatModel 会自动执行工具并将结果反馈给 LLM，形成一个内部循环：

第 1 轮调用：
  ChatModel → OpenAI API
  返回: { tool_calls: [{ name: "getWeather", args: {city:"北京"} }] }
                       │
                       ▼
  自动执行 ToolCallback.getWeather("北京") → "晴天 25°C"
                       │
                       ▼
第 2 轮调用：
  将工具结果追加到消息列表，再次调用 OpenAI API
  返回: { content: "北京今天晴天，温度25°C，适合外出。" }
                       │
                       ▼
  循环结束，返回最终响应给用户
⚠️ 注意事项

这个自动循环有最大轮次限制（默认通常为 10 轮），防止 LLM 陷入无限工具调用。如果超过限制，会抛出 ToolCallLimitExceededException。你可以在 ChatOptions 中通过 maxIterations 参数调整这个限制。

📄 错误处理与重试机制

Spring AI 使用 Spring 的 RetryTemplate 实现重试。以 OpenAiChatModel 为例：

// ============================================ // 重试策略配置（简化源码） // ============================================ private ChatResponse callWithRetry(OpenAiRequest request) { return retryTemplate.execute(context -> { try { return this.restClient.post() .uri("/v1/chat/completions") .header("Authorization", "Bearer " + apiKey) .body(request) .retrieve() .body(OpenAiResponse.class); } catch (HttpStatusCodeException e) { if (e.getStatusCode().value() == 429) { // 速率限制 → 重试 throw new TransientDataAccessException( "Rate limited", e); } if (e.getStatusCode().is5xxServerError()) { // 服务端错误 → 重试 throw new TransientDataAccessException( "Server error", e); } // 4xx 客户端错误 → 不重试，直接抛出 throw new BadOutputException(e); } }); }
✔ 要点小结

ChatModel 是 Spring AI 的"通信中枢"。它将通用的 ChatRequest 转换为供应商特定格式，通过 RestClient 发送 HTTP 请求，并在收到工具调用时自动循环执行。重试机制基于 Spring RetryTemplate，区分可重试的临时错误（429、5xx）和不可重试的客户端错误（4xx）。

📄 参数解析与类型转换

LLM 返回的工具参数是 JSON 字符串，ToolCallback 需要将其转换为 Java 类型。Spring AI 内部使用 Jackson 进行反序列化：

// ============================================ // MethodToolCallback 的执行逻辑（简化源码） // ============================================ class MethodToolCallback implements ToolCallback { private final Method toolMethod; // 反射引用 private final Object targetObject; // 工具 Bean 实例 private final ObjectMapper mapper; // JSON 处理器 @Override public String call(String toolArguments) { try { // ① 将 JSON 字符串解析为参数数组 JsonNode argsNode = mapper.readTree(toolArguments); // ② 按方法参数顺序，逐个反序列化 Parameter[] params = toolMethod.getParameters(); Object[] argValues = new Object[params.length]; for (int i = 0; i < params.length; i++) { String paramName = getParameterName(params[i]); Class<?> paramType = params[i].getType(); // 类型转换：JSON → Java 对象 argValues[i] = mapper.treeToValue( argsNode.get(paramName), paramType); } // ③ 反射调用目标方法 Object result = toolMethod.invoke(targetObject, argValues); // ④ 将返回值序列化为 JSON 字符串 return mapper.writeValueAsString(result); } catch (Exception e) { return "{\"error\": \"" + e.getMessage() + "\"}"; } } }

📄 Embedding 生成与索引

当调用 add() 时，VectorStore 内部会通过 EmbeddingModel 生成向量。以 PgVectorStore 为例：

// ============================================ // PgVectorStore.doAdd() 简化源码 // ============================================ public void doAdd(List<Document> documents) { // ① 提取文本内容 List<String> contents = documents.stream() .map(Document::getText) .collect(toList()); // ② 批量生成 Embedding 向量 EmbeddingRequest embedRequest = new EmbeddingRequest(contents); EmbeddingResponse embedResponse = embeddingModel.call(embedRequest); // ③ 构建 SQL INSERT 语句 String sql = """ INSERT INTO vector_store (id, content, embedding, metadata) VALUES ($1, $2, $3::vector, $4) """; // ④ 批量插入（使用 PG 的 vector 类型） for (int i = 0; i < documents.size(); i++) { jdbcTemplate.update(sql, documents.get(i).getId(), contents.get(i), new PgVector(embedResponse.getResult(i).getOutput()), objectMapper.writeValueAsString( documents.get(i).getMetadata()) ); } }
💡 设计亮点

Spring AI 的 VectorStore 抽象让向量数据库的具体实现对上层透明。你切换 PgVector 为 Chroma 时，只需更换依赖和配置，RAG 代码一行不用改。这就是策略模式 + 依赖倒置的威力。

📄 检索策略的实现

VectorStore 的检索不仅仅是简单的余弦相似度。Spring AI 支持多种检索策略：

检索策略层级：

1. 基础检索（Basic Similarity）
   └─ 纯向量余弦相似度，返回 topK 个结果
      适用场景：简单问答、语义搜索

2. 混合检索（Hybrid Search）
   └─ 向量相似度 + 关键词匹配（BM25）
      适用场景：需要精确关键词匹配的场景

3. 带过滤的检索（Filtered Search）
   └─ 向量检索 + 元数据过滤条件
      适用场景：按类别、时间、来源等筛选

4. 多路召回（Multi-Query RAG）
   └─ 将原始查询扩展为多个变体，分别检索后合并
      适用场景：提高召回率，减少语义偏差

5. 重排序（Re-ranking）
   └─ 初步检索后，用 Cross-Encoder 重新评分排序
      适用场景：对精度要求高的生产环境
✔ 要点小结

VectorStore 封装了向量数据库的读写细节。写入时自动调用 EmbeddingModel 生成向量；检索时支持相似度搜索、元数据过滤、混合检索等多种策略。通过统一接口抽象，切换底层向量数据库对上层代码完全透明。

// ============================================ // 自定义 ChatModel：对接自研 LLM API // ============================================ @Service public class MyCustomChatModel implements ChatModel { private final RestClient restClient; private final ObjectMapper objectMapper; public MyCustomChatModel(RestClient.Builder builder, MyLlmProperties properties) { this.restClient = builder .baseUrl(properties.getApiUrl()) .defaultHeader("X-Api-Key", properties.getApiKey()) .build(); this.objectMapper = new ObjectMapper(); } @Override public ChatResponse call(ChatRequest chatRequest) { // ① 转换消息格式 List<Map<String, Object>> messages = chatRequest.getMessages().stream() .map(this::convertMessage) .collect(Collectors.toList()); // ② 构建请求体 Map<String, Object> body = new HashMap<>(); body.put("model", "my-custom-model"); body.put("messages", messages); body.put("temperature", 0.7); // ③ 发送请求 Map<String, Object> response = restClient.post() .uri("/v1/chat") .body(body) .retrieve() .body(new ParameterizedTypeReference<Map<String, Object>>() {}); // ④ 转换为 ChatResponse String content = (String) response.get("text"); Generation generation = new Generation( new AssistantMessage(content)); return new ChatResponse( List.of(generation), new ChatCompletionMetadata(/* ... */) ); } @Override public Flux<ChatResponse> stream(ChatRequest chatRequest) { // 实现 SSE 流式响应 return Flux.from( restClient.post() .uri("/v1/chat/stream") .body(buildRequestBody(chatRequest)) .retrieve() .bodyToFlux(ServerSentEvent.class) ).map(event -> parseStreamEvent(event.getData())); } }

🛠 示例3：扩展 VectorStore 实现

如果你需要对接一个 Spring AI 尚未支持的向量数据库，可以实现 VectorStore 接口：

// ============================================ // 自定义 VectorStore：对接自研向量存储 // ============================================ @Service public class MyVectorStore implements VectorStore { private final EmbeddingModel embeddingModel; private final RestClient vectorDbClient; private final ObjectMapper objectMapper; @Override public void add(List<Document> documents) { // ① 生成 Embedding 向量 List<String> texts = documents.stream() .map(Document::getText) .collect(Collectors.toList()); EmbeddingResponse embedResp = embeddingModel.call( new EmbeddingRequest(texts)); // ② 打包数据并发送到向量数据库 for (int i = 0; i < documents.size(); i++) { Map<String, Object> record = new HashMap<>(); record.put("id", documents.get(i).getId()); record.put("text", documents.get(i).getText()); record.put("embedding", embedResp.getResult(i).getOutput()); record.put("metadata", documents.get(i).getMetadata()); vectorDbClient.post() .uri("/vectors") .body(record) .retrieve() .toBodilessEntity(); } } @Override public List<Document> similaritySearch(SearchRequest request) { // ① 查询文本向量化 float[] queryVec = embeddingModel.call( new EmbeddingRequest( List.of(request.getQuery()))) .getResult(0).getOutput(); // ② 发送到向量数据库检索 Map<String, Object> searchReq = new HashMap<>(); searchReq.put("vector", queryVec); searchReq.put("top_k", request.getTopK()); searchReq.put("threshold", request.getSimilarityThreshold()); List<Map<String, Object>> results = vectorDbClient.post() .uri("/vectors/search") .body(searchReq) .retrieve() .body(new ParameterizedTypeReference<>() {}); // ③ 转换为 Document 列表 return results.stream() .map(r -> new Document( (String) r.get("id"), (String) r.get("text"), (Map<String, Object>) r.get("metadata"))) .collect(Collectors.toList()); } }

📖 章节小结

本章我们深入拆解了 Spring AI 的"机芯"，从架构到源码分析了四个核心机制：

  • 三层架构：API 层定义契约、实现层对接 LLM、基础设施层复用 Spring 生态，通过依赖倒置实现高可扩展性
  • ChatClient：Builder 模式构建配置，递归责任链执行 Advisor 拦截，最终委托 ChatModel 执行调用
  • ChatModel：通用请求与供应商特定格式之间的双向转换，内置工具调用自动循环和重试机制
  • ToolCallback：通过反射扫描 @Tool 注解，自动生成 JSON Schema，运行时通过 Jackson + 反射完成参数转换和方法调用
  • VectorStore：封装向量数据库读写细节，写入时自动 Embedding，检索时支持多种策略

💼 面试题精选

面试题 1：ChatClient 的 defaultAdvisors 和 .advisors() 有什么区别？

答：defaultAdvisors() 在 ChatClient.Builder 上设置，会被所有后续 prompt 调用共享，适用于全局拦截逻辑（如日志、记忆管理）。而 .advisors() 在具体 prompt 链式调用中设置，仅对当前这一次调用生效。两者的 Advisor 会在调用时合并执行，default 的先执行。

面试题 2：ChatModel 在收到 LLM 返回的 tool_calls 时是如何处理的？

答：ChatModel 内部会自动进入工具调用循环：(1) 解析 LLM 返回的 tool_calls；(2) 按 name 匹配对应的 ToolCallback；(3) 调用 callback.call() 获取工具执行结果；(4) 将结果作为 ToolResponseMessage 追加到消息列表；(5) 再次调用 LLM，直到 LLM 返回普通文本响应或达到最大轮次限制。这个循环对上层调用者是透明的。

面试题 3：Spring AI 的三层架构是如何实现供应商无关性的？

答：API 层（spring-ai-model）定义了 ChatModel、EmbeddingModel 等接口，这些接口不依赖任何具体 LLM。实现层（如 spring-ai-openai）提供具体实现。基础设施层提供 HTTP 通信等通用能力。通过 Spring Boot 的自动配置机制，引入不同的 starter 就会自动创建对应的实现 Bean。业务代码只依赖 API 层接口，切换供应商只需更换 starter 依赖和配置，上层代码零修改。

面试题 4：ToolCallback 是如何将 Java 方法适配为 LLM 工具的？

答：分两个阶段。注册阶段：通过反射扫描 @Tool 注解，提取方法名、参数类型、描述信息，生成 JSON Schema 作为 ToolDefinition。执行阶段：LLM 返回 tool_call 时，通过 Jackson 将 JSON 参数反序列化为 Java 对象，再通过反射调用目标方法，最后将返回值序列化为 JSON 字符串。这本质上是一个Adapter 模式的实现。

面试题 5：VectorStore 的 add() 方法内部做了哪些事情？

答：(1) 接收 Document 列表；(2) 可选的后处理（如分块、元数据增强）；(3) 调用 EmbeddingModel 批量生成向量；(4) 将文档内容、向量、元数据打包；(5) 调用具体实现的数据库写入方法（如 SQL INSERT 或 REST API 调用）。关键是 Embedding 生成和数据库写入是解耦的，VectorStore 只负责编排流程。
