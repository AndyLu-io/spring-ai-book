第 18 章

MCP 协议深度解析

从各自为战到互联互通，Model Context Protocol 是 Agent 生态的"普通话"。掌握 MCP，让你的 AI 应用拥有"万能接口"。

⏱ 预计 1 周 📖 6 个知识点 💻 中级难度

📚 开篇故事：从方言到普通话

在 AI 应用开发的早期，每个大模型厂商都有自己的 API 格式。OpenAI 用 Function Calling，Anthropic 用 Tool Use，Google 用 Function Declarations。它们做的事情本质上一样 — 让模型调用外部工具 — 但接口完全不同。这就好比中国有几百种方言，虽然都能表达"你好"，但各地人互相听不懂。

你在广东说粤语，在四川说四川话，在上海说吴语。如果你想让全国各地的人都能顺畅交流，你需要一种"普通话"。这就是 MCP（Model Context Protocol）诞生的背景。

2024 年底，Anthropic 发布了 MCP 协议，它的目标是为 AI 应用和外部工具之间建立一个统一的通信标准。就像 HTTP 统一了 Web 通信、SQL 统一了数据库查询一样，MCP 有望统一 AI 模型与工具之间的交互方式。这不再是你用"粤语"和 AI 对话、他用"四川话"和 AI 对话的局面 — 大家都说同一种语言，互联互通。

本章将带你从协议原理到实战代码，彻底掌握 MCP 的核心机制，学会用 Spring AI 构建自己的 MCP Server 和 Client。

🌐 MCP 协议概述

MCP 是什么？

Model Context Protocol 是一个开放协议，它定义了 AI 模型（或应用）与外部工具/数据源之间的标准通信方式。MCP 让 AI 应用能够以统一的方式连接到各种数据源和工具。

比喻：Agent 的"通用翻译器"

想象你是一个只会说中文的人，要去全世界旅行。以前，你到日本要学几句日语，到法国要学几句法语，到德国要学几句德语 — 每到一个地方都要重新学。有了翻译器（MCP），你说中文，翻译器自动帮你转换成对方听得懂的语言，对方的回答也自动翻译回来。你不需要再学外语了。

MCP vs Function Calling

很多人会问：OpenAI 的 Function Calling 不就能做这件事吗？为什么还需要 MCP？

维度Function CallingMCP
定位模型 API 的工具调用能力应用与工具之间的通信协议
标准厂商私有（虽然各家大同小异）开放标准（Anthropic 发起）
生态每个厂商维护自己的工具列表一个 MCP Server 可被所有客户端复用
能力工具调用（Tools）Tools + Resources + Prompts + Sampling
连接工具定义嵌入在请求中独立的 Client-Server 架构，可动态发现
可复用每个模型 API 需要重新集成一次实现，处处使用
Function Calling 模式
┌──────────┐    不同格式     ┌──────────┐
│  GPT-4o  │ <----------> │ 工具 A   │
└──────────┘                └──────────┘
┌──────────┐    不同格式     ┌──────────┐
│ Claude   │ <----------> │ 工具 B   │
└──────────┘                └──────────┘
┌──────────┐    不同格式     ┌──────────┐
│ Gemini   │ <----------> │ 工具 C   │
└──────────┘                └──────────┘

MCP 模式
┌──────────┐                ┌──────────┐
│  GPT-4o  │ +--MCP--+     │ 工具 A   │
│  Claude  │ +--MCP--+     │ 工具 B   │  统一协议！
│  Gemini  │ +--MCP--+     │ 工具 C   │
└──────────┘                └──────────┘
💡 你知道吗？
MCP 并不替代 Function Calling，而是建立在它之上。底层仍然使用模型的工具调用能力，但 MCP 提供了一个标准化的"中间层"，让工具可以跨模型、跨平台复用。

MCP 的四大能力

  • Tools（工具） — 让模型可以执行操作，如查询数据库、调用 API、发送邮件
  • Resources（资源） — 让模型可以读取数据，如文件内容、数据库记录、API 响应
  • Prompts（提示） — 预定义的提示模板，供客户端复用
  • Sampling（采样） — 让服务器反向请求模型进行推理（较新功能）
💡 小窍门

初学 MCP，先掌握 Tools 和 Resources 就够了。Prompts 和 Sampling 是进阶功能，实际项目中使用频率相对较低。

🏗 MCP 架构

Client-Server 架构

MCP 采用经典的客户端-服务器架构。客户端（通常是 AI 应用）连接到一个或多个 MCP Server，每个 Server 暴露特定的工具和资源。

比喻：Agent 的"标准接口"

就像 USB 接口统一了电脑和外设的连接方式 — 以前键盘用 PS/2 接口，鼠标用串口，打印机用并口，每个都不一样。USB 出现后，一个接口搞定所有外设。MCP 之于 AI 工具，就像 USB 之于外设。

通信协议：JSON-RPC 2.0

MCP 使用 JSON-RPC 2.0 作为底层消息格式。这意味着所有消息都是 JSON 格式的请求-响应对。

// MCP 客户端发送的初始化请求 { "jsonrpc": "2.0", "id": 1, "method": "initialize", "params": { "protocolVersion": "2024-11-05", "capabilities": { "tools": {} }, "clientInfo": { "name": "my-ai-app", "version": "1.0.0" } } } // 服务器响应：返回可用工具列表 { "jsonrpc": "2.0", "id": 1, "result": { "protocolVersion": "2024-11-05", "capabilities": { "tools": { "listChanged": true } }, "serverInfo": { "name": "database-server", "version": "1.0.0" } } }
💡 小窍门

使用 Spring AI 时，你不需要手动处理 JSON-RPC 消息。Spring AI 的 MCP 集成已经封装了协议细节，你只需关注业务逻辑。

传输方式

MCP 支持两种主要的传输方式：

传输方式适用场景特点
stdio本地工具（CLI 命令行）通过标准输入/输出通信，最简单
SSE (HTTP)远程工具（Web 服务）通过 HTTP + Server-Sent Events 通信

⚙ MCP Server 实现

用 Spring AI 构建 MCP Server

Spring AI 提供了 MCP Server 的 Starter，让你可以用极少的代码构建一个标准的 MCP Server。

比喻：Agent 的"工具工厂"

MCP Server 就像一个"工具工厂"。工厂对外展示自己能生产哪些工具（工具列表），客户（AI 应用）来了可以查看目录（Tool Discovery），选中一个工具后直接下单使用（Tool Call）。工厂不需要知道谁在用、用在哪里，只要按标准规格生产就行。

<!-- pom.xml: 添加 MCP Server 依赖 --> <dependency> <groupId>org.springframework.ai</groupId> <artifactId>spring-ai-mcp-server-spring-boot-starter</artifactId> </dependency> <!-- 如果使用 WebFlux 传输 --> <dependency> <groupId>org.springframework.ai</groupId> <artifactId>spring-ai-mcp-server-webflux-spring-boot-starter</artifactId> </dependency>
// application.yml spring: ai: mcp: server: name: my-mcp-server version: 1.0.0 type: SYNC # SYNC 或 ASYNC sse-message-port: 8080 # SSE 端口

定义 MCP 工具

import org.springframework.ai.tool.annotation.Tool; import org.springframework.ai.tool.annotation.ToolParam; import org.springframework.stereotype.Service; @Service public class WeatherService { /** * 查询指定城市的天气信息 * 这个方法会被自动注册为 MCP 工具 */ @Tool(description = "查询指定城市的天气信息") public String getWeather( @ToolParam(description = "城市名称") String city) { // 实际项目中这里调用天气 API return String.format( "%s: 晴天，温度 25°C，湿度 60%%", city); } @Tool(description = "查询未来3天的天气预报") public String getForecast( @ToolParam(description = "城市名称") String city, @ToolParam(description = "天数") int days) { return String.format( "%s 未来 %d 天天气预报：晴转多云，温度 20-28°C", city, days); } }
⚠️ 踩坑警告

@Tool 注解的 description 非常重要！模型根据描述来决定何时调用这个工具。描述不清晰会导致模型调用错误或完全不调用。请把描述写得像给一个聪明但不了解你系统的人解释一样。

定义 MCP 资源

import org.springframework.ai.mcp.server.tool.McpServerFeatures; import org.springframework.context.annotation.Bean; import org.springframework.context.annotation.Configuration; @Configuration public class McpResourceConfig { @Bean public List<McpServerFeatures.SyncResourceSpecification> resources() { return List.of( new McpServerFeatures.SyncResourceSpecification( // 资源 URI "config://app/settings", // 资源名称 "应用配置", // 资源描述 "当前应用的配置信息", // MIME 类型 "application/json", // 读取资源的函数 (uri, params) -> { return new McpServerFeatures.SyncResourceResult( "application/json", "{\"theme\": \"dark\", \"lang\": \"zh\"}" ); } ) ); } }

运行 MCP Server

@SpringBootApplication public class McpServerApplication { public static void main(String[] args) { SpringApplication.run(McpServerApplication.class, args); } } // 启动后，MCP Server 会： // 1. 在 http://localhost:8080 暴露 SSE 端点 // 2. 自动注册所有 @Tool 注解的方法 // 3. 通过 MCP 协议响应客户端的工具发现和调用请求 
MCP Server 启动流程

       |

扫描 @Bean 资源定义 ────────────> 注册为 MCP 资源

🔌 MCP Client 集成

连接 MCP Server

在 Spring AI 中，你也可以构建 MCP Client，连接到远程或本地的 MCP Server，获取可用的工具和资源。

比喻：Agent 的"万能插头"

MCP Client 就像一个"万能插头"。不管你去哪个国家（连接哪个 MCP Server），只要插座标准一致（MCP 协议），你的电器（AI 应用）就能正常工作。你不需要为每个国家买一个转换器。

<!-- pom.xml: 添加 MCP Client 依赖 --> <dependency> <groupId>org.springframework.ai</groupId> <artifactId>spring-ai-mcp-client-spring-boot-starter</artifactId> </dependency> <!-- SSE 传输方式 --> <dependency> <groupId>org.springframework.ai</groupId> <artifactId>spring-ai-mcp-client-webflux-spring-boot-starter</artifactId> </dependency>
# application.yml spring: ai: mcp: client: sse: connections: weather-server: url: http://localhost:8080 # MCP Server 地址 openai: api-key: ${OPENAI_API_KEY}

使用 MCP 工具的 AI 应用

@Service public class McpChatService { private final ChatClient chatClient; public McpChatService(ChatClient.Builder chatClientBuilder) { // Spring AI 自动将 MCP Server 的工具注入到 ChatClient this.chatClient = chatClientBuilder .defaultTools("weather-server") // 引用 MCP Server 名称 .build(); } public String chat(String message) { return chatClient.prompt() .user(message) .call() .content(); } } // 使用示例 // 用户问："北京今天天气怎么样？" // AI 会自动： // 1. 识别需要调用天气工具 // 2. 通过 MCP 协议调用 MCP Server 的 getWeather 工具 // 3. 获取结果并组织成自然语言回答

💡 小窍门

Spring AI 的 MCP 集成是声明式的。你只需要在配置文件中指定 MCP Server 的地址，Spring AI 会自动发现工具、建立连接、处理协议。你几乎不需要写任何 MCP 协议相关的代码。

📚 MCP 资源与提示

Resources：Agent 的"知识库"

MCP Resources 让 MCP Server 可以向客户端暴露数据。与 Tools 不同（Tools 是"执行操作"），Resources 是"提供信息"。

比喻：Agent 的"知识库"

如果 Tools 是"手"（可以做事），那 Resources 就是"眼睛"（可以看到信息）。你的 AI 应用可以通过 Resources 读取配置文件、数据库内容、API 数据等，而不需要自己去连接这些系统。

Tools vs Resources

Tools（手）                 Resources（眼睛）
  ┌──────────┐               ┌──────────┐
  │ getWeather│               │ app-config│
  │ sendEmail │               │ db-schema │
  │ createFile│               │ api-docs  │
  └────┬─────┘               └────┬─────┘
       │                           │
       ▼                           ▼
  执行操作，返回结果         提供数据，返回内容
  "帮我查天气"              "给我看配置"

Prompts：预定义的"提示模板"

MCP Prompts 让 Server 可以向客户端提供预定义的提示模板，客户端可以复用这些模板而不需要自己编写。

// 在 MCP Server 中定义 Prompt @Bean public List<McpServerFeatures.SyncPromptSpecification> prompts() { return List.of( new McpServerFeatures.SyncPromptSpecification( "code-review", "代码审查提示模板", List.of( new McpServerFeatures.PromptArgument( "language", "编程语言", true), new McpServerFeatures.PromptArgument( "code", "要审查的代码", true) ), (params) -> { String lang = params.get("language"); String code = params.get("code"); return new McpServerFeatures.SyncPromptResult( List.of( new McpServerFeatures.PromptMessage( "user", "请审查以下 %s 代码：\n%s".formatted(lang, code) ) ) ); } ) ); }
💡 你知道吗？

Prompts 的价值在于标准化和复用。一个团队可以开发一套最佳实践的 Prompt 模板，通过 MCP Server 共享给所有开发者和 AI 应用，确保所有人使用相同的高质量提示词。

Sampling：反向调用模型

Sampling 是 MCP 的一个较新功能，它允许 MCP Server 反向请求客户端的模型进行推理。这在需要"人机协作"的场景中非常有用。

普通调用：Client ──────> Server（执行工具）
Sampling：Client <------ Server（请求模型推理）

Server 遇到不确定的情况 ──> 请求 Client 的模型 ──> 获得推理结果 ──> 继续执行
💡 小窍门

Sampling 目前使用场景有限，主要用于复杂的多步骤推理任务。初学阶段了解即可，实际项目中很少用到。

何时使用 MCP？

适合使用的场景

  • 需要让多个 AI 应用共享同一套工具
  • 构建可复用的工具生态
  • 需要跨模型平台使用工具
  • 工具需要独立部署和更新

暂时不需要的场景

  • 简单的单模型应用（Function Calling 足够）
  • 工具不涉及复用（只用一次）
  • 对延迟极度敏感（多一层协议有开销）
  • 团队对 MCP 协议还不熟悉

  • MCP 架构 — Client-Server 模式，使用 JSON-RPC 2.0 通信
  • MCP Server — 用 Spring AI 的 @Tool 注解快速构建
  • MCP Client — 声明式配置，自动发现和连接工具
  • 四大能力 — Tools（操作）、Resources（数据）、Prompts（模板）、Sampling（推理）
  • 生态与展望 — 社区活跃，各大厂商纷纷支持

MCP 是 AI 应用从"各自为战"走向"互联互通"的关键一步。掌握 MCP，你就拿到了 AI 工具生态的"通行证"。

🎓 面试高频题

👨‍💻 Q1: MCP 和 Function Calling 有什么区别？

Function Calling 是模型 API 提供的工具调用能力，属于模型层面的功能。MCP 是一个开放协议标准，定义了 AI 应用与工具之间的通信规范，属于应用架构层面。MCP 建立在 Function Calling 之上，提供标准化、可复用的工具生态。

👨‍💻 Q2: MCP 的四种能力分别是什么？各自的作用？

Tools 让模型执行操作（查天气、发邮件）；Resources 让模型读取数据（配置、文档）；Prompts 提供预定义的提示模板供复用；Sampling 允许 Server 反向请求模型进行推理。初学重点掌握 Tools 和 Resources。

👨‍💻 Q3: Spring AI 中如何构建 MCP Server？

引入 spring-ai-mcp-server-spring-boot-starter 依赖，在 application.yml 配置 MCP Server 信息，使用 @Tool 注解标记工具方法，Spring AI 自动将这些方法注册为 MCP 工具。通过 SSE 或 stdio 传输方式对外提供服务。
