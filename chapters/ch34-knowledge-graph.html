知识图谱与 GraphRAG 深度实战

从"向量搜索"到"关系推理" -- 掌握知识图谱构建、Neo4j 实战、GraphRAG 深度实现，让 Agent 具备真正的结构化推理能力

    📚 开篇故事：从电话簿到社交网络

    假设你刚搬来一座新城市，想找一位靠谱的牙医。如果你只有向量搜索，它的工作方式就像"按声音相似度找人" -- 它会告诉你："有个叫王医生的人，和你描述的牙医声音很像。"但你不知道王医生在哪里工作、他治过谁、谁推荐过他。

    而知识图谱则像一张社交网络图。它不仅知道每个人是谁，还知道谁认识谁、谁和谁是同事、谁推荐了谁、哪家诊所离你最近。当你问"给我推荐一个靠谱的牙医"，它能这样推理：张三推荐了王医生 -> 王医生在协和医院工作 -> 协和医院的患者满意度是 4.8 分 -> 王医生擅长种植牙。

    GraphRAG（Graph Retrieval-Augmented Generation）就是让 AI 同时利用"相似度"和"关系"来回答问题的技术。它把传统 RAG 的"在向量空间里找最相似的文档片段"升级为"在知识图谱里沿着关系路径推理，再结合向量搜索找到最相关的上下文"。

    传统 RAG 就像在图书馆里按关键词搜索书架 -- 你只能找到包含相同关键词的书。而 GraphRAG 就像一位资深图书馆员，他不仅知道哪些书和你的问题相关，还知道这些书之间的引用关系、作者关系、主题关联 -- 他能帮你构建一个完整的知识图景。

    📄 核心内容（第1节）：知识图谱基础 -- 万物皆实体，万物皆关系

    📄 什么是知识图谱

    知识图谱（Knowledge Graph）是一种用图结构来表示知识的模型。它的核心由三部分组成：

    • 实体（Entity）-- 图中的节点，代表现实世界中的"事物"。比如：一个人、一家公司、一个技能、一座城市。每个实体都有一个标签（Label）来表示它的类型。
    • 关系（Relation）-- 图中的边，连接两个实体，表示它们之间的"联系"。比如：张三"就职于"字节跳动、李四"掌握"Spring AI。每条关系都有一个类型（Type）和一个方向。
    • 属性（Property）-- 实体或关系上的附加信息。比如：张三的属性可以是 {name: "张三", age: 30}，"就职于"关系的属性可以是 {since: 2023}。
    ┌─────────────┐ works_for ┌─────────────┐ │ 张三 │ ────────────────→ │ 字节跳动 │ │ (Person) │ │ (Company) │ └─────────────┘ └─────────────┘ │ │ knows develops │ │ ▼ ▼ ┌─────────────┐ belongs_to ┌─────────────┐ │ 李四 │ ←──────────────── │ 推荐系统 │ │ (Person) │ │ (Project) │ └─────────────┘ └─────────────┘

    📄 知识图谱 vs 向量数据库 vs 关系型数据库

    三种数据存储方式各有优势，理解它们的区别是选型的关键：

    维度关系型数据库向量数据库知识图谱
    数据模型表（行列结构）向量（高维空间中的点）图（节点 + 边）
    核心能力精确查询、聚合计算语义相似度搜索关系遍历、路径推理
    查询语言SQLAPI（向量 + 过滤）Cypher / SPARQL / Gremlin
    适合场景结构化业务数据非结构化内容检索复杂关系分析、推理
    查询特点"给我所有年龄>30的用户""找和这段话最相似的文档""找到张三的所有同事的导师"
    多跳能力需要多次 JOIN不支持原生支持，一次查询可达多跳
    代表产品MySQL、PostgreSQLPinecone、Milvus、WeaviateNeo4j、Amazon Neptune、JanusGraph
    💡 选型建议

    三种数据库不是替代关系，而是互补关系。在实际项目中，往往需要同时使用：关系型数据库存储业务数据，向量数据库存储非结构化内容的语义表示，知识图谱存储实体间的复杂关系。它们通过 ID 关联，形成完整的数据架构。

    📄 知识图谱的核心价值

    知识图谱之所以在 AI 领域越来越重要，是因为它提供了三种传统数据存储难以实现的能力：

    🧠 推理能力

    通过已知事实推导出新知识。比如：张三在字节跳动工作 + 字节跳动的 CEO 是张一鸣 = 张三的 CEO 是张一鸣。这种推理能力是 LLM 最需要的"结构化背景知识"。

    🔗 关联发现

    发现看似无关的事物之间的隐藏联系。比如：两个看似无关的疾病，可能共享相同的蛋白质靶点；两个看似无关的技术，可能有共同的发明人。这种"跨域关联"是创新的重要来源。

    🗺 知识层次

    知识图谱通常分为三个层次：

    • 事实层（Fact Layer）-- 存储具体的事实。"张三就职于字节跳动"、"Spring AI 1.0 发布于 2024 年"。这是图谱的基础层。
    • 概念层（Concept Layer）-- 存储抽象的概念和分类体系。"AI 工程师是工程师的一种"、"Spring AI 是一个框架"。这是图谱的语义层。
    • 规则层（Rule Layer）-- 存储推理规则。"如果一个人是某公司的 AI 工程师，那么他可能掌握深度学习技能"。这是图谱的推理层。

    📄 知识表示方法

    知识图谱有多种表示方法，适用于不同的场景：

    RDF（Resource Description Framework）

    W3C 标准，使用"三元组"（Subject-Predicate-Object）表示知识。适合语义 Web、开放数据。查询语言为 SPARQL。优点：标准化程度高，跨平台互操作性强；缺点：查询效率较低，不适合大规模图。

    属性图（Property Graph）

    节点和边都可以有属性。Neo4j 是最流行的属性图数据库。查询语言为 Cypher。优点：直觉、灵活、查询效率高；缺点：非 W3C 标准，不同图数据库之间互操作性较差。

    💡 实践建议

    在 AI 应用开发中，我们主要使用属性图模型（Neo4j），因为它更直觉、性能更好，且与 Spring AI 有更好的集成。RDF 更适合需要与外部知识库（如 Wikidata）互操作的场景。

    📄 Cypher 查询语言基础

    Cypher 是 Neo4j 的查询语言，语法直观易学。我们从最基本的 CRUD 操作开始：

    // ========== 创建节点 ========== CREATE (p:Person {name: '张三', role: 'AI工程师'}) CREATE (c:Company {name: '字节跳动', industry: '互联网'}) CREATE (p)-[:WORKS_AT {since: 2023}]->(c) // 创建更多节点和关系 CREATE (l:Skill {name: 'Spring AI'}) CREATE (p)-[:HAS_SKILL {level: 'expert'}]->(l) // ========== 创建完整子图 ========== CREATE (p2:Person {name: '李四', role: '后端工程师'}) CREATE (p3:Person {name: '王五', role: '数据工程师'}) CREATE (s2:Skill {name: 'Neo4j'}) CREATE (p2)-[:WORKS_AT]->(c) CREATE (p3)-[:WORKS_AT]->(c) CREATE (p)-[:KNOWS]->(p2) CREATE (p2)-[:KNOWS]->(p3) CREATE (p2)-[:HAS_SKILL]->(s2) CREATE (p3)-[:HAS_SKILL]->(s2)
    // ========== 查询：找到张三的所有同事 ========== MATCH (p:Person {name: '张三'})-[:WORKS_AT]->(c:Company) <-[:WORKS_AT]-(colleague:Person) WHERE colleague <> p RETURN colleague.name, colleague.role // ========== 查询：找到与张三有共同技能的人 ========== MATCH (p:Person {name: '张三'})-[:HAS_SKILL]->(s:Skill) <-[:HAS_SKILL]-(other:Person) WHERE other <> p RETURN other.name, COLLECT(s.name) AS common_skills // ========== 查询：图算法 - 最短路径 ========== MATCH path = shortestPath( (p1:Person {name: '张三'})-[*]-(p2:Person {name: '王五'}) ) RETURN path, LENGTH(path) AS distance // ========== 查询：按路径长度查找所有路径 ========== MATCH path = allShortestPaths( (p1:Person {name: '张三'})-[*]-(p2:Person {name: '王五'}) ) RETURN [n IN nodes(path) | n.name] AS route, LENGTH(path) AS distance // ========== 查询：社区检测 - PageRank ========== CALL gds.pageRank.stream('person-graph') YIELD nodeId, score RETURN gds.util.asNode(nodeId).name AS name, score ORDER BY score DESC LIMIT 10
    📚 Cypher 语法速查

    节点模式：(variable:Label {key: value})；关系模式：-[:REL_TYPE {key: value}]->；路径变量：path = (a)-[*]-(b)；聚合函数：COLLECT、COUNT、AVG、SUM；列表推导：[x IN list | expression]；条件过滤：WHERE、AND、OR、NOT、STARTS WITH、CONTAINS。

    📄 索引和约束

    为了保证查询性能和数据完整性，我们需要为图谱创建索引和约束：

    // 创建唯一性约束（确保实体名称唯一） CREATE CONSTRAINT person_name_unique FOR (p:Person) REQUIRE p.name IS UNIQUE CREATE CONSTRAINT company_name_unique FOR (c:Company) REQUIRE c.name IS UNIQUE // 创建索引（加速查询） CREATE INDEX person_role_index FOR (p:Person) ON (p.role) CREATE INDEX skill_name_index FOR (s:Skill) ON (s.name) // 创建复合索引（多属性联合查询） CREATE INDEX person_name_role_index FOR (p:Person) ON (p.name, p.role) // 全文索引（支持文本搜索） CREATE FULLTEXT INDEX person_search FOR (p:Person) ON EACH [p.name, p.role]

    📄 图算法

    Neo4j 提供了丰富的图算法库（Graph Data Science），以下是几个最常用的算法：

    👣 最短路径

    找到两个节点之间最短的连接路径。适用于社交网络中"六度分隔"、物流路径规划等场景。

    📈 中心性分析

    PageRank、Betweenness Centrality 等算法，用于找到图谱中最"重要"的节点。适用于影响力分析、关键人物识别。

    🌐 社区检测

    Leiden、Louvain 等算法，自动发现图谱中的社区结构。这是 GraphRAG 的核心算法之一，用于将大规模图谱划分为有意义的社区。

    🔎 相似度计算

    Jaccard、Cosine Similarity 等算法，计算节点之间的相似程度。适用于推荐系统、实体对齐。

    📄 核心内容（第3节）：知识图谱构建 -- 从文本到图谱

    📄 构建流程概览

    从非结构化文本自动构建知识图谱，需要经历以下几个步骤：

    原始文档 │ ▼ 文本分块 (Chunking) │ ▼ 实体识别 (NER) ──→ 提取：人名、公司、技术、概念... │ ▼ 关系抽取 (RE) ──→ 三元组：(实体1, 关系, 实体2) │ ▼ 知识融合 (Fusion) ──→ 实体对齐、消歧、去重 │ ▼ 知识图谱 (Neo4j) ──→ 持久化存储 │ ▼ 质量验证 ──→ 人工审核、自动校验

    📄 实体识别（NER）

    使用 LLM 进行零样本实体识别，无需标注数据即可提取实体：

    @Service public class EntityExtractor { private final ChatClient chatClient; public EntityExtractor(ChatClient.Builder builder) { this.chatClient = builder.build(); } public List<Entity> extractEntities(String text) { String prompt = """ 从以下文本中提取所有实体。每个实体包含： - name: 实体名称 - type: 实体类型（Person/Company/Skill/Technology/Concept/Event） - properties: 其他属性（如角色、描述等） 以JSON数组格式返回： [{"name": "...", "type": "...", "properties": {...}}] 文本：{text} """; String result = chatClient.prompt() .user(prompt) .call() .content(); return parseEntities(result); } private List<Entity> parseEntities(String json) { try { ObjectMapper mapper = new ObjectMapper(); return mapper.readValue(json, new TypeReference<List<Entity>>() {}); } catch (Exception e) { log.error("实体解析失败", e); return List.of(); } } } // 实体数据类 public record Entity( String name, String type, Map<String, Object> properties ) {}

    📄 关系抽取与图谱构建

    在提取实体之后，我们需要抽取实体间的关系，并将结果写入 Neo4j：

    @Service public class KnowledgeGraphBuilder { private final ChatClient chatClient; private final Neo4jClient neo4jClient; private final EntityExtractor entityExtractor; public KnowledgeGraphBuilder( ChatClient.Builder builder, Neo4jClient neo4jClient, EntityExtractor entityExtractor) { this.chatClient = builder.build(); this.neo4jClient = neo4jClient; this.entityExtractor = entityExtractor; } public List<EntityRelation> extractRelations(String text) { String prompt = """ 从以下文本中提取实体和关系，以JSON格式返回： [{"entity1": "...", "relation": "...", "entity2": "...", "entity1_type": "...", "entity2_type": "..."}] 文本：{text} """; String result = chatClient.prompt() .user(prompt) .call() .content(); return parseRelations(result); } public void buildGraph(String document) { // 分段处理 List<String> chunks = splitDocument(document); for (String chunk : chunks) { List<EntityRelation> relations = extractRelations(chunk); // 写入 Neo4j for (EntityRelation rel : relations) { neo4jClient.query(""" MERGE (a:%s {name: $name1}) MERGE (b:%s {name: $name2}) MERGE (a)-[:%s]->(b) """.formatted( rel.entity1Type(), rel.entity2Type(), rel.relation())) .bind("name1", rel.entity1()) .bind("name2", rel.entity2()) .run(); } } } private List<String> splitDocument(String document) { // 按段落分块，每块不超过 500 字符 return Arrays.stream(document.split("\\n\\n")) .filter(chunk -> !chunk.isBlank()) .map(String::trim) .toList(); } } // 关系数据类 public record EntityRelation( String entity1, String entity2, String relation, String entity1Type, String entity2Type ) {}
    ⚠️ 注意事项

    实体消歧是图谱构建中最困难的环节。同一个实体可能有不同的表述（"Spring AI"、"spring-ai"、"Spring人工智能框架"），需要进行实体对齐。可以使用 LLM 进行消歧判断，或者建立别名映射表。在写入图谱前，务必先查询是否已存在相似实体。

    📄 Spring Data Neo4j 实体建模

    使用 Spring Data Neo4j 可以更优雅地操作图谱数据：

    @Node public class PersonNode { @Id @GeneratedValue private Long id; private String name; private String role; @Relationship(type = "WORKS_AT", direction = Relationship.Direction.OUTGOING) private CompanyNode company; @Relationship(type = "HAS_SKILL", direction = Relationship.Direction.OUTGOING) private List<SkillNode> skills = new ArrayList<>(); @Relationship(type = "KNOWS", direction = Relationship.Direction.OUTGOING) private List<PersonNode> connections = new ArrayList<>(); // Getters, setters... } @Node public class CompanyNode { @Id @GeneratedValue private Long id; private String name; private String industry; // Getters, setters... } @Node public class SkillNode { @Id @GeneratedValue private Long id; private String name; private String category; // Getters, setters... } // Repository 接口 public interface PersonRepository extends Neo4jRepository<PersonNode, Long> { Optional<PersonNode> findByName(String name); @Query(""" MATCH (p:Person {name: $name})-[:KNOWS*1..3]-(other:Person) RETURN DISTINCT other LIMIT 20 """) List<PersonNode> findConnections(@Param("name") String name); @Query(""" MATCH (p:Person)-[:HAS_SKILL]->(s:Skill) WHERE s.name IN $skillNames RETURN p, COLLECT(s) AS skills """) List<PersonNode> findBySkills(@Param("skillNames") List<String> skillNames); }

    📄 核心内容（第4节）：GraphRAG 深度实现 -- 让检索具备推理能力

    📄 GraphRAG 核心原理

    GraphRAG（Graph Retrieval-Augmented Generation）是微软在 2024 年提出的一种新范式。它通过图结构 + 社区检测 + 摘要生成，将传统 RAG 的检索能力提升到新的高度。

    文档 → 分块 → 实体/关系提取 → 知识图谱 ↓ 社区检测(Leiden) ↓ 社区摘要生成(LLM) ↓ ┌─────────────────┴─────────────────┐ │ │ Local Search Global Search (特定实体查询) (全局主题查询) │ │ └─────────────────┬─────────────────┘ ↓ 综合回答

    📄 社区检测与摘要生成

    GraphRAG 的核心创新在于将知识图谱划分为社区（Community），并为每个社区生成摘要：

    @Service public class GraphRagService { private final Neo4jClient neo4jClient; private final ChatClient chatClient; public GraphRagService(Neo4jClient neo4jClient, ChatClient.Builder builder) { this.neo4jClient = neo4jClient; this.chatClient = builder.build(); } // Step 1: 使用 Leiden 算法进行社区检测 public void detectCommunities() { // 安装并调用 Leiden 算法 neo4jClient.query(""" CALL gds.leiden.stream({ nodeProjection: '*', relationshipProjection: '*', communityProperty: 'communityId' }) YIELD nodeId, communityId MATCH (n) WHERE id(n) = nodeId SET n.communityId = communityId """).run(); } // Step 2: 为每个社区生成摘要 public void generateCommunitySummaries() { // 获取所有社区 List<Map<String, Object>> communities = neo4jClient.query(""" MATCH (n) WHERE n.communityId IS NOT NULL RETURN n.communityId AS communityId, COLLECT(n) AS members """) .fetch() .all(); for (Map<String, Object> community : communities) { Long communityId = (Long) community.get("communityId"); List<Map<String, Object>> members = (List<Map<String, Object>>) community.get("members"); // 构建社区描述 String description = buildCommunityDescription(members); // 使用 LLM 生成摘要 String summary = chatClient.prompt() .system("你是一个知识图谱分析专家。") .user("为以下社区生成简洁的摘要：\n{description}") .call() .content(); // 存储摘要 neo4jClient.query(""" MERGE (c:Community {id: $communityId}) SET c.summary = $summary """) .bind("communityId", communityId) .bind("summary", summary) .run(); } } private String buildCommunityDescription( List<Map<String, Object>> members) { StringBuilder sb = new StringBuilder(); for (Map<String, Object> member : members) { sb.append(String.format("- %s (%s): %s\n", member.get("name"), member.get("labels"), member.getOrDefault("description", ""))); } return sb.toString(); } }

    📄 Local Search：从特定实体出发

    Local Search 适用于针对特定实体的查询。它从查询中提取实体，在图谱中查找相关子图，结合社区上下文生成回答：

    @Service public class GraphRagService { // Local Search: 从特定实体出发 public String localSearch(String query) { // 1. 从查询中提取实体 List<String> entities = extractEntities(query); // 2. 在图谱中查找相关子图 List<Map<String, Object>> subgraph = neo4jClient.query(""" MATCH (e)-[r]-(connected) WHERE e.name IN $entities RETURN e, r, connected LIMIT 50 """) .bind("entities", entities) .fetch() .all(); // 3. 获取社区摘要作为上下文 String communityContext = getCommunityContext(entities); // 4. 获取直接邻居的详细信息 String neighborContext = formatSubgraph(subgraph); // 5. 生成回答 return chatClient.prompt() .system(""" 你是一个知识图谱问答助手。基于以下知识图谱信息回答问题。 图谱信息包含实体、关系和社区摘要。 如果图谱中没有足够信息，请明确说明。 """) .user(""" 图谱信息: {graph} 社区摘要: {community} 问题: {query} """) .call() .content(); } // 获取社区上下文 private String getCommunityContext(List<String> entities) { return neo4jClient.query(""" MATCH (e)-[:IN_COMMUNITY]->(c:Community) WHERE e.name IN $entities RETURN DISTINCT c.summary AS summary """) .bind("entities", entities) .fetch() .all() .stream() .map(m -> (String) m.get("summary")) .collect(Collectors.joining("\n---\n")); } // 格式化子图信息 private String formatSubgraph( List<Map<String, Object>> subgraph) { return subgraph.stream() .map(r -> { String from = getNodeName(r.get("e")); String to = getNodeName(r.get("connected")); String rel = getRelType(r.get("r")); return String.format("%s --[%s]--> %s", from, rel, to); }) .collect(Collectors.joining("\n")); } }

    📄 Global Search：全局主题查询

    Global Search 适用于全局性、宏观性的查询（如"AI 技术的发展趋势是什么？"）。它使用 Map-Reduce 模式，先让每个社区独立回答，再综合所有答案：

    public String globalSearch(String query) { // 1. 获取所有社区摘要 List<String> summaries = getAllCommunitySummaries(); // 2. Map 阶段：每个社区独立回答 List<String> partialAnswers = summaries.stream() .map(summary -> generatePartialAnswer(query, summary)) .toList(); // 3. Reduce 阶段：综合所有部分答案 return generateFinalAnswer(query, partialAnswers); } private String generatePartialAnswer( String query, String communitySummary) { return chatClient.prompt() .system(""" 基于以下社区摘要，判断该社区的信息是否与问题相关。 如果相关，生成部分答案；如果不相关，返回"不相关"。 社区摘要：{summary} """) .user(query) .call() .content(); } private String generateFinalAnswer( String query, List<String> partialAnswers) { // 过滤掉"不相关"的答案 List<String> relevantAnswers = partialAnswers.stream() .filter(a -> !a.equals("不相关")) .toList(); if (relevantAnswers.isEmpty()) { return "抱歉，知识图谱中没有足够信息来回答这个问题。"; } String combinedContext = String.join("\n---\n", relevantAnswers); return chatClient.prompt() .system(""" 你是一个知识图谱分析专家。基于以下来自多个社区的部分答案， 综合生成一个全面、准确的最终回答。 如果不同社区的答案存在矛盾，请指出并分析。 """) .user(""" 问题: {query} 部分答案: {answers} """) .call() .content(); }
    💡 Local vs Global 选型

    Local Search 适合精确查询："张三和李四之间有什么关系？"、"Spring AI 的核心组件有哪些？"。它快速、精准，但缺乏全局视角。Global Search 适合宏观查询："AI Agent 技术的发展趋势是什么？"、"不同团队的技术栈有什么共同点？"。它全面、深入，但成本较高（需要处理所有社区摘要）。实际项目中，可以先用 Local Search，当查询范围较广时自动切换到 Global Search。

    📄 混合检索：向量搜索 + 图谱查询

    GraphRAG 的最佳实践是将向量搜索和图谱查询结合使用，取长补短：

    @Service public class HybridRagService { private final VectorStore vectorStore; private final GraphRagService graphRagService; private final ChatClient chatClient; public String hybridSearch(String query) { // 1. 向量搜索：找到语义最相关的文档片段 SearchRequest request = SearchRequest.query(query) .withTopK(5) .withSimilarityThreshold(0.7); List<Document> docs = vectorStore.similaritySearch(request); // 2. 图谱搜索：找到实体间的关系 String graphResult = graphRagService.localSearch(query); // 3. 综合两种检索结果生成回答 String vectorContext = docs.stream() .map(Document::getContent) .collect(Collectors.joining("\n---\n")); return chatClient.prompt() .system(""" 你是一个智能问答助手。综合以下两种信息源生成回答： 1. 向量搜索结果（文档片段） 2. 知识图谱查询结果（实体关系） 优先使用图谱中的结构化关系，向量搜索提供补充上下文。 """) .user(""" 向量搜索结果: {vectorContext} 知识图谱结果: {graphResult} 问题: {query} """) .call() .content(); } }
    📚 混合检索的优势

    向量搜索擅长处理语义相似性，能找到措辞不同但含义相同的文档。图谱查询擅长处理结构化关系，能进行多跳推理。两者结合，可以覆盖更广泛的查询场景：精确的事实查询用图谱，模糊的语义查询用向量，复杂问题两者并用。

    📄 核心内容（第5节）：知识图谱 + Agent 联动 -- 结构化推理

    📄 Agent 使用知识图谱作为 Tool

    将知识图谱封装为 Agent 的 Tool，可以让 Agent 自主决定何时查询图谱、如何推理：

    @Service public class KnowledgeGraphTool { private final Neo4jClient neo4jClient; public KnowledgeGraphTool(Neo4jClient neo4jClient) { this.neo4jClient = neo4jClient; } @Tool(description = "查询知识图谱，找到实体间的关系路径。输入起始实体和目标实体。") public String queryGraph( @ToolParam(description = "起始实体名称") String startEntity, @ToolParam(description = "目标实体名称（可选）") String endEntity ) { if (endEntity != null) { // 路径查询：找到两个实体之间的路径 return neo4jClient.query(""" MATCH path = shortestPath( (a {name: $start})-[*]-(b {name: $end}) ) RETURN [n IN nodes(path) | n.name] AS path_nodes, [r IN relationships(path) | type(r)] AS path_relations """) .bind("start", startEntity) .bind("end", endEntity) .fetch() .one() .map(this::formatPath) .orElse("未找到路径"); } else { // 邻居查询：找到实体的所有直接关系 return neo4jClient.query(""" MATCH (a {name: $name})-[r]-(b) RETURN b.name AS name, type(r) AS relation, labels(b) AS type LIMIT 20 """) .bind("name", startEntity) .fetch() .all() .stream() .map(this::formatRelation) .collect(Collectors.joining("\n")); } } @Tool(description = "查找图谱中与某个主题相关的所有实体和关系。") public String searchByTopic( @ToolParam(description = "搜索关键词") String keyword ) { return neo4jClient.query(""" MATCH (n) WHERE n.name CONTAINS $keyword OR n.description CONTAINS $keyword OPTIONAL MATCH (n)-[r]-(m) RETURN n.name AS node, COLLECT({ relation: type(r), target: m.name }) AS connections LIMIT 30 """) .bind("keyword", keyword) .fetch() .all() .stream() .map(this::formatTopicResult) .collect(Collectors.joining("\n")); } private String formatPath(Map<String, Object> result) { List<String> nodes = (List<String>) result.get("path_nodes"); List<String> relations = (List<String>) result.get("path_relations"); StringBuilder sb = new StringBuilder(); for (int i = 0; i < nodes.size() - 1; i++) { sb.append(nodes.get(i)) .append(" --[") .append(relations.get(i)) .append("]--> "); } sb.append(nodes.get(nodes.size() - 1)); return sb.toString(); } private String formatRelation(Map<String, Object> result) { return String.format("%s --[%s]--> %s (%s)", "起始节点", result.get("relation"), result.get("name"), result.get("type")); } private String formatTopicResult(Map<String, Object> result) { String node = (String) result.get("node"); List<Map<String, String>> connections = (List<Map<String, String>>) result.get("connections"); StringBuilder sb = new StringBuilder(); sb.append("📌 ").append(node).append("\n"); for (Map<String, String> conn : connections) { sb.append(" → ") .append(conn.get("relation")) .append(" → ") .append(conn.get("target")) .append("\n"); } return sb.toString(); } }
    // 在 Agent 配置中注册 Tool @Configuration public class AgentConfig { @Bean public ChatClient knowledgeAgent( ChatClient.Builder builder, KnowledgeGraphTool kgTool, VectorStore vectorStore) { return builder .defaultSystem(""" 你是一个知识丰富的助手，可以使用以下工具： 1. 知识图谱查询：查询实体关系、路径、主题 2. 文档搜索：搜索相关文档内容 请根据问题选择合适的工具。 """) .defaultTools(kgTool) .defaultAdvisors( new QuestionAnswerAdvisor(vectorStore)) .build(); } }

    📄 图谱推理 Chain：多跳推理

    多跳推理是知识图谱最强大的能力之一。通过 LangChain4j 或 Spring AI 的 Chain 机制，可以实现自动化的多步推理：

    @Service public class GraphReasoningChain { private final Neo4jClient neo4jClient; private final ChatClient chatClient; /** * 多跳推理：从起点出发，沿关系路径推理 */ public String multiHopReasoning( String startEntity, String question, int maxHops) { StringBuilder context = new StringBuilder(); String currentEntity = startEntity; // 逐跳探索 for (int hop = 1; hop <= maxHops; hop++) { List<Map<String, Object>> neighbors = neo4jClient.query(""" MATCH (a {name: $name})-[r]-(b) RETURN b.name AS neighbor, type(r) AS relation, labels(b) AS type ORDER BY b.importance DESC LIMIT 5 """) .bind("name", currentEntity) .fetch() .all(); if (neighbors.isEmpty()) break; // 记录当前跳的信息 context.append(String.format("第%d跳：%s\n", hop, currentEntity)); for (Map<String, Object> n : neighbors) { context.append(String.format( " → [%s] → %s (%s)\n", n.get("relation"), n.get("neighbor"), n.get("type"))); } // 使用 LLM 决定下一步探索方向 String nextStep = chatClient.prompt() .system(""" 你是一个图谱推理助手。基于已探索的信息， 判断应该继续探索哪个方向来回答用户的问题。 返回最相关的实体名称，或返回"停止"。 """) .user(""" 已探索信息： {context} 用户问题：{question} 应该继续探索哪个实体？ """) .call() .content(); if ("停止".equals(nextStep.trim())) break; currentEntity = nextStep.trim(); } // 基于收集的上下文生成最终回答 return chatClient.prompt() .system("基于以下图谱推理路径回答问题。") .user(""" 推理路径： {context} 问题：{question} """) .call() .content(); } }
    💡 推理深度控制

    多跳推理的深度（maxHops）需要权衡：太浅可能遗漏关键信息，太深可能导致"信息爆炸"和推理偏离。建议从 maxHops=2 开始，根据查询复杂度逐步增加。同时使用 LLM 进行"剪枝"，让 Agent 自主决定何时停止探索。

    @RestController @RequestMapping("/api/qa") public class IntelligentQAController { private final ChatClient knowledgeAgent; private final HybridRagService hybridRagService; private final GraphReasoningChain reasoningChain; @PostMapping public ResponseEntity<Map<String, Object>> ask( @RequestBody QARequest request) { // 根据查询类型选择策略 String answer; if (request.type() == QAType.FACTUAL) { // 事实查询：使用图谱精确查找 answer = knowledgeAgent.prompt() .user(request.question()) .call() .content(); } else if (request.type() == QAType.EXPLORATORY) { // 探索性查询：使用多跳推理 answer = reasoningChain.multiHopReasoning( request.startEntity(), request.question(), 3 ); } else { // 复杂查询：使用混合检索 answer = hybridRagService.hybridSearch( request.question()); } return ResponseEntity.ok(Map.of( "question", request.question(), "answer", answer, "type", request.type(), "timestamp", Instant.now() )); } } public record QARequest( String question, QAType type, String startEntity ) {} public enum QAType { FACTUAL, // 事实查询 EXPLORATORY, // 探索性查询 COMPLEX // 复杂查询 }

    📄 知识更新策略

    知识图谱不是一次构建就完成的，需要持续更新。以下是几种常见的更新策略：

    🔄 增量更新

    当新文档到达时，只提取新增的实体和关系，使用 MERGE 语句合并到图谱中。MERGE 会先检查是否存在，不存在则创建，存在则更新属性。

    🔃 版本化更新

    为实体和关系添加"有效期"属性（valid_from, valid_to），过期的不删除而是标记为历史版本。这保留了知识的演变过程。

    // 增量更新示例 public void incrementalUpdate(String newDocument) { // 1. 提取新实体和关系 List<EntityRelation> newRelations = extractRelations(newDocument); // 2. 使用 MERGE 写入（避免重复） for (EntityRelation rel : newRelations) { neo4jClient.query(""" MERGE (a:%s {name: $name1}) ON CREATE SET a.createdAt = datetime() ON MATCH SET a.updatedAt = datetime() MERGE (b:%s {name: $name2}) ON CREATE SET b.createdAt = datetime() ON MATCH SET b.updatedAt = datetime() MERGE (a)-[r:%s]->(b) ON CREATE SET r.createdAt = datetime(), r.source = $source ON MATCH SET r.updatedAt = datetime() """.formatted( rel.entity1Type(), rel.entity2Type(), rel.relation())) .bind("name1", rel.entity1()) .bind("name2", rel.entity2()) .bind("source", newDocument.substring(0, Math.min(100, newDocument.length()))) .run(); } }
    ⚠️ 数据质量保障

    图谱构建过程中必须进行数据质量检查：(1) 实体名称标准化 -- 统一大小写、去除特殊字符；(2) 关系方向验证 -- 确保关系方向符合业务语义；(3) 孤立节点清理 -- 定期删除没有关系的孤立节点；(4) 循环检测 -- 防止 A knows B knows A 这样的循环关系。

    📄 性能优化

    大规模图谱需要关注查询性能：

    • 索引优化-- 为高频查询属性创建索引。对 Person.name、Company.name 等必建索引。对复合查询条件创建复合索引。
    • 查询优化-- 避免全图遍历，始终使用 LIMIT 限制结果集大小。使用 Profile 查询分析器查看执行计划，避免笛卡尔积。
    • 缓存策略-- 对频繁访问的子图（如社区摘要）进行缓存。使用 Spring Cache 或 Redis 缓存图谱查询结果。
    • 分片策略-- 当图谱规模超过单机容量时，使用 Neo4j 的 Fabric（分片）或迁移到分布式图数据库如 JanusGraph。

    📝 面试高频题

    Q1：知识图谱和向量数据库有什么区别？在什么场景下应该选择知识图谱？

    向量数据库的核心是语义相似度搜索，适合非结构化内容的检索（如文档、图片）。知识图谱的核心是关系遍历和路径推理，适合需要理解实体间复杂关系的场景。选择标准：如果查询涉及"找到最相似的"，用向量数据库；如果查询涉及"找到关系路径"、"多跳推理"、"图谱可视化"，用知识图谱。实际项目中，两者通常互补使用。

    Q2：GraphRAG 和传统 RAG 有什么区别？GraphRAG 解决了什么问题？

    传统 RAG 将文档分块后存入向量数据库，查询时进行语义搜索。它的局限是：(1) 无法进行关系推理 -- 不能回答"A 和 B 之间有什么关系"；(2) 对全局性问题（如"所有团队的技术趋势"）效果差。GraphRAG 通过构建知识图谱、社区检测和摘要生成，解决了这两个问题。Local Search 支持从特定实体出发的关系查询，Global Search 支持全局主题的宏观分析。

    Q3：如何处理知识图谱中的实体消歧问题？

    实体消歧（Entity Resolution）是图谱构建中最困难的环节。常用策略：(1) 别名映射 -- 建立实体名称的同义词表（如"Spring AI" = "spring-ai" = "Spring人工智能"）；(2) LLM 消歧 -- 使用 LLM 判断两个实体是否指向同一事物；(3) 属性匹配 -- 基于实体的属性（如邮箱、URL）进行匹配；(4) 嵌入相似度 -- 将实体名称和描述编码为向量，计算相似度。生产环境中通常组合使用多种策略。

    Q4：GraphRAG 中的社区检测有什么作用？Leiden 算法有什么优势？

    社区检测将大规模图谱划分为有意义的"社区"，每个社区包含紧密相关的实体群组。社区的核心作用是：(1) 实现图谱的层次化理解 -- 每个社区可以独立生成摘要；(2) 支持 Global Search -- 通过遍历所有社区摘要回答全局性问题；(3) 降低复杂度 -- 将大规模图谱分解为可管理的子图。Leiden 算法相比 Louvain 算法的优势是：它保证社区结构的连通性，避免产生碎片化社区，且时间复杂度更低。

    Q5：在 Spring AI 中如何将知识图谱封装为 Agent 的 Tool？需要注意什么？

    使用 Spring AI 的 @Tool 注解将图谱查询方法封装为 Tool。设计要点：(1) Tool 描述要清晰 -- 明确说明输入参数和输出格式，让 Agent 能准确选择；(2) 返回格式要简洁 -- Agent 需要从 Tool 返回中提取关键信息，过长的返回会浪费 Token；(3) 错误处理要完善 -- 当图谱中不存在目标实体时，返回友好的提示而非异常；(4) 限制结果数量 -- 使用 LIMIT 防止返回过大子图，避免 Token 超限。

    • 将提取的知识写入 Neo4j
    • 编写 Cypher 查询，找到你与某位技术专家的"最短路径"
    • 使用 Neo4j Browser 可视化图谱

    进阶：为你的知识图谱添加社区检测，并为每个社区生成中文摘要。

    练习 2：实现 Local Search + Global Search

    基于一个包含 10+ 篇文档的语料库，构建 GraphRAG 系统：

    • 使用 Spring AI 将文档内容提取为知识图谱
    • 实现 Leiden 社区检测（可使用 Neo4j GDS 插件）
    • 为每个社区使用 LLM 生成摘要
    • 实现 Local Search：从特定实体出发，查询相关子图并生成回答
    • 实现 Global Search：Map-Reduce 模式，遍历所有社区摘要生成全局回答

    对比实验：分别用传统 RAG（向量搜索）和 GraphRAG 回答 10 个测试问题，对比准确率和相关性。

    练习 3：构建知识图谱 + Agent 的智能问答系统

    综合本章所学技术，构建一个完整的智能问答系统：

    • 准备一个包含技术文档的语料库（如 Spring 官方文档片段）
    • 使用第 4 章的知识实现传统 RAG
    • 使用本章知识实现 GraphRAG
    • 将知识图谱封装为 Agent 的 Tool
    • 实现混合检索：Agent 自主决定使用图谱查询还是向量搜索
    • 添加推理链：对于复杂问题，Agent 进行多跳推理
    • 构建 REST API，支持事实查询、探索性查询和复杂查询三种模式

    挑战：添加图谱增量更新能力，当新文档到达时自动更新知识图谱。
