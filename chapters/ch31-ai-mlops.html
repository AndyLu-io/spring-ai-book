第 31 章

AI MLOps 与数据飞轮

从"一次性部署"到"持续进化" -- 掌握模型版本管理、A/B 测试、自动重训练和数据飞轮，让 Agent 在生产环境中越用越聪明。

⏱ 预计 3-4 周 📖 6 个知识点 💻 高级难度

📚 开篇故事：从手工匠人到智能工厂

一个团队花了三个月精心训练了一个客服 Agent。他们用 10 万条历史对话做微调，用精心设计的 Prompt 做 RAG，上线第一天就获得了 92% 的满意度。产品经理喜极而泣："我们的 Agent 太聪明了！"

但三个月后，用户投诉越来越多。满意度从 92% 跌到了 71%。用户开始抱怨："它连我们新出的套餐都不知道"、"上次问退款流程，它还在用旧的标准"、"它推荐的产品早就下架了"。

原因很简单：业务在变、产品在更新、用户习惯在迁移，而 Agent 的知识停留在三个月前。就像一个厨师精心学会了做川菜，但餐厅改成了粤菜馆 -- 他的手艺再好，做的菜也不是客人想要的。

团队试过手动更新知识库，但业务迭代太快，新功能每周都在上线，手动更新永远跟不上节奏。他们意识到：Agent 需要的不是一次性的"训练完就上线"，而是一个持续进化的工程体系。

这就是 MLOps 要解决的问题。MLOps 不是某个工具，而是一套完整的工程实践 -- 它让 AI 系统像活的有机体一样，能感知环境变化、自动调整策略、持续自我进化。数据飞轮（Data Flywheel）则是 MLOps 的核心引擎：用户使用产生数据，数据驱动模型改进，更好的模型吸引更多用户，更多用户产生更多数据 -- 飞轮一旦转起来，Agent 就会越用越聪明。

🎯 学习目标

在本章结束后，你将能够：

  • 理解 MLOps 的核心概念与生命周期，掌握它与传统 DevOps 的异同点
  • 设计并实现模型版本管理系统，包括模型注册表、版本策略和血缘追踪
  • 构建 A/B 测试与灰度发布体系，让数据驱动模型选择，而非拍脑袋决策
  • 实现数据飞轮闭环：从用户反馈收集、自动标注到触发重训练的完整流水线
  • 掌握生产环境的监控告警与自动回滚机制，包括数据漂移检测和模型退化防护

📄 核心内容（第1节）：MLOps 基础 -- AI 系统的 DevOps

📄 什么是 MLOps

MLOps（Machine Learning Operations）是将 DevOps 理念应用于机器学习系统的工程实践。如果说 DevOps 解决的是"如何让软件持续、可靠地交付"，那么 MLOps 解决的就是"如何让 AI 模型持续、可靠地进化"。

传统软件的核心资产是代码，代码修改后重新编译部署即可。但 AI 系统的核心资产是模型 + 数据，它们之间的依赖关系远比代码复杂：模型效果取决于数据质量，数据分布会随时间漂移，模型训练结果具有不确定性。因此，MLOps 需要一套完全不同于传统 DevOps 的流程和工具。

📄 MLOps 生命周期

MLOps 的核心是一个持续循环的生命周期，而非传统的线性流水线：

┌──────────┐ → ┌──────────┐ → ┌──────────┐ │ 数据准备 │ │ 模型训练 │ │ 模型评估 │ │ (采集/清洗) │ │ (微调/验证) │ │ (指标/对比) │ └──────────┘ └──────────┘ └──────────┘ ↑ ↓ ┌──────────┐ ← ┌──────────┐ ← ┌──────────┐ │ 反馈收集 │ │ 模型监控 │ │ 模型部署 │ │ (日志/标注) │ │ (漂移检测) │ │ (A/B/灰度) │ └──────────┘ └──────────┘ └──────────┘

每个环节都不是孤立的，而是紧密咬合的齿轮：

  • 数据准备 -- 从生产环境收集用户交互日志、反馈数据、新增知识，清洗后作为训练数据
  • 模型训练 -- 使用新数据对模型进行增量微调或完全重训练
  • 模型评估 -- 在离线测试集上评估新模型，与当前生产模型进行对比
  • 模型部署 -- 通过 A/B 测试或灰度发布将新模型推送到生产环境
  • 模型监控 -- 实时监控模型的各项指标，检测数据漂移和性能退化
  • 反馈收集 -- 收集用户对模型输出的显式和隐式反馈，闭环回到数据准备

💡 Spring AI 生态的 MLOps 定位

Spring AI 并不直接提供完整的 MLOps 工具链，但它在模型服务层和反馈收集层扮演关键角色。Spring AI 的 ChatClient、ModelClient 提供了统一的模型调用抽象，可以无缝切换底层模型；通过 Spring Boot 的 Actuator 和 Micrometer，可以轻松集成 Prometheus + Grafana 实现监控；通过 Spring 的事件机制，可以构建完整的反馈收集管道。

📄 核心内容（第2节）：模型版本管理 -- 给每个模型打上身份证

📄 为什么需要模型版本管理

想象这样一个场景：你的 Agent 上线了一个微调模型，效果不错。一周后又上线了另一个版本。两周后产品经理问："上周三用户反馈突然变差，是不是新模型的问题？"你打开服务器发现：旧模型的权重文件已经被覆盖了，训练时的超参数也没记录，数据集版本更是模糊不清。

模型版本管理解决的核心问题是：确保每一次模型变更都可追溯、可复现、可回滚。它管理的不仅是模型权重文件，还包括：

  • 模型产物 -- 权重文件、配置文件、Tokenizer、预处理逻辑
  • 训练元数据 -- 超参数、训练数据版本、训练环境、训练耗时
  • 评估指标 -- 离线评估指标、对比测试结果、人工评审记录
  • 血缘关系 -- 这个模型用了哪些数据、从哪个基础模型微调而来

📄 模型注册表设计

模型注册表（Model Registry）是 MLOps 的核心组件之一。它本质上是一个模型的生命周期管理系统，管理模型从注册到退役的全过程：

┌──────────┐ → ┌──────────┐ → ┌──────────┐ │ REGISTERED │ │ STAGING │ │ PRODUCTION│ │ (已注册) │ │ (预发布) │ │ (生产中) │ └──────────┘ └──────────┘ └──────────┘ ↓ ┌──────────┐ │ ARCHIVED │ │ (已归档) │ └──────────┘

模型版本的命名策略通常采用三件套：

  • 语义版本号 -- 如 v1.2.3，主版本号变更表示不兼容的重大更新，次版本号表示新功能，修订号表示 Bug 修复
  • 数据版本标识 -- 如 data-20240115，记录训练所用的数据集版本
  • 代码版本标识 -- 如 commit-abc1234，关联训练代码的 Git 提交

📄 Java 代码实现：模型注册服务

@Service public class ModelRegistryService { private final ModelRepository modelRepository; public ModelVersion registerModel(String modelName, String artifactPath, Map<String, String> metadata) { String version = generateVersion(modelName); ModelVersion mv = ModelVersion.builder() .name(modelName) .version(version) .artifactPath(artifactPath) .status(ModelStatus.REGISTERED) .metadata(metadata) .createdAt(Instant.now()) .build(); modelRepository.save(mv); return mv; } public ModelVersion promoteToProduction(String modelName, String version) { ModelVersion mv = modelRepository.findByNameAndVersion(modelName, version); mv.setStatus(ModelStatus.PRODUCTION); mv.setPromotedAt(Instant.now()); // 自动回滚准备：将当前生产版本归档 ModelVersion previous = modelRepository .findByNameAndStatus(modelName, ModelStatus.PRODUCTION); if (previous != null) { previous.setStatus(ModelStatus.ARCHIVED); modelRepository.save(previous); } return modelRepository.save(mv); } }
// 模型版本实体 @Entity @Builder @Data public class ModelVersion { @Id @GeneratedValue private Long id; private String name; // 模型名称 private String version; // 版本号 private String artifactPath; // 模型文件路径 @Enumerated(EnumType.STRING) private ModelStatus status; // 模型状态 @ElementCollection private Map<String, String> metadata; // 元数据 private String dataVersion; // 训练数据版本 private String gitCommitHash; // 代码版本 private Instant createdAt; private Instant promotedAt; } public enum ModelStatus { REGISTERED, // 已注册 STAGING, // 预发布 PRODUCTION, // 生产中 ARCHIVED // 已归档 }

📄 模型血缘追踪

模型血缘（Model Lineage）回答的是"这个模型从哪里来"的问题。一条完整的血缘记录包括：

  • 基础模型 -- 从哪个预训练模型开始（如 gpt-4、llama-3、qwen-2）
  • 训练数据 -- 使用了哪个版本的数据集，数据量和数据分布
  • 训练配置 -- 学习率、Epoch 数、Batch Size、LoRA rank 等超参数
  • 评估结果 -- 在哪些测试集上评估，各项指标得分
💡 血缘追踪的工程实践

在生产环境中，推荐使用三元组来标识模型血缘：(data_version, code_version, base_model_version)。这三个维度确保了实验的完全可复现性。即使一年后想复现某个模型的效果，只要这三个版本号不变，就能还原完整的训练环境。

📄 核心内容（第3节）：A/B 测试与灰度发布 -- 让数据说话

📄 A/B 测试在 AI 系统中的特殊性

传统软件的 A/B 测试关注的是"哪个版本的 UI 点击率更高"，而 AI 系统的 A/B 测试需要关注更多维度：

  • 质量维度 -- 模型的回答是否更准确、更相关、更安全
  • 延迟维度 -- 新模型的推理延迟是否在可接受范围内
  • 成本维度 -- Token 消耗、API 调用成本是否有变化
  • 用户体验维度 -- 用户是否更满意、更愿意继续使用

AI 系统的 A/B 测试还有一个特殊挑战：评估指标的模糊性。传统软件的指标（点击率、转化率）是明确的，但 AI 系统的"回答质量"往往需要人工评估或 LLM-as-Judge 来打分。

📄 分流策略

用户级分流

同一个用户始终看到同一个模型版本。使用一致性哈希，确保用户在实验期间不会在不同版本间跳转。

适用场景：需要长期观察用户行为变化的实验。

请求级分流

同一个用户的不同请求可能被路由到不同模型。每次请求独立随机分配。

适用场景：快速收集大量对比数据的短期实验。

特征级分流

根据请求的特征（如问题类型、语言、用户等级）进行差异化分流。例如：简单问题走小模型，复杂问题走大模型。

适用场景：需要精细化控制模型分配的智能路由系统。

📄 A/B 测试服务实现

@Service public class AbTestService { private final TrafficRouter trafficRouter; private final MetricsCollector metrics; public String routeRequest(String userId, String prompt) { String variant = trafficRouter.getVariant(userId); String response; long startTime = System.currentTimeMillis(); if ("control".equals(variant)) { response = callModel("gpt-4", prompt); } else { response = callModel("gpt-4-finetuned-v2", prompt); } long latency = System.currentTimeMillis() - startTime; // 记录指标 metrics.record(variant, Map.of( "latency_ms", latency, "token_count", countTokens(response), "quality_score", evaluateQuality(prompt, response) )); return response; } public AbTestResult getResults(String experimentId) { Map<String, VariantMetrics> variants = metrics.getVariants(experimentId); // 计算统计显著性 double pValue = calculatePValue( variants.get("control"), variants.get("treatment") ); return AbTestResult.builder() .experimentId(experimentId) .variants(variants) .pValue(pValue) .isSignificant(pValue < 0.05) .recommendedWinner( pValue < 0.05 ? selectWinner(variants) : "continue_testing" ) .build(); } }

📄 灰度发布策略

灰度发布是 A/B 测试在生产环境中的实践，核心思想是渐进式放量：

灰度发布三阶段 阶段 1: 金丝雀发布 (1-5% 流量) → 将新模型暴露给少量用户 → 监控 24-48 小时 → 核心指标无异常才进入下一阶段 阶段 2: 渐进式扩量 (5% → 20% → 50%) → 每个阶段观察 12-24 小时 → 任一阶段发现异常立即回滚 → 核心指标持续优于对照组 阶段 3: 全量切换 (100%) → 新模型接管所有流量 → 旧模型保留在注册表中 30 天 → 随时可一键回滚
⚠️ 灰度发布的红线

在 AI 系统的灰度发布中，除了传统的错误率和延迟指标，还必须监控安全指标：有害内容生成率、幻觉率、信息泄露风险。一旦安全指标触达阈值，必须立即回滚，无需等待人工审批。安全红线不可妥协。

📄 核心内容（第4节）：数据飞轮 -- 让 Agent 越用越聪明

📄 数据飞轮的核心循环

数据飞轮（Data Flywheel）是 MLOps 的核心引擎，其本质是一个正反馈循环：用户使用 Agent 产生数据，数据驱动模型改进，更好的模型吸引更多用户，更多用户产生更多数据。

数据飞轮闭环 用户交互 → 日志收集 → 反馈分类 → 数据标注 → 模型训练 → 部署上线 ↑ ↓ └──────────────────── 更好的用户体验 ←──────────────────────────────┘

飞轮启动的关键是冷启动策略：在用户数据不足时，如何获得第一批有价值的训练数据？常见策略包括：

  • 种子数据 -- 使用人工编写的高质量样本作为初始训练集
  • 合成数据 -- 使用强模型（如 GPT-4）为弱模型生成训练数据
  • 迁移数据 -- 从相似场景的已有数据中迁移

📄 反馈信号采集

显式反馈

用户主动提供的评价信号：

  • 点赞 / 点踩按钮
  • 评分（1-5 星）
  • 文字评价
  • "这个回答有帮助吗？"

优点：信号明确，质量高。

缺点：覆盖率低（通常只有 1-5% 的交互会产生显式反馈）。

隐式反馈

从用户行为中推断的评价信号：

  • 回答后的停留时间（短 = 可能不满意）
  • 是否重新提问（重试 = 回答不好）
  • 是否采纳了建议（点击 = 有帮助）
  • 会话是否中途放弃（放弃 = 体验差）

优点：覆盖率高，无额外用户负担。

缺点：信号噪声大，需要大量数据才能提取有效模式。

📄 主动学习与自动标注

主动学习（Active Learning）是数据飞轮的"智能调度器"。它不是盲目地标注所有数据，而是智能选择最有价值的数据进行标注：

  • 不确定性采样 -- 选择模型最不确定的样本（如两个选项概率接近的分类任务）
  • 多样性采样 -- 选择与已标注数据差异最大的样本，扩大数据覆盖范围
  • 预期模型变化 -- 选择标注后对模型改变最大的样本

自动标注则是用"强模型标注弱模型的训练数据"。例如：用 GPT-4 对用户问题进行质量评分，然后用这些评分数据训练一个轻量级的分类器，这个分类器可以直接在生产环境中运行，无需每次都调用 GPT-4。

📚 标注成本优化公式

假设人工标注一条数据的成本是 $5，GPT-4 自动标注的成本是 $0.02。如果自动标注的准确率达到人工标注的 90%，那么用自动标注 + 10% 人工审核的方式，可以在保持质量的同时将标注成本降低 30 倍。

📄 数据飞轮服务实现

@Service public class DataFlywheelService { private final FeedbackRepository feedbackRepo; private final DatasetBuilder datasetBuilder; private final ModelTrainer trainer; // 收集用户反馈 public void collectFeedback(String sessionId, String prompt, String response, FeedbackType type) { FeedbackEvent event = FeedbackEvent.builder() .sessionId(sessionId) .prompt(prompt) .response(response) .feedbackType(type) .timestamp(Instant.now()) .build(); feedbackRepo.save(event); } // 自动触发重训练评估（每天检查一次） @Scheduled(fixedDelay = 86400000) public void evaluateRetraining() { long newFeedbackCount = feedbackRepo.countSinceLastTraining(); double avgScore = feedbackRepo.getAverageScoreSinceLastTraining(); if (newFeedbackCount >= 1000 || avgScore < 0.7) { // 触发重训练 Pipeline Dataset dataset = datasetBuilder.buildFromFeedback( feedbackRepo.getSinceLastTraining() ); trainer.submitTrainingJob(dataset); } } }
// 数据集构建器 -- 从反馈到训练数据 @Service public class DatasetBuilder { private final AnnotationService annotationService; private final DataQualityFilter qualityFilter; public Dataset buildFromFeedback(List<FeedbackEvent> feedbacks) { // 1. 过滤低质量数据 List<FeedbackEvent> filtered = qualityFilter.filter(feedbacks); // 2. 对正面反馈：prompt + response 作为 SFT 训练对 List<TrainingPair> positivePairs = filtered.stream() .filter(f -> f.getFeedbackType() == FeedbackType.POSITIVE) .map(f -> new TrainingPair(f.getPrompt(), f.getResponse())) .toList(); // 3. 对负面反馈：使用强模型重新生成更好的回答 List<TrainingPair> correctedPairs = filtered.stream() .filter(f -> f.getFeedbackType() == FeedbackType.NEGATIVE) .map(f -> annotationService.generateBetterResponse(f)) .toList(); // 4. 合并数据集 List<TrainingPair> allPairs = new ArrayList<>(positivePairs); allPairs.addAll(correctedPairs); return Dataset.builder() .name("flywheel-" + Instant.now().toEpochMilli()) .pairs(allPairs) .build(); } }

📄 核心内容（第5节）：监控告警与自动回滚 -- 生产环境的守护神

📄 AI 系统特有的监控指标

传统软件监控关注的是"系统是否正常运行"，AI 系统还需要关注"模型是否还在正确地工作"。AI 系统特有的三大监控维度：

监控维度含义检测方法
数据漂移输入数据的分布发生了变化PSI、KL 散度、KS 检验
模型漂移模型预测结果的分布发生了变化预测分布对比、置信度监控
性能退化模型在关键指标上持续下降滑动窗口均值、趋势检测
⚠️ 数据漂移 vs 模型漂移

数据漂移（Data Drift）是指输入数据的分布发生了变化，但模型本身的参数没有变。比如用户突然开始问很多关于新产品的问題，而模型训练时没有见过这些产品。模型漂移（Concept Drift）是指数据和标签之间的关系发生了变化。比如"退款"这个词以前总是正面情绪（"退款很顺利"），现在变成了负面情绪（"退款太慢了"）。两者都会导致模型效果下降，但应对策略不同。

📄 漂移检测算法

三种最常用的漂移检测指标：

PSI（Population Stability Index）

衡量两个分布之间差异的指标。PSI < 0.1 表示无显著漂移，0.1-0.25 表示中度漂移，> 0.25 表示严重漂移。

KL 散度

衡量两个概率分布之间差异的信息论指标。KL 散度越大，两个分布的差异越大。注意 KL 散度是非对称的。

KS 检验（Kolmogorov-Smirnov Test）

非参数检验方法，比较两个样本的累积分布函数。KS 统计量越大，两个分布的差异越显著。

📄 漂移检测服务实现

@Service public class DriftDetector { public DriftReport checkDataDrift(List<Vector> referenceData, List<Vector> currentData) { // 计算 PSI (Population Stability Index) double psi = calculatePSI(referenceData, currentData); // 计算特征级漂移 Map<String, Double> featureDrift = new HashMap<>(); for (int i = 0; i < referenceData.get(0).dimensions(); i++) { double ks = calculateKS( extractFeature(referenceData, i), extractFeature(currentData, i) ); featureDrift.put("feature_" + i, ks); } return DriftReport.builder() .overallPSI(psi) .featureDrift(featureDrift) .isDrifting(psi > 0.2) // PSI > 0.2 表示显著漂移 .severity(psi > 0.25 ? "HIGH" : psi > 0.1 ? "MEDIUM" : "LOW") .build(); } private double calculatePSI(List<Vector> reference, List<Vector> current) { int bins = 10; double[] refDist = histogram(reference, bins); double[] curDist = histogram(current, bins); double psi = 0; for (int i = 0; i < bins; i++) { if (refDist[i] > 0 && curDist[i] > 0) { psi += (curDist[i] - refDist[i]) * Math.log(curDist[i] / refDist[i]); } } return psi; } }

📄 自动回滚策略

自动回滚是 MLOps 的"安全气囊"。当模型在生产环境中出现异常时，系统能自动将流量切换回上一个稳定版本。

@Service public class AutoRollbackService { private final DriftDetector driftDetector; private final MetricsCollector metrics; private final ModelRegistryService registryService; private final AlertService alertService; @Scheduled(fixedRate = 300000) // 每 5 分钟检查一次 public void checkAndRollback() { // 1. 检查模型性能指标 double currentAccuracy = metrics.getSlidingWindowAccuracy( Duration.ofHours(1) ); double baselineAccuracy = metrics.getBaselineAccuracy(); // 2. 检查数据漂移 DriftReport driftReport = driftDetector.checkDataDrift( metrics.getReferenceData(), metrics.getCurrentData(Duration.ofHours(1)) ); // 3. 判断是否需要回滚 boolean accuracyDropped = (baselineAccuracy - currentAccuracy) > 0.05; boolean severeDrift = "HIGH".equals(driftReport.getSeverity()); if (accuracyDropped || severeDrift) { // 4. 执行自动回滚 ModelVersion previousVersion = registryService .getPreviousProductionVersion(); if (previousVersion != null) { registryService.promoteToProduction( previousVersion.getName(), previousVersion.getVersion() ); // 5. 发送告警 alertService.sendAlert( AlertLevel.CRITICAL, String.format( "模型已自动回滚: %s → %s. 原因: %s", getCurrentModelId(), previousVersion.getVersion(), accuracyDropped ? "精度下降" : "数据漂移" ) ); } } } }
💡 回滚策略的最佳实践

自动回滚的关键是渐进而非突变。推荐策略：(1) 第一次异常 -- 发送告警，人工确认；(2) 第二次异常 -- 自动回滚 50% 流量；(3) 第三次异常 -- 全量回滚。同时，回滚后必须保留现场（日志、指标快照、输入样本），以便事后分析根因。

💡 MLOps 成熟度模型

MLOps 的成熟度分为三个级别：MLOps Level 0（手动流程）-- 模型训练和部署全靠手动操作；MLOps Level 1（自动化 Pipeline）-- 训练和部署流水线自动化，但需要人工触发；MLOps Level 2（完全自动化）-- 数据飞轮自动运转，模型自动重训练、自动评估、自动部署、自动回滚。大多数团队从 Level 0 开始，目标是在 6-12 个月内达到 Level 1。

📖 章节小结

本章我们学习了 AI MLOps 与数据飞轮的完整体系：

  • MLOps 基础 -- MLOps 是将 DevOps 理念应用于 AI 系统的工程实践，核心是"持续进化"而非"一次性部署"。它与传统 DevOps 的区别在于管理的对象从代码扩展到了模型 + 数据 + 代码
  • 模型版本管理 -- 通过模型注册表管理模型的全生命周期（注册→预发布→生产→归档），使用三件套（语义版本 + 数据版本 + 代码版本）确保可追溯性和可复现性
  • A/B 测试与灰度发布 -- 通过用户级/请求级/特征级分流策略，让数据驱动模型选择。灰度发布采用金丝雀→渐进式→全量的三阶段策略，确保安全上线
  • 数据飞轮 -- 用户交互→数据收集→反馈分类→数据标注→模型训练→部署上线的闭环。通过主动学习和自动标注降低成本，通过显式+隐式反馈提升数据覆盖率
  • 监控告警与自动回滚 -- 数据漂移（PSI、KL 散度、KS 检验）、模型漂移、性能退化三维度监控。自动回滚策略：渐进而非突变，保留现场以便分析
  • Spring AI 集成 -- 通过 Actuator、Events、Scheduling、Integration、Cloud Config 等组件，将 MLOps 实践无缝集成到 Spring Boot 生态中
💡 核心理念

MLOps 的终极目标不是"部署模型"，而是"构建一个让模型持续进化的系统"。数据飞轮一旦转起来，Agent 就不再是静态的工具，而是一个会学习、会成长的智能伙伴。记住：最好的模型不是一次性训练出来的，而是在生产环境中持续打磨出来的。

🎓 面试高频题

Q1：MLOps 和传统 DevOps 有什么区别？AI 系统的 MLOps 有什么特殊挑战？

核心区别有三点：(1) 资产管理：DevOps 管理代码，MLOps 管理代码 + 数据 + 模型三者，版本管理更复杂；(2) 测试重点：DevOps 侧重功能和集成测试，MLOps 还需要数据验证、模型评估和漂移检测；(3) 反馈闭环：DevOps 的反馈是修复 Bug，MLOps 的反馈是收集数据→重训练→重新部署，周期更长、成本更高。AI 系统的特殊挑战是数据漂移和模型退化 -- 模型上线后效果会随时间自然下降，需要持续监控和更新。

Q2：什么是模型版本管理？为什么不能只用 Git 管理模型？

模型版本管理是管理模型从注册到退役的全生命周期。Git 无法完全替代的原因：(1) 体积问题：模型权重文件通常在 GB 级别，不适合放在 Git 仓库中；(2) 元数据：模型需要记录训练数据版本、超参数、评估指标等元数据，Git 只记录代码变更；(3) 阶段流转：模型需要从注册→预发布→生产→归档的生命周期管理，Git 没有这个概念；(4) 审批流程：模型上线通常需要人工审核，需要工作流引擎支持。推荐使用 MLflow Model Registry 或自研注册表管理模型。

Q3：A/B 测试中如何判断结果是否具有统计显著性？样本量如何确定？

判断统计显著性通常使用假设检验：(1) 设定零假设（两个模型无差异）和备择假设；(2) 选择检验方法（连续变量用 t 检验，比例变量用卡方检验或 z 检验）；(3) 计算 p 值，通常以 0.05 为显著性阈值。样本量确定需要考虑：(1) 预期效应量 -- 效应越小需要越多样本；(2) 显著性水平 -- 通常取 0.05；(3) 统计功效 -- 通常取 0.8。可以使用 A/B 测试样本量计算器提前估算。在 AI 系统中，由于评估指标（如回答质量）方差较大，通常需要比传统 A/B 测试更多的样本量。

Q4：数据飞轮的冷启动问题如何解决？如何在用户数据不足时启动飞轮？

冷启动是数据飞轮最困难的阶段。常用策略：(1) 种子数据 -- 人工编写高质量样本作为初始训练集，确保覆盖核心场景；(2) 合成数据 -- 使用强模型（如 GPT-4）为弱模型生成训练数据（Self-Instruct 方法）；(3) 迁移学习 -- 从相似场景的已有数据中迁移，如从通用对话数据迁移到客服场景；(4) 规则兜底 -- 在数据不足时用规则引擎补充，确保基本功能可用；(5) 人工在环 -- 初期采用"AI 建议 + 人工确认"模式，同时积累训练数据。关键是不要追求完美，先让飞轮转起来，哪怕初始质量一般。

Q5：如何设计一个完整的 AI 系统监控方案？需要监控哪些关键指标？

完整的 AI 监控方案应覆盖四层：(1) 基础设施层 -- CPU/GPU 利用率、内存、网络延迟、服务可用性；(2) 模型性能层 -- 推理延迟（P50/P95/P99）、吞吐量、Token 消耗、错误率；(3) 模型质量层 -- 数据漂移（PSI）、模型漂移（预测分布变化）、精度退化（滑动窗口均值）；(4) 业务指标层 -- 用户满意度、任务完成率、用户留存率。告警分级：LOW（记录日志）、MEDIUM（通知团队）、HIGH（自动回滚 + 紧急通知）、CRITICAL（全量回滚 + 值班电话）。关键原则：宁可多报不可漏报，误报可以后续优化，漏报可能导致严重后果。
