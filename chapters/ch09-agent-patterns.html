第 09 章

Agent 高级模式

从简单的 ReAct 到复杂的树搜索，掌握 Agent 设计模式的精髓，让 AI 不只是"回答问题"，而是"解决问题"。

⏱ 预计 3-4 周 📖 6 个知识点 💻 高级难度
本章目录

    📚 开篇故事：从学徒到大师

    很久以前，有一个叫小明的程序员，他第一次听说 Agent 的概念时兴奋不已。他用 ReAct 模式写出了第一个 Agent — 它能思考、能行动、能观察结果。小明觉得这已经够厉害了。

    但很快他发现，这个 Agent 像一个只有一把锤子的学徒 — 无论什么问题，它都用同一种方式尝试。遇到简单问题还好，但面对复杂任务时，它经常走弯路，甚至陷入死循环。

    于是小明开始研究更高阶的模式。他学会了Plan-Execute，让 Agent 先制定计划再执行，就像一个项目经理；他学会了 Reflection，让 Agent 执行后自我反思，就像一个不断复盘的运动员；他学会了 LATS，让 Agent 同时探索多条路径，像一个经验丰富的侦探在脑中模拟多种破案方案。

    三年后，小明设计的 Agent 系统已经能自主完成复杂的软件开发任务。从学徒到大师，Agent 的进化之路，就是设计模式的进化之路。

    🔄 ReAct 进阶：反思与自我纠错

    概念比喻：Agent 的"复盘日记"

    基础 ReAct 就像一个只顾埋头干活的工人。而进阶版 ReAct 增加了反思环节 — 每完成一步，Agent 会停下来问自己："这一步做得对吗？结果符合预期吗？需要调整吗？"

    就像一个优秀的厨师不会只是按菜谱做菜，而是每一步都在尝味道、调整火候。

    进阶 ReAct 循环

  ┌──────────┐
  │ 用户请求 │
  └────┬─────┘
       v
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ 思考(Think) ───> 行动(Act) ───> 观察(Observe)
  └────┬─────┘    └──────────┘    └────┬─────┘
       ^                               |
       |         ┌──────────┐          |
       +─────────│ 反思(Reflect)│<──────────┘
                 └────┬─────┘
                      |
                      v  满意？──> 输出结果
                  不满意？──> 重新思考

    代码示例：带反思循环的 Agent

    // 带反思循环的 ReAct Agent @Service public class ReflectiveReActAgent { private final ChatClient chatClient; private final ToolCallbackResolver toolResolver; private final int maxReflections = 3; public String execute(String task) { String result = ""; List<String> history = new ArrayList<>(); for (int i = 0; i < maxReflections; i++) { // 1. 执行 ReAct 循环 result = executeReactCycle(task, history); // 2. 反思：评估结果质量 Reflection reflection = reflect(task, result, history); // 3. 记录历史，供后续参考 history.add("第" + (i+1) + "次尝试: " + result); history.add("反思: " + reflection.feedback()); // 4. 如果满意，提前退出 if (reflection.isSatisfactory()) { return result; } // 5. 不满意，将反思作为下一步的指导 task = task + "\n[自我纠正] 上次的问题: " + reflection.issues() + "\n请改进。"; } return result; // 返回最后一次结果 } private Reflection reflect(String task, String result, List<String> history) { return chatClient.prompt() .system(""" 你是一个严格的评审者。 评估以下任务的执行结果，给出： 1. 是否满意（true/false） 2. 存在的问题 3. 改进建议 用 JSON 格式回复。 """) .user("任务: " + task + "\n结果: " + result) .call() .entity(Reflection.class); } } public record Reflection( boolean satisfactory, String issues, String feedback ) {}
    💡 小窍门

    反思的最大价值在于避免重复犯错。通过将历史反思记录传入下一轮，Agent 会自动"吃一堑长一智"。

    💻 试试看

    为反思环节添加退出条件：除了"满意"之外，还可以设置"超过最大反思次数"、"结果与上一次完全相同"等退出条件，避免无限循环。

    📋 Plan-Execute 模式

    概念比喻：Agent 的"项目计划书"

    想象你是一个项目经理。你不会上来就让团队干活，而是先制定计划，分解任务，分配资源，然后按计划执行。执行过程中遇到变化，就动态调整计划。

    Plan-Execute 模式就是让 Agent 具备这种"先想后做"的能力，大幅减少不必要的工具调用和无效探索。

    Plan-Execute 架构

  ┌───────────┐
  │  用户请求  │
  └─────┬─────┘
        v
  ┌───────────────┐
  │  Planner (规划) │  ← 用 LLM 生成步骤列表
  │  Step 1: ...   │
  │  Step 2: ...   │
  │  Step 3: ...   │
  └─────┬─────────┘
        v
  ┌───────────────┐     ┌──────────────┐
  │  Executor (执行) ───> │ 观察 & 评估  │
  └─────┬─────────┘     └──────┬───────┘
        |                      |
        |  成功？─────────────> 下一步
        |  失败？──> Re-Plan (重新规划)
        v
  ┌───────────┐
  │  最终结果  │
  └───────────┘

    代码示例：计划-执行-验证循环

    @Service public class PlanExecuteAgent { private final ChatClient chatClient; public String solve(String task) { // 1. 规划阶段 Plan plan = createPlan(task); System.out.println("计划: " + plan.steps()); // 2. 执行阶段 List<StepResult> results = new ArrayList<>(); for (int i = 0; i < plan.steps().size(); i++) { String step = plan.steps().get(i); String context = buildContext(results); // 执行单个步骤 StepResult result = executeStep(step, context); results.add(result); // 3. 验证阶段 if (!result.success()) { // 执行失败，重新规划剩余步骤 Plan newPlan = replan(task, results, step, result.error()); // 用新计划替换剩余步骤 plan = newPlan; i = -1; // 从第一步重新开始 } } // 4. 汇总结果 return synthesize(task, results); } private Plan createPlan(String task) { return chatClient.prompt() .system(""" 你是一个任务规划专家。 将复杂任务分解为可执行的步骤。 每个步骤应该是原子的、可验证的。 用 JSON 数组格式返回步骤列表。 """) .user(task) .call() .entity(Plan.class); } private Plan replan(String originalTask, List<StepResult> completedSteps, String failedStep, String error) { return chatClient.prompt() .system(""" 上一个计划的某一步失败了。 请基于已完成的步骤和失败原因， 重新规划剩余的步骤。 """) .user("原始任务: " + originalTask + "\n已完成: " + completedSteps + "\n失败步骤: " + failedStep + "\n错误: " + error) .call() .entity(Plan.class); } } public record Plan(List<String> steps) {} public record StepResult(boolean success, String output, String error) {}
    ⚠️ 陷阱警告

    Plan-Execute 的 Re-Plan 频率需要控制。如果每次都重新规划整个任务，会导致大量 token 浪费。建议只对失败的步骤进行局部调整，而非全盘重来。

    💻 试试看

    实现一个支持并行执行的 Plan-Execute Agent：当某些步骤之间没有依赖关系时，同时执行多个步骤，提升效率。

    🌱 LATS（Language Agent Tree Search）

    概念比喻：Agent 的"头脑风暴"

    想象你在解一道数学题，你不是只用一种方法，而是在脑中同时尝试多种解法，然后选择最优的那一条。LATS 就是让 Agent 同时探索多条路径，像下棋一样"向前看"几步。

    它借鉴了蒙特卡洛树搜索（MCTS）的思想，通过选择 → 扩展 → 模拟 → 回溯四步循环，在探索和利用之间取得平衡。

    LATS 树搜索架构

                    ┌──────────┐
                    │  根节点   │
                    │  (任务)   │
                    └────┬─────┘
            ┌────────────┼────────────┐
            v            v            v
       ┌────────┐  ┌────────┐  ┌────────┐
       │路径 A │  │路径 B │  │路径 C │
       │ 方案1  │  │ 方案2  │  │ 方案3  │
       └───┬────┘  └───┬────┘  └───┬────┘
           |           |           |
      ┌────┼────┐      |      ┌────┼────┐
      v    v    v      v      v    v    v
    子节点 子节点 子节点  子节点  子节点 子节点 子节点
    ...    ...   ...     ...    ...   ...   ...

  评估公式: UCB = exploitation + exploration
  选择得分最高的路径继续扩展

    核心实现

    // LATS 核心节点结构 public class LATNode { String state; // 当前状态 String action; // 执行的动作 double score; // 节点得分 int visits; // 访问次数 LATNode parent; // 父节点 List<LATNode> children; // 子节点 public double ucbScore() { // UCB1 公式：平衡探索与利用 double exploitation = score / visits; double exploration = Math.sqrt( 2 * Math.log(parent.visits) / visits); return exploitation + exploration; } } // LATS 搜索器 @Service public class LATSearcher { private final ChatClient chatClient; private final int maxIterations = 10; private final int branchFactor = 3; public String search(String task) { LATNode root = new LATNode(task, "", 0, 0, null); for (int i = 0; i < maxIterations; i++) { // 1. 选择：用 UCB 找到最有潜力的节点 LATNode selected = select(root); // 2. 扩展：为选中的节点生成多个子节点 List<LATNode> children = expand(selected); selected.children = children; // 3. 模拟：评估每个子节点的质量 for (LATNode child : children) { child.score = simulate(child); child.visits++; } // 4. 回溯：将模拟结果传播回父节点 backpropagate(children.get(0)); } // 返回访问次数最多的路径（最优解） return getBestSolution(root); } private LATNode select(LATNode node) { if (node.children.isEmpty()) return node; return node.children.stream() .max(Comparator .comparingDouble(LATNode::ucbScore)) .orElse(node); } }
    💡 你知道吗？

    LATS 是 AlphaGo 思想在语言 Agent 领域的迁移。AlphaGo 用 MCTS 找到最优下法，LATS 用它找到最优解题路径。两者的核心都是在指数级的搜索空间中高效找到好方案。

    ⚠️ 陷阱警告

    LATS 的 token 消耗是 ReAct 的 3-5 倍，因为每次扩展都要调用 LLM 生成多个候选方案。只在任务复杂度高、对质量要求严格时使用。简单任务用 ReAct 足矣。

    💡 反思模式（Reflection）

    概念比喻：Agent 的"自我批评"

    一个好的作家写完文章后，一定会通读一遍，找出问题，然后修改。反思模式就是让 Agent 具备这种能力 — 生成结果后，用另一个 LLM 调用来评估质量，发现问题后自动迭代改进。

    这个模式的巧妙之处在于分离了"生成"和"评估"两个角色，就像作者和编辑是两个人，各司其职。

    反思模式工作流

  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ Generator │────>│ Evaluator│────>│  判断     │
  │ 生成内容  │     │ 评估质量  │     │ 满意？   │
  └──────────┘     └──────────┘     └────┬─────┘
        ^                               |
        |         ┌──────────┐          |
        +─────────│ Reflector │<──────────┘
                  │ 反馈改进  │  不满意
                  └──────────┘  满意 ──> 输出

    代码示例：生成-评估-反思循环

    @Service public class ReflectionAgent { private final ChatClient chatClient; private final int maxIterations = 3; public String generate(String task) { String output = ""; List<String> critiques = new ArrayList<>(); for (int i = 0; i < maxIterations; i++) { // 1. 生成 output = generateContent(task, critiques); // 2. 评估 Evaluation eval = evaluate(task, output); // 3. 反思 if (eval.score() >= 0.8) { return output; // 质量达标 } critiques.add(eval.feedback()); } return output; } private String generateContent(String task, List<String> previousCritiques) { String critiqueContext = previousCritiques.isEmpty() ? "" : "\n之前的问题:\n" + String.join("\n", previousCritiques) + "\n请避免这些问题。"; return chatClient.prompt() .user(task + critiqueContext) .call().content(); } private Evaluation evaluate(String task, String output) { return chatClient.prompt() .system(""" 你是严格的评审。评估以下输出质量。 评分 0-1，低于 0.8 需要给出具体改进建议。 用 JSON 回复: {"score": 0.9, "feedback": "..."} """) .user("任务: " + task + "\n输出: " + output) .call() .entity(Evaluation.class); } } public record Evaluation(double score, String feedback) {}
    💡 小窍门

    反思模式的关键在于评估标准要清晰。在 System Prompt 中明确定义"什么算好"、"什么算差"，否则 LLM 的评估会很模糊。可以参考 Google 的 Self-Refine 论文。

    💻 试试看

    实现一个多维度评估的反思 Agent，分别评估"准确性"、"完整性"、"可读性"三个维度，只有三个维度都达标时才输出最终结果。

    📈 模式对比与选型

    四大模式对比

    维度ReActPlan-ExecuteLATSReflection
    核心思想 思考-行动-观察 先规划后执行 树搜索多路径 生成-评估-改进
    适用场景 简单/中等任务 复杂多步骤任务 需要最优解的任务 质量要求高的任务

    延迟 低 中 高 中高

    实现复杂度 低 中 高 中
    可调试性 高（链式清晰） 高（步骤明确） 低（树结构复杂） 中（循环需跟踪）

    你的 Agent 需要解决什么类型的问题？

    💡 你知道吗？

    实际生产中，很少只用单一模式。最常见的是组合使用：Plan-Execute 做主体框架，Reflection 做质量保障，ReAct 处理单个步骤。这就是下一节要讲的"组合模式"。

    🔨 组合模式

    概念比喻：Agent 的"瑞士军刀"

    一把瑞士军刀之所以强大，不是因为它有一个完美的工具，而是因为它组合了多种工具。Agent 模式也一样 — 将 Plan-Execute、ReAct、Reflection 组合在一起，形成一个功能强大的复合 Agent。

    组合模式架构

  ┌──────────────────────────────────────┐
  │           Plan-Execute 框架          │
  │                                      │
  │  ┌──────────┐    ┌───────────────┐  │
  │  │ Planner  │───>│  Step 执行器   │  │
  │  └──────────┘    └───────┬───────┘  │
  │                          |           │
  │                    ┌─────┼─────┐    │
  │                    v     v     v    │
  │                 ReAct ReAct ReAct   │
  │                  步骤1  步骤2  步骤3  │
  │                    |     |     |    │
  │                    └─────┼─────┘    │
  │                          v           │
  │                  ┌───────────────┐   │
  │                  │ Reflection     │   │
  │                  │ 质量评估反馈  │   │
  │                  └───────┬───────┘   │
  │                          |           │
  │              不满意？──> Re-Plan     │
  │              满意？───> 输出结果      │
  └──────────────────────────────────────┘

    代码示例：组合 Agent

    @Service public class CompositeAgent { private final PlanExecuteAgent planner; private final ReflectiveReActAgent executor; private final ReflectionAgent reviewer; private final int maxRetries = 2; public String solve(String task) { // 1. Plan-Execute: 规划并执行步骤 List<StepResult> results = planner.executePlan(task); // 2. 汇总初步结果 String draft = synthesize(results); // 3. Reflection: 质量评估 for (int i = 0; i < maxRetries; i++) { String refined = reviewer.generate(task, draft); if (!refined.equals(draft)) { draft = refined; } else { break; // 没有改进了 } } return draft; } }
    ⚠️ 陷阱警告

    组合模式虽然强大，但复杂度和成本也最高。每增加一层，token 消耗和延迟都会成倍增长。务必根据实际需求选择合适的组合，不要为了"炫技"而过度设计。

    📖 章节小结

    本章我们学习了四种核心 Agent 设计模式：

    • ReAct 进阶 — 加入反思循环，让 Agent 学会"吃一堑长一智"
    • Plan-Execute — 先规划后执行，支持动态调整计划
    • LATS — 树搜索式探索，在指数级空间中高效找到最优解
    • Reflection — 生成-评估-改进循环，分离"作者"和"编辑"角色

    记住：没有最好的模式，只有最合适的模式。简单任务用 ReAct，复杂任务用 Plan-Execute，质量要求高加 Reflection，需要最优解用 LATS。

    🎓 面试高频题

    👨‍💻 Q1: ReAct 和 Plan-Execute 的核心区别是什么？

    ReAct 是逐步思考：每一步都重新思考下一步该做什么，适合简单任务。Plan-Execute 是先全局规划再执行：先制定完整计划，然后按步骤执行，适合复杂多步骤任务。ReAct 更灵活但容易走弯路，Plan-Execute 更高效但需要重新规划机制。

    👨‍💻 Q2: LATS 的 UCB 公式如何平衡探索与利用？

    UCB1 公式 = exploitation（利用）+ exploration（探索）。exploitation 项是节点的平均得分，倾向于选择已知高分节点；exploration 项随访问次数增加而减小，鼓励探索访问较少的节点。通过调节两者权重，Agent 既不会过早锁定次优路径，也不会无限制地探索低质量路径。

    👨‍💻 Q3: Reflection 模式中，如何避免评估者"自己骗自己"？

    1. 使用不同的 LLM：生成和评估用不同的模型，减少偏见。2. 明确评估标准：在 Prompt 中给出具体的评分维度和阈值。3. 引入外部验证：用单元测试、事实校验等手段客观验证。4. 限制迭代次数：避免无限循环改进。
