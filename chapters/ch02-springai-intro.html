Spring AI 入门

掌握 Spring AI 核心 API，理解 ChatClient 与 ChatModel 抽象，实现多模型适配与流式响应

⏱️ 预计 2-3 周 📖 6 个知识点 💻 中级难度

📚 开篇故事：万能遥控器的诞生

小李是一名 Java 开发者，最近领导要求他给公司的客服系统接入 AI 能力。他的第一反应是兴奋，第二反应是头疼。

兴奋是因为 AI 确实能解决客服的大量重复问题。头疼是因为他发现市面上的 AI 模型五花八门：OpenAI 的 GPT-4o 擅长多语言，Anthropic 的 Claude 在长文本分析上更强，Google 的 Gemini 多模态能力突出，而公司内部还在试用开源的 Llama 3。

"难道我要为每个模型写一套对接代码？"小李痛苦地想。光是处理不同 API 的认证方式、请求格式、响应结构，就够他喝一壶的了。更别说模型还在不断更新换代，今天用的 API 明天可能就废弃了。

就在这时，他发现了 Spring AI。Spring AI 就像一个"万能遥控器" -- 你只需要按一个按钮，它就能帮你控制电视、空调、音响，不管品牌是什么。同样地，你只需要一套 Java API，Spring AI 就能帮你对接 OpenAI、Claude、Gemini、Ollama 等任何模型，切换模型只需要改一行配置。

本章，我们就来认识这个"万能遥控器"的核心组件，学会用 Spring AI 优雅地与大语言模型对话。

🎯 学习目标

在本章结束后，你将能够：

  • 正确配置 Spring AI 项目依赖和多模型连接
  • 熟练使用 ChatClient（万能遥控器）发起对话请求
  • 深入理解 ChatModel 接口的设计哲学和抽象层次
  • 灵活适配 OpenAI、Claude、Gemini、Ollama 等多种模型
  • 独立实现 流式（Streaming）响应，打造丝滑交互体验
  • 掌握 多模态（文本+图片）输入处理

项目配置

Maven 依赖配置

Spring AI 通过 Spring Boot Starter 提供自动配置，只需要引入对应的模型依赖即可：

<!-- pom.xml --> <properties> <spring-ai.version>1.0.0</spring-ai.version> </properties> <dependencies> <dependency> <groupId>org.springframework.ai</groupId> <artifactId>spring-ai-openai-spring-boot-starter</artifactId> </dependency> </dependencies> <repositories> <repository> <id>spring-milestones</id> <url>https://repo.spring.io/milestone</url> </repository> </repositories>

配置文件

在 application.yml 中配置模型连接信息：

# application.yml spring: ai: openai: api-key: ${OPENAI_API_KEY} chat: options: model: gpt-4o temperature: 0.7 max-tokens: 2048 base-url: https://api.openai.com
Spring AI 1.0+ 使用 spring-ai 前缀替代旧的 spring.ai 前缀。配置项结构与 Spring Boot 的外部化配置完全一致。

Gradle 配置

// build.gradle.kts dependencies { implementation("org.springframework.ai:spring-ai-openai-spring-boot-starter") }

🔎 ChatClient 基础

ChatClient = "万能遥控器" 🎮

想象你家客厅有一堆遥控器：电视一个、空调一个、音响一个...每次换个设备就要找对应的遥控器。现在有人发明了万能遥控器 -- 一个遥控器控制所有设备，你只需要告诉它"我要控制电视"还是"我要控制空调"。

ChatClient 就是这个"万能遥控器"。它是 Spring AI 的核心入口，通过统一的流式 API 与任何大语言模型交互。不管底层是 OpenAI、Claude 还是本地的 Ollama，你用的都是同一套 ChatClient API。

工作原理图

你的 Java 代码
    |
    v
ChatClient <------- "万能遥控器" ------> 统一 API
    |
    +---> ChatModel (抽象接口)
    |       |
    |       +---> OpenAiChatModel  --> OpenAI API
    |       +---> AnthropicChatModel --> Claude API
    |       +---> GeminiChatModel  --> Gemini API
    |       +---> OllamaChatModel  --> 本地模型
    v
统一的响应格式 (ChatResponse)

快速上手

@Service public class AiChatService { private final ChatClient chatClient; // 万能遥控器 // 构造器注入：Spring 自动把 ChatModel "快递"过来 public AiChatService(ChatModel chatModel) { this.chatClient = ChatClient.create(chatModel); // 创建遥控器 } public String chat(String message) { return chatClient.prompt() // 按下"对话"按钮 .user(message) // 输入你的问题 .call() // 发送请求 .content(); // 获取回答 } }
💡 一句话理解
ChatClient 之于 AI 模型，就像 JDBC 之于数据库 -- 你只学一套 API，就能操作不同的后端。

ChatClient 链式调用详解

ChatClient 支持丰富的链式配置，就像拼积木一样组合出你需要的功能：

// 🔴 基础对话 -- 最简单的用法 String response = chatClient.prompt() .user("你好，介绍一下你自己") .call() .content(); // 结果: "我是 AI 助手，可以帮助你解答问题..." // 🟠 带系统提示词 -- 给 AI 一个"人设" String response = chatClient.prompt() .system("你是一个专业的Java技术顾问，回答要简洁专业") // 设定角色 .user("什么是Spring AI？") .call() .content(); // 🟢 设置模型参数 -- 精确控制输出 String response = chatClient.prompt() .system("你是一个翻译助手") .user("翻译：Hello World") .options(OpenAiChatOptions.builder() .model("gpt-4o") // 指定模型 .temperature(0.3) // 低温度 = 更精确 .maxTokens(1024) // 限制输出长度 .build()) .call() .content();
链式方法作用比喻
.system()设置系统提示词给 AI 一个"身份牌"
.user()设置用户消息你对 AI 说的话
.options()设置模型参数调节遥控器的"频道"和"音量"
.call()同步调用按一下按钮，等结果
.stream()流式调用看直播弹幕，一个字一个字蹦出来
.content()获取文本内容拿到 AI 的回答

使用 Advisor（顾问）

Advisor 是 Spring AI 的拦截器机制，类似 Spring MVC 的 Filter。它可以在请求前后添加逻辑，比如日志记录、对话记忆、安全检查等。

生活比喻

Advisor 就像机场的安检通道 -- 你（用户请求）要登机（调用 LLM）之前，必须经过安检（Advisor）。安检员会检查你的行李（日志记录）、核对你的身份（记忆管理）、甚至帮你托运行李（预处理）。

用户请求
    |
    v
Advisor 1: SimpleLoggerAdvisor  <-- 记录请求日志
    |
    v
Advisor 2: MessageChatMemoryAdvisor  <-- 注入对话历史
    |
    v
LLM 调用
    |
    v
Advisor 链处理响应
    |
    v
返回给用户
// 📄 日志顾问 - 记录所有请求和响应 String response = chatClient.prompt() .advisors(new SimpleLoggerAdvisor()) // 加上日志安检 .user("你好") .call() .content(); // 📄 对话记忆顾问 - 让 AI 记住之前说过的话 String response = chatClient.prompt() .advisors(new MessageChatMemoryAdvisor(chatMemory, "userId", 10)) // | | // 记忆存储 用户ID 保留最近10条 .user("记住我的名字叫张三") .call() .content();
💡 小窍门
Advisor 链按顺序执行，你可以同时使用多个 Advisor。建议的执行顺序：日志 → 安全检查 → 记忆管理 → RAG 检索。
💻 试试看

创建一个 ChatClient，使用 .system() 设置角色为"你是一个专业的Java面试官"，然后问它"什么是 Spring Boot 的自动配置？"。观察回答的质量和风格。

进阶挑战：试试 .temperature(0.0) 和 .temperature(1.5)，对比两次回答的差异。

ChatClient 链式调用

ChatClient 支持丰富的链式配置：

// 带系统提示词的对话 String response = chatClient.prompt() .system("你是一个专业的Java技术顾问，回答要简洁专业") .user("什么是Spring AI？") .call() .content(); // 设置模型参数 String response = chatClient.prompt() .system("你是一个翻译助手") .user("翻译：Hello World") .options(OpenAiChatOptions.builder() .model("gpt-4o") .temperature(0.3) .maxTokens(1024) .build()) .call() .content();

使用 Advisor（顾问）

Advisor 是 Spring AI 的拦截器机制，可以在请求前后添加逻辑：

// 日志顾问 - 记录所有请求和响应 String response = chatClient.prompt() .advisors(new SimpleLoggerAdvisor()) .user("你好") .call() .content(); // 对话记忆顾问 String response = chatClient.prompt() .advisors(new MessageChatMemoryAdvisor(chatMemory, "userId", 10)) .user("记住我的名字叫张三") .call() .content();
Advisor 链按顺序执行，类似 Spring MVC 的 Filter。你可以自定义 Advisor 实现特定的日志、监控或预处理逻辑。

🔌 ChatModel 接口

ChatModel 接口设计

如果说 ChatClient 是"万能遥控器"，那 ChatModel 就是遥控器背后的"万能翻译官" 🌐。不管你说中文、英文还是日文，翻译官都能帮你翻译成对方能理解的语言，然后把回答翻译回来。

ChatModel 是 Spring AI 对大语言模型的统一抽象。每个模型提供商（OpenAI、Claude、Gemini）都实现了这个接口，所以你不需要关心底层 API 的差异。

核心架构图

ChatClient --> ChatModel (统一抽象接口)
                   |
         +---------+---------+---------+
         |         |         |         |
    OpenAI    Claude    Gemini    Ollama
   |         |         |         |
  API       API       API       本地
// 📄 ChatModel 接口定义（简化版） public interface ChatModel extends Model<ChatPrompt, ChatResponse> { // 同步调用 -- 按下按钮，等待结果 ChatResponse call(ChatPrompt chatPrompt); // 流式调用 -- 看直播弹幕，一个字一个字蹦出来 Flux<ChatResponse> stream(ChatPrompt chatPrompt); } // 📄 直接使用 ChatModel 的示例 @Service public class ModelService { private final ChatModel chatModel; // 底层的"翻译官" public ChatResponse ask(String prompt) { // 手动构建 ChatPrompt（ChatClient 会自动帮你做这一步） ChatPrompt chatPrompt = new ChatPrompt( List.of( SystemMessage.of("你是一个有帮助的助手"), // 系统指令 UserMessage.of(prompt) // 用户输入 ), OpenAiChatOptions.builder().model("gpt-4o").build() // 模型选项 ); return chatModel.call(chatPrompt); // 调用模型 } }

ChatClient vs ChatModel -- 该用哪个？

对比维度ChatClientChatModel
抽象层次高层 API（遥控器）底层接口（翻译官）
使用场景日常开发（推荐）需要精细控制时
支持 Advisor✅ 内置支持❌ 需要手动实现
流式支持✅ .stream()✅ .stream()
结构化输出✅ .entity()❌ 需手动处理
推荐度⭐⭐⭐⭐⭐ 90% 场景使用⭐⭐ 10% 高级场景
💡 建议
除非你需要直接操作 ChatResponse 的元数据（如 Token 用量、模型信息），否则始终使用 ChatClient。它更简洁、更安全、功能更丰富。

核心组件关系

理解这些组件的关系，就像理解一辆汽车的零件：

  • Prompt 📝 -- 你要对 AI 说的话（包含消息列表 + 模型参数）
  • Message 💬 -- 单条消息（SystemMessage / UserMessage / AssistantMessage）
  • ChatResponse 📦 -- AI 返回的完整响应（内容 + Token 用量 + 元数据）
  • Generation 🏆 -- ChatResponse 中的单个生成结果（一次调用可能有多个）
  • ChatOptions ⚙️ -- 模型参数（temperature、model、maxTokens 等）

返回结构化对象

使用 entity() 方法，Spring AI 会自动将 LLM 的回答解析为 Java 对象，就像让 AI 帮你填表单一样。

// 🎯 定义输出结构 -- 让 AI 按这个格式回答 public record WeatherInfo( String city, // 城市名 double temperature, // 温度 String description // 天气描述 ) {} // 🎯 配置 ChatClient Bean @Bean public ChatClient chatClient(ChatModel model) { return ChatClient.builder(model) .defaultOptions(OpenAiChatOptions.builder() .model("gpt-4o") .build()) .build(); } // 🎯 自动解析为 WeatherInfo 对象 WeatherInfo weather = chatClient.prompt() .user("北京今天的天气怎么样？") .call() .entity(WeatherInfo.class); // 自动解析！ // 使用结果 -- 就像使用普通 Java 对象一样 System.out.println(weather.city()); // 北京 System.out.println(weather.temperature()); // 25.3 System.out.println(weather.description()); // 晴，微风
💡 原理揭秘
entity() 方法内部使用 BeanOutputConverter：先将 Java 类转为 JSON Schema，注入到 System Prompt 中引导 LLM 输出 JSON，再用 Jackson 反序列化为 Java 对象。Record 类型效果最好，因为字段名和类型信息会被准确映射。
💡 你知道吗？
Spring AI 的 ChatModel 接口设计借鉴了 Spring 的 RestTemplate / WebClient 模式。如果你熟悉 Spring 的 HTTP 客户端，那么 ChatModel 的 API 会非常亲切。这也是 Spring 生态"一致性"的体现。

核心组件关系

  • Prompt - 由 Message 列表和 Options 组成的请求对象
  • Message - 消息实体，包含 SystemMessage、UserMessage、AssistantMessage
  • ChatResponse - 包含生成结果、Token 用量、元数据
  • Generation - ChatResponse 中的单个生成结果
  • ChatOptions - 模型参数（temperature、model、maxTokens 等）

返回结构化对象

// 使用 Bean 输出解析器，自动将回复转为 Java 对象 public record WeatherInfo( String city, double temperature, String description ) {} @Bean public ChatClient chatClient(ChatModel model) { return ChatClient.builder(model) .defaultOptions(OpenAiChatOptions.builder() .model("gpt-4o") .build()) .build(); } // 自动解析为 WeatherInfo 对象 WeatherInfo weather = chatClient.prompt() .user("北京今天的天气怎么样？") .call() .entity(WeatherInfo.class);
entity() 方法内部使用 BeanOutputConverter，通过 JSON Schema 引导 LLM 输出结构化 JSON，再反序列化为 Java 对象。

🌐 多模型适配

模型无关 = "万能翻译官"

Spring AI 的核心设计理念是模型无关（Model Agnostic）。就像"万能翻译官"不管你是中国人、日本人还是美国人，都能帮你翻译。Spring AI 不管底层是 OpenAI、Claude 还是 Gemini，都用同一套 API。

切换模型只需要改一行配置，代码完全不需要动：

OpenAI

# application.yml spring.ai.openai: api-key: ${OPENAI_API_KEY} chat.options.model: gpt-4o
gpt-4o gpt-4o-mini o1-preview

Anthropic Claude

# application.yml spring.ai.anthropic: api-key: ${ANTHROPIC_API_KEY} chat.options.model: claude-sonnet-4-20250514
claude-sonnet-4 claude-opus-4 claude-haiku

Google Gemini

# application.yml spring.ai.google: gemini: api-key: ${GEMINI_API_KEY} chat.options.model: gemini-2.0-flash
gemini-2.0-flash gemini-2.5-pro

Ollama（本地模型）

# application.yml spring.ai.ollama: base-url: http://localhost:11434 chat.options.model: llama3
llama3 mistral qwen

运行时动态切换模型

@Service public class MultiModelService { private final ChatClient openaiClient; private final ChatClient claudeClient; public MultiModelService( @Qualifier("openAiChatModel") ChatModel openai, @Qualifier("anthropicChatModel") ChatModel claude) { this.openaiClient = ChatClient.create(openai); this.claudeClient = ChatClient.create(claude); } public String chatWithModel(String message, String provider) { ChatClient client = "openai".equals(provider) ? openaiClient : claudeClient; return client.prompt() .user(message) .call() .content(); } }
使用 @Qualifier 注入不同模型的 ChatModel Bean，结合策略模式在运行时选择模型。Spring AI 会根据依赖自动注册对应模型的 Bean。

📡 流式输出

流式输出 = "直播弹幕" 📺

想象你在网上看直播，主播说的话是一个字一个字蹦出来的，而不是等他说完一整句话才显示。这就是流式输出（Streaming）。

普通的 .call() 就像等快递 -- 你下单后要等快递员把整个包裹送来。而 .stream() 就像直播弹幕 -- AI 边想边说，用户能立即看到第一个字，不用傻等。

流式 vs 非流式对比

非流式 (.call())          流式 (.stream())
用户: "写一首诗"          用户: "写一首诗"
    |                        |
    v                        v
[等待 5 秒...]           [春] [眠] [不] [觉] [晓]...
    |                     (逐字输出，秒级响应)
    v
"春眠不觉晓..."         用户已经开始阅读了！
(一次性返回)

SSE 流式响应实现

@RestController @RequestMapping("/api/v1/chat") public class ChatStreamController { private final ChatClient chatClient; // SSE = Server-Sent Events，服务端主动推送事件给客户端 @GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE) // 声明 SSE 端点 public Flux<String> streamChat( @RequestParam String message) { return chatClient.prompt() .user(message) .stream() // 🟠 关键：使用 stream() 替代 call() .content(); // 返回 Flux（多个字符串片段） } }
💡 SSE 注意事项
  • 使用 produces = MediaType.TEXT_EVENT_STREAM_VALUE 声明 SSE 端点
  • 返回类型必须是 Flux<String>
  • 前端使用 EventSource 或 fetch API 接收流式数据
  • 记得设置合理的超时时间（建议 60 秒）

前端流式接收

// 🇨🇳 JavaScript 前端代码 -- 接收 SSE 流式数据 const eventSource = new EventSource( `/api/v1/chat/stream?message=你好` ); // 每收到一个片段就追加显示（打字机效果） eventSource.onmessage = (event) => { const chunk = event.data; // AI 的一小段回复 document.getElementById('output').textContent += chunk; }; // 连接关闭时结束 eventSource.onerror = () => { eventSource.close(); console.log('AI 说完了！'); };

带历史记录的流式对话

@RestController public class ConversationController { private final ChatClient chatClient; private final ChatMemory chatMemory; @GetMapping(value = "/chat/stream/{userId}", produces = MediaType.TEXT_EVENT_STREAM_VALUE) public Flux<String> streamWithMemory( @PathVariable String userId, @RequestParam String message) { return chatClient.prompt() .system("你是一个有帮助的助手") .advisors(new MessageChatMemoryAdvisor(chatMemory, userId)) .user(message) .stream() // 流式输出 + 对话记忆 .content(); } }
⚠️ 踩坑警告

流式输出会增加连接持有时间，在高并发场景下需要关注连接池配置。建议：(1) 为 SSE 连接设置 60 秒超时；(2) 使用 WebFlux 而非 Web MVC 处理流式端点；(3) 监控活跃连接数，防止连接耗尽。

使用 produce = MediaType.TEXT_EVENT_STREAM_VALUE 声明 SSE 端点。前端使用 EventSource 或 fetch API 接收流式数据。

前端流式接收

// JavaScript 前端代码 const eventSource = new EventSource( `/api/v1/chat/stream?message=你好` ); eventSource.onmessage = (event) => { const chunk = event.data; document.getElementById('output').textContent += chunk; }; eventSource.onerror = () => { eventSource.close(); };

带历史记录的流式对话

@RestController public class ConversationController { private final ChatClient chatClient; private final ChatMemory chatMemory; @GetMapping(value = "/chat/stream/{userId}", produces = MediaType.TEXT_EVENT_STREAM_VALUE) public Flux<String> streamWithMemory( @PathVariable String userId, @RequestParam String message) { return chatClient.prompt() .system("你是一个有帮助的助手") .advisors(new MessageChatMemoryAdvisor(chatMemory, userId)) .user(message) .stream() .content(); } }
流式输出会增加连接持有时间，在高并发场景下需要关注连接池配置和超时设置。建议为 SSE 连接设置合理的超时（如 60 秒）。

👁 多模态输入

多模态 = "五感齐全" 🌱

普通 AI 只能"听"你说话（文本输入）。但多模态 AI 五感齐全 -- 它不仅能"听"（文本），还能"看"（图片）、"读"（文档）。就像一个既能听你描述病情，又能看 X 光片的医生，诊断自然更准确。

Spring AI 支持多模态（Multimodal）输入，可以同时发送文本和图片给模型：

多模态工作流程

用户上传图片 + 输入问题
    |
    v
Spring AI
    +---> 文本部分: "这张图片里有什么？"
    +---> 图片部分: Media(image/jpeg, URL/data)
    v
多模态 LLM (GPT-4o / Claude 3.5 / Gemini)
    v
AI 回答: "这张图片显示的是一个日落场景..."

URL 图片输入

@Service public class VisionService { private final ChatClient chatClient; /** * 分析网络图片 * @param imageUrl 图片的 URL 地址 * @param question 对图片的问题 * @return AI 的分析结果 */ public String analyzeImage(String imageUrl, String question) { return chatClient.prompt() .user(prompt -> prompt .text(question) // 问题文本 .media(MimeType.IMAGE_JPEG, new URL(imageUrl))) // 图片 .call() .content(); } }

📖 章节小结

本章我们认识了 Spring AI 的核心组件：

  • ChatClient = "万能遥控器" -- 统一的 AI 对话入口，支持链式调用和 Advisor
  • ChatModel = "万能翻译官" -- 底层抽象接口，每个模型提供商各自实现
  • 多模型适配 -- 模型无关设计，切换模型只需改一行配置
  • 流式输出 = "直播弹幕" -- 逐字输出，提升用户体验
  • 多模态输入 = "五感齐全" -- 支持文本+图片等多模态输入

这些组件就像积木块 -- 单独看每个都很简单，但组合起来就能搭建出功能强大的 AI 应用。

🎓 面试高频题

👨‍💻 Q1: ChatClient 和 ChatModel 的区别是什么？

ChatModel 是底层接口，直接操作 ChatPrompt/ChatResponse。ChatClient 是高层封装，支持 Advisor 链、结构化输出（entity()）、流式调用等高级功能。90% 场景推荐使用 ChatClient，只有需要精细控制元数据时才直接用 ChatModel。

👨‍💻 Q2: Spring AI 如何实现模型无关？

通过 ChatModel 接口的统一抽象。每个模型提供商（OpenAI、Claude、Gemini）实现各自的 ChatModel，Spring AI 根据配置自动注册对应的 Bean。切换模型只需修改 application.yml 中的配置，代码无需改动。这就是 Spring 的依赖注入 + 策略模式的经典应用。

👨‍💻 Q3: 流式输出的原理是什么？为什么用 SSE？

流式输出基于 SSE（Server-Sent Events）协议，服务端通过持久连接主动向客户端推送数据。相比 WebSocket，SSE 更简单（单向通信）、更适合 AI 场景（只需要服务端向客户端推送）。Spring AI 使用 Reactor Flux 封装了 SSE 流，前端用 EventSource API 接收。

👨‍💻 Q4: Advisor 的执行顺序有什么讲究？

Advisor 按注册顺序执行。推荐顺序：日志记录 → 安全检查 → 对话记忆 → RAG 检索 → 业务逻辑。原因：日志应该最先记录原始请求；安全检查要在业务逻辑之前拦截；记忆和 RAG 要在 LLM 调用前注入上下文。
