第 03 章

RAG 检索增强生成

掌握检索增强生成的核心原理，实现基于私有知识库的智能问答系统

⏱️ 预计 2-3 周 📖 6 个知识点 💻 中级难度

📚 开篇故事：AI 的"开卷考试"

小王是一名刚入职的 Java 开发者，公司让他为客服部门搭建一个智能问答系统。客户问"你们的产品保修期是多久？"，AI 要能准确回答。

小王一开始觉得很简单 -- 直接问 ChatGPT 就行了。但很快他就发现了问题：ChatGPT 不知道他们公司的产品信息，回答的内容全是"编"出来的。更可怕的是，它编得还特别像真的，客服人员差点就把错误答案发给了客户。

这就是大语言模型的"幻觉问题"。它就像一个闭卷考试的学生 -- 虽然很聪明，但如果没学过某个知识点，它不是说"我不知道"，而是"自信地编一个答案"。

后来小王学到了一个叫 RAG 的技术。它的思路很简单：开卷考试。AI 在回答问题之前，先去翻一翻公司的产品手册、FAQ 文档、历史工单，找到相关的内容，然后再基于这些真实资料来回答。这样一来，AI 就不再是"闭着眼睛瞎编"，而是"有据可查"。

本章，我们就来学习这个让 AI 从"闭卷"变"开卷"的强大技术。

🎯 学习目标

在本章结束后，你将能够：

  • 理解 RAG 的架构原理和"先检索，再生成"的工作流程
  • 掌握 文档加载、分割和向量化的完整流程
  • 配置 向量数据库（Chroma/Milvus）存储和检索
  • 设计 高效的检索策略（混合检索、重排序等）
  • 使用 Spring AI 的 QuestionAnswerAdvisor 实现完整 RAG
  • 优化 RAG 系统的准确率和性能

🎯 RAG 解决什么问题？

大语言模型有三大"硬伤"，RAG 就是它们的"特效药"：

硬伤表现RAG 如何解决
🔴 知识截止 训练数据有时间限制，不知道"昨天"发生的事 检索最新文档，让 AI 基于最新资料回答
🔴 幻觉问题 编造看似合理但实际错误的内容 提供真实文档作为依据，让 AI "有据可查"
🔴 无私有知识 不了解企业内部文档和数据 将企业文档入库，AI 随时检索使用
💡 你知道吗？
RAG（Retrieval-Augmented Generation）最早由 Facebook AI Research 在 2020 年提出。现在已成为企业级 AI 应用的标配技术。据统计，超过 80% 的企业 AI 项目都采用了 RAG 架构。

📚 RAG 架构原理

RAG = "开卷考试" 📝

RAG（Retrieval-Augmented Generation，检索增强生成）的核心思想用一句话概括：先检索相关资料，再让 AI 基于资料回答。

就像开卷考试 -- 学生可以翻书找答案，但前提是你得有一本编好目录的教科书（向量数据库），还得知道怎么快速翻到正确的页码（向量检索）。

📄 文档加载与分割

文档加载 -- 图书管理员帮你找书 📖

第一步是把各种格式的文档"搬"进来。Spring AI 提供了丰富的 DocumentReader（图书管理员），支持 PDF、Markdown、TXT、HTML 等常见格式：

// 📄 加载 PDF 文档 -- 最常见的企业文档格式 PdfDocumentReader pdfReader = new PdfDocumentReader( new FileSystemResource("docs/manual.pdf")); List<Document> documents = pdfReader.get(); // 📄 加载 Markdown 文档 -- 技术文档首选 MarkdownDocumentReader mdReader = new MarkdownDocumentReader( new FileSystemResource("docs/readme.md")); List<Document> documents = mdReader.get(); // 📄 加载纯文本文件 TextReader textReader = new TextReader( new FileSystemResource("docs/data.txt")); List<Document> documents = textReader.get(); // 📄 加载整个目录 -- 一次性导入所有文档 DirectoryReader dirReader = new DirectoryReader( new FileSystemResource("./docs")); List<Document> allDocs = dirReader.get();

✂️ 文档分割 = "把厚书拆成卡片"

一本 500 页的书不可能直接塞进 AI 的脑袋。你需要把它拆成一张张卡片（Chunk），每张卡片包含一个完整的知识点。这就是文档分割的意义。

为什么需要分割？

一本 500 页的书
    |
    v  TokenTextSplitter (拆卡片)
500 张知识卡片 (每个 200-500 Token)
    |
    v  EmbeddingModel (翻译密码)
500 个向量 (每个 1536 维)
    |
    v  VectorStore (放进图书馆)
图书馆索引完成 ✔
// 🎯 TokenTextSplitter -- 基于 Token 数量分割（最常用） TokenTextSplitter splitter = new TokenTextSplitter( 200, // 默认每张卡片 200 Token 50, // 最小 50 Token（太短的卡片不要） 2000, // 最大 2000 Token（太长的要再拆） 5000, // 最大 Token 数（含重叠部分） true // 保留段落分隔符 ); List<Document> chunks = splitter.apply(documents); System.out.println("原始文档数: " + documents.size()); System.out.println("分割后卡片数: " + chunks.size()); // 原始文档数: 1 // 分割后卡片数: 25

分割参数调优指南

参数太小的问题太大的问题推荐值
Chunk Size 丢失上下文，语义不完整 包含过多无关信息，检索精度低 200-500 Token
Chunk Overlap 相邻卡片信息断裂 重复信息多，浪费存储 10-20%（约 50 Token）
⚠️ 踩坑警告

不要在句子中间切断！一个完整的句子被拆到两张卡片里，会导致语义断裂，检索质量严重下降。建议按段落或章节分割，而非按固定字符数硬切。

自定义分割策略

@Bean public DocumentTransformer customSplitter() { return docs -> docs.stream() .flatMap(doc -> { String content = doc.getContent(); // 按 Markdown 标题分割 -- 保留章节结构 String[] sections = content.split("(?=## )"); return Arrays.stream(sections) .map(s -> new Document(s.trim(), doc.getMetadata())); }) .toList(); }
💡 分割策略选择
  • TokenTextSplitter -- 通用首选，适合大多数场景
  • 按标题分割 -- 技术文档推荐，保留结构
  • 按段落分割 -- 长文章推荐，保持语义完整
  • 递归分割 -- 先按大标题，再按小标题，最后按段落

🔑 Embedding 向量化

Embedding = "数字密码翻译机" 🔑

人类理解语言靠"语义"，但计算机只懂数字。Embedding 就是一台"数字密码翻译机" -- 它把人类语言翻译成一串数字（向量），让计算机也能"理解"语义。

神奇的是：语义相近的文本，翻译出来的数字也相近！就像"猫"和"小猫咪"的向量很近，而"猫"和"汽车"的向量很远。

向量化原理图

人类语言                              数字向量 (1536 维)

"Spring AI 是一个 Java 框架"  -->  [0.023, -0.156, 0.891, ...]
^ 语义相近                           ^ 数字相近
"Spring Boot 自动配置"        -->  [0.019, -0.148, 0.876, ...]

"今天天气不错"                -->  [0.712, 0.334, -0.567, ...]
^ 语义差距大                         ^ 数字差距大

Spring AI Embedding 配置与使用

# application.yml - OpenAI Embedding 配置 spring.ai: openai: api-key: ${OPENAI_API_KEY} embedding: options: model: text-embedding-3-small # 性价比之王
@Service public class EmbeddingService { private final EmbeddingModel embeddingModel; // 数字密码翻译机 /** * 将文本转为向量 * @param text 输入文本 * @return 1536 维的浮点数组 */ public float[] embed(String text) { EmbeddingResponse response = embeddingModel.call( new EmbeddingPrompt(text)); // 翻译！ return response.getResult().getOutput().getVector(); } /** * 计算两段文本的语义相似度 * @return 0-1 之间的相似度分数，1 = 完全相同 */ public double similarity(String text1, String text2) { float[] v1 = embed(text1); // 翻译第一段 float[] v2 = embed(text2); // 翻译第二段 return cosineSimilarity(v1, v2); // 计算距离 } }
💡 成本对比
  • text-embedding-3-small -- $0.02/1M tokens，性价比最高
  • text-embedding-3-large -- $0.13/1M tokens，精度更高但贵 6 倍
  • BGE-M3 -- 免费（本地部署），中文效果最好

🏛 向量存储

VectorStore = "超级图书馆" 🏛

向量数据库就像一个"超级图书馆"。普通图书馆按书名、作者、ISBN 编目，你必须知道准确的信息才能找到书。而超级图书馆按"语义"编目 -- 你说"我想找关于如何让 AI 更聪明的书"，它就能帮你找到《深度学习》、《机器学习实战》等相关书籍。

Spring AI 的 VectorStore 接口抽象了向量数据库的操作，提供统一的 API。不管底层是 Chroma、Milvus 还是 Pinecone，你用的都是同一套 API。

@Service public class KnowledgeBaseService { private final VectorStore vectorStore; // 添加文档到向量数据库 public void addDocuments(List<Document> documents) { vectorStore.accept(documents); } // 相似度检索 public List<Document> search(String query, int topK) { SearchRequest request = SearchRequest.query(query) .withTopK(topK) .withSimilarityThreshold(0.7); return vectorStore.similaritySearch(request); } // 删除文档 public void deleteDocuments(List<String> ids) { vectorStore.delete(ids); } }

向量数据库选型对比

数据库特点适用场景部署方式
Chroma 轻量级、API 简洁 开发测试、小规模项目 Docker / 嵌入式
Milvus 分布式、高性能 生产环境、大规模数据 Kubernetes / Docker
Pinecone 全托管、零运维 不想自己运维的团队 云服务
SimpleVectorStore 内存存储、零配置 单元测试、原型验证 进程内
Elasticsearch 混合检索、全文搜索 已有 ES 集群的项目 Docker / 云服务

Chroma 配置（入门推荐）

# application.yml spring.ai.vectorstore.chroma: url: http://localhost:8000 collection-name: docs

Milvus 配置（生产推荐）

# application.yml spring.ai.vectorstore.milvus: host: localhost port: 19530 collection-name: docs

SimpleVectorStore（内存模式）

开发和测试时可以使用内存向量存储，无需额外部署：

@Bean public VectorStore vectorStore(EmbeddingModel embeddingModel) { return new SimpleVectorStore(embeddingModel); }
SimpleVectorStore 将数据存储在内存中，重启后数据丢失。仅适用于开发测试，生产环境必须使用持久化的向量数据库。

🔍 检索策略

检索 = "翻书找答案" 🔍

检索是 RAG 的核心环节。就像你参加开卷考试，翻书的速度和准确度决定了你的成绩。检索策略就是要做到"又快又准"地找到最相关的文档片段。

相似度检索 -- 基础用法

// 🎯 基本相似度检索 -- 最常用的检索方式 SearchRequest request = SearchRequest.query("Spring AI 的 RAG 功能") .withTopK(5) // 返回前 5 个最相关的结果 .withSimilarityThreshold(0.7); // 相似度阈值：低于 0.7 的不要 List<Document> results = vectorStore.similaritySearch(request); // 遍历检索结果 results.forEach(doc -> { System.out.println("相关内容: " + doc.getContent().substring(0, 100)); System.out.println("相似度: " + doc.getScore()); System.out.println("来源: " + doc.getMetadata().get("source")); });
参数作用推荐值调优建议
TopK 返回最相关的 K 个结果 3-10 太少可能漏掉关键信息，太多会引入噪声
SimilarityThreshold 过滤掉不相关的结果 0.5-0.8 太高会漏掉结果，太低会引入无关内容

混合检索（Hybrid Search）-- 双管齐下

纯向量检索擅长"语义搜索"，但在精确匹配（如搜产品型号 "iPhone 15 Pro Max"）上可能不够精准。混合检索同时使用向量检索和关键词检索，取长补短。

// 🎯 混合检索 + 元数据过滤 SearchRequest request = SearchRequest.query("Spring Boot 自动配置原理") .withTopK(10) .withSimilarityThreshold(0.5) .withFilterExpression("category == 'tutorial'"); // 元数据过滤：只要教程类文档 List<Document> results = vectorStore.similaritySearch(request);
💡 元数据过滤技巧

在文档入库时添加元数据（category、date、source），检索时结合过滤条件，可以大幅提高精度。比如：只检索"2024年以后"的文档，或只检索"官方文档"来源的内容。

🚀 检索策略优化进阶

策略原理效果适用场景
📝 查询重写 用 LLM 优化用户查询 提升检索质量 30%+ 用户表达模糊时
🔀 多路召回 向量检索 + 关键词检索 覆盖面更广 通用场景
🔁 重排序 Cross-Encoder 重新打分 精度提升 20%+ 对精度要求高时
🔗 Parent-Child 检索小块，返回父块上下文 上下文更完整 长文档场景
// 📝 查询重写示例 -- 用 LLM 优化用户查询 String originalQuery = "那个框架怎么配来着"; String rewrittenQuery = chatClient.prompt() .system("将用户的口语化问题改写为精确的技术搜索关键词") .user(originalQuery) .call() .content(); // rewrittenQuery = "Spring Boot 框架配置方法" // 用重写后的查询去检索，效果更好 SearchRequest request = SearchRequest.query(rewrittenQuery).withTopK(5);
💡 你知道吗？
Google 搜索使用了 200+ 个排名信号来决定搜索结果。同样，生产级 RAG 系统通常组合 3-5 种检索策略。不要指望单一策略能解决所有问题，组合拳才是王道。

🛠️ Spring AI RAG 实现

完整 RAG 系统搭建 -- 三步走

搭建一个 RAG 系统只需要三步：配图书馆、放书进去、开卷考试。

// 🎯 第一步：配置"超级图书馆"（向量数据库） @Configuration public class RagConfig { @Bean public VectorStore vectorStore(EmbeddingModel embeddingModel) { return ChromaVectorStore.builder(embeddingModel) .url("http://localhost:8000") .collectionName("knowledge-base") .build(); } // 🎯 第二步：把书放进去（加载 + 分割 + 向量化） @Bean public void loadDocuments(VectorStore vectorStore) { // 1. 加载整个目录的文档 DirectoryReader reader = new DirectoryReader( new FileSystemResource("./docs")); List<Document> docs = reader.get(); // 2. 分割成小卡片 TokenTextSplitter splitter = new TokenTextSplitter(); List<Document> chunks = splitter.apply(docs); // 3. 存入向量数据库（自动向量化） vectorStore.accept(chunks); System.out.println("Loaded " + chunks.size() + " chunks into vector store"); } }

使用 QuestionAnswerAdvisor -- 一站式 RAG

Spring AI 提供了 QuestionAnswerAdvisor，封装了完整的 RAG 流程。你只需要提供 VectorStore，它会自动完成所有步骤。

QuestionAnswerAdvisor 内部流程

  用户问题: "如何配置 RAG？"
    |
    v  1. 问题向量化
  [0.023, -0.156, ...]
    |
    v  2. 相似度检索
  检索到 5 个相关文档片段
    |
    v  3. 注入 Prompt
  "请参考以下资料回答：\n1. ...\n2. ..."
    |
    v  4. LLM 生成回答
  基于真实资料的准确回答 ✅
@Service public class RagChatService { private final ChatClient chatClient; private final VectorStore vectorStore; public RagChatService(ChatModel chatModel, VectorStore vectorStore) { this.chatClient = ChatClient.create(chatModel); this.vectorStore = vectorStore; } /** * RAG 问答 -- 一行代码搞定！ */ public String chat(String question) { return chatClient.prompt() .system("你是公司知识库助手，请基于检索到的资料回答问题。如果资料中没有相关信息，请诚实地说不知道。") .advisors(new QuestionAnswerAdvisor(vectorStore, SearchRequest.query(question) .withTopK(5) // 检索前 5 个相关片段 .withSimilarityThreshold(0.7))) // 相似度阈值 .user(question) .call() .content(); } }
💡 关键点

(1) 系统提示词中要告诉 AI "基于检索资料回答"，避免幻觉；(2) 设置相似度阈值过滤不相关结果；(3) TopK 建议 3-7，太多会引入噪声。

📖 章节小结

本章我们掌握了 RAG 的完整知识体系：

  • RAG = "开卷考试" -- 先检索相关资料，再让 AI 基于资料回答
  • 向量数据库 = "超级图书馆" -- 按语义编目，按语义检索
  • Embedding = "数字密码翻译机" -- 把文本变成计算机能理解的数字
  • 文档分割 = "把厚书拆成卡片" -- 拆成合适大小的知识单元
  • QuestionAnswerAdvisor -- Spring AI 封装的 RAG 一站式解决方案

记住 RAG 的核心公式：RAG = 检索 + 生成。检索解决"AI 不知道"的问题，生成解决"怎么表达"的问题。

🎓 面试高频题

👨‍💻 Q1: RAG 和 Fine-tuning 有什么区别？

RAG 是"开卷考试"，不改变模型本身，通过检索外部知识来增强回答。成本低、更新快、可解释性强。Fine-tuning 是"补课"，用领域数据重新训练模型。成本高、更新慢、但模型内化了知识。企业级应用通常两者结合使用。

👨‍💻 Q2: Chunk Size 如何选择？

Chunk Size 是一个权衡：太小（<100 Token）会丢失上下文，语义不完整；太大（>1000 Token）会包含过多无关信息，检索精度低。推荐 200-500 Token，Overlap 10-20%。技术文档建议按章节分割，保持语义完整性。

👨‍💻 Q3: 如何提高 RAG 的准确率？

关键优化点：(1) 文档质量 -- 清洗噪声、统一格式；(2) 分割策略 -- 按语义分割而非固定长度；(3) Embedding 模型 -- 选择适合领域的模型；(4) 检索策略 -- 混合检索 + 重排序；(5) Prompt 工程 -- 引导模型基于检索结果回答。

👨‍💻 Q4: 向量数据库如何选型？

小规模（<10 万文档）：Chroma 或 SimpleVectorStore。中规模（10-100 万）：Milvus 或 Qdrant。大规模（>100 万）：Milvus 分布式或 Pinecone（全托管）。已有 Elasticsearch 集群：直接用 ES 向量搜索。关键指标：查询延迟、存储成本、运维复杂度。
