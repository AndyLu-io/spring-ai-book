第 28 章

Agent 可观测性与监控体系

从"黑匣子"到"透明手术室" -- 为你的 Agent 系统装上全套监控仪表盘，让每一次 LLM 调用、每一次工具执行都尽在掌握

⏱ 预计 3-4 天 📖 7 个知识点 💻 中高级难度

📚 开篇故事：ICU 里的生命监护仪

想象你是一位外科医生，刚完成一台复杂的心脏搭桥手术。术后，病人被推入 ICU。在 ICU 里，床头有一台生命监护仪，屏幕上实时跳动着心电图、血压、血氧饱和度、呼吸频率 -- 每一个数字都在告诉你："病人现在状况如何。"

如果某个指标突然异常，监护仪会立即发出报警声，护士马上就能定位问题：是心率过快？还是血氧下降？还是血压骤降？没有这台监护仪，医生就只能靠"摸脉搏"这种原始手段来判断病情 -- 等到症状明显时，往往已经来不及了。

你的 AI Agent 系统就是那个"术后病人"。当它上线生产环境后，你需要一台数字世界的"生命监护仪"：实时看到每次 LLM 调用的延迟、每个工具执行的成功率、每分钱花在了哪个模型上。出了问题，你需要在用户抱怨之前就发现异常；性能下降，你需要精确定位是哪个环节的瓶颈。这就是可观测性（Observability） -- 不只是"出了问题能查日志"，而是"系统运行的每一个细节都尽在掌握"。

🎯 学习目标

在本章结束后，你将能够：

  • 区分可观测性与传统监控的本质差异，理解三大支柱（Tracing、Metrics、Logging）的协同关系
  • 设计并实现基于 OpenTelemetry 的 Agent 系统分布式追踪，覆盖 LLM 调用和工具执行全链路
  • 建立完整的 Agent 系统 Metrics 体系，包括任务指标、成本指标和质量指标
  • 设计结构化日志策略，支持对话回放、安全审计和异常排查
  • 搭建Grafana 监控仪表盘和告警策略，实现异常的主动发现和快速定位

📄 核心内容（第1节）：可观测性基础

📄 可观测性 vs 监控：不只是换个名字

监控（Monitoring）回答的是"系统是否正常运行"。比如：CPU 利用率超过 80% 报警、响应时间超过 2 秒报警。监控是被动的、预设规则的 -- 你必须提前知道什么算"不正常"。

可观测性（Observability）回答的是"系统为什么这样运行"。它允许你从外部输出推断内部状态，而不需要预先定义所有可能的问题。就像医生不仅看体温计（监控），还要做 CT、血液检查、心电图（可观测性），才能精确诊断病因。

维度监控（Monitoring）可观测性（Observability）
核心问题系统是否正常？系统为什么是这个状态？
数据来源预定义的指标和阈值Traces + Metrics + Logs 全量数据
能力边界只能发现已知问题能发现未知问题（Unknown Unknowns）
实现方式Dashboard + AlertStructured Logs + Distributed Traces + Metrics
典型工具Nagios、Zabbix、CloudWatchOpenTelemetry、Datadog、Grafana Tempo

📄 三大支柱：可观测性的"铁三角"

可观测性由三大支柱组成，它们各司其职又紧密协作，就像医院的三大检查手段 -- CT（结构化日志）、血液化验（指标）和心电图（追踪），缺一不可。

📝 Tracing（分布式追踪）

记录请求从进入到离开的完整旅程。一个用户请求可能经历：API 网关 -> Agent 编排器 -> LLM 调用 -> 工具执行 -> 结果汇总。Tracing 把这条链路上的每一个环节串起来，让你看到完整的时间线。

核心概念：Trace（一次完整请求）、Span（请求中的一个操作）、Context Propagation（跨服务传递追踪上下文）。

📈 Metrics（指标）

对系统行为的量化度量。不是记录每一次调用的详情，而是统计一段时间内的聚合数据：QPS、P99 延迟、错误率、Token 消耗总量。Metrics 是最轻量的观测手段，适合做趋势分析和告警。

核心概念：Counter（计数器）、Gauge（仪表盘）、Histogram（直方图）、Summary（摘要）。

📄 Logging（日志）

记录系统运行过程中的离散事件。一条日志就是一行文本："2026-06-02 14:30:22 [INFO] Agent completed task, tokens used: 1523, cost: $0.008"。日志是排查具体问题时最直接的手段 -- 但前提是你要有结构化的日志设计，否则海量日志只会变成"大海捞针"。

📄 Agent 系统的可观测性挑战

相比传统 Web 应用，Agent 系统的可观测性面临独特挑战：

  • 非确定性执行路径 -- 每次请求的 LLM 调用次数、工具选择、推理步骤都可能不同，传统的固定链路追踪方案不再适用
  • 成本维度 -- Token 消耗是传统 Web 应用没有的维度，每次 LLM 调用都有真实的金钱成本，需要精细化追踪
  • 长尾延迟 -- Agent 的一个任务可能包含 5-10 次 LLM 调用，总延迟可达 30-60 秒，远超传统 API 的毫秒级响应
  • 质量评估 -- 不仅要监控"系统是否运行"，还要监控"回答质量如何"，这是可观测性领域的全新维度

📄 可观测性设计原则

  Agent 可观测性设计原则

  1. 无侵入性     监控代码不应污染业务逻辑，通过 Advisor/AOP 自动采集
  2. 全链路覆盖   从用户请求到 LLM 响应，每一个环节都要可观测
  3. 成本感知     每次 LLM 调用都要记录 Token 消耗和对应成本
  4. 低开销       监控本身的性能开销不超过总请求时间的 1%
  5. 可扩展       支持自定义 Span、自定义指标、自定义日志字段
  6. 隐私合规     日志中不能包含用户敏感数据（PII），需要脱敏处理
💡 设计经验

可观测性的投入产出比遵循"复利效应"：前期搭建虽然需要 2-3 天，但后续每一次故障排查的时间会从"小时级"缩短到"分钟级"。在 Agent 系统中，一次未经观测的 LLM 调用异常可能导致用户连续遇到 5-6 次失败才最终得到正确回答，每次都白白烧了 Token 费。

📄 核心内容（第2节）：Tracing 实现

📄 分布式追踪基础：Span、Trace、Context Propagation

分布式追踪的核心思想是将一个请求的完整生命周期拆解为一系列Span（跨度），每个 Span 代表一个独立的操作（如一次 LLM 调用、一次工具执行）。所有 Span 通过 Trace ID 关联在一起，形成一棵调用树。

生活比喻：想象你在顺丰快递寄了一个包裹。Trace 就是这个包裹的运单号，从你下单到送达，经过的每一个分拣中心、每一辆运输车、每一次扫描都是一次 Span。通过运单号，你能看到包裹完整的旅程；通过每个 Span 的时间戳，你能知道包裹在哪个环节卡住了。

  一个 Agent 请求的 Trace 结构

  Trace ID: abc-123-def-456
  │
  ├── [Span] agent.execute (总耗时 8.2s)
  │    ├── [Span] llm.chat.completions (2.1s, model=gpt-4o, tokens=1523)
  │    ├── [Span] tool.search_web (1.3s, tool=web_search)
  │    │    └── [Span] http.get (1.2s, url=https://api.search...)
  │    ├── [Span] llm.chat.completions (1.8s, model=gpt-4o, tokens=2341)
  │    ├── [Span] tool.db_query (0.4s, tool=sql_executor)
  │    └── [Span] llm.chat.completions (2.5s, model=gpt-4o, tokens=3102)
  │
  └── 总 Token 消耗: 6966 | 总成本: $0.035

📄 OpenTelemetry 集成

OpenTelemetry（简称 OTel）是 CNCF 的可观测性标准框架，统一了 Tracing、Metrics 和 Logs 的采集和导出。它提供了：

  • API + SDK -- Java API 用于创建 Span 和指标，SDK 负责采样、导出、上下文传播
  • OTLP 协议 -- 标准化的数据传输协议，可将数据发送到 Jaeger、Tempo、Zipkin 等任何兼容的后端
  • 自动埋点 -- 通过 Java Agent（-javaagent:opentelemetry-javaagent.jar）自动捕获 HTTP、gRPC 等调用

对于 Agent 系统，OpenTelemetry 的优势在于它是厂商中立的 -- 今天用 Jaeger，明天可以无缝切换到 Grafana Tempo，不需要改一行代码。

📄 LLM 调用链路追踪

LLM 调用是 Agent 系统中最核心、最昂贵的操作。一次 LLM 调用的 Span 应该记录以下信息：

Span 属性说明示例
llm.model模型名称gpt-4o, claude-3-opus
llm.input_tokens输入 Token 数1523
llm.output_tokens输出 Token 数856
llm.total_tokens总 Token 数2379
llm.cost_usd本次调用成本（美元）0.0182
llm.temperature温度参数0.7
llm.finish_reason结束原因stop, length
llm.prompt_hashPrompt 的哈希值（用于去重分析）sha256:a3f2...

📄 工具调用链路追踪

工具调用是 Agent 系统的另一个关键环节。每个工具调用应该记录：

  • tool.name -- 工具名称（如 search_web、sql_query）
  • tool.input -- 工具输入参数（脱敏后）
  • tool.output_size -- 输出数据大小（字节）
  • tool.success -- 是否成功（true/false）
  • tool.error_type -- 失败时的错误类型（超时、权限不足、数据不存在等）
⚠️ 安全提醒

Span 属性中绝对不要记录用户的敏感数据（如密码、API Key、身份证号）。输入参数应做脱敏处理：密码替换为 ***，API Key 只记录前 4 位。日志和追踪数据通常会发送到集中存储，一旦泄露影响范围很大。

📄 自定义 Span 设计

对于 Agent 系统，建议设计以下自定义 Span 层级：

  Agent Span 层级设计

  └── agent.task (一个完整任务)
       ├── agent.plan (规划阶段)
       │    └── llm.chat.completions
       ├── agent.execute.step-N (第N步执行)
       │    ├── llm.chat.completions (决策：选择工具)
       │    ├── tool.{tool_name} (工具执行)
       │    └── llm.chat.completions (处理结果)
       ├── agent.evaluate (结果评估)
       │    └── llm.chat.completions
       └── agent.summarize (最终总结)
            └── llm.chat.completions
💡 采样策略建议

在高流量场景下，100% 采样会消耗大量存储。推荐自适应采样：正常请求采样率 10%（基于 Trace ID 的确定性采样），异常请求（错误、高延迟、高成本）100% 采样。这样既控制了存储成本，又保证了异常场景的完整可观测性。

📄 核心内容（第3节）：Metrics 设计

📄 Agent 系统核心指标体系

Agent 系统的指标设计需要覆盖四个维度：任务执行、LLM 交互、工具调用和成本控制。下面这张表是 Agent 系统最重要的指标清单：

维度指标名称类型说明
任务执行agent.task.totalCounter任务总数
agent.task.success_rateGauge任务完成率（成功/总数）
agent.task.durationHistogram任务总耗时分布
LLM 交互llm.request.totalCounterLLM 调用总数
llm.request.latencyHistogramLLM 调用延迟分布
llm.token.inputCounter累计输入 Token 数
llm.token.outputCounter累计输出 Token 数
工具调用tool.call.totalCounter工具调用总数
tool.call.success_rateGauge工具调用成功率
tool.call.error.countCounter按错误类型分类的错误数
成本控制cost.total_usdCounter累计总成本（美元）
cost.per_taskHistogram每任务平均成本分布
cost.per_modelCounter按模型拆分的成本

📄 Micrometer 集成

Micrometer 是 Spring Boot 的指标门面，类似 Slf4j 之于日志。它提供了统一的 API 来创建和注册指标，并自动绑定到 Prometheus、Datadog、CloudWatch 等后端。

// ============================================ // Agent Metrics Collector - 核心指标收集器 // ============================================ @Component public class AgentMetricsCollector { private final MeterRegistry registry; // 任务相关指标 private final Counter taskTotal; private final Counter taskSuccess; private final Timer taskDuration; // LLM 相关指标 private final Counter llmRequestTotal; private final Timer llmLatency; private final Counter llmInputTokens; private final Counter llmOutputTokens; // 成本相关指标 private final Counter costTotal; public AgentMetricsCollector(MeterRegistry registry) { this.registry = registry; this.taskTotal = Counter.builder("agent.task.total") .description("Total agent tasks") .register(registry); this.taskSuccess = Counter.builder("agent.task.success") .description("Successful agent tasks") .register(registry); this.taskDuration = Timer.builder("agent.task.duration") .description("Task execution duration") .publishPercentiles(0.5, 0.95, 0.99) .register(registry); this.llmRequestTotal = Counter.builder("llm.request.total") .description("Total LLM requests") .register(registry); this.llmLatency = Timer.builder("llm.request.latency") .description("LLM request latency") .publishPercentiles(0.5, 0.95, 0.99) .register(registry); this.llmInputTokens = Counter.builder("llm.token.input") .description("Total input tokens") .register(registry); this.llmOutputTokens = Counter.builder("llm.token.output") .description("Total output tokens") .register(registry); this.costTotal = Counter.builder("cost.total_usd") .description("Total cost in USD") .register(registry); } public void recordLlmCall(String model, long inputTokens, long outputTokens, double cost, Duration latency) { Tags tags = Tags.of("model", model); llmRequestTotal.increment(tags); llmLatency.record(latency, tags); llmInputTokens.increment(inputTokens, tags); llmOutputTokens.increment(outputTokens, tags); costTotal.increment(cost, tags); } public void recordTaskComplete(boolean success, Duration duration) { taskTotal.increment(); if (success) taskSuccess.increment(); taskDuration.record(duration); } }

📄 Prometheus + Grafana 可视化

将 Micrometer 指标导出到 Prometheus 并通过 Grafana 可视化是业界最主流的方案。核心配置如下：

# application.yml - Prometheus 指标导出配置 management: endpoints: web: exposure: include: health, info, prometheus, metrics metrics: export: prometheus: enabled: true tags: application: agent-service # Micrometer 指标端点: /actuator/prometheus # Prometheus 抓取配置: # scrape_configs: # - job_name: 'agent-service' # metrics_path: '/actuator/prometheus' # static_configs: # - targets: ['localhost:8080']

📄 自定义指标收集

除了基础指标，Agent 系统还需要一些业务级自定义指标：

  • agent.conversation.turns -- 对话轮次分布（帮助理解用户交互深度）
  • agent.retry.count -- 重试次数分布（反映系统稳定性）
  • agent.tool.choice -- 工具选择分布（哪些工具被调用最多）
  • agent.model.distribution -- 模型使用分布（监控模型路由策略效果）
💡 最佳实践：使用 Tag 而非独立指标

不要为每个模型、每个工具创建独立的 Counter。而是使用标签（Tag）：llm.request.total{model="gpt-4o"} 和 llm.request.total{model="claude-3-opus"}。Prometheus 的标签查询（sum by (model)）比管理数百个独立指标高效得多。

📄 核心内容（第4节）：Logging 策略

📄 结构化日志设计

传统的纯文本日志（如 INFO: Task completed）在 Agent 系统中几乎无法使用 -- 一条请求可能产生 20+ 条日志，没有结构化格式就无法做聚合分析。Agent 系统应该使用JSON 格式的结构化日志：

// 结构化日志示例 - JSON 格式 { "timestamp": "2026-06-02T14:30:22.123Z", "level": "INFO", "traceId": "abc-123-def-456", "spanId": "span-789", "service": "agent-service", "event": "llm.call.completed", "model": "gpt-4o", "inputTokens": 1523, "outputTokens": 856, "latencyMs": 2100, "costUsd": 0.0182, "userId": "user_***42", "taskId": "task-001" }

结构化日志的关键设计原则：

  • 统一 traceId -- 每条日志都携带 traceId，与 Tracing 系统打通，可以按 traceId 检索出一个请求的所有日志
  • 事件驱动命名 -- 用 llm.call.completed、tool.execution.failed 这样的动词短语作为事件名，方便日志分析引擎做聚合
  • PII 脱敏 -- 用户 ID 用 user_***42 格式，密码、Token 等字段完全不记录

📄 对话历史记录

Agent 系统的对话日志是最重要的数据资产之一，用于质量回溯、合规审计和模型微调。推荐设计：

日志类型内容保留策略用途
对话日志完整的 User/Assistant 消息交替90 天质量回溯、微调数据
决策日志Agent 每步的推理过程和工具选择30 天行为分析、调试
错误日志异常堆栈、重试信息、降级决策180 天问题排查、SLA 报告
审计日志用户身份、操作类型、时间戳365 天合规审计、安全调查

📄 安全审计日志

Agent 系统涉及用户数据和 LLM 交互，安全审计日志是合规要求。审计日志应记录：

  • 谁做了什么 -- 用户身份、操作类型（对话、工具调用、配置变更）
  • 何时做的 -- 精确到毫秒的时间戳
  • 结果如何 -- 成功/失败、拒绝（安全过滤触发）
  • 敏感操作 -- 文件访问、数据库查询、外部 API 调用等需要单独标记
⚠️ 审计日志不可删改

审计日志必须存储在只追加（Append-Only）的存储中，不允许修改或删除。推荐使用独立的审计日志服务，与业务日志分离存储。某些行业（金融、医疗）对审计日志的保留期限有法律要求。

📄 日志聚合与分析

方案一：ELK Stack

Elasticsearch + Logstash + Kibana，最成熟的日志方案。Logstash 负责日志采集和转换，Elasticsearch 存储和索引，Kibana 提供可视化查询。

优势：全文搜索能力强，生态成熟。
劣势：资源消耗大（Elasticsearch 吃内存），运维成本高。

方案二：Grafana Loki

Loki + Grafana，轻量级日志方案。Loki 只索引标签（Label），不索引日志内容，存储成本远低于 Elasticsearch。

优势：资源消耗小，与 Prometheus/Grafana 天然集成。
劣势：不支持全文搜索（只能按标签过滤后 grep）。

💡 选型建议

如果团队已经在用 Prometheus + Grafana 做 Metrics 和 Tracing，日志也用 Grafana 全家桶（Loki）是最省心的选择 -- 一个 Grafana 界面就能看到 Traces、Metrics、Logs，并且可以通过 traceId 在三者之间一键跳转。

📄 核心内容（第5节）：监控仪表盘设计

📄 仪表盘设计原则

一个好的监控仪表盘应该像飞机驾驶舱的仪表板 -- 飞行员一眼就能判断"一切正常"还是"哪个系统出了问题"。Agent 系统的仪表盘设计应遵循：

  • 层级化 -- 概览层（全局健康度） -> 领域层（任务/LLM/工具） -> 细节层（单次请求追踪）
  • 黄金指标优先 -- 首屏只展示最重要的 4-5 个指标，避免信息过载
  • 时间窗口对比 -- 支持"现在 vs 昨天同时段"、"现在 vs 上周同时段"的对比视图
  • 下钻能力 -- 从概览图表点击可跳转到对应的 Trace 详情

📄 告警策略设计

告警是监控体系的"最后一公里" -- 没有告警的仪表盘就是"摆在那里的装饰品"。Agent 系统的告警策略应分三级：

P0 紧急告警

  • 任务成功率 < 80%（持续 5 分钟）
  • LLM API 连续失败（错误率 > 50%）
  • 系统 QPS 突降至 0

通知方式：电话 + 短信 + 钉钉/飞书

P1 警告告警

  • P95 延迟 > 30s（持续 10 分钟）
  • 每小时成本超过预算 120%
  • 工具错误率 > 10%

通知方式：钉钉/飞书 + 邮件

P2 信息告警

  • 新错误类型首次出现
  • 模型切换事件（降级/升级）
  • 日均成本趋势偏差 > 30%

通知方式：邮件 / Slack

📄 异常检测与根因分析

除了阈值告警，Agent 系统还需要智能异常检测。传统阈值告警只能发现"已知的异常模式"，而智能检测能发现"未知的异常"：

  • 基线对比 -- 自动学习过去 7 天的正常模式，当当前值偏离基线 > 2 个标准差时告警
  • 关联分析 -- 当 LLM 延迟上升时，自动关联分析是否与 Token 使用量、并发请求数相关
  • 根因推断 -- 利用 Trace 数据构建调用链拓扑图，自动定位"是 LLM 慢了还是工具慢了"

📄 自定义 Tracing Advisor

// ============================================ // Agent Tracing Advisor - 自动追踪每次 LLM 调用 // ============================================ @Component public class AgentTracingAdvisor implements CallAdvisor, StreamAdvisor { private final Tracer tracer; private final AgentMetricsCollector metrics; public AgentTracingAdvisor(Tracer tracer, AgentMetricsCollector metrics) { this.tracer = tracer; this.metrics = metrics; } @Override public CallResponse aroundCall(CallAdvisorChain chain, CallRequest request) { String userMessage = extractUserMessage(request); return tracer.spanBuilder("agent.llm.call") .setAttribute("agent.type", "chat") .setAttribute("agent.user_message_length", userMessage.length()) .startSpan() .use(span -> { long start = System.nanoTime(); CallResponse response = chain.nextCall(request); long latencyMs = (System.nanoTime() - start) / 1_000_000; // 提取 Token 使用量和成本信息 var usage = extractUsage(response); double cost = calculateCost(usage); // 设置 Span 属性 span.setAttribute("llm.input_tokens", usage.inputTokens()); span.setAttribute("llm.output_tokens", usage.outputTokens()); span.setAttribute("llm.cost_usd", cost); span.setAttribute("llm.latency_ms", latencyMs); // 记录 Metrics metrics.recordLlmCall( usage.model(), usage.inputTokens(), usage.outputTokens(), cost, Duration.ofMillis(latencyMs) ); return response; }); } @Override public String getName() { return "AgentTracingAdvisor"; } @Override public int getOrder() { return Ordered.HIGHEST_PRECEDENCE; // 最高优先级，最先执行 } }

📚 章节小结

本章为你的 Agent 系统搭建了一套完整的可观测性体系：

  • Tracing -- 基于 OpenTelemetry 实现全链路追踪，覆盖 LLM 调用、工具执行、Agent 编排的每一个环节，通过 Span 属性记录 Token 消耗和成本
  • Metrics -- 通过 Micrometer 建立四维指标体系（任务、LLM、工具、成本），导出到 Prometheus + Grafana 进行可视化和告警
  • Logging -- 设计结构化 JSON 日志，支持 traceId 串联、PII 脱敏、审计合规，并通过 Loki 或 ELK 进行聚合分析
  • 仪表盘与告警 -- 分层级设计监控面板，建立 P0/P1/P2 三级告警策略，结合智能异常检测实现问题的主动发现

记住：可观测性不是"锦上添花"，而是生产环境的必备基础设施。没有可观测性的 Agent 系统，就像没有仪表盘的飞机 -- 你可能飞得起来，但你不知道自己在哪里、飞多高、油还剩多少。

👨‍💻 面试题解析

👨‍💻 Q1: 可观测性的三大支柱是什么？它们在 Agent 系统中分别解决什么问题？

答：三大支柱是 Tracing、Metrics 和 Logging：
(1) Tracing（分布式追踪）：记录一个请求从进入系统到返回结果的完整调用链路。在 Agent 系统中，一个任务可能包含多次 LLM 调用和工具执行，Tracing 将这些操作串联成一棵调用树，帮助理解"请求经过了哪些步骤、每步耗时多少"；
(2) Metrics（指标）：对系统行为的量化度量，如 QPS、延迟分布、错误率、Token 消耗量。Metrics 是最轻量的观测手段，适合做趋势分析、容量规划和告警。在 Agent 系统中，还需要追踪成本指标（每请求成本、每模型成本）；
(3) Logging（日志）：记录系统运行中的离散事件。结构化日志记录了每次 LLM 调用的详细参数、工具执行的输入输出、Agent 的决策过程。用于问题排查、安全审计和对话回放。
三者的协同：Metrics 发现异常 -> Tracing 定位到具体的调用链路 -> Logging 查看详细上下文。这个流程就像"报警 -> 定位故障节点 -> 查看详细日志"。

👨‍💻 Q2: OpenTelemetry 在 Spring AI 中如何集成？为什么选择 OTel 而不是直接用 Zipkin/Jaeger？

答：集成方式有两种：
(1) Java Agent 方式：启动时添加 -javaagent:opentelemetry-javaagent.jar，自动捕获 HTTP、gRPC、JDBC 等调用，零代码侵入；
(2) Spring Boot Starter 方式：引入 opentelemetry-spring-boot-starter，通过 Spring Bean 自动配置 OTel SDK，支持自定义 Span 和 Metrics。

选择 OTel 而非直接用 Zipkin/Jaeger 的原因：
(1) 厂商中立：OTel 是 CNCF 标准，数据格式统一。今天用 Jaeger，明天可以无缝切换到 Grafana Tempo 或 Datadog，不需要改代码；
(2) 统一三大支柱：OTel 同时支持 Traces、Metrics、Logs 的采集和导出，不需要为每种数据类型单独集成不同的库；
(3) 生态丰富：几乎所有主流 APM 工具都支持 OTLP 协议，集成成本极低。

👨‍💻 Q3: Agent 系统的 Metrics 指标设计应该覆盖哪些维度？如何避免指标爆炸？

答：Agent 系统的指标设计应覆盖四个维度：
(1) 任务执行维度：任务总数、成功率、总耗时分布、重试次数；
(2) LLM 交互维度：调用次数、延迟分布（P50/P95/P99）、输入/输出 Token 数、错误率；
(3) 工具调用维度：各工具调用次数、成功率、平均延迟、错误分类；
(4) 成本控制维度：累计总成本、每任务成本、按模型拆分的成本、成本趋势。

避免指标爆炸的策略：
(1) 使用 Tag 而非独立指标：如 llm.request.total{model="gpt-4o"} 而非为每个模型创建独立的 Counter；
(2) 控制 Tag 基数：模型名、工具名等 Tag 的唯一值数量应有上限（建议 < 100），否则 Prometheus 的时间序列数会爆炸；
(3) 分层聚合：原始指标在应用层采集，聚合后的指标在 Prometheus 层计算（如 sum(rate(llm.request.total[5m])) by (model)）。

👨‍💻 Q4: 如何设计 Agent 系统的告警策略？如何避免"告警疲劳"？

答：告警策略设计要点：
(1) 分级告警：P0（紧急，电话+短信）、P1（警告，即时通讯+邮件）、P2（信息，邮件）。不同级别对应不同的通知渠道和响应时效；
(2) 多窗口多次数：避免瞬时抖动触发告警。如"P95 延迟 > 30s 持续 10 分钟内出现 3 次"才告警，而不是"单次超过就告警"；
(3) 智能基线：基于历史数据自动计算正常范围，而不是写死固定阈值。工作日和周末、白天和夜间的正常值差异很大。

避免告警疲劳的策略：
(1) 告警收敛：相关告警合并（如"LLM 超时"和"LLM 错误率高"可能是同一个根因），避免一次性收到 10 条相关告警；
(2) 静默规则：在计划维护窗口自动静默告警；
(3) 定期 Review：每月分析告警命中率，关闭从未触发的无用告警，调整误报率高的告警阈值。

👨‍💻 Q5: 在 Agent 系统中，如何实现 Traces、Metrics、Logs 三者之间的关联？

答：三者关联的核心是统一的 Trace ID：
(1) Traces -> Logs：在日志框架中配置 MDC（Mapped Diagnostic Context），将 traceId 和 spanId 注入到每条日志中。在 Grafana Loki 中，可以按 traceId="abc-123" 检索出一个请求的所有日志；
(2) Traces -> Metrics：通过 Micrometer 的 Trace integration，在 Metrics 中自动携带 traceId。点击 Metrics 图表中的异常点，可以直接跳转到对应的 Trace 详情；
(3) Logs -> Traces：在日志中记录 traceId，点击日志条目可以跳转到 Grafana Tempo 查看完整调用链路；
(4) 实现要点：OpenTelemetry Java Agent 会自动将 traceId 注入到 SLF4J MDC 中。如果使用 Spring Boot，引入 io.micrometer:micrometer-tracing-bridge-otel 即可自动打通。

在 Grafana 中，一个典型的排障流程：Metrics 面板发现延迟异常 -> 点击异常时间段 -> 查看该时间段的 Traces 列表 -> 点击某条 Trace 查看调用链路 -> 点击某个 Span 查看对应的 Logs。

✍️ 练习 1：搭建完整的可观测性基础设施

为你的 Agent 项目搭建 Tracing + Metrics + Logging 全套可观测性基础设施：

  • 使用 Docker Compose 部署 Grafana + Tempo + Loki + Prometheus（参考 Grafana 官方 docker-compose 示例）
  • 配置 OpenTelemetry SDK，将 Traces 导出到 Tempo，Metrics 导出到 Prometheus
  • 在 Agent 服务中实现 AgentTracingAdvisor，为每次 LLM 调用自动创建 Span
  • 配置 logback-spring.xml 输出 JSON 格式日志到 Loki
  • 在 Grafana 中创建 Dashboard，包含至少 4 个面板：QPS、P95 延迟、Token 消耗趋势、成本趋势
✍️ 练习 2：实现告警策略

基于 Prometheus Alertmanager 实现 Agent 系统的告警策略：

  • 编写至少 3 条 Prometheus 告警规则（任务成功率、P95 延迟、成本阈值）
  • 配置 Alertmanager 路由规则，P0 告警发送到 Webhook（模拟电话通知），P1 告警发送到邮件
  • 实现告警收敛逻辑：相同告警在 5 分钟内只通知一次
  • 测试告警流程：模拟 LLM 延迟飙升，验证从异常发生到收到告警的完整链路
✍️ 练习 3：实现对话日志回放系统

构建一个对话日志的回放和分析系统：

  • 设计对话日志的存储 Schema（包含 traceId、userId、messages 数组、metrics 摘要）
  • 实现一个 REST API，支持按 traceId 查询完整的对话历史（包括每次 LLM 调用的 Token 消耗和延迟）
  • 实现一个简单的"对话回放"页面，按时间线展示 Agent 的每一步操作：用户输入 -> Agent 决策 -> 工具调用 -> LLM 响应
  • 添加对话质量标注功能：支持人工标记每次对话的质量评分（1-5 分），用于后续的模型评估
