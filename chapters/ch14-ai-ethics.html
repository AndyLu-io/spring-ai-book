第 14 章

AI 伦理与治理

理解 AI 伦理的核心原则，掌握主流合规框架要求，学会在 Agent 系统中实施安全防护与伦理治理策略

预计 1-2 周 6 个知识点 中级难度

📚 开篇故事

想象你去一家医院看病。医生开药之前，一定会先问清你的过敏史、正在服用的药物，然后按照诊疗规范来开处方。这套"先确认、再执行、可追溯"的流程，不是为了限制医生的医术，而是为了保护患者的安全。

AI 系统就像一位拥有"超能力"的医生 — 它可以在几秒钟内影响成千上万的人。如果没有伦理规范，这位"AI 医生"可能会因为训练数据中的偏见而歧视某些群体，因为缺乏透明度而做出无法解释的决策，因为没有问责机制而让错误无人承担。

2024 年，某银行的 AI 信贷审批系统被曝出对女性申请人的拒贷率比同等条件的男性高出 23%。调查发现，训练数据中 10 年前的历史数据本身就带有性别偏见。这不是 AI "作恶"，而是 AI 忠实地放大了人类社会已有的不公。

AI 伦理与治理，就是给 AI 这位"超级医生"建立一套职业道德规范和监管制度。它确保 AI 在强大的同时，是公平的、透明的、可问责的。本章将带你从原则到实践，掌握构建负责任 AI 系统的完整知识体系。

🎯 学习目标

在本章结束后，你将能够：

  • 理解伦理原则 — 掌握公平性、透明性、可问责性、隐私保护等 AI 伦理核心原则
  • 掌握偏见检测 — 学会识别训练数据偏见、输出偏见，并应用缓解策略
  • 熟悉合规框架 — 了解 EU AI Act、中国《生成式 AI 管理办法》、GDPR 等主流法规要求
  • 实施可解释性 — 掌握决策轨迹可视化、审计日志设计等可解释 AI 技术
  • 代码实战 — 能用 Spring AI 实现内容审核 Advisor、偏见检测工具和审计日志

📄 第一节：AI 伦理基础

概念比喻：AI 的"职业道德守则"

如果说 AI 是一位拥有超能力的员工，那么 AI 伦理就是它的职业道德守则。就像医生有希波克拉底誓言、律师有职业伦理规范一样，AI 系统也需要一套行为准则来确保它在追求效率的同时，不伤害人类的权益。

AI 伦理的四大核心原则，就像交通规则中的"红灯停、绿灯行" — 简单却不可或缺：

公平性（Fairness）：确保 AI 不歧视

公平性要求 AI 系统对所有群体一视同仁，不因性别、种族、年龄等因素产生差别对待。

公平性度量维度

  ┌─────────────────────────────────────────────┐
  │              公平性度量框架                    │
  ├─────────────┬───────────────┬───────────────┤
  │  群体公平    │  个体公平      │  反事实公平    │
  ├─────────────┼───────────────┼───────────────┤
  │ 各群体获得   │ 相似个体获得   │ 改变受保护属性  │
  │ 相同结果     │ 相似结果       │ 结果应不变     │
  ├─────────────┼───────────────┼───────────────┤
  │ 统计均等     │ 李普西茨连续性  │ 反事实检验     │
  │ 机会均等     │               │               │
  └─────────────┴───────────────┴───────────────┘

公平性的数学定义

  • 统计均等（Statistical Parity）：P(预测=1|群体A) = P(预测=1|群体B)，即各群体获得正向预测的概率相同
  • 机会均等（Equal Opportunity）：P(预测=1|实际=1, 群体A) = P(预测=1|实际=1, 群体B)，即在真正合格的个体中，各群体获得正向预测的概率相同
  • 预测均等（Predictive Parity）：P(实际=1|预测=1, 群体A) = P(实际=1|预测=1, 群体B)，即在获得正向预测的个体中，各群体真正合格的概率相同
⚠️ 重要提醒

上述三种公平性度量在数学上不可能同时满足（除非基础率相同或预测完全准确）。因此，公平性选择本质上是一个社会价值判断，需要根据具体业务场景来决定。

透明性（Transparency）：让决策可理解

透明性要求 AI 系统的决策过程可以被理解、被解释、被验证。就像法官判案必须写出判决理由一样，AI 给出结论时也需要说明"为什么"。

透明性的三个层次

  • 模型透明：公开模型的架构、训练数据来源、性能指标。开源模型天然具备这个优势
  • 决策透明：对单个预测结果提供解释。例如："贷款被拒是因为：1) 收入低于中位数 2) 信用历史不足 3) 负债比偏高"
  • 系统透明：公开 AI 系统的使用目的、影响范围、已知局限性和风险
// 透明性原则在 API 设计中的体现 public record ExplainableResult<T>( T prediction, // 预测结果 List<Explanation> factors, // 决策因素及权重 double confidence, // 置信度 List<String> warnings, // 已知局限性 String modelVersion, // 使用的模型版本 Instant timestamp // 决策时间戳 ) {} public record Explanation( String factor, // 影响因素名称 String detail, // 具体描述 double weight, // 权重（-1.0 到 1.0） String direction // "positive" 或 "negative" ) {}

可问责性（Accountability）：明确责任归属

可问责性要求 AI 系统的每一个决策都能追溯到具体的责任主体。出了问题，不能说"是 AI 自己做的"。

AI 可问责性链路

  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
  │ 数据提供者 │──>│ 模型训练者 │──>│ 系统部署者 │──>│ 最终用户   │
  └──────────┘   └──────────┘   └──────────┘   └──────────┘
       │              │              │              │
       ▼              ▼              ▼              ▼
  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
  │ 数据质量   │   │ 模型偏见  │   │ 系统滥用  │   │ 使用不当  │
  │ 责任       │   │ 责任     │   │ 责任      │   │ 责任      │
  └──────────┘   └──────────┘   └──────────┘   └──────────┘

可问责性的实施要素

  • 审计日志：记录每一次 AI 决策的输入、输出、使用模型版本和时间戳
  • 版本管理：每次模型更新都有版本号和变更说明，支持回滚
  • 人机协作：高风险决策保留人工审核环节，AI 提供建议，人类做最终决定
  • 投诉机制：建立用户反馈和申诉渠道，有明确的处理流程和时限

隐私保护（Privacy）：数据最小化原则

隐私保护要求 AI 系统在数据收集、存储、使用各环节都遵循最小化原则 — 只收集必要的数据，只使用授权的范围，只保留需要的时长。

关键技术手段

  • 数据脱敏：在训练前对敏感字段进行匿名化或假名化处理
  • 差分隐私：在数据或模型中加入校准噪声，防止从输出反推个体信息
  • 联邦学习：数据不出本地，只交换模型参数，从根本上避免数据集中
  • 同态加密：在加密数据上直接计算，无需解密
💡 实践建议

在 Agent 系统中，每次工具调用都可能涉及数据外传。务必在 Agent 的系统提示中明确数据处理边界，并在工具层面实现数据过滤和脱敏。

📄 第二节：偏见检测与缓解

偏见的来源：AI 如何"继承"人类偏见

偏见不是 AI 自己产生的，而是从三个环节"学"来的：

AI 偏见传播链

  ┌──────────────┐
  │ 社会历史偏见  │  历史数据本身就带有歧视性
  └──────┬───────┘
         v
  ┌──────────────┐
  │ 训练数据偏见  │  数据采集不均衡、标注偏差
  └──────┬───────┘
         v
  ┌──────────────┐
  │ 模型学习偏见  │  特征选择不当、过拟合特定模式
  └──────┬───────┘
         v
  ┌──────────────┐
  │ 输出放大偏见  │  线上部署后反馈循环加剧
  └──────────────┘

典型偏见类型

  • 采样偏见（Sampling Bias）：训练数据不能代表目标人群。例如，语音识别系统主要用英语母语者数据训练，对非母语口音识别率显著下降
  • 标签偏见（Labeling Bias）：人工标注时引入标注者的主观偏见。例如，"有攻击性的"内容标注标准因文化而异
  • 特征偏见（Feature Bias）：使用与受保护属性高度相关的代理特征。例如，邮政编码可以间接反映种族分布
  • 确认偏见（Confirmation Bias）：模型倾向于"验证"训练数据中已存在的模式

偏见检测方法

检测偏见就像给 AI 做"体检"，需要从数据和输出两个维度进行系统性检查。

训练数据偏见检测

@Service public class BiasDetector { /** * 检查训练数据中的群体分布均衡性 * 统计均等差异度 = |P(Y=1|A=0) - P(Y=1|A=1)| * 差异度 > 0.1 即认为存在显著偏见 */ public BiasReport checkGroupBalance( List<TrainingSample> data, String protectedAttribute, String targetAttribute) { // 按受保护属性分组 var groups = data.stream() .collect(Collectors.groupingBy( s -> s.features().get(protectedAttribute))); var rates = new HashMap<String, Double>(); for (var entry : groups.entrySet()) { long positiveCount = entry.getValue().stream() .filter(s -> "1".equals( s.labels().get(targetAttribute))) .count(); double rate = (double) positiveCount / entry.getValue().size(); rates.put(entry.getKey(), rate); } // 计算最大差异度 double maxDiff = rates.values().stream() .mapToDouble(r1 -> rates.values().stream() .mapToDouble(r2 -> Math.abs(r1 - r2)) .max().orElse(0.0)) .max().orElse(0.0); return new BiasReport( protectedAttribute, rates, maxDiff, maxDiff > 0.1 ? Severity.HIGH : maxDiff > 0.05 ? Severity.MEDIUM : Severity.LOW ); } }

输出偏见检测

/** * 检测模型输出是否对不同群体存在差异化对待 * 方法：构造仅受保护属性不同的对照样本，比较输出差异 */ public OutputBiasReport checkOutputBias( ChatClient chatClient, List<Pair<String, String>> controlPairs) { var divergences = new ArrayList<Double>(); for (var pair : controlPairs) { String outputA = chatClient.prompt() .user(pair.getLeft()).call().content(); String outputB = chatClient.prompt() .user(pair.getRight()).call().content(); // 用嵌入向量计算输出相似度 double similarity = computeSimilarity( outputA, outputB); divergences.add(1.0 - similarity); } double avgDivergence = divergences.stream() .mapToDouble(d -> d).average().orElse(0.0); return new OutputBiasReport( controlPairs.size(), avgDivergence, avgDivergence > 0.3 ); }

偏见缓解策略

偏见缓解就像给 AI "矫正视力"，根据干预时机分为三个阶段：

① 预处理（Pre-processing）

在训练数据上做文章：

  • 重采样：过采样少数群体或欠采样多数群体，使分布均衡
  • 重标签：用更公平的标注标准重新标记数据
  • 特征变换：使用对抗学习去除受保护属性的信息

② 处理中（In-processing）

在训练过程中干预：

  • 约束优化：在损失函数中加入公平性约束项
  • 对抗训练：训练一个"公平性判别器"来消除偏见
  • 正则化：对偏见方向的梯度进行惩罚

③ 后处理（Post-processing）

在模型输出阶段修正：

  • 阈值调整：为不同群体设置不同的决策阈值，以达到机会均等
  • 输出校准：根据公平性度量对预测概率进行后校准
  • 拒绝选项：对于置信度低的边缘案例，转为人工审核
💡 选择策略

预处理成本最低、侵入性最小，适合大多数场景。处理中效果最好但需要访问模型内部，通常只有模型训练者才能使用。后处理适用于已部署的黑盒模型。

📄 第三节：合规框架

概念比喻：AI 的"交通法规"

如果说伦理原则是"开车要有道德"，那么合规框架就是写在法律里的交通规则 — 红灯停、限速 120、必须系安全带。各国政府正在为 AI 制定类似的"交通法规"，违反者将面临罚款甚至刑事责任。

中国《生成式人工智能服务管理暂行办法》

2023 年 8 月 15 日正式实施，是中国 AI 领域最重要的专项法规。

要求具体内容对 Agent 开发的影响
内容安全 生成内容不得含有颠覆国家政权、恐怖主义、歧视等内容 必须实现输入/输出内容审核机制
数据合规 训练数据不得侵犯知识产权，个人信息处理需遵守《个保法》 数据来源必须合法合规，需建立数据溯源机制
用户协议 明确服务协议和使用规范，告知用户 AI 生成属性 输出内容需标注"AI 生成"，用户协议需包含风险提示
投诉处理 建立投诉举报机制，及时处理违规内容 需实现内容举报和快速响应流程
安全评估 按规定进行安全评估和算法备案 上架前需通过安全评估，按要求备案算法
标识义务 对生成的图片、视频等内容进行标识 需在元数据中嵌入 AI 生成标识

EU AI Act：全球最严 AI 法案

2024 年 8 月正式生效的欧盟《人工智能法案》，是全球首个对 AI 进行系统性监管的法律。它采用基于风险的分级管理：

EU AI Act 风险分级体系

  ┌─────────────────────────────────────────────┐
  │        不可接受风险（禁止）                  │
  │  社会信用评分、实时远程生物识别（执法除外）     │
  ├─────────────────────────────────────────────┤
  │        高风险（严格监管）                     │
  │  信用评估、招聘筛选、司法辅助、医疗诊断        │
  │  要求：风险评估 + 数据治理 + 人类监督 + 日志  │
  ├─────────────────────────────────────────────┤
  │        有限风险（透明度义务）                 │
  │  聊天机器人、情感识别系统                      │
  │  要求：告知用户正在与 AI 交互                  │
  ├─────────────────────────────────────────────┤
  │        最小风险（自由使用）                   │
  │  垃圾邮件过滤、AI 游戏、推荐系统               │
  └─────────────────────────────────────────────┘

对 Agent 系统的关键要求

  • 高风险 Agent：必须进行 Conformity Assessment（合格评估），建立质量管理体系
  • 通用 AI（GPAI）：基础模型提供商需披露训练数据摘要、进行红队测试
  • 透明度：用户必须知道他们正在与 AI 交互，AI 生成内容必须可识别
  • 人工监督：高风险 AI 系统必须有人工干预和否决机制

GDPR 对 AI 的要求

欧盟《通用数据保护条例》虽然不是 AI 专项法律，但对 AI 系统有深远影响：

  • 第 22 条 — 自动化决策：数据主体有权不受仅基于自动化处理的决策约束，有权获得人工干预、表达观点和质疑决定
  • 第 13-14 条 — 信息权：必须告知用户自动化决策的存在、逻辑、意义和预期后果
  • 第 15 条 — 访问权：用户有权获得关于其个人数据如何用于自动化决策的信息
  • 第 25 条 — 隐私设计：AI 系统必须在设计阶段就考虑数据保护
📚 实践要点

即使你的服务不在欧盟，只要处理欧盟居民的数据，就必须遵守 GDPR。对于全球化 Agent 产品，建议默认按 GDPR 标准设计。

📄 第四节：可解释性（XAI）

概念比喻：AI 的"透明窗口"

想象你走进一家餐厅的开放式厨房 — 你能看到厨师的每一个操作步骤，食材从哪里来、怎么处理、放了什么调料。可解释性就是给 AI 系统装上这样一个透明窗口，让用户和监管者都能看清决策的全过程。

决策轨迹可视化

对于 Agent 系统，可解释性不仅包括模型本身的可解释性，还包括推理链路的可追溯性 — Agent 为什么选择这个工具？中间推理了哪些步骤？

Agent 决策轨迹可视化

  用户输入: "北京明天天气怎么样？"
    │
    ├─ 意图识别: WeatherQuery (confidence: 0.95)
    │     └─ 提取参数: {city: "北京", date: "明天"}
    │
    ├─ 工具选择: WeatherAPI (reason: "需要实时天气数据")
    │     └─ 候选工具: WeatherAPI(0.93), SearchAPI(0.45), Calculator(0.02)

置信度表达机制

AI 系统应该诚实地告诉用户"我有多确定"，而不是对所有问题都给出一样自信的回答。

// 置信度分级与用户展示映射 public enum ConfidenceLevel { @ConfidenceRange(min = 0.9, max = 1.0) HIGH("确定", "基于可靠数据源，结果高度可信"), @ConfidenceRange(min = 0.7, max = 0.9) MEDIUM("较大可能", "基于可用信息，建议进一步验证"), @ConfidenceRange(min = 0.5, max = 0.7) LOW("仅供参考", "信息有限，建议人工核实"), @ConfidenceRange(min = 0.0, max = 0.5) UNRELIABLE("无法确定", "缺乏足够信息，建议咨询专业人士"); private final String label; private final String disclaimer; // 嵌入输出中的置信度标注 public String annotate(String response) { return response + "\n\n---\n[" + label + "] " + disclaimer; } }

审计日志设计

审计日志是 AI 可问责性的技术基石。每一条日志都是一份"证据"，记录了 AI 系统在何时、何地、对谁、做了什么决策。

// Agent 审计日志数据模型 public record AuditLogEntry( String traceId, // 请求追踪ID String sessionId, // 会话ID Instant timestamp, // 时间戳 String userId, // 用户标识（脱敏后） String action, // 操作类型 String input, // 输入内容 String output, // 输出内容 String modelId, // 模型版本 double confidence, // 置信度 List<String> toolsUsed, // 调用的工具列表 List<SafetyCheck> safetyChecks, // 安全检查结果 Map<String, Object> metadata // 扩展元数据 ) {} // 安全检查记录 public record SafetyCheck( String checkType, // 检查类型 boolean passed, // 是否通过 String detail, // 详情 Instant checkedAt // 检查时间 ) {}
💡 审计日志设计原则

1) 不可篡改：日志写入后不应被修改或删除，建议使用追加写入模式。2) 最小化：只记录必要信息，敏感数据需脱敏。3) 可检索：按 traceId、userId、时间范围等维度可快速查询。4) 合规留存：根据当地法律要求确定保留时长（通常 6 个月至 3 年）。

基于 Spring AI Advisor 模式实现的内容审核拦截器，在每次 LLM 调用前后自动进行安全检查。

// ===== 内容审核 Advisor ===== @Component public class ContentModerationAdvisor implements CallAdvisor { private final ChatClient safetyClassifier; private final AuditLogger auditLogger; private static final Set<String> BLOCKED_CATEGORIES = Set.of( "violence", "hate_speech", "illegal_activity", "personal_attack", "political_sensitivity" ); @Override public String getName() { return "ContentModerationAdvisor"; } @Override public AdvisResponse aroundCall( CallAdvisorChain chain, ChatClientRequest request, CallOptions options) { String userInput = extractUserMessage(request); // 1. 输入内容审核 ModerationResult inputCheck = moderateContent(userInput, "input"); if (inputCheck.blocked()) { auditLogger.logSafetyEvent( extractTraceId(request), "INPUT_BLOCKED", inputCheck.category(), userInput ); return new AdvisResponse( buildBlockedResponse(inputCheck.category()), request ); } // 2. 调用下游（正常流程） AdvisResponse response = chain.nextCall(request, options); // 3. 输出内容审核 String assistantOutput = extractAssistantContent(response); ModerationResult outputCheck = moderateContent(assistantOutput, "output"); if (outputCheck.blocked()) { auditLogger.logSafetyEvent( extractTraceId(request), "OUTPUT_BLOCKED", outputCheck.category(), assistantOutput ); return new AdvisResponse( buildBlockedResponse(outputCheck.category()), request ); } // 4. 记录审核通过的审计日志 auditLogger.logNormalRequest( extractTraceId(request), userInput, assistantOutput ); return response; } private ModerationResult moderateContent( String text, String phase) { String result = safetyClassifier.prompt() .system(""" 你是一个内容安全分类器。判断文本是否包含 以下违规类别：violence, hate_speech, illegal_activity, personal_attack, political_sensitivity, safe。 只返回 JSON 格式： {"category": "类别", "safe": true/false, "confidence": 0.0-1.0} """) .user(text) .call() .content(); return parseModerationResult(result); } } public record ModerationResult( String category, boolean safe, boolean blocked, double confidence ) {}

// ===== 审计日志服务 ===== @Service public class AuditLogger { private final AuditLogRepository repository; private final ObjectMapper objectMapper; public void logNormalRequest( String traceId, String input, String output) { var entry = new AuditLogEntry( traceId, getCurrentSessionId(), Instant.now(), anonymizeUserId(getCurrentUserId()), "CHAT_COMPLETION", anonymizePII(input), anonymizePII(output), getCurrentModelVersion(), calculateConfidence(output), getToolsUsed(), List.of(), Map.of("phase", "normal") ); repository.save(entry); } public void logSafetyEvent( String traceId, String eventType, String category, String content) { var entry = new AuditLogEntry( traceId, getCurrentSessionId(), Instant.now(), anonymizeUserId(getCurrentUserId()), eventType, anonymizePII(content), null, // 被拦截的请求没有输出 getCurrentModelVersion(), 0.0, List.of(), List.of(new SafetyCheck( category, false, "Blocked by " + eventType, Instant.now() )), Map.of("phase", "safety") ); repository.save(entry); } // PII 脱敏处理 private String anonymizePII(String text) { if (text == null) return null; // 手机号脱敏 text = text.replaceAll( "(1[3-9]\\d)\\d{4}(\\d{4})", "$1****$2" ); // 身份证脱敏 text = text.replaceAll( "(\\d{6})\\d{8}(\\d{4})", "$1********$2" ); // 邮箱脱敏 text = text.replaceAll( "(\\w{2})\\w+@(\\w+\\.\\w+)", "$1***@$2" ); return text; } }

3. 偏见检测工具

// ===== Agent 偏见检测工具 ===== @Service public class BiasDetectionTool { /** * 注册为 Spring AI Tool，Agent 可以主动调用 * 在生成关键内容前自动检测潜在偏见 */ @Tool(description = "检测文本内容是否存在偏见，" + "返回偏见类型、严重程度和建议") public BiasCheckResult detectBias( @ToolParam(description = "待检测的文本") String text, @ToolParam(description = "上下文场景描述") String context) { // 基于规则的快速检测 List<BiasIndicator> indicators = ruleBasedCheck(text); // 基于 LLM 的深度分析 String analysis = biasAnalyzer.prompt() .system(""" 分析以下文本是否存在偏见。考虑： 1. 性别/种族/年龄/地域等维度的刻板印象 2. 语言中的隐含假设 3. 信息呈现的选择性偏差 返回 JSON 格式的分析结果。 """) .user("上下文: " + context + "\n文本: " + text) .call() .content(); return buildBiasCheckResult(indicators, analysis); } private List<BiasIndicator> ruleBasedCheck(String text) { var indicators = new ArrayList<BiasIndicator>(); // 性别刻板印象检测 if (containsGenderStereotype(text)) { indicators.add(new BiasIndicator( BiasType.GENDER_STEREOTYPE, Severity.MEDIUM, "检测到可能的性别刻板印象表述" )); } // 年龄偏见检测 if (containsAgeBias(text)) { indicators.add(new BiasIndicator( BiasType.AGE_BIAS, Severity.LOW, "检测到可能的年龄相关偏见" )); } return indicators; } } public record BiasCheckResult( List<BiasIndicator> indicators, String llmAnalysis, boolean hasBias, Severity overallSeverity, String suggestion ) {}

// ===== 配置类：为 Agent 注入伦理治理能力 ===== @Configuration public class EthicalAgentConfig { @Bean public ChatClient ethicalChatClient( ChatClient.Builder builder, ContentModerationAdvisor moderationAdvisor, AuditLoggingAdvisor auditAdvisor) { return builder .defaultAdvisors( // 安全检查在最外层 auditAdvisor, moderationAdvisor ) .defaultTools(new BiasDetectionTool()) .defaultSystem(""" 你是一个负责任的 AI 助手。 请遵循以下原则： 1. 对所有用户群体保持公平和尊重 2. 当你不确定时，明确表示不确定 3. 不对受保护属性（性别、种族、年龄等） 做出假设性判断 4. 涉及专业领域（医疗、法律、金融）时， 建议用户咨询专业人士 """) .build(); } }

当用户通过 ethicalChatClient 发送请求时，会自动经过：审计日志记录 → 内容安全审核 → 模型生成 → 输出安全审核 → 偏见检测工具可选调用。整个过程对用户透明，但每一步都被记录和监控。

📖 章节小结

  • 公平性、透明性、可问责性、隐私保护 — 这四大原则是 AI 伦理的基石。公平性确保不歧视，透明性让决策可理解，可问责性明确责任归属，隐私保护遵循数据最小化
  • 偏见源于数据和模型 — 训练数据的采样偏差、标签偏差、特征偏差都会导致模型输出偏见。缓解策略分为预处理、处理中和后处理三个阶段
  • 合规是底线 — 中国《生成式 AI 管理办法》要求内容安全审核、数据合规和算法备案；EU AI Act 按风险分级进行监管；GDPR 要求自动化决策可解释和可干预
  • 可解释性是信任的基础 — 通过决策轨迹可视化、置信度表达和审计日志设计，让用户和监管者都能理解 AI 的决策过程
  • Spring AI 提供了落地工具 — Advisor 模式天然适合实现内容审核和审计日志，Tool 机制可以注册偏见检测等伦理工具

🎓 面试高频题

👨‍💻 Q1: 什么是 AI 中的公平性？如何在技术上度量和实现？

答：公平性确保 AI 系统不对不同群体产生歧视性输出。技术上常用三种度量：统计均等（各群体获得正向预测的概率相同）、机会均等（真正合格的个体中各群体获得正向预测的概率相同）、预测均等（获得正向预测的个体中各群体真正合格的概率相同）。注意这三者在数学上不可同时满足。实现上分为预处理（数据重采样）、处理中（约束优化、对抗训练）和后处理（阈值校准）。实际选择取决于业务场景的风险容忍度和合规要求。

👨‍💻 Q2: EU AI Act 如何对 AI 系统进行风险分级？不同级别的要求有什么区别？

答：EU AI Act 将 AI 系统分为四个风险级别：不可接受风险（如社会信用评分，直接禁止）、高风险（如信用评估、招聘筛选，需要合格评估、质量管理体系、人类监督）、有限风险（如聊天机器人，需告知用户正在与 AI 交互）、最小风险（如垃圾邮件过滤，自由使用）。对于 Agent 系统，如果是高风险场景，必须实现人工干预机制、完整的决策日志和可解释性。

👨‍💻 Q3: 在 Agent 系统中，如何设计有效的审计日志？需要记录哪些关键信息？

答：审计日志应记录：traceId（追踪ID）、timestamp（时间戳）、userId（脱敏后的用户标识）、input/output（输入输出内容，PII需脱敏）、modelId（模型版本）、confidence（置信度）、toolsUsed（调用的工具列表）、safetyChecks（安全检查结果）。设计原则：不可篡改（追加写入）、PII脱敏（符合GDPR）、可检索（按多维度索引）、合规留存（根据法规确定保留时长）。Spring AI中可通过Advisor模式在每次调用前后自动记录。

👨‍💻 Q4: 中国《生成式AI服务管理暂行办法》对开发者有哪些核心要求？

答：核心要求包括：1) 内容安全 — 必须实现输入/输出内容审核，不得生成违规内容；2) 数据合规 — 训练数据不得侵犯知识产权，个人信息处理遵守《个保法》；3) 标识义务 — AI生成的内容需标注，图片视频需嵌入元数据标识；4) 用户告知 — 明确告知用户AI生成属性，服务协议包含风险提示；5) 投诉机制 — 建立投诉举报和快速响应流程；6) 安全评估 — 按规定进行安全评估和算法备案。

👨‍💻 Q5: 如何在 Spring AI 中实现 Agent 的内容安全审核？

答：推荐使用 Spring AI 的 Advisor 模式。实现一个 ContentModerationAdvisor 实现 CallAdvisor 接口，在 aroundCall 方法中：1) 提取用户输入，调用安全分类器进行输入审核；2) 调用 chain.nextCall 执行正常流程；3) 对输出内容进行安全审核；4) 记录审计日志。通过 ChatClient.defaultAdvisors 注册，所有请求自动经过审核。安全分类器可以用专门的LLM实现，也可以用关键词匹配+正则作为快速层，LLM作为深度分析层。

✍️ 练习 1：构建公平性检测数据集

目标：构建一组对照样本来测试你的 AI 系统是否存在偏见。

步骤：
1) 选择一个业务场景（如简历筛选、贷款审批、内容推荐）
2) 构建 10 组对照样本对，每对仅在受保护属性上不同（如姓名暗示不同性别，其他条件相同）
3) 用你的 Agent 系统处理这些样本，记录输出结果
4) 计算输出差异度，判断是否存在偏见
5) 如果存在偏见，尝试应用一种缓解策略并验证效果

进阶：尝试从 3 个不同维度（性别、年龄、地域）分别构建对照集，综合评估系统的公平性。
