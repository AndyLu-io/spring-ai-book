第 30 章

企业级 Agent 平台架构

从"单体 Agent"到"Agent 即服务" — 掌握多租户隔离、编排引擎、权限审计、Agent 市场和 SDK 网关设计，构建可规模化的 Agent 平台。

⏱ 预计 4-5 周 📖 6 个知识点 💻 专家级难度
本章目录

    📚 开篇故事：从路边摊到连锁帝国

    你开发了一个很棒的客服 Agent——它能自动回答用户问题、处理退换货、甚至进行情感安抚。技术团队非常满意，产品经理也非常满意。

    但问题很快来了。销售部找过来说："我们也想要一个 Agent，帮销售团队自动生成客户跟进邮件和报价单。"法务部说："能不能做一个合同审查 Agent，自动识别合同中的风险条款？"HR 部说："我们想用 Agent 筛选简历，自动评估候选人匹配度。"

    每个部门的需求各不相同：数据敏感度不同（HR 的薪资数据绝不能让销售部看到），权限要求不同（法务 Agent 需要调用外部法律数据库，而客服 Agent 不需要），模型偏好不同（销售部想用 GPT-4 获得最佳文案质量，客服部想用本地模型控制成本）。

    如果为每个部门单独开发一套系统，成本巨大且维护噩梦。如果共用一套系统，又面临隔离、定制、权限等一大堆问题。

    这就是企业级 Agent 平台要解决的核心问题——像麦当劳一样，用一套标准化的体系支撑千变万化的本地需求。麦当劳不会为每个城市单独发明一套做汉堡的方法，而是建立了一套完整的标准化体系：统一的供应链、统一的操作流程、统一的品质控制，但允许每个门店根据当地口味做微调（比如在中国卖油条、在日本卖照烧堡）。

    企业级 Agent 平台正是这样一个"标准化 + 定制化"的系统。它让企业只需要建设一次基础设施，就能让所有部门快速创建、部署和管理自己的 Agent，同时保证数据隔离、安全合规和成本可控。

    📄 核心内容（第1节）：平台架构全景 — 从 Agent 到 Agent-as-a-Service

    平台化的核心价值

    为什么要将 Agent 从"单体应用"升级为"平台服务"？核心价值体现在三个维度：

    • 标准化 — 统一的 Agent 开发、部署、监控流程。不同团队使用相同的 SDK、相同的 API 规范、相同的运维工具，降低沟通和维护成本
    • 可复用 — Prompt 模板、Tool 注册、RAG 引擎、记忆存储等能力作为平台级服务，每个 Agent 不需要从零构建，而是"组装"现有组件
    • 可治理 — 统一的权限控制、审计日志、成本核算、质量监控。企业能在一个 dashboard 上看到所有 Agent 的运行状态和资源消耗

    五层架构设计

    一个成熟的企业级 Agent 平台通常采用五层架构：

    ┌───────────────────────────────────────────────────────────────────┐
│                    接入层 (Gateway)                        │
│         API Gateway / SDK / Web Console / CLI             │
└──────────────────────────────────────────────────────────────────────┘          ┳
              │
┌───────────────────────────────────────────────────────────────────┐
│                    编排层 (Orchestration)                  │
│    Agent 编排引擎 / 工作流引擎 / 多Agent调度器              │
└──────────────────────────────────────────────────────────────────────┘          ┳
              │
┌───────────────────────────────────────────────────────────────────┐
│                    能力层 (Capability)                     │
│   Prompt管理 / RAG引擎 / Tool Registry / Memory Store     │
└──────────────────────────────────────────────────────────────────────┘          ┳
              │
┌───────────────────────────────────────────────────────────────────┐
│                    数据层 (Data)                           │
│   向量数据库 / 知识图谱 / 会话存储 / 日志审计              │
└──────────────────────────────────────────────────────────────────────┘          ┳
              │
┌───────────────────────────────────────────────────────────────────┐
│                    基础设施层 (Infra)                     │
│   模型服务 / GPU集群 / 消息队列 / 对象存储                 │
└──────────────────────────────────────────────────────────────────────┘

    各层职责明确：

    • 接入层 — 统一流量入口。API Gateway 负责认证、限流、路由；SDK 为开发者提供便捷的接入方式；Web Console 提供可视化管理界面
    • 编排层 — Agent 的"大脑"。编排引擎负责调度 Agent 的执行流程，工作流引擎支持多步骤复杂任务，多 Agent 调度器实现 Agent 之间的协作
    • 能力层 — Agent 的"四肢"。Prompt 管理中心统一管理提示词模板，RAG 引擎提供知识检索能力，Tool Registry 管理所有可用工具，Memory Store 提供记忆存储
    • 数据层 — Agent 的"记忆"。向量数据库存储语义索引，知识图谱存储结构化关系，会话存储保存对话历史，日志审计记录所有操作
    • 基础设施层 — 平台的"地基"。模型服务管理 LLM 的部署和调用，GPU 集群提供计算资源，消息队列实现异步通信，对象存储管理文件和媒体

    与传统微服务架构的对比

    维度传统微服务Agent 平台
    核心单元Service（无状态函数）Agent（有状态、有目标、有工具）
    通信方式同步 RPC / HTTP异步消息 + 流式响应
    状态管理无状态或外部存储内置记忆系统（短期/长期）
    错误处理重试、熔断、降级自主推理、工具切换、回退策略
    可观测性日志 + 指标 + 链路追踪日志 + 指标 + 链路追踪 + Agent Trace
    扩展方式水平扩展 Service 实例扩展模型服务 + Agent 并发

    关键设计原则

    松耦合

    每个 Agent 独立开发、独立部署、独立扩缩。Agent 之间通过标准协议通信，不共享内部状态。新增 Agent 不影响已有 Agent。

    高内聚

    Agent 相关的所有能力（Prompt、Tool、Memory、RAG）聚合在一个配置单元中。一个 Agent 的完整定义包含：提示词、可用工具、记忆策略、权限范围。

    可观测

    每个 Agent 的每次调用都可追踪：输入了什么、调用了哪些工具、推理了哪些步骤、输出了什么。成本、延迟、质量三维监控。

    可扩展

    水平扩展：同一 Agent 可部署多个实例处理并发。垂直扩展：可切换更强的模型。能力扩展：可动态注册新的 Tool 和 RAG 知识库。

    💡 架构选型建议

    不是所有企业都需要从第一天就构建完整的五层平台。渐进式建设是更务实的策略：先从接入层 + 能力层开始（一个 API Gateway + Spring AI SDK），验证 Agent 场景后，再逐步添加编排层、数据层和基础设施层。避免过度设计——平台是长出来的，不是画出来的。

    📄 核心内容（第2节）：多租户隔离 — 一套系统服务千个团队

    多租户隔离模型

    多租户是企业级平台的基石。根据隔离程度和成本，有三种主流模型：

    隔离模型隔离级别成本适用场景
    共享数据库逻辑隔离（tenant_id 字段）最低中小租户、开发测试环境
    独立 SchemaSchema 级隔离中等中大型租户、数据敏感度中等
    独立数据库物理隔离最高大型企业、金融/医疗等高安全场景
    📚 Agent 平台的特殊性

    Agent 平台的多租户比传统 SaaS 更复杂，因为除了数据隔离，还需要隔离 Prompt（提示词策略）、Tool（工具权限）、Memory（对话记忆）和Model（模型访问权限）。一个租户的 Agent 不能看到另一个租户的 Prompt 模板、不能调用另一个租户的 Tool、不能读取另一个租户的对话历史。

    Agent 级隔离设计

    @Entity @Table(name = "tenants") public class Tenant { @Id private String id; private String name; private TenantTier tier; // FREE, PRO, ENTERPRISE @OneToMany private List<AgentConfig> agents; @Embedded private ResourceQuota quota; } @Embeddable public class ResourceQuota { private int maxAgents; private int maxRequestsPerDay; private int maxTokensPerMonth; private int maxConcurrentSessions; private List<String> allowedModels; } @Service public class TenantIsolationService { public AgentContext createContext(String tenantId, String agentId) { Tenant tenant = tenantRepo.findById(tenantId) .orElseThrow(() -> new TenantNotFoundException(tenantId)); // 检查配额 checkQuota(tenant); // 构建隔离上下文 return AgentContext.builder() .tenantId(tenantId) .agentId(agentId) .modelRouter(selectModelRouter(tenant.getTier())) .vectorStore(tenantIsolatedVectorStore(tenantId)) .toolRegistry(filteredToolRegistry(tenantId)) .memoryStore(isolatedMemoryStore(tenantId)) .build(); } private void checkQuota(Tenant tenant) { ResourceUsage usage = usageTracker.getCurrentUsage(tenant.getId()); ResourceQuota quota = tenant.getQuota(); if (usage.getRequestsToday() >= quota.getMaxRequestsPerDay()) { throw new QuotaExceededException("Daily request limit exceeded"); } } }

    隔离上下文中的每个组件都按照租户维度进行了隔离：

    • Model Router — 根据租户等级路由到不同的模型服务。FREE 用户走共享模型池，PRO 用户走专用实例，ENTERPRISE 用户走私有部署
    • Vector Store — 每个租户有独立的向量集合（Collection），通过 namespace 或 metadata filter 实现数据隔离
    • Tool Registry — 每个租户只能看到和使用自己注册的 Tool，通过 ACL（访问控制列表）过滤
    • Memory Store — 对话历史按租户隔离存储，绝不会跨租户泄露

    资源配额与限流

    平台需要对每个租户的资源使用进行精确控制：

    @Component public class TenantRateLimiter { private final RedisTemplate<String, String> redis; public boolean tryAcquire(String tenantId, QuotaType type) { String key = "quota:{tenantId}:{type}:" + LocalDate.now(); long current = redis.opsForValue().increment(key); if (current == 1) { // 设置过期时间为当天结束 redis.expire(key, Duration.ofHours(24)); } ResourceQuota quota = getQuota(tenantId); return current <= quota.getLimit(type); } } public enum QuotaType { REQUESTS_PER_DAY, // 每日请求数 TOKENS_PER_MONTH, // 每月 Token 数 CONCURRENT_SESSIONS, // 并发会话数 TOOL_CALLS_PER_HOUR // 每小时工具调用数 }
    ⚠️ 限流策略要点

    限流不是简单地拒绝超限请求，而是分级响应：接近配额时返回警告 Header（X-RateLimit-Remaining），超过配额时返回 429 状态码并附带 Retry-After，严重超限时临时冻结租户并发送告警通知。同时要区分"硬限流"（不可突破）和"软限流"（可临时突破但会额外计费）。

    📄 核心内容（第3节）：Agent 编排引擎 — 可视化 + 代码混合编排

    编排范式

    企业级平台需要支持多种编排范式，满足不同角色的需求：

    范式使用者优势劣势
    声明式（YAML/JSON）非技术人员、运营直观、可版本管理、可视化编辑表达能力有限、调试困难
    命令式（代码）开发者灵活、可调试、可测试学习成本高、需要编程能力
    可视化（拖拽）产品经理、业务分析师零代码、所见即所得复杂逻辑表达困难、版本管理弱
    💡 最佳实践：混合编排

    成熟的平台通常支持三种范式的混合使用：用 YAML 定义 Agent 的整体流程骨架，用代码实现复杂逻辑节点，用可视化界面做流程预览和简单调整。三者共享同一套执行引擎。

    DSL 设计：Agent 工作流定义

    使用 YAML 定义 Agent 工作流，让非技术人员也能理解和编辑：

    # agent-workflow.yaml workflow: name: customer-service version: "2.1" steps: - id: classify type: llm_call model: gpt-4 prompt: "对用户问题进行分类：{input}" output: category - id: route type: condition switch: - case: category == "technical" goto: tech_support - case: category == "billing" goto: billing - default: goto: general - id: tech_support type: agent agent: tech-support-agent tools: [knowledge_base, ticket_system] - id: billing type: agent agent: billing-agent tools: [payment_api, invoice_service] - id: general type: agent agent: general-support-agent tools: [faq_base] - id: response type: output format: | {"answer": "{{step.answer}}", "category": "{{category}}"}

    这段 DSL 定义了一个客服工作流：先用 LLM 对用户问题分类，然后根据分类结果路由到不同的专业 Agent，最后格式化输出。整个流程清晰、可维护、可版本管理。

    运行时引擎

    @Service public class WorkflowEngine { private final Map<String, WorkflowDefinition> definitions; private final LlmClient llmClient; private final ToolRegistry toolRegistry; public WorkflowResult execute(String workflowId, Map<String, Object> input) { WorkflowDefinition wf = definitions.get(workflowId); ExecutionContext ctx = new ExecutionContext(input); for (WorkflowStep step : wf.getSteps()) { switch (step.getType()) { case LLM_CALL -> { String result = llmClient.call( step.getModel(), step.getPrompt(ctx) ); ctx.setVariable(step.getOutput(), result); } case CONDITION -> { String nextStep = evaluateCondition(step, ctx); ctx.setCurrentStep(nextStep); } case AGENT -> { AgentExecutor agent = agentRegistry.getAgent(step.getAgent()); ToolCallback[] tools = resolveTools(step.getTools()); String result = agent.execute(ctx, tools); ctx.setVariable("answer", result); } case TOOL_CALL -> { ToolCallback tool = toolRegistry.getTool(step.getTool()); String result = tool.call(step.getParams(ctx)); ctx.setVariable(step.getOutput(), result); } } } return WorkflowResult.from(ctx); } }

    版本管理与回滚

    工作流的版本管理是平台可靠性的关键保障：

    • 语义化版本 — 遵循 SemVer 规范（major.minor.patch）。DSL 结构变更是 major，新增步骤是 minor，提示词调整是 patch
    • 灰度发布 — 新版本先分配 5% 流量验证，确认无异常后逐步放量到 100%。支持 A/B 测试对比新旧版本效果
    • 一键回滚 — 任何版本都保留完整快照（DSL + 配置 + 依赖版本），回滚操作在秒级完成
    • 审计追踪 — 每次版本变更记录：谁修改了什么、什么时候修改的、修改的原因

    📄 核心内容（第4节）：权限与审计 — 企业合规的底线

    RBAC 在 Agent 平台的应用

    基于角色的访问控制（RBAC）是 Agent 平台权限管理的基础。典型的 Agent 平台角色设计：

    角色权限范围典型用户
    Platform Admin全局管理：租户管理、基础设施配置、全局策略平台运维团队
    Tenant Admin租户内管理：Agent CRUD、团队成员管理、配额查看部门负责人
    Agent DeveloperAgent 开发：创建/编辑 Agent、配置 Prompt 和 Tool开发工程师
    Agent User使用 Agent：与 Agent 对话、查看历史记录终端用户
    Auditor只读审计：查看所有操作日志和使用统计合规审计人员

    ABAC：细粒度属性控制

    RBAC 提供角色级控制，但企业场景往往需要更细粒度的基于属性的访问控制（ABAC）。ABAC 的决策基于四个维度：

    • 主体属性 — 用户的角色、部门、安全等级、地理位置
    • 资源属性 — Agent 的类型、数据敏感度、所属租户
    • 操作属性 — 读取、修改、执行、导出
    • 环境属性 — 时间、IP 地址、设备类型、网络环境
    @Service public class AgentAccessControl { public boolean checkAccess(AccessContext ctx) { // 角色检查（RBAC） if (!hasRequiredRole(ctx.getUser(), ctx.getRequiredRole())) { return false; } // 属性检查（ABAC） // 1. 数据敏感度匹配 if (ctx.getAgent().getDataClassification() == Classification.CONFIDENTIAL && ctx.getUser().getSecurityLevel() < SecurityLevel.L3) { return false; } // 2. 时间窗口限制 if (ctx.getAgent().hasTimeRestriction() && !isWithinAllowedHours(ctx.getAgent())) { return false; } // 3. 地理位置限制 if (ctx.getAgent().isGeoRestricted() && !isAllowedRegion(ctx.getIp())) { return false; } return true; } }

    操作审计

    审计是企业合规的刚性要求。每一次 Agent 交互都必须完整记录——谁在什么时候用 Agent 做了什么、输入了什么、输出了什么：

    @Aspect @Component public class AgentAuditAspect { private final AuditLogRepository auditRepo; @Around("@annotation(auditable)") public Object audit(ProceedingJoinPoint joinPoint, Auditable auditable) throws Throwable { AuditLog log = AuditLog.builder() .userId(SecurityContextHolder.getCurrentUserId()) .tenantId(SecurityContextHolder.getCurrentTenantId()) .action(auditable.action()) .resource(joinPoint.getSignature().toShortString()) .timestamp(Instant.now()) .build(); try { Object result = joinPoint.proceed(); log.setStatus("SUCCESS"); log.setResponseSummary(summarize(result)); return result; } catch (Exception e) { log.setStatus("FAILURE"); log.setError(e.getMessage()); throw e; } finally { auditRepo.save(log); // 异步发送到审计日志系统 auditEventPublisher.publish(log); } } }

    数据脱敏

    Agent 在处理用户请求时可能接触到敏感信息（身份证号、银行卡号、手机号等）。在数据进入 LLM 之前必须进行脱敏处理：

    @Service public class DataMaskingService { // 敏感信息正则 private static final Map<Pattern, String> MASKING_RULES = Map.of( Pattern.compile("\\d{18}[0-9X]"), "身份证号", Pattern.compile("\\d{16,19}"), "银行卡号", Pattern.compile("1[3-9]\\d{9}"), "手机号" ); public String maskSensitiveData(String text) { String masked = text; for (Map.Entry<Pattern, String> rule : MASKING_RULES.entrySet()) { Matcher matcher = rule.getKey().matcher(masked); masked = matcher.replaceAll(m -> { String original = m.group(); return original.substring(0, 3) + "*".repeat(original.length() - 6) + original.substring(original.length() - 3); }); } return masked; } // 还原脱敏数据（在 LLM 返回后） public String unmaskSensitiveData(String maskedText, MaskingContext ctx) { for (MaskedEntry entry : ctx.getEntries()) { maskedText = maskedText.replace(entry.getMasked(), entry.getOriginal()); } return maskedText; } }
    ⚠️ 脱敏策略要点

    脱敏不是简单地删除敏感信息，而是替换为占位符并在 LLM 返回后还原。例如，用户输入"我的身份证号是 110101199001011234"，脱敏后发送给 LLM 的是"我的身份证号是 110***1234"，LLM 返回后再还原为完整的身份证号。这样既保护了隐私，又不影响 Agent 的理解能力。

    合规框架

    GDPR（欧盟通用数据保护条例）

    Agent 平台必须支持：数据主体访问权（用户可导出自己的对话记录）、被遗忘权（用户可要求删除所有对话数据）、数据可携带权（数据格式标准化）、数据处理同意（明确告知并获取用户同意）。

    数据安全法（中国）

    Agent 平台需要：数据分类分级（按敏感度分为一般/重要/核心三级）、数据出境评估（跨境调用 LLM 需要评估）、日志留存（操作日志至少保存 6 个月）、安全审计（定期进行安全评估）。

    📄 核心内容（第5节）：Agent 市场与 SDK — 生态化扩张

    Agent 市场设计

    Agent 市场是平台生态的核心——它让 Agent 的创建者和使用者高效连接。核心功能模块：

    • 注册与发布 — 开发者提交 Agent 定义（Prompt + Tool + 配置），平台自动进行安全扫描、质量评估和兼容性测试后发布
    • 发现与搜索 — 按领域（客服、销售、法务）、能力（文本处理、数据分析、图像理解）、评分（用户评价、调用成功率）多维度检索
    • 评分与评价 — 使用效果评分（回答质量、响应速度）+ 人工评价（易用性、稳定性），形成 Agent 的信誉体系
    • 版本管理 — Agent 发布后可以迭代更新，用户可以选择锁定特定版本或自动升级到最新稳定版
    • 计费与结算 — 按调用次数、Token 消耗或订阅制收费，支持 Agent 开发者的收入分成

    📖 章节小结

    • 平台化是 Agent 规模化的必经之路 — 从单体 Agent 到 Agent-as-a-Service，平台化解决了标准化、可复用、可治理三大核心挑战
    • 五层架构是行业最佳实践 — 接入层、编排层、能力层、数据层、基础设施层，各层职责明确、松耦合、可独立扩展
    • 多租户隔离是平台的基石 — 不仅要隔离数据，还要隔离 Prompt、Tool、Memory 和 Model 访问权限。配额管理和限流是成本控制的关键
    • 编排引擎让 Agent 流程可视化 — DSL（YAML）+ 代码 + 可视化的混合编排模式，满足不同角色的需求。版本管理和灰度发布保障可靠性
    • 权限审计是企业合规的底线 — RBAC + ABAC 组合实现细粒度控制，操作审计记录所有行为，数据脱敏保护用户隐私
    • Agent 市场 + SDK 构建生态 — 市场让 Agent 创建者和使用者高效连接，好的 SDK 让开发者 5 分钟接入平台
    • 渐进式建设是最务实的策略 — 从 MVP 开始，按阶段迭代。平台是长出来的，不是画出来的

    👨‍💻 面试高频题

    👨‍💻 Q1: 企业级 Agent 平台和普通的 Agent 应用有什么本质区别？

    答：本质区别在于规模化的治理能力。普通 Agent 应用是"能跑就行"，关注的是单个 Agent 的功能实现。企业级平台需要解决的是"如何让 1000 个 Agent 在同一套系统中安全、高效、可控地运行"。核心差异体现在四个维度：(1) 多租户隔离——不同部门的 Agent 数据、配置、资源完全隔离；(2) 统一治理——权限控制、审计日志、成本核算、质量监控；(3) 标准化接入——SDK、API 规范、开发流程让不同团队高效协作；(4) 生态化扩展——Agent 市场、插件机制、第三方集成。平台不是简单的 Agent 集合，而是一个"Agent 操作系统"。

    👨‍💻 Q2: 多租户场景下，如何实现 Agent 的数据隔离？有哪些方案？

    答：数据隔离需要从四个层面考虑：
    (1) 数据存储层——三种模型可选：共享数据库（逻辑隔离，用 tenant_id 字段过滤）、独立 Schema（Schema 级隔离，适合中等安全要求）、独立数据库（物理隔离，适合金融/医疗等高安全场景）；
    (2) Agent 配置层——Prompt 模板、Tool 注册、System Prompt 等按租户隔离存储，通过 namespace 或 metadata 区分；
    (3) 运行时层——Agent 执行时注入租户上下文（TenantContext），Vector Store 通过 metadata filter 隔离，Memory Store 按 tenant_id 分区；
    (4) 模型访问层——不同租户可以路由到不同的模型实例（共享池/专用实例/私有部署），API Key 按租户独立管理。

    👨‍💻 Q3: Agent 编排引擎的设计有哪些关键考量？DSL 和代码编排如何融合？

    答：编排引擎设计的关键考量：
    (1) 执行语义——需要明确定义每种步骤类型（LLM Call、Condition、Agent、Tool Call）的输入输出契约；
    (2) 错误处理——工作流中的某一步失败时，需要支持重试、跳过、回退等策略；
    (3) 状态管理——长时间运行的工作流需要持久化中间状态，支持断点续传；
    (4) 并发控制——并行分支的执行和结果合并。

    DSL 和代码融合的最佳实践：用 YAML 定义流程骨架（步骤顺序、条件分支、Agent 路由），用代码实现复杂节点（数据转换、外部 API 调用、自定义逻辑）。两者共享同一套 ExecutionContext，DSL 中可以引用代码实现的自定义 StepHandler。这样非技术人员可以编辑流程骨架，开发者可以实现复杂逻辑。

    👨‍💻 Q4: 在 Agent 平台中，如何设计有效的审计系统？

    答：Agent 平台的审计系统需要记录比传统系统更丰富的信息：
    (1) 调用审计——谁、什么时候、调用了哪个 Agent、输入了什么、输出了什么、消耗了多少 Token；
    (2) 工具审计——Agent 调用了哪些 Tool、Tool 的输入输出、Tool 的执行耗时；
    (3) 推理审计——Agent 的思考过程（Chain of Thought）、决策路径、中间步骤；
    (4) 安全审计——权限变更、敏感数据访问、异常行为检测。

    技术实现：使用 AOP（切面）拦截 Agent 调用，异步写入审计日志（避免影响响应延迟）。审计日志存储在独立的存储系统中（如 Elasticsearch），支持全文检索和聚合分析。关键设计：审计日志不可篡改（追加写入、签名验证），保留足够长（至少 6 个月，金融行业要求 5 年），可导出（满足监管要求）。

    👨‍💻 Q5: 如果要从零开始建设企业级 Agent 平台，你会如何规划技术选型和建设路线？

    答：技术选型和建设路线规划：
    技术选型：(1) 核心框架——Spring AI（Java 生态）或 LangChain（Python 生态）；(2) 数据存储——PostgreSQL（关系数据）+ Redis（缓存/限流）+ Milvus/Qdrant（向量存储）；(3) 消息队列——Kafka（异步调用/审计日志）；(4) 容器编排——Kubernetes（弹性扩缩）；(5) 监控——Prometheus + Grafana + OpenTelemetry。

    建设路线：(1) 第一阶段（MVP）——Agent CRUD + 单租户 + 基础调用，验证核心场景；(2) 第二阶段（多租户）——隔离 + RBAC + 审计 + 限流，支撑多部门；(3) 第三阶段（生态）——DSL 编排 + Agent 市场 + SDK + 文档，构建开发者生态；(4) 第四阶段（智能化）——自动扩缩 + A/B 测试 + 智能路由 + 成本优化。每个阶段 2-3 个月，渐进式迭代。
