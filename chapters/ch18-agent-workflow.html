第 13 章

Agent 工作流编排

从简单线性流程到复杂 DAG 工作流，学会用 LangGraph 编排多 Agent 协作，构建生产级 AI 应用。

⏱ 预计 3-4 周 📖 7 个知识点 💻 高级难度

📚 开篇故事：从流水线到智能工厂

小明的第一个 AI 应用是一个简单的聊天机器人 — 用户问一句，AI 答一句。就像一个单工位的手工作坊，一次只能处理一个任务。

后来需求升级了：用户说"帮我写一篇关于 Spring AI 的博客"，这需要先调研、再写大纲、然后逐段写作、最后校对修改。一个 Agent 搞不定，需要多个 Agent 协作。小明尝试用简单的函数调用链来串联它们，但很快就陷入了"意大利面条式代码"的泥潭 — 线程同步、错误处理、状态管理，每个都是噩梦。

直到他发现了 LangGraph。这个框架就像给 AI 工厂装上了智能调度系统 — 你只需要定义"哪些工位做什么"、"工位之间怎么传递半成品"、"出了问题怎么处理"，LangGraph 就能自动编排整个流程。从手工作坊到智能工厂，这就是工作流编排的力量。

🔧 工作流基础概念

概念比喻：Agent 的"流水线工厂"

工作流就像一条工厂流水线：

  • 节点（Node） = 工位 — 每个工位完成特定任务
  • 边（Edge） = 传送带 — 将半成品从一个工位传到下一个
  • 状态（State） = 流水线上的产品 — 在工位之间传递的数据
  • 条件分支 = 质检站 — 根据检查结果决定下一步去哪个工位

🧩 LangGraph 入门

概念比喻：Agent 的"乐高积木"

LangGraph 的核心思想是状态图（State Graph）：你定义状态（数据在节点间传递的内容），定义节点（每个步骤做什么），定义边（节点之间怎么连接）。框架自动处理执行调度、状态同步、错误恢复。

// 使用 Spring AI + LangGraph 概念构建工作流 // 注意：Spring AI 目前通过 Advisor 机制支持类似功能 // 1. 定义工作流状态 public class WorkflowState { private String userInput; private String analyzedIntent; private String processedData; private String finalOutput; private List<String> errors = new ArrayList<>(); private int retryCount = 0; // getters/setters... } // 2. 定义节点（每个节点是一个处理步骤） @Service public class OrderWorkflowNodes { private final ChatClient chatClient; // 节点 1：意图分析 public WorkflowState analyzeIntent(WorkflowState state) { String intent = chatClient.prompt() .system("分析用户意图，返回分类：查询/下单/投诉/其他") .user(state.getUserInput()) .call().content(); state.setAnalyzedIntent(intent.trim()); return state; } // 节点 2：订单处理 public WorkflowState processOrder(WorkflowState state) { String result = chatClient.prompt() .system("根据用户意图处理订单") .user(state.getAnalyzedIntent() + ": " + state.getUserInput()) .call().content(); state.setProcessedData(result); return state; } // 节点 3：生成回复 public WorkflowState generateResponse(WorkflowState state) { String response = chatClient.prompt() .system("用友好的语气回复用户") .user(state.getProcessedData()) .call().content(); state.setFinalOutput(response); return state; } } // 3. 编排工作流 @Service public class OrderWorkflow { private final OrderWorkflowNodes nodes; public String execute(String userInput) { WorkflowState state = new WorkflowState(); state.setUserInput(userInput); // 线性工作流：analyzeIntent -> processOrder -> generateResponse state = nodes.analyzeIntent(state); state = nodes.processOrder(state); state = nodes.generateResponse(state); return state.getFinalOutput(); } }
💡 小窍门

Spring AI 的 Advisor 机制（如 RequestResponseAdvisor）可以实现类似工作流的拦截和增强。对于更复杂的 DAG 工作流，可以考虑集成 LangGraph4j（Java 版 LangGraph）。

🔌 条件路由

概念比喻：Agent 的"智能分拣"

条件路由就像快递分拣中心 — 根据包裹的目的地（意图），自动送到不同的处理线路。简单查询走快速通道，复杂问题走深度处理通道。

// 条件路由实现 @Service public class SmartRouter { private final ChatClient chatClient; public String route(String input) { // 1. 意图分类 String intent = classifyIntent(input); // 2. 根据意图路由到不同处理链 return switch (intent) { case "simple_query" -> handleSimpleQuery(input); case "complex_task" -> handleComplexTask(input); case "complaint" -> handleComplaint(input); default -> handleDefault(input); }; } private String classifyIntent(String input) { return chatClient.prompt() .system(""" 将用户输入分类为以下之一： - simple_query: 简单查询（天气、定义、翻译） - complex_task: 复杂任务（写文章、分析数据） - complaint: 投诉或问题反馈 只返回分类名称。 """) .user(input) .call().content().trim(); } private String handleComplexTask(String input) { // 复杂任务：先规划再执行 String plan = createPlan(input); String result = executePlan(plan); return reviewAndRefine(result); } }

⚙ 并行执行

概念比喻：Agent 的"多线程工厂"

有些任务之间没有依赖关系，可以同时执行。就像工厂里同时在多个工位上组装不同的零件，最后再组装到一起。并行执行能大幅减少总延迟。

// 并行执行多个独立的 Agent 任务 @Service public class ParallelWorkflow { private final ChatClient chatClient; private final ExecutorService executor = Executors.newFixedThreadPool(4); public String analyzeDocument(String document) { // 三个独立的分析任务，并行执行 CompletableFuture<String> summaryFuture = CompletableFuture.supplyAsync(() -> generateSummary(document), executor); CompletableFuture<String> keywordsFuture = CompletableFuture.supplyAsync(() -> extractKeywords(document), executor); CompletableFuture<String> sentimentFuture = CompletableFuture.supplyAsync(() -> analyzeSentiment(document), executor); // 等待所有任务完成 CompletableFuture.allOf( summaryFuture, keywordsFuture, sentimentFuture ).join(); // 汇总结果 String summary = summaryFuture.join(); String keywords = keywordsFuture.join(); String sentiment = sentimentFuture.join(); return chatClient.prompt() .system("将以下分析结果整合为一份报告") .user("摘要: " + summary + "\n关键词: " + keywords + "\n情感: " + sentiment) .call().content(); } private String generateSummary(String doc) { return chatClient.prompt() .system("生成 100 字以内的文档摘要") .user(doc).call().content(); } private String extractKeywords(String doc) { return chatClient.prompt() .system("提取 5 个核心关键词") .user(doc).call().content(); } private String analyzeSentiment(String doc) { return chatClient.prompt() .system("分析文档的情感倾向：积极/中性/消极") .user(doc).call().content(); } }

⚠️ 陷阱警告

并行调用 LLM API 会同时消耗多个请求配额。如果 API 有并发限制（rate limit），需要控制并行度。建议使用信号量（Semaphore）限制同时执行的请求数。

🧑 人机协作节点

概念比喻：Agent 的"人工审核关卡"

有些决策太重要了，不能完全交给 AI。就像银行的大额转账需要人工审批一样，工作流中的关键节点可以设置"人工审核关卡"，让人类参与决策。

// Human-in-the-Loop 工作流 @Service public class HumanInLoopWorkflow { private final ChatClient chatClient; private final ApprovalService approvalService; public String processWithApproval(String task) { // 1. Agent 生成草案 String draft = chatClient.prompt() .system("根据任务生成方案") .user(task).call().content(); // 2. 发送人工审核 ApprovalRequest request = new ApprovalRequest( UUID.randomUUID().toString(), task, draft, Instant.now().plus(Duration.ofHours(24)) ); approvalService.requestApproval(request); // 3. 等待审核结果（异步回调或轮询） ApprovalResult result = approvalService .waitForApproval(request.id()); // 4. 根据审核结果处理 return switch (result.status()) { case APPROVED -> result.feedback().isEmpty() ? draft : reviseWithFeedback(draft, result.feedback()); case REJECTED -> handleRejection(task, result.reason()); case TIMEOUT -> handleTimeout(task); }; } private String reviseWithFeedback(String draft, String feedback) { return chatClient.prompt() .system("根据审核反馈修改方案") .user("原方案: " + draft + "\n反馈: " + feedback) .call().content(); } } public record ApprovalRequest( String id, String task, String draft, Instant deadline ) {} public record ApprovalResult( ApprovalStatus status, String feedback, String reason ) {}
💡 你知道吗？

Human-in-the-Loop 不是 AI 能力不足的表现，而是负责任 AI 的最佳实践。即使是 OpenAI 的 o1 模型，在涉及医疗、法律、金融等高风险领域时，也建议设置人工审核节点。

🛡️ 错误处理与重试

概念比喻：Agent 的"容错机制"

流水线上某个工位出了故障，好的工厂不会整条线停摆，而是自动重试、降级处理、或切换到备用方案。Agent 工作流也需要同样的容错能力。

// 工作流错误处理与重试 @Service public class ResilientWorkflow { private final ChatClient chatClient; private final int maxRetries = 3; public String execute(String task) { String lastError = ""; for (int attempt = 1; attempt <= maxRetries; attempt++) { try { // 尝试执行 return doExecute(task); } catch (RateLimitException e) { // 限流：指数退避 long waitTime = (long) Math.pow(2, attempt) * 1000; Thread.sleep(waitTime); lastError = "限流，等待 " + waitTime + "ms"; } catch (ToolException e) { // 工具错误：尝试备用工具 lastError = e.getMessage(); if (attempt == maxRetries) { return fallbackExecution(task); } } catch (Exception e) { lastError = e.getMessage(); if (attempt == maxRetries) { return gracefulDegradation(task, lastError); } } } return gracefulDegradation(task, lastError); } // 降级方案：用更简单的模型 private String gracefulDegradation(String task, String error) { return "抱歉，系统暂时无法完成该任务。" + "错误: " + error + "\n建议：请稍后重试或联系人工客服。"; } // 备用方案：使用缓存的历史结果 private String fallbackExecution(String task) { return "使用缓存的历史结果回答..."; } }

核心代码实现

// 电商订单工作流 @Service public class OrderWorkflow { private final ChatClient chatClient; private final OrderService orderService; private final InventoryService inventoryService; private final NotificationService notificationService; private final ApprovalService approvalService; public String processOrder(String userInput, String userId) { // 1. 意图识别 String intent = classifyIntent(userInput); // 2. 条件路由 return switch (intent) { case "query" -> handleQuery(userInput, userId); case "order" -> handleNewOrder(userInput, userId); case "return" -> handleReturn(userInput, userId); case "complaint" -> handleComplaint(userInput, userId); default -> handleUnknown(userInput); }; } private String handleNewOrder(String input, String userId) { // 并行：查库存 + 风控检查 CompletableFuture<InventoryResult> inventoryFuture = CompletableFuture.supplyAsync(() -> inventoryService.check(input)); CompletableFuture<RiskResult> riskFuture = CompletableFuture.supplyAsync(() -> orderService.checkRisk(userId)); CompletableFuture.allOf(inventoryFuture, riskFuture).join(); InventoryResult inventory = inventoryFuture.join(); RiskResult risk = riskFuture.join(); // 检查结果 if (!inventory.available()) { return "抱歉，" + inventory.itemName() + " 已售罄。"; } if (risk.blocked()) { return "订单需要人工审核，我们会在 24 小时内联系您。"; } // 创建订单 Order order = orderService.create(input, userId); // 并行：通知用户 + 更新库存 CompletableFuture.runAsync(() -> notificationService.notifyOrderCreated(order)); CompletableFuture.runAsync(() -> inventoryService.decrement(input)); return "订单创建成功！订单号: " + order.id(); } }
💡 小窍门

生产级工作流的关键要素：1. 状态持久化（流程中断后可恢复）；2. 超时控制（避免无限等待）；3. 审计日志（每步操作可追溯）；4. 熔断器（防止级联故障）。

📖 章节小结

Agent 工作流编排是从"简单 Agent"到"生产系统"的关键一步：

  • 工作流基础 — 节点、边、状态、条件分支，构建 DAG 执行图
  • LangGraph — 状态图框架，自动处理调度和状态同步
  • 条件路由 — 根据意图动态选择处理路径
  • 并行执行 — 独立任务同时执行，降低总延迟
  • 人机协作 — 关键决策节点引入人工审核
  • 错误处理 — 重试、降级、熔断，保证系统韧性
  • 生产级工作流 — 状态持久化、超时控制、审计日志

记住：好的工作流不是最复杂的，而是最可靠的。先保证正确性，再优化性能。

🎓 面试高频题

👨‍💻 Q1: LangGraph 和普通函数调用链有什么区别？

函数调用链是线性的、硬编码的，流程改变需要改代码。LangGraph 是图结构的、可配置的，流程通过节点和边定义，支持条件分支、并行、循环。LangGraph 还内置了状态管理、检查点、可视化等能力，适合复杂的工作流编排。

👨‍💻 Q2: 如何处理工作流中某个节点长时间无响应？

1. 设置超时：每个节点设置最大执行时间，超时自动中断。2. 心跳检测：节点定期发送心跳，超时未收到则判定为卡死。3. 熔断器：连续失败多次后自动跳过该节点，使用降级方案。4. 死信队列：将卡死的任务放入队列，后续人工处理。

👨‍💻 Q3: 人机协作节点如何设计才能不影响用户体验？

1. 异步处理：不要让用户等待审核，先返回"处理中"，审核完成后通知。2. 智能预判：只有高风险操作才触发人工审核，低风险自动通过。3. 超时兜底：设置审核超时（如 24 小时），超时后自动降级处理。4. 批量审核：将多个待审核项聚合，减少审核人员的工作量。
