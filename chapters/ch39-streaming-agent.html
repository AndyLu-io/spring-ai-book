第 39 章

实时流式与事件驱动 Agent

从"等待回答"到"实时对话" — 掌握 SSE、WebSocket、事件驱动架构，构建低延迟、高交互的实时 Agent 体验。

⏱ 预计 2-3 周 📖 5 个知识点 💻 高级难度

📚 开篇故事：从寄信到视频通话

你开发了一个智能客服 Agent，用户提出问题后，Agent 开始思考、调用工具、组织语言，最后一次性返回一个完整的回答。整个过程可能需要 10 秒、30 秒，甚至几分钟。

用户盯着空白的屏幕，心里想："它是在思考还是挂了？要不要刷新页面？"这种体验就像寄信——你把问题投进邮箱，然后焦急地等待回信。你不知道信在路上走了多远，不知道对方有没有在处理，你只能等。

但如果 Agent 能一边思考一边输出呢？就像你看到对方正在打字——"正在检索知识库..."、"找到 3 条相关信息..."、"正在组织回答..."、"根据以上分析..."——你能实时看到 Agent 的"思维过程"。这不是等待，而是视频通话：你看到对方的表情变化、点头思考、开口回答，整个过程是连续的、可感知的。

更进一步，如果 Agent 不只是回答你的问题，还能主动推送信息呢？比如你的监控 Agent 发现服务器 CPU 飙升，不需要你问，它会主动通知你："检测到异常，CPU 使用率达到 92%，建议立即扩容。"这就像多人会议室——多个参与者（Agent、用户、外部系统）在同一个空间里自由交流，每个人都能主动发言，而不是排队等别人说完。

这三种交互模式——"寄信"（同步请求-响应）、"视频通话"（流式输出）、"会议室"（事件驱动）——代表了 Agent 交互的进化方向。本章将带你从传统的"等待回答"模式，走向实时流式和事件驱动的 Agent 架构，让 Agent 真正"活"起来。

📄 核心内容（第1节）：流式输出基础 — SSE 与 WebSocket

三种通信模式对比

在构建流式 Agent 之前，先理解三种基本的 HTTP 通信模式：

通信模式原理延迟适用场景
HTTP 短连接客户端发请求 → 服务端处理 → 返回完整响应 → 断开高（等待完整响应）表单提交、文件下载
HTTP 长连接客户端发请求 → 服务端长时间处理 → 返回结果中（但无中间反馈）报表生成、批量处理
流式传输客户端发请求 → 服务端持续推送数据块 → 完成后断开低（首字节即呈现）LLM 输出、实时搜索
💡 关键指标：TTFT（Time To First Token）

流式输出的核心价值在于降低 TTFT。传统同步模式下，用户需要等待完整的响应生成（可能 10-30 秒），而流式模式下，第一个 token 通常在 500ms 内就开始到达。感知延迟大幅降低，用户体验从"等待"变为"观看"。

SSE（Server-Sent Events）：单向流式

SSE 是 HTTP 协议内建的单向推送机制，服务端通过一个持久连接持续向客户端发送数据。它是 LLM 流式输出的最佳选择——因为 LLM 的输出本质上是单向的（服务端生成，客户端展示），不需要客户端中途发送数据。

客户端                  服务端
  │                      │
  │── HTTP Request ─────→│
  │                      │── 调用 LLM
  │←── SSE: token 1 ────│
  │←── SSE: token 2 ────│
  │←── SSE: token 3 ────│
  │←── SSE: [DONE] ─────│
  │                      │

SSE 的优势

  • 基于 HTTP — 无需额外协议，现有基础设施（Nginx、CDN、防火墙）天然支持
  • 自动重连 — 浏览器 EventSource API 内建断线重连机制
  • 简单易用 — 服务端只需设置 Content-Type: text/event-stream，客户端只需 new EventSource()
  • 无状态 — 每个 SSE 连接就是一个普通 HTTP 请求，易于负载均衡

SSE 的数据格式

// SSE 事件格式（每行以 "data:" 开头，事件以空行分隔） data: {"token": "根据", "index": 0} data: {"token": "以上", "index": 1} data: {"token": "分析", "index": 2} data: [DONE] // 支持的字段：data（数据）、event（事件类型）、id（事件ID）、retry（重连间隔） // 示例：带事件类型 event: tool_call data: {"tool": "search", "args": {"query": "天气"}} event: message data: {"token": "北京今天晴"}

WebSocket：双向通信

WebSocket 通过一次 HTTP 握手升级为全双工通信通道，适合需要客户端和服务端双向实时交互的场景。

WebSocket 适用场景

  • 多轮对话 Agent — 用户可以在 Agent 输出过程中追加指令、中断、修改方向
  • 协作编辑 Agent — 多个用户和 Agent 共同编辑同一份文档
  • 实时游戏/模拟 — Agent 作为游戏NPC，需要即时响应玩家操作
  • 投票/确认 Agent — Agent 在执行关键操作前需要实时征求用户确认

WebSocket 的代价

  • 有状态连接 — 服务端需要维护连接状态，水平扩展需要粘性会话或消息广播
  • 基础设施要求高 — 需要支持 WebSocket 的反向代理、负载均衡器
  • 断线处理复杂 — 需要手动实现心跳、重连、状态恢复
  • 安全性考量 — 长连接更容易被滥用，需要额外的认证和限流机制

协议选型决策

在 Agent 系统中，选择 SSE 还是 WebSocket 取决于具体的交互需求：

决策因素选择 SSE选择 WebSocket
数据流向单向（服务端 → 客户端）双向（客户端 ↔ 服务端）
交互需求只需展示 Agent 输出需要中途干预/追加输入
基础设施标准 HTTP 基础设施即可需要 WebSocket 支持的代理
并发规模大量并发连接（单向轻量）中等并发（有状态连接）
复杂度低（浏览器原生支持）中高（需处理心跳、重连）
💡 实战建议

对于大多数 LLM Agent 场景，SSE 是首选。理由：LLM 输出是单向的、基础设施更简单、并发扩展更容易。只有当你确实需要客户端在 Agent 输出过程中实时追加信息时，才需要 WebSocket。很多"需要 WebSocket"的场景，其实可以通过"用户先提交输入 → Agent 流式输出 → 用户再次提交新输入"的模式用 SSE 实现。

SSE 流式输出实战：Controller 层

@RestController @RequestMapping("/api/chat") public class StreamingChatController { private final ChatClient chatClient; @GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE) public Flux<ServerSentEvent<String>> streamChat( @RequestParam String prompt, @RequestParam(defaultValue = "gpt-4") String model) { return chatClient.prompt() .user(prompt) .model(model) .stream() .content() .map(token -> ServerSentEvent.<String>builder() .id(UUID.randomUUID().toString()) .event("message") .data(token) .build()) .concatWith(Flux.just( ServerSentEvent.<String>builder() .event("done") .data("[DONE]") .build() )); } }
💡 为什么用 concatWith 发送 DONE 信号？

客户端需要一个明确的信号来知道 Agent 已经输出完毕。SSE 没有原生的"结束"事件，所以我们用 concatWith 在正常 token 流结束后追加一个 [DONE] 事件。客户端收到这个事件后关闭连接、重新启用输入框、记录最终结果。这是 SSE 流式 API 的标准实践。

📄 核心内容（第2节）：Spring AI 流式 API 深度实战

ChatClient 流式调用详解

Spring AI 的 ChatClient 提供了多种流式 API，适配不同的使用场景：

  • .stream().content() — 返回 Flux<String>，每个元素是一个 token。最简单的流式调用
  • .stream().chatResponse() — 返回 Flux<ChatResponse>，包含完整的响应元数据（模型选择、finish reason 等），适合需要检查元信息的场景
  • .stream().entity() — 流式输出 + 自动反序列化为 Java 对象，适合结构化输出场景
💡 流式与同步的切换

Spring AI 的 ChatClient 同步和流式 API 使用相同的 Builder 模式构建，只需将 .call() 替换为 .stream() 即可切换。这使得在开发阶段用同步调试、生产环境切换流式变得非常简单。

流式 + Tool Calling 组合

当 Agent 同时使用流式输出和工具调用时，需要在 token 流中区分"文本输出"和"工具调用"两种事件。以下是完整的实现方案：

@Service public class StreamingAgentService { private final ChatClient chatClient; private final ToolCallbackRegistry toolRegistry; // 流式 Agent（带工具调用） public Flux<AgentEvent> streamWithTools(String prompt, String sessionId) { return chatClient.prompt() .system("你是一个智能助手，可以使用工具回答问题。") .user(prompt) .tools(toolRegistry.getTools(sessionId)) .stream() .chatResponse() .map(response -> { // 检查是否有工具调用 if (response.getResult().getOutput().hasToolCalls()) { return AgentEvent.toolCall( response.getResult().getOutput().getToolCalls() ); } // 普通文本输出 return AgentEvent.text( response.getResult().getOutput().getText() ); }) .onErrorResume(e -> Flux.just(AgentEvent.error(e.getMessage()))); } }

AgentEvent 类定义

// 统一的 Agent 事件类型 public record AgentEvent( EventType type, // TEXT, TOOL_CALL, TOOL_RESULT, ERROR, DONE String sessionId, String content, // 文本内容或工具调用 JSON Instant timestamp ) { public enum EventType { TEXT, TOOL_CALL, TOOL_RESULT, ERROR, DONE } public static AgentEvent text(String content, String sessionId) { return new AgentEvent(EventType.TEXT, sessionId, content, Instant.now()); } public static AgentEvent toolCall(List<ToolCall> calls, String sessionId) { return new AgentEvent(EventType.TOOL_CALL, sessionId, JsonUtils.toJson(calls), Instant.now()); } public static AgentEvent error(String message, String sessionId) { return new AgentEvent(EventType.ERROR, sessionId, message, Instant.now()); } public static AgentEvent done(String sessionId) { return new AgentEvent(EventType.DONE, sessionId, "", Instant.now()); } }

前端 SSE 消费示例

前端使用浏览器原生 EventSource API 或 fetch + ReadableStream 消费 SSE：

// 方式一：EventSource API（推荐，自动重连） const eventSource = new EventSource( `/api/chat/stream?prompt=` + encodeURIComponent(userInput) ); eventSource.addEventListener('message', (e) => { const data = JSON.parse(e.data); if (data.type === 'TEXT') { appendToChat(data.content); // 逐字显示 } else if (data.type === 'TOOL_CALL') { showToolProgress(data.content); // 显示工具调用进度 } else if (data.type === 'ERROR') { showError(data.content); } }); eventSource.addEventListener('done', () => { eventSource.close(); enableInput(); }); // 方式二：fetch + ReadableStream（更灵活） async function streamChat(prompt) { const response = await fetch(`/api/chat/stream?prompt=${prompt}`); const reader = response.body.getReader(); const decoder = new TextDecoder(); while (true) { const { done, value } = await reader.read(); if (done) break; const text = decoder.decode(value); // 解析 SSE 格式的文本，提取 data 字段 const lines = text.split('\n'); for (const line of lines) { if (line.startsWith('data: ')) { appendToChat(line.slice(6)); } } } }
💡 打字机效果优化

为了让 Agent 输出看起来更自然，可以实现"打字机效果"——逐字显示而非一次性插入。使用 requestAnimationFrame 控制渲染节奏，避免一次性 DOM 更新导致的卡顿。同时，使用虚拟滚动技术处理长对话，防止 DOM 节点过多导致的性能问题。

错误处理与重连机制

流式连接可能因为网络波动、服务端重启等原因中断。生产级流式 Agent 需要健壮的错误处理和重连机制：

@Service public class StreamingErrorHandler { // 服务端：连接断开时的清理和通知 @ControllerAdvice public static class StreamExceptionHandler { @ExceptionHandler(IOException.class) public ResponseEntity<String> handleDisconnect(IOException e) { // 记录断连日志，但不报警 log.info("SSE client disconnected: {}", e.getMessage()); return ResponseEntity.ok("Stream interrupted"); } @ExceptionHandler(LLMServiceException.class) public Flux<AgentEvent> handleLLMError(LLMServiceException e) { return Flux.just( AgentEvent.error(e.getMessage(), e.getSessionId()), AgentEvent.done(e.getSessionId()) ); } } }
⚠️ 生产环境注意事项

SSE 连接在生产环境中容易被中间层（Nginx、API Gateway、负载均衡器）超时断开。务必配置足够的超时时间（建议 300 秒以上），并在服务端实现心跳机制（每隔 15-30 秒发送一个空的 SSE 注释行 ":" 保活）。

📄 核心内容（第3节）：事件驱动架构 — 让 Agent 响应世界

事件驱动 vs 请求驱动

传统的 Agent 交互是请求驱动的：用户发请求 → Agent 处理 → 返回响应。整个流程由用户发起，Agent 被动响应。事件驱动则是 Agent 主动感知和响应外部变化：

维度请求驱动事件驱动
发起方用户主动发起请求系统/用户/定时器均可触发
响应方式同步等待或轮询异步推送通知
耦合度客户端和服务端紧密耦合发布者和订阅者松耦合
可扩展性受连接数限制天然支持多消费者
典型场景聊天对话、问答监控告警、定时巡检、数据同步

事件类型体系

一个完整的事件驱动 Agent 系统需要定义四类事件：

  • 用户事件（User Events） — 用户输入、反馈、确认、取消。这是最基本的事件类型，驱动 Agent 的对话交互
  • 系统事件（System Events） — 监控告警、系统状态变化、配置更新。Agent 作为"哨兵"响应系统异常
  • 定时事件（Scheduled Events） — 周期性巡检、报表生成、数据同步。Agent 按计划主动执行任务
  • 外部事件（External Events） — 第三方 Webhook、消息队列消息、文件变更。Agent 集成外部生态

Spring AI + Spring Cloud Stream 集成

Spring Cloud Stream 为事件驱动 Agent 提供了消息通道的抽象，让 Agent 可以通过 Kafka、RabbitMQ、Redis Stream 等不同消息中间件收发事件。

// 事件定义 public record AgentEvent( String eventType, // USER_INPUT, TOOL_RESULT, SYSTEM_ALERT, SCHEDULED String sessionId, String payload, Instant timestamp ) {} // 事件驱动 Agent @Service public class EventDrivenAgent { private final ChatClient chatClient; private final StreamBridge streamBridge; @StreamListener("agent-input") public void handleEvent(AgentEvent event) { String response = switch (event.eventType()) { case "USER_INPUT" -> handleUserInput(event); case "TOOL_RESULT" -> handleToolResult(event); case "SYSTEM_ALERT" -> handleSystemAlert(event); case "SCHEDULED" -> handleScheduled(event); default -> "Unknown event type"; }; // 发布响应事件 streamBridge.send("agent-output", MessageBuilder.withPayload( new AgentEvent("RESPONSE", event.sessionId(), response, Instant.now()) ).build()); } // 定时事件：Agent 主动巡检 @Scheduled(fixedRate = 300000) // 每5分钟 public void periodicCheck() { List<MonitorAlert> alerts = monitorService.checkAll(); for (MonitorAlert alert : alerts) { streamBridge.send("agent-input", new AgentEvent("SYSTEM_ALERT", "monitor", alert.toJson(), Instant.now())); } } }

事件总线设计

事件总线是事件驱动 Agent 的"神经中枢"，负责事件的路由、分发和持久化：

┌───────────────────────────────────────────────────────────────────┐
│                    事件总线（Event Bus）                    │
│  用户输入  系统告警  定时任务  Webhook  Agent输出  │
└───────────────────────────────────────────────────────────────────┘

  ↑  ↑       ↑       ↑      ↓
  Web  Mobile  Infra  Cron   Dashboard
  App  App     Agent  Job    Monitor

事件总线的三种实现方案

方案一：Spring Cloud Stream（推荐）

使用 Kafka 或 RabbitMQ 作为消息中间件。支持消息持久化、消费者组、分区、重试。适合生产环境。

方案二：Redis Pub/Sub

使用 Redis 的发布/订阅功能。延迟极低（微秒级），但不支持消息持久化和消费者组。适合对延迟敏感但允许丢消息的场景。

⚠️ 事件风暴防护

事件驱动系统容易出现"事件风暴"——一个事件触发多个消费者，每个消费者又产生新事件，导致事件数量指数级增长。务必设计事件抑制和去重机制：为每个事件分配唯一 ID，消费者记录已处理的事件 ID；设置事件优先级和限流；为"扇出"场景（一个事件触发多个 Agent）设置最大扇出数。

📄 核心内容（第4节）：长连接管理与生产实践

连接池管理与心跳保活

流式 Agent 在生产环境中面临的核心挑战是长连接的生命周期管理。连接可能因为各种原因中断，必须有完善的保活和重连机制。

@Configuration public class StreamConfig { @Bean public WebClient webClient() { return WebClient.builder() .clientConnector(new ReactorClientHttpConnector( HttpClient.create() .responseTimeout(Duration.ofSeconds(30)) .keepAlive(true) )) .build(); } @Bean public ConnectionPool connectionPool() { // 限制最大连接数，防止连接泄漏 return new ConnectionPool( 50, // 最大空闲连接 Duration.ofMinutes(5), // 空闲连接保活时间 Timeout.ofSeconds(30) // 连接超时 ); } }

客户端断线重连策略

客户端在 SSE 连接断开后，需要智能重连。简单的立即重连可能导致服务端在故障期间收到大量重连请求（重连风暴）。推荐使用指数退避 + 抖动策略：

@Service public class ResilientSseClient { private final WebClient webClient; private final RetryRegistry retryRegistry; public Flux<String> connectWithRetry(String url) { return webClient.get() .uri(url) .accept(MediaType.TEXT_EVENT_STREAM) .retrieve() .bodyToFlux(String.class) .retryWhen(Retry.backoff(3, Duration.ofSeconds(1)) .filter(ex -> ex instanceof IOException)) .timeout(Duration.ofMinutes(30)) .onErrorResume(e -> { log.error("SSE connection failed", e); return Flux.error(e); }); } }
💡 重连状态恢复

断线重连时，最重要的不是重新建立连接，而是恢复状态。SSE 协议支持通过 Last-Event-ID 头部告诉服务端"我上次收到的事件 ID 是什么"，服务端应该从该事件之后重新推送。在 Agent 场景中，这意味着用户重连后可以继续看到之前的对话流，而不是重新开始。

水平扩展：Redis Pub/Sub 跨实例消息

在多实例部署环境下，流式 Agent 需要跨实例的消息传递能力。当用户连接到实例 A，但 Agent 处理在实例 B 上完成时，需要通过消息通道将 token 推送回实例 A。

@Service public class DistributedStreamService { private final RedisTemplate<String, String> redisTemplate; private final Map<String, Sinks.Many<String>> activeStreams = new ConcurrentHashMap<>(); // 订阅 Redis 频道，将消息分发给对应的 SSE 连接 @PostConstruct public void init() { RedisMessageListenerContainer container = new RedisMessageListenerContainer(); container.addMessageListener((message, pattern) -> { String channel = new String(message.getChannel()); String body = new String(message.getBody()); Sinks.Many<String> sink = activeStreams.get(channel); if (sink != null) { sink.tryEmitNext(body); // 推送到本地 SSE 连接 } }, new PatternTopic("agent:stream:*")); } // 发布 token 到 Redis 频道 public void publishToken(String sessionId, String token) { redisTemplate.convertAndSend("agent:stream:" + sessionId, token); } }

性能优化：背压控制与批量发送

流式 Agent 的性能瓶颈通常不在于 LLM 生成速度，而在于 token 的网络传输效率。以下优化措施可以显著提升性能：

  • 背压控制（Backpressure） — 当客户端处理速度跟不上服务端生成速度时，使用 Reactor 的背压策略（如 onBackpressureBuffer）缓冲 token，避免内存溢出。设置合理的缓冲区大小（如 1024 个 token），超出后丢弃旧 token
  • 批量发送 — 将多个 token 合并为一个 SSE 事件发送，减少 HTTP 开销。例如每 5 个 token 或每 50ms（取先到者）合并一次发送，网络请求数减少 5 倍
  • Gzip 压缩 — 对 SSE 响应启用 Gzip 压缩，通常可以减少 60-80% 的传输体积。在 Spring Boot 中配置 server.compression.enabled=true
  • 连接复用 — 使用 HTTP/2 的多路复用能力，多个 SSE 连接共享同一个 TCP 连接，减少握手开销
// 背压控制示例 return chatClient.prompt() .user(prompt) .stream() .content() .onBackpressureBuffer(1024) // 缓冲最多 1024 个 token .bufferTimeout(5, Duration.ofMillis(50)) // 每 5 个 token 或 50ms 合并 .map(tokens -> String.join("", tokens)) // 合并为字符串 .map(batch -> ServerSentEvent.<String>builder() .event("message") .data(batch) .build());

完整服务端实现

将前面所有知识点整合，构建一个生产级的实时流式 Agent 服务：

@RestController @RequestMapping("/api/agent") public class StreamingAgentController { private final StreamingAgentService agentService; @GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE) public Flux<ServerSentEvent<String>> streamAgent( @RequestParam String prompt, @RequestParam(defaultValue = "gpt-4") String model, @RequestHeader(name = "X-Session-Id", required = false) String sessionId) { String sid = sessionId != null ? sessionId : UUID.randomUUID().toString(); return agentService.streamWithTools(prompt, sid) .map(event -> ServerSentEvent.<String>builder() .id(UUID.randomUUID().toString()) .event(event.type().name().toLowerCase()) .data(JsonUtils.toJson(event)) .build()) .concatWith(Flux.just( ServerSentEvent.<String>builder() .event("done") .data("[DONE]") .build() )) .doOnError(e -> log.error("Stream error for session {}", sid, e)) .doFinally(signal -> log.info("Stream finished: {} for session {}", signal, sid)); } }

客户端连接管理器

生产级客户端需要管理连接生命周期、重连策略、状态恢复：

// JavaScript 客户端连接管理器 class AgentStreamClient { constructor(baseUrl) { this.baseUrl = baseUrl; this.sessionId = crypto.randomUUID(); this.retryCount = 0; this.maxRetries = 5; this.lastEventId = null; } async connect(prompt, onMessage, onDone, onError) { const url = `${this.baseUrl}/api/agent/stream?prompt=` + encodeURIComponent(prompt); const headers = { 'X-Session-Id': this.sessionId }; if (this.lastEventId) { headers['Last-Event-ID'] = this.lastEventId; } try { const response = await fetch(url, { headers }); const reader = response.body.getReader(); const decoder = new TextDecoder(); this.retryCount = 0; while (true) { const { done, value } = await reader.read(); if (done) break; this.parseSSE(decoder.decode(value), onMessage, onDone); } } catch (e) { if (this.retryCount < this.maxRetries) { const delay = Math.min(1000 * 2 ** this.retryCount, 30000); this.retryCount++; setTimeout(() => this.connect(prompt, onMessage, onDone, onError), delay); } else { onError(e); } } } }

📖 章节小结

  • 流式输出是 Agent 体验升级的关键 — 从"等待完整响应"到"逐字实时输出"，TTFT 从秒级降低到毫秒级，用户体验从"等待"变为"观看"
  • SSE 是 LLM 流式输出的最佳协议 — 基于标准 HTTP、无需额外基础设施、浏览器原生支持。WebSocket 只在需要双向交互时才使用
  • Spring AI 的流式 API 简洁强大 — .stream().content() 返回 token 流，.stream().chatResponse() 返回完整响应元数据，与 Tool Calling 天然兼容
  • 事件驱动让 Agent 从被动变主动 — 四类事件（用户、系统、定时、外部）覆盖全场景，Spring Cloud Stream 提供跨中间件的消息抽象
  • 生产级流式需要完善的连接管理 — 心跳保活、指数退避重连、Last-Event-ID 状态恢复、Redis Pub/Sub 跨实例消息分发
  • 性能优化不能忽视 — 背压控制防止内存溢出，批量发送减少 HTTP 开销，Gzip 压缩减少传输体积，HTTP/2 多路复用减少连接数
  • 事件风暴是事件驱动系统的头号威胁 — 必须设计事件去重、限流、优先级控制和最大扇出数，防止系统被自己的事件淹没

👨‍💻 面试高频题

👨‍💻 Q1: SSE 和 WebSocket 在 Agent 场景中各自的优势和适用场景？

答：SSE 的核心优势是简单和轻量——基于标准 HTTP 协议，无需额外基础设施，浏览器原生 EventSource API 支持自动重连。它天然适合 LLM 的流式输出，因为 LLM 输出是单向的（服务端生成 token，客户端展示）。WebSocket 的核心优势是全双工通信——客户端可以在 Agent 输出过程中实时追加信息、中断操作、发送确认。适用场景包括：多轮对话 Agent（用户需要中途修改方向）、协作编辑 Agent（多方实时交互）、需要用户实时确认的 Agent（执行危险操作前征求同意）。实战建议：90% 的 Agent 场景用 SSE 就够了。如果需要"双向"，可以设计"提交-输出-再提交"的 SSE 模式替代 WebSocket，降低架构复杂度。

👨‍💻 Q2: 流式 Agent 如何处理 Tool Calling？流式 token 和工具调用如何在同一个流中共存？

答：核心挑战是 LLM 在输出过程中可能随时决定调用工具，而工具调用不是 token，不能简单地拼接到文本流中。解决方案是事件类型化：定义统一的 AgentEvent 类型，包含 TEXT（普通 token）、TOOL_CALL（工具调用请求）、TOOL_RESULT（工具执行结果）、ERROR（错误）、DONE（结束）五种类型。前端根据事件类型分别处理：TEXT 事件追加到聊天界面，TOOL_CALL 事件显示"正在使用 XX 工具..."的进度提示，TOOL_RESULT 事件可以展示工具执行结果。Spring AI 的 .stream().chatResponse() 返回完整的 ChatResponse 对象，可以从中提取 hasToolCalls() 判断是否有工具调用，实现文本和工具调用的统一流式处理。

👨‍💻 Q3: 在多实例部署下，如何保证 SSE 连接的水平扩展？

答：核心问题是 SSE 连接是有状态的——用户连接到实例 A，但 Agent 处理可能在实例 B 上完成。解决方案有两种：
(1) 粘性会话（Sticky Session）——负载均衡器将同一用户的请求始终路由到同一实例。简单但扩展性有限，实例故障时连接丢失；
(2) 消息广播（推荐）——使用 Redis Pub/Sub 或 Kafka 做跨实例消息分发。Agent 处理完成后，将 token 发布到以 sessionId 为 key 的消息通道，所有实例订阅该通道。用户连接在哪个实例上，那个实例就负责将 token 推送给客户端。
关键设计：Redis channel 命名规范 agent:stream:{sessionId}，每个 SSE 连接建立时订阅对应 channel，断开时取消订阅。使用 Sinks.Many 作为 Reactor 到 Redis 消息的桥接。

👨‍💻 Q4: 如何防止事件驱动 Agent 系统出现事件风暴？

答：事件风暴是指一个事件触发多个消费者，每个消费者又产生新事件，导致事件数量指数级增长。防护措施包括四层：
(1) 事件去重——为每个事件分配全局唯一 ID，消费者维护已处理 ID 集合（使用 Redis Set 或 Bloom Filter），重复事件直接丢弃；
(2) 事件限流——对每种事件类型设置速率限制（如 Token Bucket 算法），超过阈值的事件进入队列排队处理；
(3) 最大扇出数——限制一个事件最多触发的子事件数量（如最多 10 个），超出后合并或丢弃低优先级事件；
(4) 事件优先级——定义事件优先级（CRITICAL > HIGH > NORMAL > LOW），系统繁忙时只处理高优先级事件。
架构层面：使用 Kafka 的消费者组机制保证同一事件只被一个 Agent 实例处理，避免重复处理。

👨‍💻 Q5: 流式 Agent 的背压控制具体如何实现？什么时候会触发背压？

答：背压（Backpressure）是指数据生产速度超过消费速度时的流量控制机制。在流式 Agent 中，触发场景包括：LLM 生成速度很快（每秒 100+ token）但客户端网络带宽有限、客户端前端渲染慢（DOM 更新卡顿）、多个 Agent 同时输出竞争带宽。

Reactor 提供了多种背压策略：onBackpressureBuffer(n)（缓冲 n 个 token 后丢弃旧的）、onBackpressureDrop()（直接丢弃新 token）、onBackpressureLatest()（只保留最新一个 token）。推荐使用 bufferTimeout(5, Duration.ofMillis(50)) 做批量发送——每 5 个 token 或 50ms 合并为一个 SSE 事件，既减少 HTTP 开销又控制了发送频率。同时设置合理的缓冲区大小（如 1024），超过后丢弃最旧的 token，确保流不会因为消费慢而阻塞服务端。
