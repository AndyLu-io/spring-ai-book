开源模型本地部署

从依赖云端 API 到把模型跑在自己的机器上，本地部署让你拥有完全的控制权。从"租房"到"买房"，这是 AI 开发者的必经之路。

    📚 开篇故事：从租房到买房

    想象你刚来到一座新城市。最开始你选择租房 — 不需要操心装修、维修，拎包入住，很方便。但渐渐地，你发现租房有几个问题：房东随时可能涨租（API 价格变动），有些房间不让改造（数据隐私限制），隔壁邻居的噪音你无法控制（共享资源的不稳定性）。

    于是你开始考虑买房 — 把模型部署在自己的服务器上。买房意味着你拥有完全的控制权：想怎么装修就怎么装修（自定义模型），数据放在自己家里（数据不出内网），不用担心房东涨租（成本可控）。当然，买房也有挑战：需要首付（GPU 硬件投入）、需要装修（模型部署配置）、需要维护（持续运维）。

    2023-2024 年，开源大模型迎来了爆发期。Meta 的 Llama、阿里的 Qwen、Mistral AI 的 Mistral、Google 的 Gemma — 这些模型的质量已经逼近甚至在某些场景下超过了闭源模型。配合 Ollama、vLLM 等推理引擎，在本地运行大模型已经从"极客玩具"变成了"工程实践"。

    本章将带你从模型选型到部署上线，掌握本地运行大模型的完整技能栈。无论你是想降低成本、保护隐私，还是想深入理解模型原理，这章都是你的"购房指南"。

    🛒 开源模型生态

    模型的"超市"：主流开源 LLM 对比

    开源模型就像一个大型超市，货架上摆满了各种选择。你需要根据自己的"预算"（硬件条件）、"口味"（应用场景）和"需求量"（推理吞吐）来挑选。

    比喻：模型的"超市"

    走进超市，你会发现：Llama 像是"综合大卖场"（通用能力强，什么都卖），Qwen 像是"中文精品店"（中文表现突出），Mistral 像是"欧洲精品柜"（效率高，性价比好），Gemma 像是"Google 精选"（轻量级，适合入门）。

    模型系列发布者参数规模中文能力许可证推荐场景
    Llama 3 Meta 8B / 70B / 405B 良好 Llama License 通用场景，生态最丰富
    Qwen 2.5 阿里 0.5B ~ 72B 优秀 Apache 2.0 中文场景首选
    Mistral Mistral AI 7B / 8x7B / Large 一般 Apache 2.0 高效推理，资源受限
    Gemma 2 Google 2B / 9B / 27B 一般 Gemma License 轻量部署，入门学习

    💡 小窍门

    选型建议：中文场景优先考虑 Qwen 2.5 或 DeepSeek；通用场景首选 Llama 3；资源受限（显存 < 8GB）考虑 Gemma 2 2B 或 Qwen 2.5 0.5B；追求推理质量选 DeepSeek-V3/R1。

    参数规模与硬件需求

    模型参数越多，能力越强，但需要的硬件也越多。一个简单的估算公式：

    显存需求估算（FP16 精度）

模型参数量    =>    最低显存需求
─────────────────────────────────
1B  参数      =>    ~2 GB
3B  参数      =>    ~6 GB
7B  参数      =>    ~14 GB
13B 参数      =>    ~26 GB
70B 参数      =>    ~140 GB

公式：显存 ≈ 参数量(字节) × 2（FP16）
例：7B × 10^9 × 2 bytes ≈ 14 GB

量化后（INT4）：显存需求减半！
7B 参数 → ~4 GB（INT4 量化后）
    硬件配置可运行模型推理速度
    8GB 显存（RTX 4060）7B (INT4) / 3B (FP16)~30 tokens/s
    16GB 显存（RTX 4080）7B (FP16) / 13B (INT4)~40 tokens/s
    24GB 显存（RTX 4090）13B (FP16) / 70B (INT4)~50 tokens/s
    48GB 显存（A6000）70B (INT4) / 70B (FP16)~30 tokens/s
    CPU only（32GB RAM）7B (Q4)~5 tokens/s
    ⚠️ 踩坑警告

    纯 CPU 推理虽然可用，但速度很慢（约 5 tokens/s），不适合交互式应用。如果有条件，至少用一张消费级 GPU。对于生产环境，建议使用专业 GPU（A100/H100）或 GPU 集群。

    📦 Ollama 快速入门

    什么是 Ollama？

    Ollama 是一个轻量级的本地模型运行工具，它让运行开源大模型变得像安装一个 App 一样简单。

    比喻：模型的"一键安装器"

    Ollama 就像手机上的"应用商店"。你想用微信？打开商店，点击安装，几秒钟就能用了。Ollama 对大模型做了同样的事情 — 你不需要配置 CUDA、不需要编译推理框架、不需要处理模型格式。只需一条命令，模型就能跑起来。

    # macOS / Linux 安装 Ollama curl -fsSL https://ollama.com/install.sh | sh # 运行一个模型（首次运行会自动下载） ollama run llama3.1:8b # 就这么简单！模型会自动下载并启动对话 # 其他常用命令 ollama list # 查看已下载的模型 ollama pull qwen2.5:7b # 下载模型但不运行 ollama rm llama3.1:8b # 删除模型 ollama show llama3.1:8b # 查看模型信息

    Ollama API 调用

    Ollama 内置了兼容 OpenAI 格式的 API，这意味着你几乎不需要修改代码就能切换到本地模型。

    # Ollama 启动后默认监听 http://localhost:11434 # 方式 1：使用 Ollama 原生 API curl http://localhost:11434/api/chat -d '{ "model": "llama3.1:8b", "messages": [ {"role": "user", "content": "你好，请介绍一下自己"} ] }' # 方式 2：使用 OpenAI 兼容 API curl http://localhost:11434/v1/chat/completions -d '{ "model": "llama3.1:8b", "messages": [ {"role": "user", "content": "你好，请介绍一下自己"} ] }' # 方式 3：使用 Python SDK import requests response = requests.post( "http://localhost:11434/api/chat", json={ "model": "llama3.1:8b", "messages": [ {"role": "user", "content": "用Python写一个快速排序"} ] } ) print(response.json()["message"]["content"])
    💡 你知道吗？

    Ollama 的 OpenAI 兼容 API 让迁移变得非常简单。你只需要把 api-key 改为空（本地不需要认证），把 base-url 改为 http://localhost:11434/v1，其他代码完全不需要改。

    🌟 Spring AI + Ollama 集成

    配置 Spring AI 连接 Ollama

    Spring AI 提供了 Ollama 的 Starter，让你像使用 OpenAI 一样轻松使用本地模型。

    比喻：本地模型的"遥控器"

    Spring AI 的 Ollama 集成就像一个"遥控器"。电视（Ollama）已经摆在那了，你只需要拿起遥控器（Spring AI），按下按钮（调用 API），就能控制电视换台、调音量、开关机。你不需要走到电视前面去手动操作。

    <!-- pom.xml --> <dependency> <groupId>org.springframework.ai</groupId> <artifactId>spring-ai-ollama-spring-boot-starter</artifactId> </dependency>
    # application.yml spring: ai: ollama: base-url: http://localhost:11434 chat: options: model: llama3.1:8b # 使用的模型 temperature: 0.7 num-ctx: 4096 # 上下文窗口大小 embedding: options: model: nomic-embed-text # 嵌入模型（RAG 用）

    完整示例：本地 AI 助手

    @RestController @RequestMapping("/api/chat") public class LocalChatController { private final ChatClient chatClient; public LocalChatController(ChatClient.Builder builder) { this.chatClient = builder .defaultSystem("你是一个友好的 Java 开发助手，使用中文回答。") .build(); } @PostMapping public Map<String, String> chat(@RequestBody Map<String, String> req) { String message = req.get("message"); String reply = chatClient.prompt() .user(message) .call() .content(); return Map.of( "reply", reply, "model", "llama3.1:8b (local)" ); } @PostMapping("/stream") public Flux<String> stream(@RequestBody Map<String, String> req) { return chatClient.prompt() .user(req.get("message")) .stream() .content(); } }

    与 RAG 结合：本地知识库

    @Service public class LocalRagService { private final ChatClient chatClient; private final VectorStore vectorStore; public LocalRagService(ChatClient.Builder builder, VectorStore vectorStore) { this.chatClient = builder.build(); this.vectorStore = vectorStore; } public String query(String question) { // 1. 从向量库检索相关文档 SearchRequest searchRequest = SearchRequest.query(question) .withTopK(3); // 2. 构建 RAG Prompt return chatClient.prompt() .system("你是技术文档助手，根据检索到的内容回答问题。") .user(question) .advisors(new QuestionAnswerAdvisor(vectorStore, searchRequest)) .call() .content(); } } // 配置：使用 Ollama 的嵌入模型 + 向量数据库 // spring.ai.ollama.embedding.options.model=nomic-embed-text // 所有数据留在本地，完全离线可用！
    💡 小窍门

    本地 RAG 的最大优势是数据完全不出内网。对于企业内部文档、敏感数据等场景，这是非常重要的安全特性。Ollama 的 nomic-embed-text 是一个优秀的本地嵌入模型，质量接近 OpenAI 的 text-embedding-3-small。

    🚀 vLLM 高性能推理

    vLLM：模型的"涡轮增压"

    vLLM 是一个高性能的 LLM 推理引擎，它的核心创新是 PagedAttention 技术，大幅提升了推理速度和吞吐量。

    比喻：模型的"涡轮增压"

    如果说普通推理引擎是自然吸气发动机，那 vLLM 就是加了涡轮增压的发动机。同样的排量（GPU），能输出更大的马力（吞吐量）。PagedAttention 就像涡轮增压器 — 通过更聪明的内存管理，让 GPU 的计算能力得到更充分的利用。

    传统推理 vs vLLM

传统方式：预分配连续内存

│ 请求1: [████████████░░░░░░░░░░] │  大量内存浪费！
│ 请求2: [██░░░░░░░░░░░░░░░░░░░░] │  预分配了但没用完
│ 请求3: [██████░░░░░░░░░░░░░░░░] │
└─────────────────────────────────┘
  利用率：~30%

PagedAttention：分页动态分配
┌─────────────────────────────────┐
│ 请求1: [█][█][█][░][░]           │  按需分配！

  利用率：~90%

    📦 模型量化与优化

    什么是量化？

    量化是将模型的权重从高精度（FP16，16 位浮点）压缩到低精度（INT8、INT4）的技术。就像把 FLAC 无损音乐压缩成 MP3 — 文件小了很多，但音质损失很小。

    比喻：模型的"压缩术"

    想象你有一本厚厚的百科全书（FP16 模型），放在书架上占了很大空间（显存）。量化就像把这本书扫描成电子版（INT4），内容几乎一样，但体积缩小了 4 倍。你可以在同样的书架上放 4 本电子百科全书。

    量化精度对比

精度        位数    大小(7B模型)    质量损失    适用场景
──────────────────────────────────────────────────
FP32        32位    ~28 GB         无损失      科学计算
FP16/BF16   16位    ~14 GB         几乎无      GPU 训练
INT8         8位    ~7 GB          微小        生产推理
INT4         4位    ~3.5 GB        轻微        本地部署

质量保持率（大致参考）
FP16 ████████████████████████████ 100%
INT8 ███████████████████████████░  ~98%
INT4 █████████████████████████░░░  ~95%

    主流量化格式

    格式工具特点推荐场景
    GGUF llama.cpp / Ollama 兼容性最好，CPU/GPU 都支持 本地部署首选
    GPTQ AutoGPTQ GPU 推理优化，速度较快 GPU 服务器部署
    AWQ AutoAWQ 比 GPTQ 更快，质量相当 高吞吐推理
    GGML llama.cpp (旧) 已被 GGUF 取代 不推荐新项目使用
    FP8 vLLM H100/H200 原生支持 最新 GPU 硬件
    # Ollama 使用量化模型（GGUF 格式） # Ollama 会自动下载量化版本 ollama run llama3.1:8b-q4_K_M # 4-bit 量化，推荐 ollama run llama3.1:8b-q8_0 # 8-bit 量化，质量更好 ollama run llama3.1:8b # 默认 Q4_K_M 量化 # 手动量化（使用 llama.cpp） ./quantize ./models/llama-3.1-8b-f16.gguf \ ./models/llama-3.1-8b-q4_K_M.gguf \ Q4_K_M
    ⚠️ 踩坑警告

    量化会损失少量质量。对于需要精确数值计算的场景（如金融、科学计算），建议使用 INT8 或更高精度。对于一般对话和文本生成，INT4 的质量损失通常可以接受。建议在量化后进行质量测试。

    混合部署代码示例

    // 根据场景动态选择模型后端 @Service public class HybridChatService { private final ChatClient openaiClient; private final ChatClient ollamaClient; public HybridChatService( @Qualifier("openaiChatClient") ChatClient openaiClient, @Qualifier("ollamaChatClient") ChatClient ollamaClient) { this.openaiClient = openaiClient; this.ollamaClient = ollamaClient; } public String chat(String message, boolean sensitive) { if (sensitive) { // 敏感数据走本地模型 return ollamaClient.prompt() .user(message) .call() .content(); } else { // 一般任务走云端 API（质量更好） return openaiClient.prompt() .user(message) .call() .content(); } } }

    📖 章节小结

    本章我们掌握了本地部署大模型的完整技能栈：

    • 模型选型 — 了解 Llama、Qwen、Mistral、Gemma 等主流开源模型的特点
    • Ollama — 一键安装、运行模型，兼容 OpenAI API
    • Spring AI + Ollama — 本地模型的 Spring AI 集成，支持 RAG
    • vLLM — 高性能推理引擎，PagedAttention 大幅提升吞吐
    • 量化技术 — GGUF、GPTQ、AWQ 等格式，用质量换空间
    • 选型策略 — 根据数据敏感度、用量、硬件条件选择部署方案

    本地部署不是"替代"云端 API，而是"补充"。掌握两种能力，才能在不同场景下做出最优选择。

    🎓 面试高频题

    👨‍💻 Q1: 如何估算本地部署一个 LLM 需要多少显存？

    FP16 精度下，显存 ≈ 参数量（字节）× 2。例如 7B 参数模型需要约 14GB 显存。使用 INT4 量化后，显存需求减半到约 7GB。还需预留 20-30% 显存用于 KV Cache 等运行时开销。

    👨‍💻 Q2: Ollama 和 vLLM 的主要区别是什么？

    Ollama 定位个人/开发使用，安装简单，支持 CPU 推理，使用 GGUF 格式。vLLM 定位生产环境，基于 PagedAttention 实现高吞吐，支持批量推理和高并发，需要 GPU 硬件。两者都提供 OpenAI 兼容 API。

    👨‍💻 Q3: 什么是 PagedAttention？为什么它能提升推理性能？

    PagedAttention 将 KV Cache 分成固定大小的"页"，像操作系统的虚拟内存一样动态分配。传统方式需要为每个请求预分配连续的最大内存（造成浪费），PagedAttention 按需分配，内存利用率从约 30% 提升到 90%，从而支持更多并发请求。
